# 安全认证面试指南

这份材料面向 Java 后端大厂面试中的安全认证专题，重点覆盖认证、授权、Session、Cookie、JWT、OAuth2、SSO、Spring Security、RBAC、接口签名、密码存储、CSRF、CORS、越权、Token 续期、网关鉴权和项目落地。回答时不要只背名词，要讲清楚“谁是谁、能不能进、能访问什么、凭证怎么传、风险怎么兜底”。

## 0. 高频知识地图

| 主题 | 面试常问点 | 回答关键词 |
| --- | --- | --- |
| 认证与授权 | Authentication 和 Authorization 区别 | 你是谁、你能做什么 |
| Session / Cookie | 登录态怎么保存 | 服务端会话、客户端标识、SameSite、HttpOnly |
| JWT | 无状态 Token 怎么设计 | Header、Payload、Signature、过期时间、刷新 |
| OAuth2 | 第三方授权流程 | Authorization Code、Access Token、Refresh Token |
| SSO | 单点登录怎么实现 | 统一认证中心、票据、跨系统信任 |
| Spring Security | 过滤器链、认证管理器、上下文 | SecurityFilterChain、Authentication、SecurityContext |
| RBAC | 权限模型怎么设计 | 用户、角色、权限、资源、数据权限 |

| 排障 | 登录失效、401/403、跨域失败 | 认证失败、权限不足、Token 过期、配置错误 |

## 1. 认证和授权有什么区别？

**一句话回答**：认证解决“你是谁”，授权解决“你能访问什么”。

| 概念 | 关注点 | 典型动作 |
| --- | --- | --- |
| 认证 Authentication | 身份是否可信 | 登录、校验密码、校验 Token、解析用户身份 |
| 授权 Authorization | 是否有权限 | 判断角色、菜单、按钮、接口、数据范围 |

**三点总结**：

1. 认证通常发生在请求进入系统的前半段，先得到用户身份。
2. 授权必须建立在认证之后，否则不知道给谁判断权限。
3. 认证成功不代表有权限访问所有资源，仍要做接口权限和数据权限校验。

**面试表达**：

> 我会把认证和授权分开讲。认证是确认请求是谁发的，比如用户名密码、短信验证码、JWT、OAuth2 登录；授权是拿到身份后判断能不能访问某个接口、菜单或数据。生产问题里 401 通常偏认证失败，403 通常偏权限不足。

## 2. Session 和 Cookie 的关系是什么？

**一句话回答**：Session 是服务端保存的登录状态，Cookie 是浏览器保存并随请求携带的客户端标识，常见做法是 Cookie 存 `sessionId`，服务端根据 `sessionId` 找 Session。

**核心流程**：

1. 用户登录成功，服务端创建 Session。
2. 服务端返回 `Set-Cookie: JSESSIONID=xxx`。
3. 浏览器后续请求自动携带 Cookie。
4. 服务端根据 `JSESSIONID` 找到用户登录态。

**安全细节**：

- `HttpOnly`：避免 JavaScript 读取 Cookie，降低 XSS 窃取风险。
- `Secure`：只在 HTTPS 下传输。
- `SameSite`：降低 CSRF 风险。
- Session 要设置过期时间，退出登录时要主动失效。

**面试表达**：

> Cookie 不是登录态本身，它只是把一个标识带回服务端。真正的登录态存在服务端 Session 里。优点是服务端可控，退出和踢人比较方便；缺点是分布式场景要解决 Session 共享，比如用 Redis 或者改成 Token 方案。

## 3. Session 登录和 JWT 登录有什么区别？

**一句话回答**：Session 是服务端有状态登录，JWT 是客户端携带自包含 Token 的无状态登录。

| 维度 | Session | JWT |
| --- | --- | --- |
| 状态保存 | 服务端保存 Session | Token 自包含用户信息和签名 |
| 扩展性 | 需要共享 Session 或粘性会话 | 服务端只校验签名，更容易横向扩展 |
| 退出登录 | 服务端删除 Session 即可 | 需要黑名单、短过期或版本号 |
| 安全风险 | Session 劫持、CSRF | Token 泄露、过期控制、无法天然撤销 |
| 适合场景 | 传统 Web、后台管理 | 前后端分离、移动端、微服务网关 |

**面试表达**：

> Session 的优势是服务端可控，JWT 的优势是无状态和扩展方便。JWT 不是天然更安全，它一旦泄露，在过期前就可能被使用，所以要配合 HTTPS、短过期、Refresh Token、黑名单或 tokenVersion 机制。

## 4. JWT 的结构是什么？为什么不能存敏感信息？

**一句话回答**：JWT 由 Header、Payload、Signature 三部分组成，Payload 只是 Base64Url 编码，不是加密，所以不能存密码、手机号明文、身份证等敏感信息。

| 部分 | 作用 |
| --- | --- |
| Header | Token 类型和签名算法 |
| Payload | 用户 ID、租户、角色、过期时间等声明 |
| Signature | 使用密钥对前两部分签名，防篡改 |

**关键点**：

- JWT 可以防篡改，不等于防泄露。
- Payload 可以被客户端解码查看。
- 应只放必要身份和权限摘要，敏感信息放服务端查询。
- 必须设置 `exp`，避免长期有效 Token。

**面试表达**：

> JWT 的签名保证的是别人不能改 Payload 后继续通过校验，但 Payload 本身不是密文。面试里我会强调：JWT 里最多放 userId、tenantId、scope、过期时间这类必要声明，不放密码、身份证、手机号明文等敏感数据。

## 5. JWT 如何实现退出登录和踢人？

**一句话回答**：JWT 无状态方案不能像 Session 一样天然删除登录态，退出和踢人通常通过短过期、黑名单、用户 tokenVersion 或服务端会话表实现。

**常见方案**：

- 短过期 Access Token + Refresh Token：降低泄露窗口。
- Redis 黑名单：退出或踢人时把 Token 的 `jti` 放入黑名单，到期自动删除。
- tokenVersion：用户表或缓存维护版本号，改密码、踢人时递增版本，旧 Token 失效。
- 服务端会话表：Token 仍携带会话 ID，服务端可控制会话状态。

**面试表达**：

> JWT 无状态是优点，也是退出登录的难点。生产上我不会给 Access Token 设置很长有效期，而是短过期加 Refresh Token；如果需要强制踢人，就用黑名单、tokenVersion 或服务端会话状态来兜底。

## 6. Access Token 和 Refresh Token 有什么区别？

**一句话回答**：Access Token 用来访问业务接口，生命周期短；Refresh Token 用来换新的 Access Token，生命周期长但使用频率低、保护要求更高。

| Token | 作用 | 生命周期 | 风险控制 |
| --- | --- | --- | --- |
| Access Token | 访问业务资源 | 短，例如 15-60 分钟 | 泄露窗口小 |
| Refresh Token | 刷新访问凭证 | 长，例如 7-30 天 | 存储更安全、可轮换、可撤销 |

**关键设计**：

- Refresh Token 不应频繁传给普通业务接口。
- 刷新时可以做轮换，旧 Refresh Token 立即失效。
- 服务端记录 Refresh Token 状态，支持退出、踢人和风险控制。
- 刷新失败要引导重新登录，而不是无限重试。

**面试表达**：

> Access Token 面向业务访问，所以短期有效；Refresh Token 面向续期，所以生命周期长但要更严格保护。这样既减少用户频繁登录，又能把 Token 泄露风险控制在较短窗口内。

## 7. OAuth2 解决什么问题？授权码模式怎么走？

**一句话回答**：OAuth2 解决第三方应用在不获取用户密码的情况下，经过用户授权访问受保护资源的问题；授权码模式是服务端应用最常用也更安全的模式。

**授权码模式流程**：

1. 用户访问客户端应用，客户端跳转到授权服务器。
2. 用户在授权服务器登录并授权。
3. 授权服务器带着 `code` 回调客户端。
4. 客户端后端用 `code + client_secret` 换 Access Token。
5. 客户端用 Access Token 访问资源服务器。

**关键点**：

- `code` 短期有效且只能使用一次。
- `client_secret` 只放服务端，不能暴露给浏览器。
- 回调地址必须白名单校验，防止授权码被劫持。
- 对前端公共客户端要配合 PKCE。

**面试表达**：

> OAuth2 的核心不是登录，而是授权。授权码模式的安全点在于浏览器只拿到一次性 code，真正换 Token 的动作在后端完成，client_secret 不暴露给前端。现在前端或移动端还会配合 PKCE 防止 code 被截获。

## 8. OAuth2 和 OIDC 有什么区别？

**一句话回答**：OAuth2 主要做授权，OIDC 是基于 OAuth2 扩展出来的身份认证协议，增加了 ID Token，用于表达用户身份。

| 协议 | 核心目的 | 关键凭证 |
| --- | --- | --- |
| OAuth2 | 授权访问资源 | Access Token |
| OIDC | 身份认证 + 授权 | ID Token + Access Token |

**面试表达**：

> OAuth2 主要回答“第三方应用能不能访问资源”，OIDC 进一步回答“当前登录用户是谁”。所以做第三方登录时，严格讲通常用的是 OIDC 或者厂商基于 OAuth2 扩展出来的登录协议。

## 9. SSO 单点登录怎么实现？

**一句话回答**：SSO 通过统一认证中心让多个业务系统共享一次登录结果，用户在认证中心登录后，其他系统通过票据或 Token 信任这个身份。

**典型流程**：

1. 用户访问业务系统 A，A 发现未登录，跳转认证中心。
2. 用户在认证中心登录成功。
3. 认证中心生成登录态，并带票据回跳 A。
4. A 用票据向认证中心换取用户身份，建立本系统登录态。
5. 用户访问业务系统 B 时，B 跳转认证中心，认证中心发现已登录，直接签发票据。

**常见风险**：

- 回调地址必须校验白名单。
- 票据短期有效且一次性使用。
- 单点退出要通知或清理各业务系统会话。
- 多域名 Cookie 要谨慎处理，避免扩大泄露面。

**面试表达**：

> SSO 的关键是把认证能力收敛到统一认证中心，业务系统不直接处理密码，只信任认证中心签发的票据或 Token。这样用户只登录一次，多系统都能识别身份。

## 10. Spring Security 的核心原理是什么？

**一句话回答**：Spring Security 的核心是过滤器链，请求进入业务 Controller 前先经过一组安全过滤器，完成认证、授权、异常处理和安全上下文维护。

**核心组件**：

| 组件 | 作用 |
| --- | --- |
| `SecurityFilterChain` | 定义哪些请求经过哪些安全过滤器 |
| `Authentication` | 表示认证请求或认证成功后的用户身份 |
| `AuthenticationManager` | 负责执行认证 |
| `UserDetailsService` | 加载用户信息 |
| `PasswordEncoder` | 密码加密和校验 |
| `SecurityContext` | 保存当前请求的认证信息 |

**面试表达**：

> Spring Security 不是在 Controller 里做 if 判断，而是在请求进入业务逻辑前通过过滤器链完成安全处理。认证成功后会把 Authentication 放进 SecurityContext，后续授权判断和业务代码都可以拿到当前用户身份。

## 11. Spring Security 里 401 和 403 有什么区别？

**一句话回答**：401 表示未认证或认证失败，403 表示已经认证但权限不足。

| 状态码              | 含义       | 典型原因                  |
| ---------------- | -------- | --------------------- |
| 401 Unauthorized | 没有有效身份   | Token 缺失、过期、签名错误、登录失败 |
| 403 Forbidden    | 身份有效但无权限 | 角色不足、接口权限不足、数据权限不足    |

**排查思路**：

1. 先看请求是否携带 Token / Cookie。
2. 再看 Token 是否过期、签名是否正确、用户是否被禁用。
3. 如果认证成功，再看接口权限表达式、角色前缀、资源权限和数据权限。
4. 网关和后端都做鉴权时，要看是哪一层返回的状态码。

**面试表达**：

> 401 是“你还没证明你是谁”，403 是“知道你是谁，但你不能访问”。线上排查时我会先区分认证失败还是授权失败，再看是网关拦的、Spring Security 拦的，还是业务数据权限拦的。

## 12. RBAC 权限模型怎么设计？

**一句话回答**：RBAC 通过用户、角色、权限建立关系，用户绑定角色，角色绑定权限，从而降低直接给用户分配权限的维护成本。

**基础模型**：

| 表 | 作用 |
| --- | --- |
| user | 用户 |
| role | 角色，例如管理员、运营、客服 |
| permission | 权限，例如菜单、按钮、接口 |
| user_role | 用户和角色关系 |
| role_permission | 角色和权限关系 |

**扩展点**：

- 菜单权限：控制页面入口。
- 按钮权限：控制页面操作。
- 接口权限：控制 API 调用。
- 数据权限：控制能看哪些组织、城市、站点、订单。
- 多租户：权限判断要带 `tenantId`，防止跨租户越权。

**面试表达**：

> RBAC 解决的是权限分配复杂度。用户不直接挂大量权限，而是挂角色，角色再挂权限。实际项目里还要区分菜单、按钮、接口和数据权限，尤其是多租户或多组织系统，不能只校验接口权限，还要校验数据范围。

## 13. 数据权限和接口权限有什么区别？

**一句话回答**：接口权限判断能不能调用某个 API，数据权限判断调用 API 后能看到或操作哪些数据。

**举例**：

- 接口权限：用户是否能访问 `/order/list`。
- 数据权限：用户访问 `/order/list` 后，只能看自己城市、自己站点或自己组织下的订单。

**实现方式**：

- 在查询条件里追加组织、城市、站点、用户范围。
- 使用数据权限拦截器或 MyBatis 插件统一注入条件。
- 对关键写操作在业务层再次校验资源归属。
- 避免只在前端隐藏按钮，后端必须兜底。

**面试表达**：

> 接口权限只能说明你能进这个门，数据权限决定你进门后能看哪一排货架。很多越权问题不是没有登录，而是 A 用户通过改参数访问了 B 用户的数据，所以后端查询和写操作都要校验数据归属。

## 14. 什么是越权？水平越权和垂直越权有什么区别？

**一句话回答**：越权是用户访问了自己不该访问的资源；水平越权是同级用户互相访问数据，垂直越权是低权限用户访问高权限功能。

| 类型 | 例子 | 防护 |
| --- | --- | --- |
| 水平越权 | 用户 A 修改订单 ID 查看用户 B 的订单 | 校验资源归属、数据范围 |
| 垂直越权 | 普通用户调用管理员接口 | 角色/权限校验 |

**面试表达**：

> 越权防护不能依赖前端隐藏入口。后端每个敏感接口都要校验当前用户是否有功能权限，涉及具体资源时还要校验资源归属，比如订单是否属于当前用户、站点是否在当前组织范围内。

## 15. 密码为什么不能明文存储？应该怎么存？

**一句话回答**：密码不能明文存储，应该使用带盐的慢哈希算法，例如 BCrypt、Argon2、PBKDF2，登录时对输入密码做同样校验，而不是解密数据库密码。

**关键点**：

- 哈希不是加密，不能反向解密。
- 加盐可以防止彩虹表攻击。
- BCrypt 这类慢哈希可以增加暴力破解成本。
- 不同用户应有不同盐。
- 密码重置不要返回旧密码，而是走重置流程。

**面试表达**：

> 密码存储的底线是不可逆。数据库里不应该有明文密码，也不应该用可逆加密保存密码。常见做法是 BCrypt 这类带盐慢哈希，登录时用 PasswordEncoder 校验输入密码和哈希是否匹配。

## 16. 接口签名一般怎么设计？解决什么问题？

**一句话回答**：接口签名通过 `appKey + secret + timestamp + nonce + body` 计算签名，用来防篡改、防伪造，并配合时间戳和 nonce 防重放。

**典型参数**：

| 参数 | 作用 |
| --- | --- |
| appKey | 调用方身份 |
| timestamp | 请求时间，限制有效窗口 |
| nonce | 随机数，防止重复请求 |
| bodyHash | 请求体摘要，防止参数被改 |
| sign | 用 secret 计算出的签名 |

**校验流程**：

1. 校验 `appKey` 是否存在、是否启用。
2. 校验 timestamp 是否在允许窗口内。
3. 校验 nonce 是否已使用，已使用则拒绝。
4. 服务端用相同规则重新计算签名。
5. 签名一致才进入业务逻辑。

**面试表达**：

> 接口签名不是代替 HTTPS，而是在 HTTPS 之外增加调用方身份、防篡改和防重放能力。尤其是开放平台、支付回调、供应商对接这类场景，签名、时间戳、nonce 和幂等要一起设计。

## 17. 如何防止重放攻击？

**一句话回答**：重放攻击是攻击者截获一次合法请求后重复发送，防护要靠短时间窗口、nonce 唯一性、签名和业务幂等。

**防护手段**：

- timestamp 限制请求有效期，例如 5 分钟。
- nonce 或 requestId 存 Redis，窗口期内只能使用一次。
- 签名覆盖关键参数和请求体，防止内容被篡改。
- 写接口要有业务幂等，避免重复扣款、重复下单。

**面试表达**：

> 防重放不能只靠签名，因为签名请求本身也可能被原样复制。需要 timestamp 限制时间窗口，再用 nonce 或 requestId 保证同一窗口内只能用一次，最后业务层还要用幂等兜底。

## 18. CSRF 是什么？怎么防？

**一句话回答**：CSRF 是攻击者利用浏览器自动携带 Cookie 的特性，诱导已登录用户在不知情的情况下向目标网站发起请求。

**防护方式**：

- 使用 CSRF Token，服务端校验请求中的随机 Token。
- Cookie 设置 `SameSite=Lax` 或 `Strict`。
- 重要操作使用 POST/PUT/DELETE，不用 GET 执行写操作。
- 校验 Origin / Referer。
- 前后端分离使用 Authorization Header 携带 Token 时，CSRF 风险相对降低，但仍要关注 Cookie 场景。

**面试表达**：

> CSRF 的关键点是浏览器会自动带 Cookie。攻击者不一定知道你的 Cookie，也能借浏览器身份发请求。所以传统 Cookie 登录要加 CSRF Token、SameSite、Origin 校验，写操作不能用 GET。

## 19. CORS 跨域是什么？为什么不能随便配 `*`？

**一句话回答**：CORS 是浏览器同源策略下的跨域访问控制机制，服务端通过响应头声明允许哪些来源访问；不能随便配 `*`，否则会扩大接口暴露面，带凭证请求也不能和 `*` 一起使用。

**关键响应头**：

- `Access-Control-Allow-Origin`：允许的来源。
- `Access-Control-Allow-Methods`：允许的方法。
- `Access-Control-Allow-Headers`：允许的请求头。
- `Access-Control-Allow-Credentials`：是否允许携带 Cookie 等凭证。

**面试表达**：

> CORS 是浏览器安全限制，不是后端接口权限。生产上我不会全站无脑 `*`，而是按域名白名单配置；如果允许携带 Cookie，还要更谨慎，因为这等于允许指定前端来源带登录态访问后端。

## 20. XSS 和认证安全有什么关系？

**一句话回答**：XSS 会让攻击者在用户浏览器里执行脚本，可能窃取 Token、发起用户身份下的请求或篡改页面，所以认证凭证的存储和输出编码都很重要。

**防护方式**：

- 输出编码和 HTML 转义，避免把用户输入当脚本执行。
- Cookie 设置 `HttpOnly`，降低 Token 被 JS 读取风险。
- 前端避免直接渲染不可信 HTML。
- 配置 CSP 限制脚本来源。
- 敏感操作二次校验或风控。

**面试表达**：

> XSS 一旦发生，攻击者就站在用户浏览器里了。即使 Token 没被直接读走，也可能借用户身份发请求。所以认证安全不能只看后端 Token 校验，还要配合 HttpOnly、输出编码、CSP 和敏感操作校验。

## 21. 网关鉴权和服务内鉴权怎么分工？

**一句话回答**：网关适合做统一认证、Token 解析、黑白名单、限流和粗粒度权限，服务内必须保留细粒度业务权限和数据权限校验。

| 层级 | 适合做什么 |
| --- | --- |
| 网关 | 登录态校验、Token 解析、路由白名单、统一限流、基础权限 |
| 服务内 | 业务角色、资源归属、数据权限、状态机校验 |

**关键点**：

- 不能只依赖网关鉴权，服务可能被内部调用、定时任务或其他服务绕过网关。
- 网关可以把用户 ID、租户、角色等可信上下文传给后端。
- 后端要防止调用方伪造用户头，内部链路可配合签名、mTLS 或网关清洗 Header。

**面试表达**：

> 网关负责统一入口安全，服务内负责业务安全。比如网关确认 Token 有效，但订单服务仍要判断这张订单是不是当前用户或当前站点能操作。否则只要内部调用绕过网关，就可能产生越权。

## 22. 微服务之间调用如何传递用户身份？

**一句话回答**：微服务调用可以通过网关解析 Token 后传递用户上下文，也可以内部继续传递 Token；关键是保证上下文可信、可追踪、不能被外部伪造。

**常见做法**：

- 网关解析外部 Token，向后端传递 userId、tenantId、roles、traceId。
- 服务间调用通过 Feign / Dubbo 拦截器透传用户上下文。
- 内部 Header 要由网关统一清洗，避免外部用户伪造。
- 对高风险内部接口增加服务身份认证、签名或 mTLS。
- 异步 MQ 场景要把操作人、租户、来源系统写入消息体。

**面试表达**：

> 用户身份透传不是简单把 Header 一路传下去。外部请求进网关后要先清洗不可信 Header，再由网关写入可信用户上下文。服务间调用可以通过拦截器透传，但关键写操作仍要在业务服务里做权限和数据归属校验。

## 23. 多租户系统的认证授权要注意什么？

**一句话回答**：多租户系统必须把 `tenantId` 作为认证、授权、数据查询和缓存 Key 的核心维度，防止跨租户访问。

**关键点**：

- Token 中可以携带当前租户或租户列表，但要防篡改。
- 查询条件必须带租户隔离，不能只靠前端传参。
- 缓存 Key 要带 `tenantId`，避免租户间缓存污染。
- 权限判断要在租户维度内进行，同一用户在不同租户角色可能不同。
- 管理后台切换租户要重新校验权限。

**面试表达**：

> 多租户安全最怕漏掉 tenantId。认证时要知道用户属于哪些租户，授权时要按当前租户判断角色，查数据和查缓存时也要带 tenantId。否则很容易出现用户登录是对的，但数据跨租户泄露。

## 24. 登录接口如何防暴力破解？

**一句话回答**：登录防暴力破解要从账号、IP、设备、验证码、限流、失败次数和风控多个维度控制，不能只靠密码复杂度。

**常见方案**：

- 同账号连续失败锁定或增加等待时间。
- 同 IP 高频登录限流。
- 异常设备、异地登录触发验证码或二次验证。
- 登录失败提示不要暴露“账号存在但密码错”等过细信息。
- 后台记录审计日志，支持告警和追踪。

**面试表达**：

> 登录接口是高风险入口，我会做账号维度和 IP 维度的双限流，再结合验证码、失败次数、设备指纹和审计日志。提示语也要克制，不能帮助攻击者枚举账号。

## 25. 验证码和短信验证码有哪些安全风险？

**一句话回答**：验证码要防机器绕过、暴力尝试和重复使用；短信验证码还要防短信轰炸、验证码泄露、手机号枚举和验证码长期有效。

**设计要点**：

- 验证码短期有效，使用后立即失效。
- 同手机号、同 IP、同设备限制发送频率。
- 校验次数有限，失败过多自动失效。
- 短信验证码不写入日志明文。
- 重要操作最好结合登录态、图形验证码或风控策略。

**面试表达**：

> 短信验证码不是只要能发出去就行。要控制发送频率、有效期、校验次数和日志脱敏，否则容易被刷短信、撞验证码，甚至因为日志泄露导致账号风险。

## 26. 安全认证相关线上问题怎么排查？

**一句话回答**：先区分认证失败、授权失败、跨域失败、Token 过期和上下文丢失，再沿网关、认证服务、业务服务、缓存和数据库逐层排查。

**排查清单**：

1. 看状态码：401、403、5xx、CORS 浏览器报错分别代表不同方向。
2. 看请求头：Authorization、Cookie、Origin、traceId 是否存在。
3. 看 Token：是否过期、签名算法、密钥版本、issuer、audience 是否匹配。
4. 看用户状态：是否禁用、角色是否变更、tokenVersion 是否变化。
5. 看权限配置：接口路径、角色前缀、白名单、网关路由是否一致。
6. 看缓存：Session、黑名单、权限缓存、租户缓存是否过期或污染。

**面试表达**：

> 认证问题我不会上来就改代码。先看是 401 还是 403，确认是哪一层返回；再看 Token、Cookie、网关转发头、用户状态和权限缓存。很多问题是网关和服务内规则不一致、Token 过期、角色缓存没刷新或跨域配置导致的。

## 27. 如何结合项目讲安全认证？

**一句话回答**：项目表达不要只说“用了 JWT”，要讲登录态如何生成、网关和服务如何校验、权限模型怎么设计、Token 如何续期和退出、越权如何防。

**可套项目表达**：

> 在项目里，我参与过用户登录态和接口权限治理。整体思路是入口统一认证，网关负责解析 Token、校验黑白名单和透传用户上下文，业务服务负责接口权限、数据权限和资源归属校验。权限模型上用用户、角色、权限、组织或租户维度来做 RBAC 扩展；Token 设计上使用短期 Access Token 配合 Refresh Token，并通过 Redis 黑名单或 tokenVersion 支持退出登录和踢人。对外部接口或供应商回调用签名、timestamp、nonce 和幂等键防篡改、防重放。线上排查时会按 401/403、网关日志、Token 解析、权限缓存和业务数据归属逐层定位。

**项目追问准备**：

- 你们登录态是 Session 还是 JWT？为什么这么选？
- 权限是菜单权限、接口权限还是数据权限？
- Token 泄露怎么办？退出登录和踢人怎么实现？
- 网关鉴权后，服务内还要不要鉴权？
- 多租户或多组织数据怎么隔离？
- 第三方接口如何防重放和防篡改？

## 高频追问速记

| 问题 | 速记回答 |
| --- | --- |
| 认证和授权区别？ | 认证是你是谁，授权是你能做什么。 |
| 401 和 403 区别？ | 401 是未认证或认证失败，403 是认证成功但权限不足。 |
| JWT 安全吗？ | 可防篡改但不防泄露，Payload 不是加密，必须短过期并保护密钥。 |
| JWT 怎么退出登录？ | 短过期、黑名单、tokenVersion 或服务端会话状态。 |
| Session 和 JWT 怎么选？ | 传统 Web 可用 Session，前后端分离和微服务常用 JWT，但要补撤销机制。 |
| OAuth2 解决什么？ | 解决第三方应用不拿用户密码也能被授权访问资源。 |
| SSO 核心是什么？ | 统一认证中心签发票据或 Token，业务系统信任认证中心。 |
| RBAC 怎么设计？ | 用户挂角色，角色挂权限，再扩展菜单、按钮、接口和数据权限。 |
| 水平越权怎么防？ | 后端校验资源归属和数据范围，不能只靠前端隐藏。 |
| 密码怎么存？ | BCrypt/Argon2/PBKDF2 等带盐慢哈希，不存明文和可逆密文。 |
| 接口签名防什么？ | 防伪造、防篡改，配合 timestamp 和 nonce 防重放。 |
| CSRF 是什么？ | 利用浏览器自动带 Cookie 发起用户不知情请求。 |
| CORS 是权限吗？ | 不是业务权限，是浏览器跨域访问控制。 |
| 网关鉴权够不够？ | 不够，服务内仍要做业务权限和数据权限。 |
| 多租户安全重点？ | 认证、授权、查询、缓存都必须带 tenantId 隔离。 |

## 面试答题模板

**被问安全认证时可以按这个顺序回答**：

1. 先区分认证、授权、数据权限。
2. 再说明登录态方案：Session、JWT、OAuth2、SSO。
3. 讲 Token 生命周期：签发、校验、续期、退出、踢人。
4. 讲权限模型：用户、角色、权限、资源、租户或组织。
5. 讲接口安全：签名、防重放、幂等、HTTPS。
6. 讲 Web 风险：CSRF、CORS、XSS、Cookie 安全属性。
7. 最后结合项目：网关统一认证，服务内细粒度鉴权，日志审计和可观测排查。

**一句面试总括**：

> 安全认证不能只理解成登录接口。完整方案要覆盖身份可信、权限边界、Token 生命周期、服务间身份透传、接口防篡改防重放、数据权限和线上审计。网关负责统一入口安全，业务服务负责最终资源和数据权限兜底。
