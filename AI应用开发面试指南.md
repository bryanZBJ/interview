# AI 应用开发岗位面试指南与能力地图

本指南面向 Java 后端开发转 AI 应用开发岗位的面试准备，重点不是训练大模型算法，而是讲清楚如何把大模型能力接入真实业务系统，做成可上线、可观测、可控成本、可持续迭代的 AI 应用。

---

## 1. 什么是 AI 应用开发？

AI 应用开发可以理解为：**把大模型能力和业务系统结合起来，构建能理解用户意图、调用工具、检索知识、生成结果并完成业务闭环的应用系统**。

它不是简单调用一次大模型 API，而是围绕模型构建一整套工程能力：

```text
用户输入
  -> 权限校验 / 意图识别 / 上下文组织
  -> RAG 检索 / 工具调用 / 工作流编排
  -> 大模型推理
  -> 结构化输出 / 引用来源 / 人工确认
  -> 日志审计 / 效果评估 / 成本监控
```

典型场景包括：

1. **企业知识库问答**：基于公司文档、制度、接口文档、工单记录回答问题。
2. **智能客服**：理解用户问题，查询订单、售后、工单系统，再生成回复。
3. **代码助手**：理解代码仓库、生成代码、解释报错、辅助 Code Review。
4. **数据分析助手**：自然语言转 SQL，查询报表，解释指标变化。
5. **办公自动化 Agent**：自动写周报、整理会议纪要、创建任务、发送通知。
6. **业务运营助手**：根据用户行为、订单、库存、营销数据生成策略建议。

**面试表达**

> AI 应用开发不是单纯训练模型，而是把大模型接入业务系统。核心工作包括模型接入、Prompt 设计、RAG 知识库、工具调用、工作流编排、效果评估、安全控制和工程化上线。它更接近“后端工程 + 大模型能力 + 业务产品”的结合。

### 1.1. 核心术语白话解释

如果刚接触 AI 应用开发，先不要被英文术语吓住。大部分概念都可以用“后端系统 + 搜索 + 模型调用”的思路理解。

#### 1. RAG 是什么？

RAG 全称是 Retrieval-Augmented Generation，中文一般叫**检索增强生成**。

白话解释：**先查资料，再让大模型根据查到的资料回答**。

大模型本身不一定知道公司内部文档、业务规则、接口说明、订单记录，所以不能直接问它：

```text
公司报销制度是什么？
这个接口怎么调用？
某个工单为什么失败？
```

更可靠的做法是：

```text
用户提问
  -> 系统先去知识库/文档库检索相关内容
  -> 把检索到的片段交给大模型
  -> 大模型基于这些片段组织答案
  -> 返回答案和引用来源
```

可以类比成：**大模型是会总结的人，RAG 是先帮它把相关资料找出来**。

典型例子：

```text
用户：公司的年假规则是什么？
系统：先检索 HR 制度文档
模型：根据检索到的制度片段总结年假规则，并标注来源
```

**面试口径**

> RAG 是检索增强生成，核心是先从企业知识库中检索相关资料，再把资料作为上下文交给大模型生成答案。它主要解决模型不了解企业私有知识、知识更新不及时和回答缺少依据的问题。

#### 2. Embedding 是什么？

Embedding 可以理解为：**把一段文字转换成一组数字向量**。

比如：

```text
"订单退款规则"
  -> [0.12, -0.33, 0.87, ...]
```

为什么要转成数字？因为计算机可以通过向量距离判断两段文本语义是否接近。

例如：

```text
"订单退款规则"
"用户怎么申请退款"
```

这两句话字面不完全一样，但语义接近，向量距离就可能比较近。

**面试口径**

> Embedding 是把文本转换成向量表示，用于语义检索。RAG 中会先把文档片段转成向量存入向量库，用户提问时也转成向量，再找语义最相近的文档片段。

#### 3. 向量数据库是什么？

向量数据库可以理解为：**专门存储和检索向量的数据库**。

普通 MySQL 擅长：

```sql
WHERE user_id = 100
WHERE status = 'SUCCESS'
```

向量数据库擅长：

```text
找出和这个问题语义最相近的 5 段文档
```

常见向量库或向量检索方案包括 Milvus、Pinecone、Weaviate、Qdrant、Elasticsearch 向量检索、Redis Vector 等。

**面试口径**

> 向量数据库用于存储文档片段的 embedding，并支持相似度检索。用户提问时，把问题也转成向量，到向量库中找最相似的文档片段，再交给大模型回答。

#### 4. Prompt 是什么？

Prompt 就是**给模型的指令和上下文**。

它不只是用户输入的一句话，而是后端拼出来的一整段任务说明：

```text
你是一个企业知识库助手。
只能根据提供的资料回答。
如果资料中没有答案，请回答“暂未找到依据”。
请用 Markdown 输出。

资料：
...

用户问题：
...
```

**面试口径**

> Prompt 是应用和大模型之间的任务协议，用来告诉模型角色、任务、约束、上下文和输出格式。生产系统中 Prompt 应该模板化、版本化，并结合结构化输出和后端校验保证稳定性。

#### 5. Token 是什么？

Token 可以理解为模型处理文本时的基本单位，粗略可以理解成“字词片段”。

模型调用成本、上下文长度都和 token 有关：

```text
输入越长 -> token 越多 -> 成本越高 -> 响应越慢
```

所以 RAG 不能把整本文档都塞给模型，只能检索最相关的片段。

**面试口径**

> Token 是模型处理文本的基本单位，影响上下文窗口、调用成本和响应速度。AI 应用要控制 Prompt 长度，只传必要上下文，避免无效 token。

#### 6. 上下文窗口是什么？

上下文窗口是指：**模型一次请求最多能看多少内容**。

如果超过模型上限，就会截断或报错。即使模型支持很长上下文，也不代表应该无限塞内容，因为成本和延迟都会上升。

**面试口径**

> 上下文窗口决定模型一次能处理多少输入内容。RAG 的意义之一就是从大量文档中筛出少量相关片段，避免把无关内容全部塞进上下文。

#### 7. Tool Calling / Function Calling 是什么？

Tool Calling 可以理解为：**模型判断需要调用哪个工具，并生成调用参数；真正执行由后端完成**。

例如用户说：

```text
帮我查一下用户 1001 最近的订单
```

模型不应该自己查数据库，而是输出类似：

```json
{
  "tool": "queryOrder",
  "arguments": {
    "userId": 1001,
    "range": "recent"
  }
}
```

然后后端做：

```text
权限校验 -> 参数校验 -> 调订单接口 -> 返回结果 -> 模型总结
```

**面试口径**

> Tool Calling 的本质是模型生成结构化调用意图和参数，业务后端负责真正执行。这样既能让模型使用业务系统能力，又能保留权限、参数校验、幂等和审计控制。

#### 8. Agent 是什么？

Agent 可以理解为：**能根据目标自己规划步骤、选择工具、观察结果并继续执行的 AI 助手**。

普通问答：

```text
问一句 -> 答一句
```

Agent：

```text
理解目标
  -> 判断要调用哪个工具
  -> 调工具
  -> 看结果
  -> 决定下一步
  -> 输出最终结论
```

例如：

```text
帮我分析这个客户为什么最近投诉变多
```

Agent 可能会依次查询：

```text
订单记录 -> 工单记录 -> 售后记录 -> 客服聊天记录 -> 总结原因
```

**面试口径**

> Agent 是让模型具备一定任务规划和工具选择能力的应用形态。它适合路径不固定、多系统查询和复杂排查类任务；固定流程优先用 workflow，因为 workflow 更可控、更容易测试和审计。

#### 9. Workflow 是什么？

Workflow 是**固定步骤的工作流编排**。

比如：

```text
上传合同
  -> OCR 识别
  -> 提取关键字段
  -> 风险规则校验
  -> 大模型生成摘要
  -> 人工确认
```

每一步都提前定义好，模型只负责其中某些环节。

**面试口径**

> Workflow 适合流程固定、风险较高、需要强控制的业务。Agent 更灵活，但不如 workflow 稳定。生产系统里通常优先 workflow，必要时再引入受控 Agent。

#### 10. 幻觉是什么？

幻觉是指：**模型一本正经地编造不存在的信息**。

比如公司文档里没有某个制度，模型却编出一条规则。

降低幻觉的方法：

1. RAG 检索可信资料。
2. 要求模型基于引用来源回答。
3. 没有资料时允许回答“不知道”。
4. 后端对关键字段做规则校验。
5. 高风险结论人工确认。

**面试口径**

> 幻觉是大模型生成不真实内容的问题。AI 应用不能只靠 Prompt 解决幻觉，而要结合 RAG、引用来源、无证据拒答、结构化校验和人工兜底。

#### 11. 结构化输出是什么？

结构化输出是让模型按固定格式返回，比如 JSON。

例如：

```json
{
  "riskLevel": "HIGH",
  "reason": "用户多次退款且订单金额异常",
  "needHumanReview": true
}
```

这样后端才能稳定解析，而不是每次从自然语言里猜。

**面试口径**

> 结构化输出是让模型按固定 schema 返回结果，方便后端解析、校验和进入业务流程。关键业务场景不能只依赖自然语言回答。

#### 12. 模型网关是什么？

模型网关可以理解为：**统一管理多个模型调用的后端网关**。

它负责：

1. 选择模型。
2. 超时和重试。
3. 限流和配额。
4. 记录 token 成本。
5. 模型降级。
6. 日志审计。

**面试口径**

> 模型网关是 AI 应用的模型调用入口，用来统一处理多模型路由、超时、重试、限流、成本统计和日志审计，避免业务代码直接散落调用各个模型接口。

## 2. AI 应用开发和算法工程有什么区别？

| 方向 | 核心目标 | 主要能力 | 常见产出 |
| :--- | :--- | :--- | :--- |
| 算法 / 模型训练 | 让模型本身更强 | 深度学习、训练、微调、数据集、GPU、模型评测 | 模型、权重、训练方案 |
| AI 应用开发 | 把模型用到业务里 | 后端工程、模型 API、RAG、Agent、工具调用、权限、监控 | AI 产品、智能助手、业务系统能力 |
| AI 平台工程 | 支撑模型和应用规模化运行 | 推理服务、网关、模型管理、向量库、观测平台、成本治理 | 模型平台、推理平台、AI 网关 |

对于 Java 后端开发来说，转 AI 应用开发更现实的路径通常是第二类：**用已有模型做业务应用落地**。

**面试表达**

> 算法工程更关注模型怎么训练得更好，AI 应用开发更关注模型怎么在业务系统里稳定、安全、低成本地产生价值。比如做一个知识库问答系统，我不一定训练模型，但要解决文档切分、向量检索、权限过滤、Prompt 组织、引用来源、幻觉控制、效果评估和服务稳定性问题。

## 3. 典型 AI 应用系统架构

一个生产级 AI 应用通常不是“前端直接调模型”，而是有完整后端链路：

```text
[前端/客户端]
      |
      v
[AI 应用后端]
  - 用户认证与权限
  - 会话管理
  - Prompt 模板
  - RAG 检索
  - 工具调用编排
  - 模型网关
  - 结果解析
  - 日志与审计
      |
      +--> [大模型 API / 本地模型]
      +--> [向量数据库 / 搜索引擎]
      +--> [业务系统 API]
      +--> [MySQL / Redis / MQ]
      +--> [监控 / 评估 / 告警]
```

### 3.1. 核心模块

1. **模型接入层**
   - 支持 OpenAI、通义、文心、DeepSeek、Claude、本地模型等。
   - 统一封装模型调用、超时、重试、流式响应、模型降级。

2. **Prompt 管理层**
   - 管理系统提示词、角色设定、输出格式、示例、约束条件。
   - 支持版本化、灰度、AB 实验。

3. **RAG 检索层**
   - 文档解析、切分、Embedding、向量检索、关键词检索、重排序。
   - 解决模型不知道企业私有知识的问题。

4. **工具调用层**
   - 让模型调用订单查询、工单创建、库存查询、CRM、日历、邮件等业务 API。
   - 重点是权限、参数校验、幂等、审计、人工确认。

5. **结果处理层**
   - 结构化 JSON 输出解析。
   - 引用来源返回。
   - 敏感信息过滤。
   - 失败兜底和人工接管。

6. **评估与观测层**
   - 记录输入、检索结果、模型输出、延迟、token、成本、用户反馈。
   - 定期评估准确率、召回率、幻觉率和业务完成率。

## 4. 面试 AI 应用开发岗位需要具备什么能力？

### 4.1. 后端工程能力

AI 应用本质上仍然是线上业务系统，所以后端基础不能弱：

1. Java / Spring Boot / Spring Cloud 基础扎实。
2. 熟悉接口设计、鉴权、限流、幂等、异常处理。
3. 熟悉 MySQL、Redis、MQ、ES 等基础设施。
4. 能设计稳定的异步任务、补偿任务、审计日志。
5. 能处理高并发、超时、重试、降级、熔断。

**面试表达**

> AI 应用不是只调模型接口，后端工程能力仍然很重要。模型调用可能慢、贵、不稳定，还可能输出不确定结果，所以要有超时、重试、降级、缓存、日志审计和人工兜底。

### 4.2. 大模型基础能力

需要理解以下概念：

| 概念 | 面试要点 |
| :--- | :--- |
| Token | 模型处理文本的基本单位，影响上下文长度和调用成本 |
| Context Window | 模型一次能看到的上下文上限，RAG 不能无限塞文档 |
| Temperature | 控制输出随机性，越高越发散，越低越稳定 |
| Top-p | 控制候选词采样范围，和 temperature 都影响创造性 |
| Embedding | 把文本转换成向量，用于语义检索 |
| Function Calling / Tool Calling | 模型按 schema 生成工具调用参数，由应用后端真正执行 |
| Streaming | 流式返回，提升用户体验，但要处理连接中断和增量渲染 |
| Structured Output | 让模型按 JSON schema 等结构化格式输出，方便程序解析 |

**面试表达**

> 我需要理解模型不是传统确定性接口。它有 token 成本、上下文窗口、随机性和幻觉问题，所以应用层要通过 Prompt、检索增强、工具调用、结构化输出和评估体系来约束它。

### 4.3. Prompt 工程能力

Prompt 不是玄学文案，而是应用协议的一部分。

一个可维护的 Prompt 通常包括：

1. 角色：你是谁。
2. 任务：你要完成什么。
3. 背景：业务上下文是什么。
4. 约束：不能做什么，必须遵守什么规则。
5. 输出格式：JSON、Markdown、表格或固定字段。
6. 示例：给出 few-shot 示例提升稳定性。
7. 安全边界：遇到无权限、无证据、无法判断时如何处理。

**面试表达**

> Prompt 在工程里应该版本化、模板化、可评估，而不是写死在代码里。对于关键业务场景，我会要求模型输出结构化 JSON，并在后端做参数校验和兜底。

### 4.4. RAG 知识库能力

RAG 的核心是：**不要让模型凭记忆回答，而是先检索可信知识，再基于检索内容回答**。

典型流程：

```text
文档采集
  -> 文档解析
  -> 文本切分 chunk
  -> 生成 embedding
  -> 写入向量库
  -> 用户提问
  -> 查询向量库 / 关键词检索
  -> 重排序 rerank
  -> 拼接上下文
  -> 大模型生成答案
  -> 返回引用来源
```

面试重点：

1. chunk 怎么切：按标题、段落、语义、长度综合切。
2. 检索怎么做：向量检索 + 关键词检索混合效果更稳。
3. 怎么减少幻觉：只允许基于检索证据回答，返回引用来源。
4. 怎么处理权限：先按用户权限过滤文档，再检索或检索后过滤。
5. 怎么评估效果：看召回率、答案准确率、引用命中率、用户反馈。

**面试表达**

> RAG 不是简单把文档丢给模型，而是文档解析、切分、向量化、检索、重排序、Prompt 组织和答案评估的一整套链路。真正上线时还要考虑文档权限、增量更新、引用来源、缓存和幻觉控制。

### 4.5. Agent 与工具调用能力

Agent 可以理解为：模型不仅回答问题，还能根据目标选择工具、调用系统、观察结果并继续决策。

典型工具调用流程：

```text
用户：帮我查这个客户最近 3 个月异常订单
  -> 模型判断需要调用 queryOrderTool
  -> 生成工具参数：customerId、timeRange、status
  -> 后端校验权限和参数
  -> 调订单系统
  -> 返回工具结果给模型
  -> 模型总结异常原因和建议
```

关键点：

1. 工具 schema 要清晰，参数类型、枚举、必填字段要明确。
2. 模型只负责“决定调什么、生成参数”，真正执行必须由后端完成。
3. 高风险动作要人工确认，比如退款、删除、发通知。
4. 工具调用要有权限、幂等、审计、限流。
5. Agent 不适合所有场景，固定流程优先用 workflow。

**面试表达**

> 工具调用不是让模型直接操作数据库，而是让模型生成符合 schema 的调用意图和参数，后端再做权限校验、参数校验、幂等和审计后执行。对于高风险操作，我会加入人工确认和可回滚机制。

### 4.6. 评估与可观测能力

AI 应用最大的问题是结果不完全确定，所以必须有评估体系。

常见指标：

| 指标 | 说明 |
| :--- | :--- |
| 检索召回率 | 正确知识是否被检索出来 |
| 答案准确率 | 回答是否符合事实和业务规则 |
| 幻觉率 | 是否编造不存在的信息 |
| 引用命中率 | 答案是否基于引用来源 |
| 工具调用成功率 | Agent 是否选对工具并传对参数 |
| 延迟 P95/P99 | 用户体验和系统稳定性 |
| Token 成本 | 每次调用消耗和整体成本 |
| 用户反馈 | 点赞、点踩、人工纠错 |

**面试表达**

> AI 应用不能只看 demo 效果，要看线上评估。RAG 要评估检索召回和答案准确率，Agent 要评估工具选择和参数正确率，整体还要看延迟、成本、失败率和用户反馈。

### 4.7. 安全与合规能力

AI 应用要特别关注：

1. Prompt Injection：用户诱导模型忽略系统规则。
2. 数据越权：用户问到了自己无权访问的文档或订单。
3. 敏感信息泄露：手机号、身份证、密钥、内部系统信息。
4. 幻觉风险：编造政策、金额、操作结论。
5. 高风险动作：自动退款、删除数据、发送外部消息。

治理手段：

1. 权限前置：检索和工具调用前先做权限校验。
2. 证据约束：没有检索证据就回答“不确定”。
3. 输出过滤：敏感信息脱敏。
4. 人工确认：高风险动作必须二次确认。
5. 审计日志：记录输入、输出、工具调用和执行结果。

**面试表达**

> AI 应用安全的核心是不能把模型当成可信执行主体。模型输出要经过业务规则、权限、参数和风险校验，尤其是工具调用和企业知识库场景，必须防越权、防注入、防敏感信息泄露。

## 5. 高频面试题与答案

### Q1：AI 应用开发的核心难点是什么？

**答案**

核心难点不是“把模型 API 调通”，而是让模型在真实业务系统里稳定地产生价值。

常见难点：

1. 模型输出不确定，需要结构化输出和兜底。
2. 模型知识不包含企业私有数据，需要 RAG。
3. 模型可能幻觉，需要证据约束和引用来源。
4. 工具调用可能误操作，需要权限、参数校验、人工确认。
5. 模型调用成本高，需要缓存、限流、模型分级。
6. 效果难衡量，需要评估集和线上反馈闭环。

**面试表达**

> AI 应用开发的难点在工程化闭环：怎么让模型接入业务知识、调用业务工具、输出可解析结果，并且在权限、安全、成本、延迟和效果评估上可控。

### Q2：RAG 和微调有什么区别？什么时候用 RAG？

**答案**

| 方案 | 适合解决 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| RAG | 企业知识、实时知识、文档问答 | 数据更新快，可返回引用来源，成本相对低 | 检索链路复杂，依赖知识质量 |
| 微调 | 风格、格式、特定任务模式 | 输出风格更稳定，特定任务效果好 | 不适合频繁更新事实知识，训练成本高 |

一般企业知识库、政策制度、接口文档、订单知识、FAQ 更适合先用 RAG。

**面试表达**

> 如果问题是模型不知道企业内部知识，我优先用 RAG；如果问题是模型输出风格、格式或特定任务能力不稳定，再考虑微调。RAG 更适合知识频繁更新且需要引用来源的场景。

### Q3：怎么降低大模型幻觉？

**答案**

1. RAG 检索可信知识。
2. Prompt 中要求“只能基于提供材料回答”。
3. 返回引用来源。
4. 没有证据时允许回答“不知道”。
5. 对关键字段做规则校验。
6. 高风险结论交给人工确认。
7. 构建评估集，持续评估幻觉率。

**面试表达**

> 降低幻觉不能只靠一句 Prompt，而要靠检索证据、引用来源、结构化输出校验、无证据拒答和人工兜底。对于业务关键场景，模型只负责生成建议，最终执行要经过规则和权限校验。

### Q4：Function Calling / Tool Calling 的本质是什么？

**答案**

本质是让模型按预定义 schema 生成工具调用参数，后端根据参数真正执行。

模型不应该直接操作数据库或系统。

```text
模型：我需要调用 queryOrder，参数是 userId=1001
后端：校验权限、参数、限流、审计
后端：真正调用订单系统
后端：把结果返回给模型总结
```

**面试表达**

> Tool Calling 的本质是模型生成结构化调用意图，业务后端负责真正执行。它把自然语言理解和业务系统执行解耦，既能增强模型能力，也能保留权限、幂等和审计控制。

### Q5：Agent 和 Workflow 怎么选？

**答案**

| 方案 | 特点 | 适合场景 |
| :--- | :--- | :--- |
| Workflow | 流程固定、步骤明确 | 审批、报表生成、固定客服流程 |
| Agent | 目标明确但路径不固定，模型自主选择工具 | 复杂排查、多系统查询、开放式助手 |

生产系统里通常优先 workflow，只有流程不固定且收益明显时才引入 Agent。

**面试表达**

> 我不会为了 Agent 而 Agent。固定业务流程优先用 workflow，可控、可测、可审计；只有任务路径不确定，需要模型动态选择工具时，才考虑 Agent，并且要限制工具范围和动作权限。

### Q6：AI 应用怎么做性能和成本优化？

**答案**

1. 模型分级：简单任务用小模型，复杂任务用强模型。
2. 缓存：相同问题、相同检索结果、常见 FAQ 可缓存。
3. 减少上下文：只传必要文档，避免无效 token。
4. 流式输出：降低首字延迟。
5. 超时和降级：模型慢或失败时返回兜底答案。
6. 批量 embedding：离线处理文档向量。
7. 限流和配额：按用户、应用、租户控制成本。

**面试表达**

> 成本优化核心是少调、调小、少传、可缓存。工程上会做模型分级、Prompt 压缩、检索结果裁剪、缓存、限流和成本监控。

### Q7：Java 后端怎么设计一个 RAG 服务？

**答案**

可以拆成几类服务：

1. `DocumentIngestService`：文档接入、解析、切分。
2. `EmbeddingService`：调用 embedding 模型生成向量。
3. `VectorStoreService`：写入和查询向量库。
4. `RetrievalService`：向量检索、关键词检索、重排序。
5. `PromptService`：拼接系统提示词、用户问题、检索上下文。
6. `ModelGateway`：统一模型调用、重试、超时、流式响应。
7. `AiAuditService`：记录输入、输出、引用、成本和反馈。

**面试表达**

> 我会把 RAG 做成独立链路：文档离线入库，查询在线检索。离线负责解析、切分、向量化和索引；在线负责权限过滤、检索、重排序、Prompt 拼接、模型调用和审计。这样可扩展，也方便定位问题。

### Q8：如何处理 AI 应用中的权限问题？

**答案**

权限要进入检索和工具调用链路，而不是只在页面控制。

1. 用户提问前校验身份。
2. 检索时只查用户有权限的知识库和文档。
3. 工具调用前校验接口权限和数据权限。
4. 模型输出前做敏感信息过滤。
5. 所有工具调用写审计日志。

**面试表达**

> AI 应用里的权限不能只靠 Prompt 约束。RAG 要做文档级权限过滤，工具调用要做接口级和数据级权限校验，输出还要做脱敏和审计。

## 6. AI 应用开发岗位准备路线

### 第一阶段：补齐大模型基础

需要掌握：

1. Token、上下文窗口、temperature、top-p。
2. Prompt 模板、few-shot、结构化输出。
3. Embedding、向量相似度、向量库。
4. Function Calling / Tool Calling。
5. RAG 基本流程。

输出成果：

1. 写一个命令行或 Web 聊天 Demo。
2. 支持流式输出。
3. 支持 JSON 格式输出。

### 第二阶段：做一个 RAG 知识库项目

项目要求：

1. 支持上传 PDF / Markdown / Word。
2. 文档切分、向量化、入库。
3. 支持相似度检索和引用来源。
4. 支持用户反馈。
5. 记录每次问答的 token、延迟、模型、引用文档。

面试亮点：

1. chunk 策略。
2. 混合检索。
3. 幻觉控制。
4. 权限过滤。
5. 评估集。

### 第三阶段：做一个工具调用 Agent 项目

项目要求：

1. 定义 3 到 5 个业务工具，比如查订单、查库存、创建任务、发通知。
2. 工具参数用 JSON schema 描述。
3. 后端执行前做权限和参数校验。
4. 高风险动作加入人工确认。
5. 记录工具调用链路。

面试亮点：

1. 工具选择正确率。
2. 参数校验。
3. 幂等设计。
4. 审计日志。
5. Agent 和 workflow 的取舍。

### 第四阶段：工程化与上线能力

需要补齐：

1. 模型网关：多模型路由、降级、限流。
2. 成本监控：按用户、租户、接口统计 token。
3. Prompt 版本管理。
4. 评估集和回归测试。
5. 日志脱敏和合规。

## 7. 简历项目可以怎么写？

### 项目 1：企业知识库智能问答系统

**项目描述**

基于 Spring Boot + 向量数据库 + 大模型 API 构建企业知识库问答系统，支持文档上传、自动切分、向量化检索、引用来源返回和用户反馈闭环。

**可写亮点**

1. 设计文档解析和 chunk 切分流程，提升检索命中率。
2. 引入向量检索 + 关键词检索混合召回，提高专业术语和语义问题的覆盖。
3. Prompt 中强制要求基于引用材料回答，无证据时拒答，降低幻觉。
4. 记录 token、延迟、引用文档和用户反馈，支持后续评估优化。
5. 对知识库文档做权限过滤，防止跨部门数据泄露。

**面试表达**

> 这个项目不是简单接模型，而是实现了完整 RAG 链路。离线侧负责文档解析、切分、向量化和入库；在线侧负责权限过滤、检索、重排序、Prompt 拼接、模型调用和审计。为了降低幻觉，我要求答案必须引用检索来源，没有证据就拒答。

### 项目 2：业务系统智能助手

**项目描述**

基于大模型 Tool Calling 能力构建业务智能助手，支持自然语言查询订单、库存、工单和客户信息，并根据结果生成处理建议。

**可写亮点**

1. 将订单查询、库存查询、工单创建封装为模型可调用工具。
2. 使用 JSON schema 约束工具参数，后端执行前做权限和参数校验。
3. 高风险操作加入人工确认，防止模型误操作。
4. 工具调用全链路记录审计日志，便于问题追踪。
5. 对模型超时和工具失败做降级处理。

**面试表达**

> 我把模型定位为意图理解和参数生成层，真正执行业务动作的是后端工具服务。所有工具调用都要经过权限、参数、幂等和审计校验，高风险动作必须人工确认。
