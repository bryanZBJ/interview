## 1. ZooKeeper 是什么？适合解决什么问题？

**一句话回答**：ZooKeeper 是一个分布式协调服务，核心价值不是存大数据，而是提供强一致的小数据存储、节点监听、临时节点和顺序节点，用来做注册发现、配置管理、分布式锁、Leader 选举等协调类场景。

**核心考点**：

- ZooKeeper 更像“协调中心”，不是业务数据库。
- 存储的是少量元数据，例如服务地址、配置版本、锁节点、选主信息。
- 通过 ZNode 树形结构组织数据。
- 通过 Watcher 机制通知客户端节点变化。
- 通过临时节点感知客户端存活状态。

**三点总结**：

1. ZooKeeper 适合一致性要求高、数据量小、变更不特别频繁的协调场景。
2. ZooKeeper 不适合高频大数据读写，也不适合当缓存或业务存储。
3. 面试回答时要把“协调”讲清楚：注册、锁、选主、配置本质都是节点状态协调。

**面试表达**：

> ZooKeeper 本质上是分布式协调组件，我不会把它当数据库用。它更适合保存服务注册信息、锁节点、配置版本、Leader 信息这类小而关键的数据，通过临时节点、顺序节点和 Watcher 机制帮助多个服务实例在分布式环境下达成一致。

## 2. ZooKeeper 的整体架构是什么？

**一句话回答**：ZooKeeper 集群由 Leader、Follower、Observer 组成，Leader 负责写请求协调和事务提案，Follower 参与投票并处理读请求，Observer 只同步数据和处理读请求，不参与投票。

| 角色 | 作用 | 是否参与投票 |
| --- | --- | --- |
| Leader | 处理写请求，发起事务提案，协调多数派提交 | 是 |
| Follower | 参与 Leader 选举和写事务投票，也能处理读请求 | 是 |
| Observer | 同步 Leader 数据，分担读流量，不参与投票 | 否 |

**关键细节**：

- 写请求最终都要经过 Leader 协调。
- 事务提交需要超过半数节点确认，所以集群通常部署奇数台。
- Observer 可以提高读扩展能力，但不提升写入多数派能力。
- 客户端连接任意节点都可以工作，写请求会被转发给 Leader。

**三台生产集群怎么组成**：

如果生产只部署 3 台 ZooKeeper，通常是：

```text
1 个 Leader + 2 个 Follower
```

一般不建议在 3 台里放 Observer。原因是 ZooKeeper 的写入、选主和事务提交都依赖多数派。3 台机器的多数派是：

```text
floor(3 / 2) + 1 = 2
```

也就是说，Leader 提交写事务时，至少要拿到自己 + 1 个 Follower 的确认。如果部署成：

```text
1 个 Leader + 1 个 Follower + 1 个 Observer
```

真正参与投票的节点只有 Leader 和 Follower 两台，任意一个投票节点故障后就无法形成多数派，写入和重新选主能力都会很脆弱。

| 角色 | 是否参与投票 | 是否参与事务提交确认 | 适合放在 3 节点集群吗 |
| --- | --- | --- | --- |
| Leader | 是 | 是 | 必须有 |
| Follower | 是 | 是 | 需要，3 节点一般放 2 个 |
| Observer | 否 | 否 | 一般不放，适合更大集群扩展读能力 |

注意 Leader 不是固定机器，而是选举出来的。三台集群正常运行时表现为 1 Leader + 2 Follower；当前 Leader 挂掉后，剩下两个 Follower 只要还能通信，就能重新选出一个 Leader。

**面试表达**：

> ZooKeeper 是典型主从协调架构。Leader 负责写事务和提案，Follower 参与投票，Observer 用来扩展读能力但不参与选举。生产如果只部署三台，一般就是 1 个 Leader 加 2 个 Follower，不建议放 Observer，因为 Observer 不参与投票。三节点集群多数派是 2，只有 3 台都作为投票节点，才能在挂 1 台时仍然具备选主和写入能力。Observer 更适合在 5 台、7 台这类较大集群里扩展读能力。

## 3. 为什么 ZooKeeper 集群一般部署奇数台？

**一句话回答**：因为 ZooKeeper 的写提交和选主都依赖多数派，奇数台和相邻偶数台在容错能力上通常相同，但偶数台成本更高、投票更复杂。

**多数派计算公式**：

```text
多数派 = floor(投票节点总数 / 2) + 1
```

也可以理解成：必须超过一半节点同意，才能选出 Leader 或提交写事务。

| 节点数 | 多数派 | 最多容忍故障数 |
| --- | --- | --- |
| 3 台 | 2 台 | 1 台 |
| 4 台 | 3 台 | 1 台 |
| 5 台 | 3 台 | 2 台 |
| 6 台 | 4 台 | 2 台 |

**三点总结**：

1. 3 台能容忍 1 台故障，4 台也只能容忍 1 台故障，所以 4 台性价比不高。
2. 5 台能容忍 2 台故障，是更高可用级别的常见选择。
3. 节点越多，写事务投票和同步成本越高，不是越多越好。

**面试表达**：

> ZooKeeper 依赖多数派机制，多数派公式是 `floor(n / 2) + 1`。3 台需要 2 台确认，4 台需要 3 台确认，但 3 台和 4 台都只能容忍 1 台故障，所以 4 台性价比不高。线上一般选 3 或 5 台，核心是平衡容错能力和写入投票成本。

## 4. ZooKeeper 的 ZNode 有哪些类型？

**一句话回答**：ZNode 主要分持久节点、临时节点、持久顺序节点、临时顺序节点，面试最常考临时节点和临时顺序节点。

| 节点类型   | 生命周期             | 常见用途           |
| ------ | ---------------- | -------------- |
| 持久节点   | 显式删除前一直存在        | 配置目录、固定元数据     |
| 临时节点   | Session 断开后删除    | 服务注册、存活探测      |
| 持久顺序节点 | 持久存在且带递增序号       | 全局有序任务、事件记录    |
| 临时顺序节点 | Session 断开删除且带序号 | 分布式锁、Leader 选举 |

**关键细节**：

- 临时节点不能创建子节点。
- 顺序节点由 ZooKeeper 自动追加单调递增序号。
- 服务注册常用临时节点，因为实例宕机后节点会自动消失。
- 公平锁常用临时顺序节点，序号最小的客户端获得锁。

**面试表达**：

> 我会重点讲临时节点和临时顺序节点。临时节点绑定客户端 Session，适合做服务存活感知；临时顺序节点既能自动释放，又有全局顺序，所以适合做公平分布式锁和 Leader 选举。

## 5. ZooKeeper 的 Watcher 机制是什么？

**一句话回答**：Watcher 是 ZooKeeper 的事件监听机制，客户端可以监听节点数据或子节点变化，变化发生后服务端向客户端发送通知，客户端再重新读取最新数据并重新注册监听。

**核心流程**：

1. 客户端读取节点时注册 Watcher。
2. 节点数据或子节点发生变化。
3. ZooKeeper 服务端向客户端推送事件通知。
4. 客户端收到通知后重新拉取数据。
5. 如果还要继续监听，需要再次注册 Watcher。

**常见追问**：

- Watcher 是一次性的吗？是，一次触发后需要重新注册。
- Watcher 推送的是数据还是事件？主要是事件通知，不是完整数据。
- Watcher 会不会丢？Session 断开、网络抖动、客户端处理不及时都可能导致感知延迟，所以业务要能重新拉取全量状态。

**面试表达**：

> ZooKeeper Watcher 不是持续订阅流，而是一次性事件通知。客户端收到通知后应该重新拉取节点数据，并重新注册 Watcher。生产上不能只依赖通知内容做决策，要以重新读取到的 ZooKeeper 状态为准。

## 6. ZooKeeper 的 Session 机制怎么理解？

**一句话回答**：Session 是客户端和 ZooKeeper 集群之间的会话，客户端通过心跳维持 Session，超过超时时间后临时节点会被删除，依赖它的注册信息、锁信息也会自动释放。

**关键细节**：

- 客户端连接断开不一定马上 Session 失效。
- 只要在超时时间内重连成功，Session 可以继续保持。
- Session 过期后，临时节点删除，Watcher 会被触发。
- Session 超时时间太短容易误删，太长会导致故障感知变慢。

**面试表达**：

> ZooKeeper 的临时节点不是 TCP 连接断开就立即删除，而是和 Session 生命周期绑定。短暂网络抖动时客户端可能重连成功并恢复会话；只有超过 Session timeout，ZooKeeper 才认为客户端真正失效并删除临时节点。

## 7. ZooKeeper 的 ZAB 协议是什么？

**一句话回答**：ZAB 是 ZooKeeper 的原子广播协议，用来保证写事务按全局顺序复制到多数派节点，并在 Leader 故障后通过选举和恢复保证数据一致。

**通俗理解**：

ZAB 可以理解成 ZooKeeper 集群里的一套“写入同步规则”。ZooKeeper 所有写请求都要经过 Leader，Leader 给每个写事务分配一个全局递增的事务 ID，也就是 `zxid`，再把这个事务广播给 Follower。只有多数派节点确认后，这个事务才算提交成功。

它解决两个核心问题：

1. **正常运行时怎么写一致**：所有写事务按 Leader 分配的 `zxid` 顺序广播、确认、提交，保证各节点最终按同样顺序应用事务。
2. **Leader 挂了之后怎么恢复一致**：重新选出的 Leader 必须拥有最新的已提交事务，并保证已提交事务不丢、未提交事务不乱提交。

**一次写请求的大致流程**：

```text
客户端发起写请求
  -> 请求转发到 Leader
  -> Leader 生成 proposal，并分配 zxid
  -> Leader 把 proposal 广播给 Follower
  -> Follower 写入本地事务日志并 ack
  -> Leader 收到多数派 ack
  -> Leader 提交事务并通知 Follower commit
  -> 客户端收到写成功
```

**追问：如果有个 Follower 没有 ack 呢？**

不要求所有 Follower 都 ack，只要达到多数派就可以提交。没有 ack 的 Follower 可能是网络慢、磁盘慢、GC 卡顿、短暂断连，它会在后续通过 Leader 的同步机制把缺失的事务补上。

以 3 台集群为例：

```text
1 Leader + 2 Follower
多数派 = 2

Leader 自己算 1 票
只要再收到 1 个 Follower ack
就达到多数派 2，可以提交
```

也就是说，3 台集群里如果一个 Follower 没 ack，另一个 Follower ack 了，写请求仍然可以成功提交。如果两个 Follower 都没有 ack，Leader 只有自己 1 票，达不到多数派，写请求就不能提交成功。

以 5 台集群为例：

```text
1 Leader + 4 Follower
多数派 = 3

Leader 自己算 1 票
只要再收到 2 个 Follower ack
就达到多数派 3，可以提交
```

所以 5 台集群可以容忍部分 Follower 没 ack，只要多数派够，写请求就可以提交；没跟上的 Follower 后面会根据 `zxid` 和事务日志向 Leader 补齐数据。

**核心阶段**：

1. **消息广播**：Leader 为写事务生成 proposal，发送给 Follower。
2. **多数派确认**：超过半数 Follower ack 后，Leader 提交事务。
3. **崩溃恢复**：Leader 故障后重新选主，新 Leader 保证已提交事务不丢、未提交事务不乱。

**容易混淆点**：

- ZAB 是 ZooKeeper 为主从事务复制和崩溃恢复设计的原子广播协议。
- ZAB 强调主备模式下的原子广播和崩溃恢复。
- ZooKeeper 对所有写操作分配 zxid，用来标识事务顺序。

**面试表达**：

> ZAB 解决的是 ZooKeeper 写事务在集群里的顺序复制问题。所有写请求由 Leader 生成全局递增的事务 ID，也就是 zxid，再通过多数派确认提交。Leader 挂掉后，新 Leader 会基于 zxid 做恢复，保证已提交的不丢、未提交的不被错误提交。

## 8. ZooKeeper 的 Leader 选举过程怎么讲？

**一句话回答**：ZooKeeper 选主主要比较 zxid 和 serverId，优先选择事务更新程度最高的节点作为 Leader，如果 zxid 相同，再比较 serverId。

**核心规则**：

1. 每个节点发起投票，推荐自己或更合适的节点。
2. 比较候选节点的 zxid，zxid 越大表示数据越新。
3. zxid 相同则比较 serverId，serverId 大的优先。
4. 某个候选者获得超过半数投票后成为 Leader。

**追问：Leader 选举不需要达到多数派吗？**

需要。ZooKeeper 选举 Leader 也必须获得多数派投票，不是某台节点自己觉得自己是 Leader 就可以对外工作。

多数派公式仍然是：

```text
多数派 = floor(投票节点数 / 2) + 1
```

以 3 台投票节点为例：

```text
ZK1、ZK2、ZK3
多数派 = 2

某个候选者拿到 2 票，才能成为 Leader
如果只剩 1 台节点，它可以投自己，但只有 1 票，达不到多数派，不能成为有效 Leader
```

以 5 台投票节点为例：

```text
多数派 = 3
候选者必须拿到 3 票，才能成为 Leader
```

这里的“票”只来自参与投票的节点，也就是 Leader / Follower 这类 voting server。Observer 不参与 Leader 选举，也不计入多数派。

所以可以把 ZooKeeper 的多数派理解成两层：

1. **选举阶段**：候选 Leader 必须拿到多数派投票，才能成为合法 Leader。
2. **写入阶段**：Leader 提交写事务时，也必须拿到多数派 ack，才能提交成功。

这也是 ZooKeeper 防脑裂的关键：网络分区后，少数派既选不出合法 Leader，也提交不了写事务。

**追问：如果达不到多数派会怎样？**

要分两种场景看：

| 场景 | 达不到多数派的结果 |
| --- | --- |
| 选举阶段 | 选不出合法 Leader，集群不能对外提供写能力 |
| 已有 Leader 写入阶段 | Leader 收不到足够 ack，写事务不能提交成功 |

比如 3 台集群只剩 1 台可通信：

```text
投票节点数 = 3
多数派 = 2
当前可通信节点 = 1

结果：达不到多数派，不能选出有效 Leader，也不能提交写事务
```

如果原来已经有 Leader，但网络分区后 Leader 所在分区只剩自己 1 台，它也不能继续“自封有效”。因为提交写事务时拿不到多数派 ack，后续也会进入重新选举或不可用状态。

这时 ZooKeeper 对写请求的表现就是不可用：新 Provider 不能可靠注册，配置和路由不能提交，分布式锁节点也不能创建成功。读请求是否还能从某个节点读到旧数据，要看客户端连接和配置，但这不能代表集群还能正常提交状态变更。

**面试表达**：

> Leader 选举不是随便选一台，而是候选者必须拿到多数派投票才能成为合法 Leader。投票时会优先比较 zxid，zxid 大说明事务进度更新；如果 zxid 相同再比较 serverId。这样既能尽量选出数据最新的节点，也能保证网络分区时少数派选不出 Leader，避免脑裂。

## 9. ZooKeeper 如何实现分布式锁？

**一句话回答**：ZooKeeper 通常用临时顺序节点实现公平分布式锁，客户端在锁目录下创建临时顺序节点，序号最小者获得锁，其他客户端监听自己前一个节点，前一个节点删除后再尝试获取锁。

**流程**：

1. 客户端在 `/locks/order` 下创建临时顺序节点。
2. 获取同目录下所有子节点并排序。
3. 如果自己的序号最小，获得锁。
4. 如果不是最小，监听前一个节点。
5. 前一个节点删除后，重新判断自己是否最小。
6. 业务执行完删除自己的节点；客户端宕机时 Session 过期自动删除。

**为什么监听前一个节点**：

- 避免所有等待者都监听最小节点。
- 避免锁释放时大量客户端同时被唤醒。
- 降低羊群效应。

**面试表达**：

> ZooKeeper 锁的关键是临时顺序节点。临时节点保证客户端异常退出后锁能自动释放，顺序节点保证公平性。等待锁时不要所有客户端都监听当前最小节点，而是监听自己的前一个节点，这样锁释放时只唤醒一个候选者，避免羊群效应。

## 10. Redis 分布式锁和 ZooKeeper 分布式锁怎么选？

**一句话回答**：Redis 锁性能更高、实现更轻，适合大多数短临界区业务；ZooKeeper 锁一致性和自动释放语义更强，适合更看重安全性和公平性的协调场景。

| 对比项  | Redis 锁          | ZooKeeper 锁        |
| ---- | ---------------- | ------------------ |
| 性能   | 高                | 相对低                |
| 自动释放 | 依赖 TTL / 看门狗     | Session 过期自动删除临时节点 |
| 公平性  | 默认不保证            | 临时顺序节点天然更容易做公平     |
| 一致性  | 主从切换时要谨慎         | CP 语义更强            |
| 适合场景 | 秒杀、幂等、防重复提交、短业务锁 | 选主、强协调、低频关键锁       |

**三点总结**：

1. 高并发短锁优先 Redis / Redisson，但必须有过期时间、唯一 value、Lua 解锁和幂等兜底。
2. 更强调一致性、公平性和自动释放，可以考虑 ZooKeeper。
3. 分布式锁不是最终防线，核心资金、库存、状态流转还要靠数据库约束、状态机和补偿。

**面试表达**：

> 我一般不会绝对说哪个更好，而是看业务取舍。Redis 锁性能高，适合高频短临界区；ZooKeeper 锁更偏强协调，临时顺序节点天然支持公平和自动释放。无论用哪种锁，最终都不能替代数据库唯一约束、幂等和状态机。

## 11. ZooKeeper 做注册中心的原理是什么？

**一句话回答**：服务提供者启动后在 ZooKeeper 指定路径下创建临时节点，消费者监听服务目录变化并拉取服务列表，提供者宕机后临时节点自动删除，消费者感知后刷新本地地址列表。

**注册中心里放什么**：

ZooKeeper 做注册中心时，里面通常不放业务数据，只放少量服务元数据，例如服务名、实例地址、协议、端口、权重、版本、分组等。可以抽象成这样的目录：

```text
/registry
  /order-service
    /providers
      /192.168.1.10:20880
      /192.168.1.11:20880
    /consumers
      /192.168.2.20
```

不同框架路径会不同，比如 Dubbo 早期常见路径类似：

```text
/dubbo/com.xxx.OrderService/providers/provider-url-encoded
/dubbo/com.xxx.OrderService/consumers/consumer-url-encoded
/dubbo/com.xxx.OrderService/configurators
/dubbo/com.xxx.OrderService/routers
```

面试里不需要死背具体路径，关键是理解：**服务名下面挂 Provider 地址列表，Consumer 订阅这个服务目录**。

**Dubbo 和 ZooKeeper 配合使用的原理**：

Dubbo 把 ZooKeeper 当成注册中心，ZooKeeper 不直接参与 RPC 调用，只负责保存和通知服务元数据。真正的业务请求仍然是 Consumer 直接调用 Provider。

```text
Provider 启动
  -> 暴露 Dubbo 服务
  -> 向 ZooKeeper 注册 provider URL
  -> ZooKeeper 保存临时节点

Consumer 启动
  -> 订阅指定接口的 providers 目录
  -> 拉取 provider URL 列表
  -> 在本地生成代理和 Invoker 列表
  -> 按负载均衡策略直连 Provider 发起 RPC
```

以 `OrderService` 为例，Dubbo 在 ZooKeeper 中通常会按接口维度组织节点：

```text
/dubbo
  /com.xxx.OrderService
    /providers
      /dubbo%3A%2F%2F192.168.1.10%3A20880%2Fcom.xxx.OrderService...
      /dubbo%3A%2F%2F192.168.1.11%3A20880%2Fcom.xxx.OrderService...
    /consumers
      /consumer%3A%2F%2F192.168.2.20%2Fcom.xxx.OrderService...
    /routers
    /configurators
```

这里的 provider 节点本质是 URL 编码后的服务地址和参数，里面会包含协议、IP、端口、接口名、版本、分组、权重、超时时间、序列化方式等。Consumer 拿到这些 URL 后，会在本地构建可调用的 Invoker 集合，后续请求不需要经过 ZooKeeper。

**Dubbo + ZooKeeper 的使用步骤**：

1. 部署 ZooKeeper 集群，例如 `zk1:2181,zk2:2181,zk3:2181`。
2. Provider 工程引入 Dubbo 和 ZooKeeper 注册中心依赖。
3. 配置注册中心地址，例如 `dubbo.registry.address=zookeeper://zk1:2181?backup=zk2:2181,zk3:2181`。
4. 配置 Dubbo 协议端口，例如 `dubbo.protocol.name=dubbo`、`dubbo.protocol.port=20880`。
5. Provider 用 `@DubboService` 暴露接口实现，启动后把服务 URL 注册到 `/providers`。
6. Consumer 用 `@DubboReference` 引用远程接口，启动后订阅 `/providers`、`/routers`、`/configurators`。
7. Consumer 收到 Provider 列表后，在本地生成代理，调用时按负载均衡策略选择一台 Provider。
8. Provider 扩容时，新实例注册临时节点，Consumer 收到 Watcher 通知后刷新本地 Invoker 列表。
9. Provider 宕机时，ZooKeeper Session 过期，临时节点删除，Consumer 重新拉取列表并摘除故障实例。
10. 如果 ZooKeeper 短暂不可用，已获取的 Consumer 本地缓存通常还能继续调用已有 Provider，但新服务注册、地址变更和配置推送会受影响。

**Spring Boot 常见配置示例**：

```yaml
dubbo:
  application:
    name: order-provider
  registry:
    address: zookeeper://zk1:2181?backup=zk2:2181,zk3:2181
  protocol:
    name: dubbo
    port: 20880
```

```java
@DubboService(version = "1.0.0")
public class OrderServiceImpl implements OrderService {
    // 暴露给 Consumer 调用
}
```

```java
@DubboReference(version = "1.0.0")
private OrderService orderService;
```

**面试追问：调用链上 ZooKeeper 会不会成为性能瓶颈？**

一般不会。ZooKeeper 只参与服务注册、订阅和变更通知，不参与每一次 RPC 调用。Consumer 拿到 Provider 地址后会缓存在本地，业务请求是 Consumer 直连 Provider。真正需要关注的是 ZooKeeper 不可用时新实例无法注册、服务列表无法及时更新，以及 Consumer 本地缓存是否能兜住已有调用。

**面试追问：Provider 扩容时，Consumer 能感知到吗？怎么做到的？**

能感知到。Provider 新实例启动后，会把自己的服务 URL 注册到 ZooKeeper 的 `/providers` 目录下；Consumer 之前已经订阅了这个接口的 providers 目录，所以 ZooKeeper 子节点发生变化时，会通过 Watcher 通知 Consumer。

Consumer 收到通知后，不是只相信这条事件本身，而是重新拉取 providers 目录下的完整地址列表，然后和本地缓存做刷新，重新生成或更新本地 Invoker 列表。后续请求再走负载均衡时，新扩容的 Provider 就会进入可选实例集合。

可以按这条链路回答：

```text
新 Provider 启动
  -> 暴露 Dubbo 服务
  -> 向 ZooKeeper /providers 目录创建临时节点
  -> ZooKeeper 触发子节点变更通知
  -> Consumer 收到 Watcher 事件
  -> Consumer 重新拉取 provider URL 全量列表
  -> 刷新本地 Directory / Invoker 列表
  -> 后续调用通过负载均衡打到新 Provider
```

这里有两个细节要补一句：

1. Watcher 只负责通知“服务列表变了”，不直接携带完整 Provider 列表，所以 Consumer 要重新全量拉取。
2. Consumer 调用时依赖本地缓存的 Invoker 列表，不是每次请求都访问 ZooKeeper，所以扩容感知是异步刷新，本身通常有一个很短的传播延迟。

**本地 Invoker 列表是什么？**

可以把 Invoker 理解成 Consumer 端对某个 Provider 的一次“可调用封装”。它不是单纯的 IP 字符串，而是 Dubbo 根据 provider URL 构建出来的调用对象，里面包含服务地址、协议、接口、版本、分组、超时、序列化、连接信息等。

比如 ZooKeeper 里有两个 Provider URL：

```text
dubbo://192.168.1.10:20880/com.xxx.OrderService?version=1.0.0
dubbo://192.168.1.11:20880/com.xxx.OrderService?version=1.0.0
```

Consumer 拉取到这些 URL 后，会在本地转换成类似这样的结构：

```text
OrderService 本地 Directory
  -> Invoker(192.168.1.10:20880)
  -> Invoker(192.168.1.11:20880)
```

后续执行 `orderService.createOrder()` 时，Consumer 不是去 ZooKeeper 查地址，而是从本地 Directory 里的 Invoker 列表中，通过负载均衡选一个 Invoker，再由这个 Invoker 发起远程 Dubbo 调用。

面试里可以简化成一句话：**本地 Invoker 列表就是 Consumer 缓存的可调用 Provider 实例列表，Dubbo 负载均衡就是在这个列表里选一台机器调用。**

**Dubbo 的负载均衡怎么设置？**

Dubbo 的负载均衡本质发生在 Consumer 侧，因为 Consumer 已经拿到了本地 Invoker 列表，调用时需要从多个 Invoker 里选一个。常见策略有：

| 策略 | 配置值 | 适合场景 |
| --- | --- | --- |
| 随机 | `random` | 默认常用策略，配合权重使用，整体简单稳定 |
| 轮询 | `roundrobin` | 请求处理耗时比较均匀的服务 |
| 最少活跃调用 | `leastactive` | 不同机器处理速度不一致，希望优先打到更空闲实例 |
| 最短响应时间 | `shortestresponse` | 希望优先选择响应更快的实例，适合延迟敏感接口 |
| 一致性 Hash | `consistenthash` | 同一参数希望尽量落到同一台机器，例如本地缓存命中场景 |

常见配置方式有三种：

**1. 在单个引用上设置**：

```java
@DubboReference(loadbalance = "leastactive", version = "1.0.0")
private OrderService orderService;
```

这个配置只影响当前 Consumer 对 `OrderService` 的调用。

**2. 在方法级别设置**：

```java
@DubboReference(
    version = "1.0.0",
    methods = @Method(name = "createOrder", loadbalance = "consistenthash")
)
private OrderService orderService;
```

方法级别更细，适合某个接口里不同方法调用特征差异很大的情况。比如查询接口用 `leastactive`，按用户维度有本地缓存的接口可以考虑 `consistenthash`。

**3. 在全局 Consumer 维度设置**：

```yaml
dubbo:
  consumer:
    loadbalance: leastactive
```

全局配置会影响当前应用里默认的 Dubbo 消费行为。生产上更建议先用默认策略，只有在明确知道接口调用特征时，再对重点接口做引用级或方法级调整。

**配置优先级怎么记？**

面试里可以按“越具体优先级越高”来记：**方法级 > 引用级 > Consumer 全局默认值**。另外，虽然有些配置也可以从 Provider 侧作为服务参数下发，但真正做选择的是 Consumer，因为 Invoker 列表和负载均衡逻辑都在 Consumer 本地。

**负载均衡面试表达**：

> Dubbo 负载均衡是在 Consumer 侧做的，因为 Consumer 从注册中心拿到 Provider 列表后，会在本地维护 Invoker 列表。调用时 Dubbo 根据配置的 loadbalance 策略，从这些 Invoker 里选一个发起 RPC。配置上可以放在 `@DubboReference` 引用级，也可以放在方法级，或者通过 `dubbo.consumer.loadbalance` 做全局默认。一般默认 random 就够用，如果接口耗时差异大可以考虑 leastactive，想让同一类请求尽量打到同一台机器可以考虑 consistenthash。

**扩容感知面试表达**：

> Provider 扩容后，Consumer 是能感知的。因为 Consumer 启动时会订阅对应接口的 providers 目录，新 Provider 启动后会在 ZooKeeper 上创建临时节点，触发目录变更通知。Consumer 收到 Watcher 事件后会重新拉取完整 Provider URL 列表，刷新本地 Invoker 缓存。后续调用再经过 Dubbo 负载均衡时，新实例就会参与流量分配。这里要注意，ZooKeeper 不参与每次 RPC，它只做注册、订阅和变更通知，真正调用还是 Consumer 直连 Provider。

**核心流程**：

1. Provider 启动，创建临时节点：`/service/order/instance-1`。
2. 节点内容保存实例元数据，例如 IP、端口、协议、权重、版本。
3. Consumer 启动，读取 `/service/order` 下的实例列表。
4. Consumer 把服务地址缓存在本地，再按负载均衡策略选择实例调用。
5. Consumer 对服务目录注册 Watcher，监听子节点变化。
6. 新 Provider 上线时创建临时节点，Consumer 收到通知后重新拉取列表。
7. Provider 正常下线时主动删除节点；异常宕机时 Session 过期后临时节点自动删除。
8. Consumer 收到变化事件后刷新本地地址列表，后续请求不再路由到已下线实例。

**为什么用临时节点**：

- 临时节点和客户端 Session 绑定。
- Provider 正常运行时，通过心跳维持 Session。
- Provider 宕机、进程挂掉、网络断开超过 Session timeout 后，ZooKeeper 会删除对应临时节点。
- 这样注册中心能自动感知实例失效，不需要 Provider 自己一定能发“下线通知”。

**Consumer 为什么要本地缓存**：

- 每次调用都查 ZooKeeper 性能太差，也会把注册中心打成调用链瓶颈。
- 正常做法是启动时拉取一次服务列表，本地缓存，调用时直接从缓存选实例。
- Watcher 只负责通知“服务列表可能变化了”，收到通知后再重新拉取全量列表并更新本地缓存。

**和普通健康检查的区别**：

| 方式 | 存活判断 | 特点 |
| --- | --- | --- |
| ZooKeeper 临时节点 | Session 是否存活 | 强依赖 ZK 会话，节点自动删除，偏强一致 |
| 心跳上报 | Provider 定期向注册中心报活 | 更偏服务发现平台，通常更关注可用性和最终一致 |

**生产注意**：

- Watcher 是一次性事件通知，不是可靠消息队列，收到通知后要重新注册 Watcher。
- 注册中心客户端必须有本地缓存，避免 ZooKeeper 短暂不可用导致服务完全不可调用。
- Session timeout 不能太短，否则网络抖动会导致实例频繁上下线；也不能太长，否则故障摘除太慢。
- 服务注册路径要区分环境、命名空间、分组、版本，避免测试服务和生产服务串在一起。
- ZooKeeper 更偏 CP，做服务注册时一致性强，但注册中心短暂不可用可能影响新实例注册和服务列表更新。

**面试表达**：

> ZooKeeper 做注册中心，本质是用 ZNode 存服务元数据。Provider 启动后在服务目录下创建临时节点，节点里保存 IP、端口、协议、版本等信息；Consumer 启动时读取这个目录下的 Provider 列表，缓存在本地，并注册 Watcher 监听目录变化。Provider 宕机后，Session 过期，临时节点会被 ZooKeeper 自动删除；Consumer 收到事件后重新拉取全量地址列表，刷新本地缓存。这里的关键点是临时节点负责实例存活感知，Watcher 负责变化通知，本地缓存负责调用性能和注册中心不可用时的降级。

## 12. 注册中心为什么不能只依赖 ZooKeeper Watcher？

**一句话回答**：Watcher 只是一次性事件通知，网络抖动、客户端重连和事件合并都可能导致感知延迟，所以消费者必须本地缓存、定期校验，并在收到通知后全量拉取最新服务列表。

**生产做法**：

- 客户端维护本地服务列表缓存。
- Watcher 触发后重新全量拉取服务目录。
- 启动时如果注册中心不可用，可以用本地快照做降级。
- 配合心跳、健康检查、连接失败摘除和重试策略。

**面试表达**：

> 注册中心客户端不能把 Watcher 当成可靠消息队列。Watcher 只告诉我“可能变了”，真正的状态要重新拉取。生产上还会保留本地缓存和快照，避免注册中心短暂不可用时整个调用链直接瘫痪。

## 13. ZooKeeper 做配置中心有什么问题？

**一句话回答**：ZooKeeper 可以做简单配置中心，但不适合承载大配置、高频变更、复杂灰度和权限审计，现代项目更常用 Apollo、Nacos、Spring Cloud Config 等专门配置中心。

**能做什么**：

- 存储少量配置。
- 监听配置节点变化。
- 客户端感知变化后刷新本地配置。

**主要问题**：

- 不适合存大文本或频繁更新。
- 配置发布、灰度、回滚、审计能力弱。
- Watcher 一次性触发，需要客户端正确重注册。
- 配置模型不如专门配置中心友好。

**面试表达**：

> ZooKeeper 能做配置监听，但我不会把它设计成完整配置平台。配置中心除了存值，还需要灰度、回滚、权限、审计、版本管理和可视化发布，这些能力 Apollo 或 Nacos 更合适。ZooKeeper 更适合保存小而关键的协调状态。

## 14. Nacos 和 ZooKeeper 作为注册中心有什么区别？

**一句话回答**：ZooKeeper 偏 CP，强调一致性；Nacos 同时支持 AP 和 CP，更贴近微服务注册配置场景，提供服务发现、健康检查、配置管理和控制台能力。

| 对比项   | ZooKeeper       | Nacos                    |
| ----- | --------------- | ------------------------ |
| 定位    | 分布式协调服务         | 服务发现 + 配置中心              |
| 一致性倾向 | CP              | AP / CP 可选               |
| 健康检查  | 依赖临时节点和 Session | 心跳、临时实例、持久实例             |
| 配置管理  | 基础能力            | 原生配置中心能力                 |
| 易用性   | 偏底层             | 更贴近 Spring Cloud / 微服务治理 |

**面试表达**：

> ZooKeeper 是通用协调组件，做注册中心是它的一个应用；Nacos 则是面向微服务治理设计的注册配置平台。服务注册这种场景很多时候更看重可用性和最终一致，所以 Nacos 的 AP 模型会更贴近互联网业务；但强一致协调类场景，ZooKeeper 仍然有价值。

## 15. Eureka、Nacos、ZooKeeper、Consul 怎么选？

**一句话回答**：选型看业务对一致性、可用性、生态和运维成本的要求；Spring Cloud 老项目常见 Eureka，国内 Spring Cloud Alibaba 常见 Nacos，强协调常用 ZooKeeper，跨语言和服务网格场景可考虑 Consul。

| 组件        | 特点                    | 适合场景                            |
| --------- | --------------------- | ------------------------------- |
| Eureka    | AP、自我保护、客户端缓存强        | Spring Cloud 老体系、可用性优先          |
| Nacos     | 注册配置一体、AP/CP 可选、控制台完善 | Spring Cloud Alibaba、Dubbo、配置中心 |
| ZooKeeper | CP、强协调、临时节点和顺序节点      | Dubbo 注册、分布式协调、锁、选主             |
| Consul    | 健康检查、KV、跨语言、服务网格生态    | 多语言服务发现、基础设施治理                  |

**面试表达**：

> 注册中心选型不是背组件优缺点，而是看业务诉求。服务发现通常更怕注册中心短暂不可用导致全站故障，所以 AP 和客户端缓存很重要；但如果是分布式锁、选主这类强协调问题，就更看重 CP 和一致性。

## 16. Eureka 的自我保护机制是什么？

**一句话回答**：Eureka 自我保护是在短时间内大量服务心跳丢失时，不立即剔除实例，避免网络抖动或注册中心自身异常导致大量健康服务被误删。

**核心理解**：

- Eureka 偏 AP，宁可保留可能失效的实例，也避免误删大量健康实例。
- 客户端本地有服务列表缓存，注册中心短暂不可用时仍可调用。
- 自我保护可能带来调用到坏实例的问题，需要客户端重试、超时和熔断兜底。

**面试表达**：

> Eureka 自我保护的思路是宁可短时间保留一些可疑实例，也不要因为网络抖动把大量正常服务摘掉。它牺牲的是列表实时准确性，换来注册中心故障时的整体可用性，所以客户端侧必须配合超时、重试和熔断。

## 17. Nacos 临时实例和持久实例有什么区别？

**一句话回答**：临时实例依赖心跳维持，心跳超时后会被剔除；持久实例更适合静态服务或非心跳模式，实例不会因为短暂心跳失败就直接删除。

**常见理解**：

- 临时实例适合普通微服务实例，上线下线频繁。
- 持久实例适合固定 IP、基础设施服务、需要人工维护状态的场景。
- 临时实例更偏自动发现，持久实例更偏人工管理和稳定配置。

**面试表达**：

> 大多数 Spring Cloud 微服务注册到 Nacos 会用临时实例，通过心跳表示存活。持久实例更适合固定服务或需要人工维护的场景，不会因为短暂心跳丢失就自动删除。

## 18. Nacos 配置中心的长轮询机制怎么理解？

**一句话回答**：客户端向 Nacos 发起配置监听请求，如果配置没有变化，服务端会挂起一段时间；期间配置变化则立即返回，否则超时后返回，客户端再发起下一轮请求。

**为什么不用纯短轮询**：

- 短轮询会频繁请求服务端，浪费资源。
- 长轮询能在变化时尽快感知，又减少无效请求。
- 客户端仍需保存本地快照，避免配置中心不可用时启动失败。

**面试表达**：

> Nacos 配置监听不是客户端每秒暴力查一次，而是长轮询。没有变化时服务端先挂起请求，有变化就立即返回，超时再进入下一轮。这样兼顾实时性和服务端压力。

## 19. CAP 理论在注册中心里怎么落地？

**一句话回答**：分布式系统必须考虑网络分区，注册中心通常在 CP 和 AP 之间取舍；服务发现更偏可用性，协调锁和选主更偏一致性。

**CAP 分别代表什么**：

| 字母 | 含义 | 面试解释 |
| --- | --- | --- |
| C | Consistency，一致性 | 多个节点看到的数据结果一致，不能出现两个客户端读到互相矛盾的状态。 |
| A | Availability，可用性 | 每个请求都能在合理时间内拿到非错误响应，不因为部分节点故障就整体不可用。 |
| P | Partition Tolerance，分区容错性 | 网络分区、节点之间通信失败时，系统仍然能继续运行。 |

**关键理解**：

- 在分布式系统里，网络分区一定要考虑，所以 P 基本是必须保留的前提。
- 真正的取舍通常发生在 C 和 A 之间：网络分区时，是拒绝部分请求来保证一致性，还是继续提供服务但允许短暂不一致。
- 所以面试里不要说“三选二”说得太机械，更准确的是“在发生网络分区时，在一致性和可用性之间做取舍”。

**典型取舍**：

- ZooKeeper 偏 CP：网络分区时没有多数派的一侧不可写，保证一致性。
- Eureka 偏 AP：注册中心异常时优先保证服务列表可用和客户端缓存可用。
- Nacos 支持 AP / CP：可以根据服务实例类型和场景选择。

**ZooKeeper 为什么是 CP**：

- ZooKeeper 的核心场景是分布式协调，例如配置元数据、分布式锁、Leader 选举、服务注册元数据。
- 这些场景最怕多个节点看到不一致的关键状态，例如两个客户端同时认为自己拿到了锁，或者网络分区后出现两个 Leader。
- ZooKeeper 通过 ZAB 协议、Leader 写入、多数派确认来保证事务顺序和一致性。
- 当集群发生网络分区时，只有能形成多数派的一侧可以继续对外提供写能力；少数派无法提交事务，宁可不可写，也不返回可能破坏一致性的结果。
- 因此 ZooKeeper 保留了 C 和 P，牺牲的是网络分区场景下部分节点的可用性，也就是偏 CP。

**追问：Dubbo 有本地缓存，ZooKeeper 挂了也能继续调用，为什么还说 ZooKeeper 偏 CP？**

这里要区分两个层面：

| 层面 | 看什么 | 结论 |
| --- | --- | --- |
| ZooKeeper 组件本身 | 注册、删除、变更通知、写入一致性 | 偏 CP |
| Dubbo 调用体系 | Consumer 已缓存 URL 后能否继续调用已有 Provider | 可用性较高 |

ZooKeeper 偏 CP，说的是 **ZooKeeper 集群自身在网络分区或多数派不足时，会拒绝写入和状态变更，优先保证注册数据一致**。比如没有多数派时，新 Provider 不能可靠注册，旧 Provider 下线也不能可靠通知，路由规则和动态配置也可能无法更新。

**怎么理解“没有多数派时拒绝写入和状态变更”？**

ZooKeeper 的写入不是单机写成功就算成功，而是走类似这样的流程：

```text
客户端发起写请求
  -> 请求到达 Leader
  -> Leader 生成事务提案
  -> Follower 参与确认
  -> 超过半数投票节点确认
  -> Leader 提交事务并通知客户端成功
```

多数派公式是：

```text
多数派 = floor(投票节点数 / 2) + 1
```

以 3 台 ZooKeeper 为例，多数派是 2。只要 Leader 能拿到自己加 1 个 Follower 的确认，就能提交写入；如果网络分区后某一侧只有 1 台机器，它就没有多数派，不能提交写入。

```text
3 台集群：ZK1、ZK2、ZK3
多数派：2 台

分区 A：ZK1、ZK2  -> 有多数派，可以选 Leader，可以写
分区 B：ZK3       -> 没有多数派，不能选 Leader，不能写
```

如果网络分区分成三份，也不是看“分了几份”，而是看每一份里有没有多数派。

**3 台集群分成 1 + 1 + 1**：

```text
3 台集群：ZK1、ZK2、ZK3
多数派：2 台

分区 A：ZK1  -> 1 台，没有多数派，不能写
分区 B：ZK2  -> 1 台，没有多数派，不能写
分区 C：ZK3  -> 1 台，没有多数派，不能写
```

这种情况下三个分区都没有多数派，所以整个 ZooKeeper 集群对写请求不可用，不能可靠注册新 Provider，也不能提交配置、路由、锁节点等状态变更。

**5 台集群分成 3 + 1 + 1**：

```text
5 台集群：ZK1、ZK2、ZK3、ZK4、ZK5
多数派：3 台

分区 A：ZK1、ZK2、ZK3  -> 3 台，有多数派，可以选 Leader，可以写
分区 B：ZK4            -> 1 台，没有多数派，不能写
分区 C：ZK5            -> 1 台，没有多数派，不能写
```

这时只有拥有 3 台的分区能继续工作，两个单节点分区不能写。反过来，如果 5 台分成 2 + 2 + 1，每个分区都达不到多数派 3，那么所有分区都不能提交写入。

**会不会出现两个分区都有多数派，导致写冲突？**

正常不会。因为“多数派”要求超过半数，而网络分区里的节点集合是互不重叠的。两个互不重叠的分区不可能同时都超过半数。

用数字看更直观：

```text
5 台集群，多数派是 3
如果两个分区都要有多数派，至少需要 3 + 3 = 6 台
但集群总共只有 5 台，所以不可能

6 台集群，多数派是 4
如果两个分区都要有多数派，至少需要 4 + 4 = 8 台
但集群总共只有 6 台，所以不可能
```

这就是多数派机制防脑裂的核心：**任意两个多数派集合一定会有交集**。同一个 ZooKeeper 节点在同一个选举轮次里不会同时给两个 Leader 投票，也不会让两个互相独立的 Leader 都拿到合法多数派。

**如果每个分区都有 3 台 ZooKeeper 呢？**

要先看它们是不是同一个 ZooKeeper ensemble。

如果它们是同一个 9 节点 ZooKeeper 集群，被网络分成 `3 + 3 + 3`，多数派不是 3，而是：

```text
多数派 = floor(9 / 2) + 1 = 5
```

所以每个分区虽然都有 3 台，但都达不到 5 台多数派，三个分区都不能提交写入：

```text
9 台集群，多数派是 5

分区 A：3 台 -> 没有多数派，不能写
分区 B：3 台 -> 没有多数派，不能写
分区 C：3 台 -> 没有多数派，不能写
```

如果每个分区里的 3 台本来就是一套独立 ZooKeeper 集群，那它们当然都可以在自己的 3 节点集群里形成多数派。但这已经不是“同一个 ZooKeeper 集群发生网络分区”，而是“三套独立注册中心”。如果 Dubbo 客户端被配置成有的连 A 集群、有的连 B 集群、有的连 C 集群，就可能出现注册数据不一致，但这属于注册中心配置和环境隔离问题，不是 ZooKeeper quorum 算法内部允许了多个多数派。

如果面试官问“那两个分区都说自己有多数派怎么办”，通常只有几类异常情况：

1. 不是同一个 ZooKeeper 集群，配置了两套不同的投票节点列表。
2. 运维误配置，节点的 `server.id`、集群列表或数据目录混乱。
3. 不是严格意义的网络分区，而是客户端、DNS、代理层把请求打到了不同环境。

在同一个配置正确的 ZooKeeper ensemble 里，多数派机制本身就是为了避免两个分区同时提交写入。

所谓“拒绝写入和状态变更”，就是这类操作不会被少数派节点成功提交：

- 新 Provider 注册临时节点。
- Provider 下线删除节点。
- 配置节点更新。
- 路由规则、权重规则变更。
- 分布式锁节点创建。

这样做的目的，是避免网络分区两边都各自写成功，后面网络恢复时出现两份互相冲突的注册数据或锁状态。ZooKeeper 宁愿让少数派不可写，也不允许状态被写乱。

Dubbo 可用性高，是因为 Dubbo Consumer 有本地缓存。只要 Consumer 之前已经订阅并缓存了 Provider URL，即使 ZooKeeper 暂时不可用，已有 Consumer 仍然可以继续直连已有 Provider 发起 RPC。

但这并不说明 ZooKeeper 是 AP，因为 ZooKeeper 不可用期间，下面这些能力会受影响：

1. 新 Provider 扩容后可能无法注册，Consumer 感知不到新实例。
2. Provider 下线后，如果变更通知失败，Consumer 可能短时间继续拿旧地址调用。
3. 新 Consumer 启动时，如果本地没有缓存，可能拿不到 Provider 列表。
4. 动态路由、权重、配置变更等依赖注册中心通知的能力会延迟或失效。

所以更准确的说法是：**ZooKeeper 作为注册中心本身偏 CP；Dubbo 在它上面通过本地缓存、重试、超时和容错，把“已发现服务的调用可用性”做高了。**

**面试表达**：

> CAP 里的 C 是一致性，A 是可用性，P 是分区容错性。分布式系统必须考虑网络分区，所以真正取舍通常是在分区发生时选 C 还是选 A。ZooKeeper 偏 CP，因为它服务的是分布式协调场景，比如锁、选主、配置元数据，这些场景不能接受两个客户端同时拿到锁或出现两个 Leader。它通过 ZAB 和多数派提交保证一致性，网络分区时少数派不可写，牺牲部分可用性来保证状态不乱。Dubbo 配合 ZooKeeper 时，Consumer 本地缓存确实能让已有调用继续跑，但这提高的是 Dubbo 调用链的容错，不改变 ZooKeeper 自身“多数派不足就拒绝状态变更”的 CP 特征。

## 20. BASE 和最终一致性怎么结合组件回答？

**一句话回答**：BASE 是互联网系统常见的最终一致思路，核心是允许短暂不一致，通过消息、补偿、幂等、状态机和对账让系统最终收敛。

**BASE 分别代表什么**：

| 缩写 | 含义 | 面试解释 |
| --- | --- | --- |
| BA | Basically Available，基本可用 | 系统异常时不保证所有功能都完美可用，但核心链路要尽量可用，例如降级、限流、排队、返回兜底结果。 |
| S | Soft State，软状态 | 系统允许存在短暂中间态，例如订单已创建但库存消息还没消费，或者支付成功但履约状态还在同步中。 |
| E | Eventually Consistent，最终一致性 | 不要求每个时刻都强一致，但通过 MQ、重试、补偿、幂等、状态机和对账最终收敛到一致状态。 |

**和 ACID 的区别**：

| 思路 | 典型场景 | 特点 |
| --- | --- | --- |
| ACID | 单库事务、本地强一致写入 | 要么全成功，要么全失败，强调强一致和事务隔离。 |
| BASE | 跨服务、跨系统、长链路业务 | 允许短暂不一致，用异步、补偿和对账换取可用性、吞吐和解耦。 |

**典型例子：订单和库存最终一致**：

1. 订单服务先在本地事务里创建订单。
2. 通过 MQ 发送库存扣减或库存预占消息。
3. 库存服务异步消费消息，执行扣减、预占或释放。
4. 消费端用业务唯一键、状态机或唯一索引保证幂等。
5. 如果 MQ 投递、消费或库存处理失败，通过重试、死信、补偿任务和对账修复。
6. 订单创建成功的一瞬间，订单和库存可能短暂不一致，但最终会收敛。

**组件配合**：

- 注册中心保证服务可发现，不保证业务状态一致。
- MQ 负责异步解耦和最终投递。
- Redis 承担缓存、限流、短期状态。
- MySQL 通过唯一索引、事务和状态机兜底。
- XXL-Job / 定时任务做补偿和对账。

**面试表达**：

> BASE 是分布式系统里常见的最终一致性思想。它不追求每个时刻都强一致，而是在保证核心功能基本可用的前提下，允许系统存在短暂中间状态，再通过 MQ、幂等、重试、补偿任务和对账机制，让数据最终收敛一致。比如订单、库存、支付、履约这类跨服务链路，核心写入用本地事务兜住，跨服务状态用消息和补偿收敛，消费端必须幂等，最后通过对账发现并修复异常状态。

## 21. 什么是脑裂？ZooKeeper 如何避免脑裂？

**一句话回答**：脑裂是集群因为网络分区出现多个主或多份不一致决策的情况，ZooKeeper 通过多数派机制避免少数派继续对外提交写事务。

**关键点**：

- Leader 必须获得多数派支持才能提交写事务。
- 网络分区后，少数派无法形成多数派，不能继续提交。
- 新 Leader 必须通过选举产生，并基于 zxid 保证数据恢复。

**面试表达**：

> ZooKeeper 防脑裂靠多数派。即使网络分区，只有能拿到超过半数节点支持的一侧才能选出 Leader 并提交事务；少数派即使还活着，也不能继续对外提交写请求。

## 22. 分布式配置更新如何保证安全？

**一句话回答**：配置更新要有版本、灰度、校验、回滚、本地快照和变更审计，不能直接把线上配置当普通 KV 随意覆盖。

**安全发布流程**：

1. 配置修改先校验格式和取值范围。
2. 生成新版本，记录发布人、时间和变更 diff。
3. 小流量或指定实例灰度生效。
4. 观察核心指标和错误日志。
5. 全量发布或一键回滚。
6. 客户端保存本地快照，配置中心异常时可降级启动。

**面试表达**：

> 配置中心最怕“改一行配置，全站事故”。所以我会强调版本、灰度、回滚和审计。客户端侧还要有本地快照和默认值，不能因为配置中心短暂不可用就导致服务启动失败或运行中断。

## 23. 服务注册成功但消费者调用不到，怎么排查？

**排查路径**：

1. **看注册中心**：实例是否注册、命名空间/分组/服务名是否一致。
2. **看健康状态**：实例是否被标记为 unhealthy，心跳是否正常。
3. **看消费者缓存**：本地服务列表是否刷新，是否读到旧列表。
4. **看网络连通**：消费者到提供者 IP/端口是否可达。
5. **看负载均衡**：是否路由到下线实例、权重为 0、标签不匹配。
6. **看协议配置**：Dubbo/Spring Cloud 版本、序列化、超时、接口版本是否一致。

**面试表达**：

> 我会先确认“注册成功”是不是消费者能看到。注册中心控制台看到实例只是第一步，还要看命名空间、分组、健康状态、消费者本地缓存和实际网络连通。很多问题不是注册失败，而是消费者订阅的服务名或分组不一致。

## 24. 配置中心改了配置但服务没生效，怎么排查？

**排查路径**：

1. 配置是否发布到正确环境、命名空间、应用名和分组。
2. 客户端是否监听了正确 dataId / key。
3. 应用是否支持动态刷新，还是只在启动时读取。
4. `@Value`、配置类、线程池参数等是否真的能热更新。
5. 本地快照是否覆盖了远端配置。
6. 配置格式是否解析失败但被默认值吞掉。
7. 是否需要重启或触发 RefreshScope。

**面试表达**：

> 配置没生效不一定是配置中心问题。我要先确认发布目标是否正确，再看客户端是否监听正确配置，最后看代码层是否支持动态刷新。有些参数启动时就固化了，比如线程池核心参数、静态变量或初始化后的 Bean，不是配置变了就自动生效。

## 25. 分布式锁一直释放不了，怎么排查？

**排查路径**：

1. 锁 key 或 ZNode 路径是否设计过粗，导致大量业务争同一把锁。
2. 业务代码是否在异常分支漏掉释放锁。
3. Redis 锁是否设置了过长 TTL 或看门狗一直续期。
4. ZooKeeper 客户端 Session 是否仍然存活，临时节点是否未删除。
5. 是否把远程 RPC、长事务、大循环放进锁内。
6. 是否缺少锁等待超时、降级和告警。

**面试表达**：

> 分布式锁问题我会先看锁粒度和释放逻辑，再看持锁期间是否执行了慢 RPC 或长事务。Redis 锁重点看 TTL、看门狗和 finally 解锁；ZooKeeper 锁重点看 Session 是否还活着、临时节点是否正常删除。

## 26. 分布式组件如何做监控告警？

**一句话回答**：分布式组件监控要同时看集群健康、客户端行为和业务影响，不能只看进程是否存活。

**监控维度**：

- ZooKeeper：Leader 状态、节点数、连接数、请求延迟、未同步 Follower、磁盘和快照。
- Nacos：实例数、心跳延迟、配置发布失败、长轮询堆积、集群节点状态。
- 注册中心客户端：注册失败、订阅失败、本地缓存为空、调用无可用实例。
- 配置中心客户端：拉取失败、解析失败、动态刷新失败、配置版本落后。
- 分布式锁：等待时间、持锁时间、失败次数、超时次数。

**面试表达**：

> 我会把组件监控和业务监控串起来。比如注册中心不只是看 Nacos 或 ZooKeeper 进程活着，还要看客户端是否注册成功、消费者是否拿到实例、调用是否出现 no provider。组件健康最终要落到业务调用是否可用。

## 27. 面试时如何把分布式组件和项目经验结合？

**回答结构**：

1. **业务背景**：例如履约、库存、订单、支付、调度、营销。
2. **分布式问题**：实例动态上下线、配置动态调整、并发互斥、跨服务一致性。
3. **组件选型**：注册中心、配置中心、Redis / ZooKeeper 锁、MQ、定时补偿。
4. **风险兜底**：本地缓存、超时重试、熔断降级、幂等、对账、告警。
5. **结果表达**：减少人工处理、降低故障影响、提升稳定性和发布效率。

**示例表达**：

> 在履约或库存类项目里，我会把分布式组件讲成一条链路：服务间调用依赖注册中心发现实例；规则开关、灰度策略和降级阈值放配置中心；并发修改库存或分配规则时用短粒度分布式锁保护关键区；跨服务状态通过 MQ、状态机、幂等和补偿任务保证最终一致。这样回答比单独背 ZooKeeper 或 Nacos 原理更贴近项目。

## 高频追问速记

| 问题 | 速答 |
| --- | --- |
| ZooKeeper 为什么适合做分布式锁？ | 临时节点自动释放，顺序节点保证公平，Watcher 负责唤醒等待者。 |
| ZooKeeper 锁如何避免羊群效应？ | 每个等待者只监听自己的前一个节点，而不是都监听最小节点。 |
| ZooKeeper 是 CP 还是 AP？ | 更偏 CP，网络分区时少数派不可写，优先保证一致性。 |
| 服务注册更适合 CP 还是 AP？ | 多数业务更偏 AP，短暂旧列表可用重试和熔断兜底。 |
| 配置中心为什么要本地快照？ | 防止配置中心不可用时服务无法启动或运行中断。 |
| Nacos 和 ZooKeeper 最大区别？ | Nacos 是注册配置平台，ZooKeeper 是通用协调服务。 |
| Redis 锁安全吗？ | 工程上常用，但要唯一 value、Lua 解锁、TTL/看门狗、幂等兜底，强一致场景要谨慎。 |
| 脑裂怎么避免？ | 通过多数派机制，只有多数派一侧能选主和提交。 |
| Watcher 是可靠消息吗？ | 不是，Watcher 是一次性事件通知，收到后要重新拉取数据。 |

## 面试答题模板

### 原理题模板

> 这个组件主要解决的是分布式协调问题。以 ZooKeeper 为例，它通过 ZNode、临时节点、顺序节点、Watcher 和 ZAB 协议实现一致性元数据管理。面试里我会重点讲它适合小数据、强协调、低频关键状态，不适合高频大数据读写。

### 选型题模板

> 我不会只按技术名气选组件，而是看场景。如果是服务注册，通常更看重可用性、客户端缓存和健康检查，Nacos/Eureka 更常见；如果是分布式锁、Leader 选举这类强协调问题，ZooKeeper 更合适。

### 项目落地模板

> 在项目里，注册中心负责服务发现，配置中心负责动态开关和灰度参数，Redis 或 ZooKeeper 锁负责短临界区互斥，MQ 和补偿任务负责跨服务最终一致。所有组件都不是单点答案，最终还要配合超时、重试、幂等、降级、监控和对账。

### 排障题模板

> 我会先确认组件自身状态，再确认客户端订阅和本地缓存，最后回到业务调用链。比如服务发现问题，要依次看注册中心实例、命名空间/分组、健康状态、消费者缓存、网络连通、负载均衡和接口版本，避免只盯控制台状态。
