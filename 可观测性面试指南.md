# 可观测性面试指南

这份材料面向 Java 后端线上排障和稳定性面试，重点覆盖日志、指标、链路追踪、OpenTelemetry、Prometheus、Grafana、SkyWalking/Zipkin、告警、SLO、慢接口定位和项目落地。回答时建议按“先保现场 -> 分层定位 -> 指标验证 -> 根因修复 -> 复盘沉淀”来讲。

## 0. 高频知识地图

| 主题 | 高频问题 | 回答关键词 |
| --- | --- | --- |
| 三件套 | Log / Metric / Trace 区别 | 日志细节、指标趋势、链路路径 |
| Trace | traceId、spanId、上下游耗时 | 跨服务调用、采样、上下文传播 |
| Metrics | QPS、RT、错误率、饱和度 | RED、USE、P95/P99、Prometheus |
| Logs | 结构化日志、MDC、脱敏 | traceId、业务主键、异常栈 |
| 告警 | 阈值、聚合、抑制、升级 | 误报、漏报、SLO、错误预算 |
| 排障 | 慢接口、错误率升高、CPU 高 | 网关、应用、JVM、DB、Redis、MQ、RPC |

## 1. 什么是可观测性？

**一句话回答**：可观测性是通过日志、指标和链路追踪理解系统内部状态的能力，让线上问题能被发现、定位、解释和复盘。

**三类信号**：

- Logs：发生了什么，保留细节。
- Metrics：趋势如何，适合监控和告警。
- Traces：请求经过哪里，每一段耗时多少。

**面试表达**：

> 可观测性不是装个日志框架，而是让系统出问题时能回答三个问题：哪里坏了、为什么坏、影响多大。日志看细节，指标看趋势和影响面，Trace 看跨服务路径和耗时分布。

## 2. 日志、指标、链路追踪有什么区别？

| 类型 | 优点 | 局限 | 典型用途 |
| --- | --- | --- | --- |
| 日志 | 信息最详细 | 成本高，难聚合 | 查异常栈、业务参数、关键状态 |
| 指标 | 聚合快，适合告警 | 细节少 | QPS、RT、错误率、资源水位 |
| Trace | 串联调用链 | 采样和接入成本 | 慢接口、跨服务定位 |

**面试表达**：

> 如果接口慢，我先看指标确认影响范围，再看 Trace 找慢在哪个服务，最后看日志确认具体业务参数和异常栈。三者组合起来才是完整排障链路。

## 3. RED 和 USE 指标怎么理解？

**一句话回答**：RED 关注请求服务质量，USE 关注资源健康状态。

| 模型 | 指标 | 场景 |
| --- | --- | --- |
| RED | Rate、Errors、Duration | 接口、服务、RPC |
| USE | Utilization、Saturation、Errors | CPU、内存、磁盘、网络、线程池 |

**面试表达**：

> 服务接口我用 RED：请求量、错误率、耗时；资源排障我用 USE：利用率、饱和度、错误。比如接口慢既要看 P99，也要看线程池队列、DB 连接池和 CPU 是否饱和。

## 4. traceId 和 spanId 是什么？

**一句话回答**：traceId 标识一次完整请求链路，spanId 标识链路中的一个调用片段；多个 span 组成一条 trace。

**关键点**：

- 入口生成或接收 traceId。
- 每个 RPC/HTTP/DB 调用形成 span。
- 上下游通过请求头传播上下文。
- 日志要打印 traceId，方便从日志跳到 Trace。

**面试表达**：

> traceId 是整条链路的请求 ID，spanId 是链路中的一段。比如网关到订单服务、订单到库存服务、库存到 Redis，每段都是 span。日志里带 traceId 后，排障时能从一条错误日志串到完整调用链。

## 5. OpenTelemetry 是什么？

**一句话回答**：OpenTelemetry 是可观测性数据采集和标准化框架，用统一方式采集 traces、metrics、logs，并导出到不同后端系统。

**核心价值**：

- 标准化埋点和上下文传播。
- 降低绑定单一厂商的成本。
- 支持自动 instrumentation 和手动埋点。
- 通过 Collector 做接收、处理和导出。

**面试表达**：

> OpenTelemetry 的价值是统一采集标准。应用侧按 OTel 规范产生 Trace、Metric、Log，后端可以接 Prometheus、Jaeger、Tempo、Elastic、云厂商平台等，不必把业务代码绑死在某个监控产品上。

## 6. Prometheus 的核心模型是什么？

**一句话回答**：Prometheus 是以时间序列为核心的监控系统，通过拉取目标暴露的 metrics，存储带标签的时间序列，并用 PromQL 查询和告警。

**关键概念**：

- metric name：指标名。
- labels：维度标签，例如 service、instance、method。
- scrape：定时拉取。
- PromQL：查询语言。
- Alertmanager：告警管理。

**面试表达**：

> Prometheus 适合做指标监控。应用暴露 `/metrics`，Prometheus 定时拉取，按标签维度存储时间序列。告警时用 PromQL 判断错误率、P99、CPU 或队列长度是否超过阈值。

## 7. Grafana 做什么？

**一句话回答**：Grafana 主要用于可视化和分析指标、日志、Trace 等数据，常和 Prometheus、Loki、Tempo、Elasticsearch 等数据源配合。

**面试表达**：

> Grafana 不是采集器，它更像可视化和分析平台。Prometheus 负责存指标，Grafana 负责把 QPS、错误率、P95/P99、线程池和 JVM 指标做成仪表盘，帮助排障和容量评估。

## 8. SkyWalking、Zipkin、Jaeger、Tempo 有什么区别？

**一句话回答**：它们都可用于分布式链路追踪，差异主要在生态、存储、接入方式、查询能力和运维成本。

| 工具 | 特点 |
| --- | --- |
| SkyWalking | Java 生态常见，APM 能力较完整 |
| Zipkin | 轻量老牌链路追踪 |
| Jaeger | CNCF 生态常见，追踪能力成熟 |
| Tempo | Grafana 生态，常和 Loki/Prometheus/Grafana 组合 |

**面试表达**：

> 选 Trace 工具我会看团队生态。Java 单体到微服务排障常见 SkyWalking；云原生体系可以用 Jaeger/Tempo；关键不是名字，而是 traceId 能不能贯穿网关、应用、RPC、MQ 和数据库访问。

## 9. Java 服务应该采集哪些指标？

**应用指标**：

- QPS、错误率、P50/P95/P99。
- HTTP 状态码分布。
- RPC 调用耗时和错误。
- 线程池活跃数、队列长度、拒绝数。
- 连接池活跃连接、等待连接数。
- MQ 消费 Lag、消费失败数。

**JVM 指标**：

- 堆内存、非堆、直接内存。
- GC 次数、GC 时间。
- 线程数、死锁。
- 类加载数。

**面试表达**：

> Java 服务我最关注接口 RED 指标、JVM GC、线程池、连接池和下游调用。因为很多慢接口不是代码慢，而是线程池满、DB 连接池等不到连接，或 GC 停顿导致 RT 抖动。

## 10. 日志怎么设计才方便排障？

**关键原则**：

- 结构化日志，关键字段固定。
- 必须包含 traceId、业务主键、用户/租户、接口名。
- 异常日志保留堆栈。
- 敏感字段脱敏。
- 关键状态流转打印前后状态。
- 日志级别分清 debug/info/warn/error。

**面试表达**：

> 好日志不是越多越好，而是关键字段稳定。比如工单号、库存 token、traceId、状态前后值必须有；手机号、身份证要脱敏。这样线上定位时可以从业务主键直接串起完整链路。

## 11. 如何设计告警？

**一句话回答**：告警要基于用户影响和系统风险设计，避免只按单点指标拍阈值；同时要有分级、抑制、聚合和升级机制。

**告警类型**：

- 错误率突增。
- P95/P99 超阈值。
- QPS 异常跌落或突增。
- MQ Lag 持续增长。
- 线程池队列满或拒绝。
- DB/Redis 连接池耗尽。
- JVM Full GC 频繁。

**面试表达**：

> 告警要避免两个极端：一是太敏感天天误报，二是用户已经投诉了才发现。我会按业务影响分级，比如核心下单错误率、支付回调失败、MQ 积压持续增长都要高优先级；单机 CPU 短暂抖动可以聚合后再告警。

## 12. SLI、SLO、SLA 怎么理解？

| 概念 | 含义 |
| --- | --- |
| SLI | 服务水平指标，如成功率、延迟 |
| SLO | 内部目标，如 99.9% 请求 300ms 内完成 |
| SLA | 对外承诺，通常涉及赔付或合同 |

**面试表达**：

> SLI 是指标，SLO 是目标，SLA 是承诺。比如接口成功率是 SLI，月成功率 99.9% 是 SLO，对客户承诺达不到要赔付就是 SLA。工程上我会用 SLO 驱动告警和容量治理。

## 13. 慢接口怎么排查？

**排查路径**：

1. 看指标：是单接口慢还是全站慢。
2. 看 Trace：慢在网关、应用、DB、Redis、MQ 还是 RPC。
3. 看 JVM：GC、线程池、锁等待。
4. 看数据库：慢 SQL、索引、连接池。
5. 看缓存：命中率、热点 key、大 key、Redis 延迟。
6. 看下游：超时、重试、熔断。
7. 看发布：最近代码、配置、流量变化。

**面试表达**：

> 慢接口我不会只看业务代码，而是先用指标判断影响面，用 Trace 定位慢在哪一段，再用日志和具体组件指标确认根因。比如 Trace 显示库存服务慢，再看库存服务的线程池、DB 慢 SQL 和 Redis 延迟。

## 14. CPU 100% 怎么结合可观测性排查？

**步骤**：

- 指标确认是单机还是集群普遍。
- 找最忙进程和线程。
- `jstack` / Arthas 看线程栈。
- 结合 Trace 看是否某接口流量突增。
- 看 GC 指标，排除频繁 GC。
- 看日志是否有异常循环或重试风暴。

**面试表达**：

> CPU 高要把系统指标和应用现场结合。监控告诉我哪台机器高，线程栈告诉我哪个线程在跑，Trace 和日志告诉我是哪个接口或任务触发。这样比只看 top 更容易定位根因。

## 15. MQ 积压如何监控和告警？

**关键指标**：

- Consumer Lag。
- 消费 TPS。
- 消费失败数。
- 重试/死信数量。
- 单条消息处理耗时。
- 下游 DB/RPC 耗时。

**面试表达**：

> MQ 积压告警不能只看队列长度，还要看增长趋势和消费能力。如果 Lag 持续增长，说明生产速度大于消费速度；要继续看是消费者线程不足、下游慢、单分区热点还是消费失败重试。

## 16. 如何给核心链路做仪表盘？

**核心面板**：

- 入口 QPS、成功率、P95/P99。
- 每个下游 RPC 的 RT 和错误率。
- DB 慢 SQL、连接池。
- Redis 命中率、延迟、热 key。
- MQ 生产/消费 TPS、Lag。
- JVM GC、线程池、CPU、内存。
- 业务指标：下单成功率、支付成功率、履约状态流转成功率。

**面试表达**：

> 核心链路仪表盘要把技术指标和业务指标放在一起。只看 CPU 没意义，要能看到下单成功率、库存预约成功率、MQ 积压和下游 RT，这样故障时才能判断用户影响和技术根因。

## 17. 采样会不会导致问题查不到？

**一句话回答**：会有风险，所以 Trace 采样要结合错误优先、慢请求优先和关键链路全采样策略。

**建议**：

- 普通请求按比例采样。
- 错误请求尽量保留。
- 慢请求尽量保留。
- 核心交易链路提高采样率。
- 日志中保留 traceId，必要时从日志补链路。

**面试表达**：

> Trace 全量采集成本很高，但采样过低会错过问题。我会对普通流量采样，对错误和慢请求尽量保留，核心链路提高采样率。这样兼顾成本和排障效果。

## 18. 如何结合项目讲可观测性？

**回答结构**：

1. 入口生成 traceId，全链路透传。
2. 日志打印业务主键和状态流转。
3. Prometheus 采集接口、JVM、线程池、连接池、MQ 指标。
4. Grafana 做核心链路仪表盘。
5. 告警按业务影响分级。
6. 故障后复盘补指标、补日志、补自动化修复。

**面试表达**：

> 在履约链路里，我会让工单号、库存 token 和 traceId 贯穿所有服务。Trace 看跨服务耗时，日志看状态前后值，指标看 QPS、错误率、P99、线程池和 MQ Lag。这样库存预约失败或工单状态不同步时，可以快速定位是哪一段出了问题。
