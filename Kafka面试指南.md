# Kafka 面试指南

这份材料面向 Java 后端大厂面试中 Kafka 高频题，重点覆盖 Topic、Partition、Replica、ISR、ACK、Producer、Consumer Group、Rebalance、Offset、幂等、事务、Exactly Once、消息积压和项目落地。回答时建议按“业务场景 -> Kafka 机制 -> 风险点 -> 兜底方案”来讲。

## 0. 高频知识地图

| 主题 | 面试常问点 | 回答关键词 |
| --- | --- | --- |
| 架构 | Broker、Topic、Partition、Replica | 分区并行、Leader/Follower、副本容灾 |
| Producer | 发送流程、ACK、批量、压缩、重试 | batch、linger.ms、acks、retries、idempotence |
| Consumer | Consumer Group、Offset、Rebalance | 分区分配、提交位点、重复消费、消费积压 |
| 可靠性 | 消息不丢、不重复、有序 | acks=all、ISR、幂等、事务、业务幂等 |
| 性能 | 为什么快、如何调优 | 顺序写、Page Cache、零拷贝、批量发送 |
| Exactly Once | 幂等生产者、事务、read_committed | Kafka 内部精确一次，不等于业务全链路精确一次 |
| 排障 | 积压、乱序、重复、Broker 故障 | Lag、分区数、消费线程、慢逻辑、限流降级 |

## 1. Kafka 是什么？适合解决什么问题？

**一句话回答**：Kafka 是高吞吐、分布式、可持久化的事件流平台，常用于日志收集、异步解耦、削峰填谷、事件驱动和数据管道。

**核心场景**：

- 用户行为日志、埋点、监控日志采集。
- 订单、库存、履约状态事件异步驱动。
- 高峰流量削峰，把同步压力转为异步消费。
- 数据同步管道，例如 MySQL Binlog 到数仓或搜索引擎。

**面试表达**：

> Kafka 的核心价值是把高吞吐事件流稳定存下来，并让多个消费者按自己的进度消费。它适合异步解耦、日志采集和数据管道，但不适合强实时、强事务、需要同步返回结果的核心链路。

## 2. Kafka 的整体架构是什么？

**一句话回答**：Kafka 集群由多个 Broker 组成，消息按 Topic 分类，每个 Topic 拆成多个 Partition，每个 Partition 有 Leader 和 Follower 副本，Producer 写 Leader，Consumer 从 Leader 拉取消息。

| 概念 | 作用 |
| --- | --- |
| Broker | Kafka 服务节点，负责存储和读写消息 |
| Topic | 消息主题，逻辑分类 |
| Partition | Topic 的物理分片，提供并行读写能力 |
| Replica | 分区副本，提高容灾能力 |
| Leader | 分区主副本，处理读写 |
| Follower | 从副本，复制 Leader 数据 |
| Controller | 管理分区 Leader 选举、元数据变化 |

**面试表达**：

> Kafka 的并发能力来自 Partition。Topic 只是逻辑概念，真正存储和并行读写的是 Partition。每个 Partition 有多个副本，但同一时刻只有 Leader 处理读写，Follower 负责同步，Leader 挂了再从可用副本里选新 Leader。

## 3. Kafka 为什么吞吐高？

**一句话回答**：Kafka 吞吐高主要因为顺序写磁盘、Page Cache、批量发送、零拷贝、分区并行和拉取式消费模型。

**关键点**：

- 顺序追加写日志，避免随机写磁盘。
- 利用操作系统 Page Cache 缓存文件数据。
- Producer 批量发送，减少网络请求次数。
- Broker 发送文件数据时可利用零拷贝减少用户态/内核态复制。
- Topic 分区后可以在多个 Broker 和消费者之间并行处理。

**面试表达**：

> Kafka 快不是因为不落盘，而是因为它把磁盘写成顺序追加日志，并充分利用 Page Cache、批量和零拷贝。再加上 Partition 并行，整体吞吐会很高。

## 4. Topic 和 Partition 怎么设计？

**一句话回答**：Topic 按业务事件类型划分，Partition 按吞吐、并行度、顺序性和未来扩展来设计，不是越多越好。

**设计原则**：

- 按业务语义建 Topic，例如 `order_status_event`、`inventory_event`。
- 需要同一业务键有序时，用业务 key 路由到同一 Partition。
- Partition 数决定同一 Consumer Group 最大并行消费数。
- 分区太少会限制吞吐，太多会增加文件句柄、元数据、Leader 选举和 Rebalance 成本。

**面试表达**：

> Partition 数要从吞吐和并行度倒推。如果一个消费者实例只能处理 1000 条/秒，目标 10000 条/秒，至少要 10 个有效分区和对应消费能力。但如果业务要求同一订单有序，就要用 orderId 作为 key，把同一订单事件打到同一分区。

## 5. Kafka 如何保证同一业务维度消息有序？

**一句话回答**：Kafka 只能保证单个 Partition 内有序，不能保证 Topic 全局有序；要保证同一订单或同一工单有序，需要用业务 key 把相关消息路由到同一个 Partition。

**注意点**：

- 同一 `orderId` 使用同一个 key。
- 消费端同一分区内单线程顺序处理最稳。
- 开多线程消费时要按业务 key 二次分发，避免并发乱序。
- 分区数变更可能影响 key 到分区的映射，需要谨慎。

**面试表达**：

> Kafka 的有序边界是 Partition。业务上我一般只追求同一订单或同一库存 token 的局部有序，而不是全局有序。Producer 用业务 key 固定分区，Consumer 同一分区顺序处理，才能保证这条链路的顺序。

## 6. Kafka 的 ACK 机制怎么理解？

**一句话回答**：ACK 控制 Producer 认为消息发送成功需要 Broker 达到什么确认程度，常见有 `acks=0`、`acks=1`、`acks=all`。

| 配置 | 含义 | 风险 |
| --- | --- | --- |
| `acks=0` | 不等 Broker 确认 | 吞吐高但可能丢消息 |
| `acks=1` | Leader 写入成功即返回 | Leader 宕机且副本未同步时可能丢 |
| `acks=all` | ISR 副本确认后返回 | 可靠性高，延迟更高 |

**面试表达**：

> 关键业务消息我会倾向 `acks=all`，并配合合适的副本数和最小 ISR，保证消息不是只写到 Leader 内存就返回。日志类低价值消息可以根据吞吐和丢失容忍度放宽。

## 7. ISR 是什么？有什么用？

**一句话回答**：ISR 是和 Leader 保持同步的副本集合，Kafka 通过 ISR 判断哪些副本有资格参与可靠确认和故障切换。

**关键点**：

- Leader 一定在 ISR 中。
- Follower 落后过多会被踢出 ISR。
- `acks=all` 通常要求 ISR 中满足条件的副本确认。
- `min.insync.replicas` 可以约束最少同步副本数。

**面试表达**：

> ISR 不是所有副本，而是当前同步进度跟得上的副本。它决定了消息可靠确认和 Leader 故障切换的安全边界。核心消息要结合 `acks=all` 和 `min.insync.replicas`，避免只剩一个副本还继续写。

## 8. Kafka 消息会丢吗？怎么保证不丢？

**一句话回答**：Kafka 可以通过配置和业务兜底降低丢失风险，但端到端不丢要从 Producer、Broker、Consumer 三段设计。

**三段保证**：

- Producer：`acks=all`、重试、幂等生产者、发送失败记录本地表或告警。
- Broker：合理副本数、`min.insync.replicas`、磁盘监控、禁止不安全选主。
- Consumer：业务处理成功后再提交 Offset，失败重试或进入死信/补偿。

**面试表达**：

> 消息不丢不能只看 Kafka。生产端要确认发送成功，Broker 要有副本和 ISR 保护，消费端要处理成功后再提交位点。关键业务还要有本地消息表、补偿任务和对账兜底。

## 9. Kafka 为什么会重复消费？怎么处理？

**一句话回答**：Kafka 默认更接近至少一次投递，消费者处理成功但提交 Offset 前宕机，或 Rebalance 后重新分配分区，都可能导致重复消费。

**解决方案**：

- 消费逻辑必须幂等。
- 用业务唯一键去重，例如订单号、事件 ID、流水号。
- 数据库唯一索引兜底。
- 状态机判断当前状态是否允许流转。
- 处理成功后再提交 Offset。

**面试表达**：

> Kafka 重复消费是正常现象，不能指望消息系统完全替业务去重。我会让消费者按业务主键幂等，比如事件表唯一索引、状态机条件更新、处理日志表，保证重复消息来了也不会重复扣库存或重复改状态。

## 10. Offset 是什么？自动提交和手动提交怎么选？

**一句话回答**：Offset 是 Consumer 在 Partition 上的消费进度；自动提交简单但可能丢或重复，关键业务更推荐处理成功后手动提交。

| 方式 | 优点 | 风险 |
| --- | --- | --- |
| 自动提交 | 简单，代码少 | 可能业务未成功但位点已提交 |
| 手动提交 | 可控，处理成功后提交 | 实现复杂，可能重复消费 |

**面试表达**：

> 核心业务我会手动提交 Offset，确保业务落库或状态更新成功后再提交。即使重复消费，也能靠幂等兜住；但如果提前提交后业务失败，就可能造成消息逻辑丢失。

## 11. Consumer Group 是什么？

**一句话回答**：Consumer Group 是 Kafka 的消费分组，同一个组内多个消费者共同消费一个 Topic 的分区；同一个 Partition 在同一时刻只能分配给组内一个消费者。

**关键点**：

- 同组内：负载均衡，共同消费。
- 不同组：广播语义，各自独立消费。
- 消费者数量超过分区数，多出来的消费者会空闲。

**面试表达**：

> Consumer Group 决定消费并行模型。同一个组内是竞争消费，不同组之间是各自独立消费。想提升同组消费并行度，光加消费者不够，还要有足够 Partition。

## 12. Rebalance 是什么？为什么让人头疼？

**一句话回答**：Rebalance 是 Consumer Group 内分区重新分配过程，消费者上下线、订阅变化、心跳超时、分区变化都可能触发；期间可能暂停消费并带来重复消费。

**风险**：

- 消费短暂停顿。
- 分区被重新分配，未提交 Offset 的消息可能被重复处理。
- 频繁 Rebalance 会导致吞吐抖动。

**优化**：

- 控制消费者处理时间，避免心跳超时。
- 合理配置 `max.poll.interval.ms`、`session.timeout.ms`。
- 避免频繁扩缩容。
- 使用静态成员或更稳定的分配策略。

**面试表达**：

> Rebalance 本质是组内分区重新分配。它不是错误，但频繁发生会让消费停顿、重复消费和 Lag 抖动。线上我会重点看消费者是否处理太慢、是否频繁重启、心跳和 poll 参数是否合理。

## 13. Kafka 消息积压怎么排查？

**排查路径**：

1. 看 Lag：哪个 Topic、哪个 Group、哪个 Partition 积压。
2. 看生产速率是否突然升高。
3. 看消费端 RT、线程池、DB、Redis、RPC 是否变慢。
4. 看 Rebalance 是否频繁发生。
5. 看单分区热点，是否 key 分布不均。
6. 看下游限流或故障，是否消费端被迫阻塞。
7. 临时扩容消费者，但不能超过有效分区并行度。

**面试表达**：

> 积压不是一上来就加机器。先看 Lag 分布，是所有分区都慢还是某几个热点分区慢；再看消费者处理逻辑是不是 DB 或 RPC 慢。如果是分区数不足，加消费者也没用；如果是下游慢，要先保护下游并做降级或异步补偿。

## 14. Kafka 如何处理消息乱序？

**一句话回答**：先明确有序范围，Kafka 只保证单分区有序；跨分区乱序需要业务设计解决，例如按业务 key 固定分区、状态机校验或延迟重试。

**处理方式**：

- 同一业务实体固定 key。
- 消费端按状态机判断能否处理。
- 先到的后置事件可以延迟重试。
- 用版本号、时间戳或序列号做幂等和顺序校验。

**面试表达**：

> 乱序要先问“要求什么范围有序”。订单系统一般只要求同一订单有序，用 orderId 固定分区即可。跨订单全局有序成本太高，通常没有必要。

## 15. Kafka 幂等生产者是什么？

**一句话回答**：幂等生产者通过 Producer ID 和序列号让同一生产者对同一分区的重复发送不会产生重复消息，用来解决生产端重试导致的重复写入。

**注意点**：

- 解决的是生产端重试重复，不等于消费端业务幂等。
- 通常和重试、ACK 配合使用。
- 不能替代业务唯一键和消费者幂等。

**面试表达**：

> Kafka 幂等生产者解决的是 Producer 重试时同一条消息写入多次的问题，但它不保证消费者业务只执行一次。消费端仍然要按业务主键做幂等。

## 16. Kafka 事务和 Exactly Once 怎么理解？

**一句话回答**：Kafka 事务可以把多分区写入和消费位点提交放进一个事务，配合幂等生产者实现 Kafka 内部的 Exactly Once；但业务端到端精确一次仍要靠数据库事务、幂等和状态机兜底。

**典型场景**：

- 读取 Topic A，处理后写入 Topic B，同时提交 A 的 Offset。
- 要么输出消息和 Offset 一起成功，要么一起失败。

**面试表达**：

> Kafka 的 Exactly Once 主要指 Kafka 流处理内部的精确一次，比如消费 A、生产 B、提交 Offset 这几步原子化。它不等于“订单不会重复创建”。只要涉及外部数据库或第三方系统，业务幂等仍然必须做。

## 17. Kafka 和 RocketMQ 怎么区别？

| 对比项 | Kafka | RocketMQ |
| --- | --- | --- |
| 典型强项 | 日志、事件流、数据管道、高吞吐 | 业务消息、事务消息、延迟消息 |
| 存储模型 | Partition 日志 | Topic + Queue |
| 消费模型 | Consumer Group 拉取 | Consumer Group，业务消息语义更强 |
| 延迟消息 | 需要额外设计或新能力支持 | 原生延迟消息更常见 |
| 顺序消息 | 单分区有序 | 同一队列有序 |

**面试表达**：

> Kafka 更像高吞吐事件流平台，适合日志和数据管道；RocketMQ 更偏业务消息，事务消息、延迟消息这些业务语义更突出。项目选型不能只看吞吐，要看业务需要什么消息能力。

## 18. Kafka 如何结合项目经验来讲？

**回答结构**：

1. 业务场景：订单、库存、履约状态、日志采集、数据同步。
2. Topic 设计：按事件类型和消费者职责拆分。
3. 分区设计：按吞吐和局部有序要求决定。
4. 可靠性：ACK、ISR、幂等、手动提交 Offset。
5. 业务兜底：幂等表、状态机、补偿任务、告警。

**面试表达**：

> 如果在履约场景里用 Kafka，我会把工单状态、库存 token、站点变更这些事件按 Topic 拆开，同一工单用工单号作为 key 保证局部有序。生产端用可靠发送，消费端处理成功后提交 Offset，并通过状态机和幂等表防重复，异常消息进入补偿和告警。

## 高频追问速记

| 问题 | 速答 |
| --- | --- |
| Kafka 为什么快？ | 顺序写、Page Cache、零拷贝、批量、分区并行。 |
| Kafka 保证全局有序吗？ | 不保证，只保证单 Partition 内有序。 |
| 消费者数能否大于分区数？ | 可以，但多出来的消费者在同一组内会空闲。 |
| ACK 怎么选？ | 关键消息倾向 `acks=all`，日志类可按容忍度放宽。 |
| ISR 是什么？ | 和 Leader 同步的副本集合，是可靠确认和故障切换边界。 |
| 为什么重复消费？ | 处理成功但 Offset 未提交、Rebalance、重试等。 |
| 如何解决重复消费？ | 业务幂等、唯一索引、状态机、处理日志表。 |
| Exactly Once 等于业务精确一次吗？ | 不等于，外部 DB 和业务动作仍要幂等。 |
