# Java 互联网公司常见业务场景面试题与解决方案

这份材料按互联网 Java 面试中最常见的业务场景整理，适合用来回答“你遇到过哪些复杂业务问题”“高并发下怎么设计”“如何保证一致性和可靠性”这类问题。回答时不要只背技术名词，最好按“业务目标 -> 核心矛盾 -> 方案设计 -> 风险兜底”来讲。

## 1. 高并发扣减库存如何防止超卖？

**常见问法**

- 秒杀或下单时，多个用户同时购买同一个商品，怎么保证库存不被扣成负数？
- Redis 扣库存和 MySQL 扣库存分别适合什么场景？

**解决方案**

低并发或普通交易场景，可以用 MySQL 条件更新：

```sql
UPDATE sku_stock
SET available_stock = available_stock - #{num}
WHERE sku_id = #{skuId}
  AND available_stock >= #{num};
```

通过影响行数判断是否扣减成功。这个方案依赖数据库行锁和条件更新，简单可靠，不会超卖，但热点 SKU 会产生行锁竞争。

高并发秒杀场景，一般用 Redis + Lua 先做原子预扣：

```lua
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock < tonumber(ARGV[1]) then
    return 0
end
redis.call('DECRBY', KEYS[1], ARGV[1])
return 1
```

预扣成功后写入 MQ，异步创建订单、扣减 DB 或写库存流水。订单取消、支付超时、创建失败时，需要回补库存，并通过库存流水和定时对账保证最终一致。

**关键细节**

- 扣减必须是原子操作，不能先查库存再扣库存。
- Redis 扣减成功不代表交易最终成功，需要订单状态闭环。
- 需要防止重复扣减，可以用业务单号、订单号、用户活动维度做幂等。
- 热点 SKU 可以做库存分桶，降低单 key 或单行 DB 压力。

**面试表达**

> 普通场景我会用数据库条件更新保证不超卖；秒杀场景会把库存前置到 Redis，用 Lua 做原子扣减，再通过 MQ 削峰异步落库。整体上通过库存流水、订单状态、超时回补和定时对账保证最终一致性。

## 1.1. 如何设计与实施 Redis 库存预热？

**常见问法**

- Redis 预热库存怎么做？
- 预热库存如何防止缓存雪崩和击穿？
- 活动进行中，运营临时追加或减少库存，Redis 和 DB 如何协同？

**解决方案**

库存预热是秒杀和大促场景下保护 DB 的核心手段。具体实施方案如下：

1. **确定预热时机**：
   - **定时任务预热（推荐）**：使用 XXL-Job 定期（如每 10 分钟）扫描即将开始的秒杀活动，提前 30 分钟将库存载入 Redis。
   - **后台发布触发**：运营在管理后台上架或发布秒杀活动时，通过发送 MQ 消息异步触发库存预热。
   - **手动触发**：后台提供预热按钮，供大促前人工核对并手动预热。
2. **选择数据结构**：
   - 使用 Redis **String** 结构：Key 设计为 `promo:stock:{activityId}:{skuId}`，Value 为库存数。
   - 避免使用 Hash 结构（如整个活动一个 Key），防止所有 SKU 路由到同一个 Redis 节点，导致单节点 CPU 过载（热 Key 问题）。
3. **安全写入逻辑**：
   - 使用分布式锁（如 Redisson 锁住 `lock:preheat:{activityId}:{skuId}`）防止多实例重复预热。
   - 先通过 `EXISTS` 校验是否已存在缓存。若活动已开始，绝对不要用 `SET` 直接覆盖，防止冲掉高并发扣减中的实时库存。
   - 批量查询 DB 并通过 Redis **Pipeline（管道）** 批量写入，降低网络开销。
   - 设置过期时间（TTL）为 `活动结束时间 + 24小时`，并加上随机偏移值（1~10分钟）防止雪崩。

**关键细节**

- **热 Key 售罄降级**：对于超热点 SKU，在 JVM 本地内存（如 Caffeine）缓存一个售罄标记。一旦 Redis 库存扣为 0，广播更新所有本地缓存，后续请求在本地拦截，防止大量无效请求持续压迫 Redis。
- **预热失败 Fail-Fast**：如果商品库存没有预热成功，API 层直接报错返回“系统繁忙/售罄”，绝对不允许回退到 DB 查询，坚决保护数据库。
- **动态修改库存**：
  - **追加**：使用 `INCRBY key delta` 增加 Redis 库存，并异步更新 DB。
  - **减少**：使用 Lua 脚本先校验当前 Redis 库存是否充足，充足则扣减，最后异步更新 DB。

**面试表达**

> 我们会在活动开始前，通过 XXL-Job 分批将库存数据以 Pipeline 方式预热到 Redis，选用 String 结构以分散集群槽位。为防雪崩，TTL 会加入随机偏差。一旦发生预热遗漏或失败，系统会执行 Fail-Fast 而不穿透到 DB。对超热点 SKU 结合本地缓存的售罄标记拦截无效流量，保证 Redis 本身的安全。

## 2. 秒杀系统怎么设计？

**常见问法**

- 几十万人同时抢一个商品，系统怎么扛住？
- 秒杀如何防刷、限流、削峰、避免超卖？

**解决方案**

秒杀系统要分层削峰：

1. 前端做活动静态化、倒计时、按钮置灰，减少无效请求。
2. 网关层做限流、黑名单、验证码、人机校验。
3. 应用层做用户维度幂等，例如一人一单。
4. Redis 预热库存，用 Lua 原子扣减。
5. 扣减成功后写 MQ，异步创建订单。
6. 订单超时未支付后通过延迟消息或定时任务关闭订单并回补库存。
7. 后台通过库存流水、订单流水、支付流水做对账。

**关键细节**

- 秒杀接口路径可动态化，降低脚本刷接口概率。
- 用户请求可以先进入排队状态，避免所有请求直接打到订单系统。
- 秒杀库存要与普通库存隔离，避免活动流量影响主交易链路。
- MQ 消费端要幂等，因为消息可能重复投递。

**面试表达**

> 秒杀不是单纯加 Redis，而是从入口限流、防刷、库存预扣、MQ 削峰、订单异步创建、超时回补和对账补偿一起设计。核心目标是保护主链路，不超卖，同时允许最终一致。

## 2.1. 秒杀库存怎么保证数据最终一致性？

**常见问法**

- 秒杀库存怎么保证数据一致性？
- Redis 扣减成功但 DB 扣减失败了怎么办？
- 如何保证 Redis 和 DB 库存的最终一致？

**解决方案**

在高并发秒杀场景下，系统无法承受数据库事务锁的开销，通常采用 **Redis 预扣减（真理边界） + MQ 异步落库（最终一致）** 的方案。为了保证两端库存绝对不乱，必须做闭环设计：

1. **第一防线：Redis Lua 原子预减**
   - 所有库存扣减请求首先在 Redis 执行，使用 Lua 脚本保证“读库存 -> 校验 -> 扣库存”的原子性，确保 Redis 层面绝对不超卖（锁住库存的上限）。
2. **第二防线：发送 RocketMQ 事务消息**
   - 扣减成功后创建订单，必须保证“订单创建”与“发送扣减 DB 消息”的原子性。
   - 使用 RocketMQ 事务消息：先发半消息（Half Message） → 执行本地创建订单事务 → 提交/回滚事务消息。若本地事务回滚，消息取消发送，同时把扣减的 Redis 库存加回去。
3. **第三防线：下游幂等扣减 DB**
   - MQ 消费者（库存服务）消费消息扣减 MySQL 真实库存时，可能由于网络抖动重复消费。
   - 解决方案：在 MySQL 中使用**库存扣减流水表**，以 `order_no` 作为**唯一索引**。每次扣减 DB 前向流水表插记录，插成功才执行扣减 SQL，插失败（唯一键冲突）则直接丢弃消息。
4. **第四防线：超时未支付关单与回补**
   - 用户抢到秒杀资格但未在规定时间（如 15 分钟）内完成支付，订单系统发起关单，发送“回退库存”消息到 MQ。
   - 消费者执行 MySQL 库存增加，并更新流水状态为“已退回”；同时往 Redis 执行 `INCRBY` 回补库存。
5. **第五防线：离线定时对账（最终校准）**
   - 定时任务（如 XXL-Job）定期核对：`MySQL 剩余库存 == 初始库存 - 已支付订单数 - 占用中（未支付）订单数`。
   - 配合 Redis 库存与 DB 对账，若发现对不齐，发出告警并通过人工/程序工具进行校准。

**关键细节**

- **防范超卖是核心**：Redis 的 Lua 脚本是控制商品数量的硬边界，而 RocketMQ 事务消息是确保本端业务与下发消息强绑定的纽带。两者缺一不可。
- **Fail-Fast**：一旦 Redis 库存扣为 0，后续请求在 Redis 层面直接返回“已售罄”，完全不经过 MQ 和 DB。

**面试表达**

> 我们系统通过“Redis 原子预减”保证不超卖，并使用“RocketMQ 事务消息”确保订单创建与库存扣减消息的发送实现强绑定。对于下游消费端，通过“库存流水表唯一索引”解决重复消费导致的二次扣减，如果用户超时未支付则触发关单和双端（Redis & DB）库存回补，最终通过“离线对账任务”进行库存核对和校准，实现高吞吐下的库存最终一致性。

## 2.2. 高并发库存异步下单时，前端如何实时展示结果？

**常见问法**

- Redis 预扣成功后，订单是 MQ 异步创建的，前端怎么知道订单是否真的下成功？
- 如果最终一致性要等几秒，用户端怎么做实时状态反馈？
- 秒杀场景是返回订单号，还是返回排队中？

**核心思路**

高并发库存场景里，不能为了让前端立刻看到“下单成功”，就把库存、订单、支付、营销全部放到一个同步强一致链路里。更好的做法是：

```text
后端同步完成：资格校验 + Redis 库存预占
后端异步完成：订单创建 + DB 库存落库 + 营销通知 + 补偿对账
前端实时展示：已受理 -> 库存已锁定 -> 订单生成中 -> 成功/失败
```

也就是说，**实时反馈不等于同步完成所有业务动作**。用户应该实时看到处理进度，最终结果由订单状态确认。

**完整流程**

1. 用户点击抢购，前端按钮置灰，防止重复点击。
2. 网关做限流、防刷、风控、人机校验。
3. 后端执行 Redis Lua，原子完成库存判断、一人一单、库存预扣。
4. 如果库存不足，直接返回 `SOLD_OUT`。
5. 如果重复抢购，直接返回 `DUPLICATE`。
6. 如果预扣成功，生成 `requestId`，返回“库存已锁定，订单生成中”。
7. 后端发送 MQ，下游异步创建订单、写库存流水、扣减 DB 库存。
8. 订单服务处理完成后，把结果写入 Redis 或订单表。
9. 前端通过短轮询、SSE 或 WebSocket 获取最终结果。
10. 成功则跳转支付页；失败则提示失败，并由后端回补库存。

**用户端看到的状态**

| 后端阶段 | 前端展示 | 说明 |
| :--- | :--- | :--- |
| 请求刚提交 | 正在提交... | 前端按钮置灰，避免重复提交 |
| Redis 库存不足 | 已售罄 | 直接失败，不进入 MQ |
| 重复抢购 | 请勿重复抢购 | 用户维度幂等拦截 |
| Redis 预扣成功 | 库存已锁定，订单生成中 | 说明抢购资格已拿到，但订单还没最终创建 |
| MQ 排队中 | 排队处理中，请稍候 | 系统正在削峰处理 |
| 订单创建成功 | 下单成功，请支付 | 返回 `orderId`，跳转支付页 |
| 订单创建失败 | 下单失败，库存已释放 | 后端要回补 Redis / DB 库存 |
| 长时间无结果 | 订单处理中，请稍后到订单列表查看 | 前端停止无限等待，最终以订单中心为准 |

**接口设计**

抢购接口不建议直接返回“订单创建成功”，而是返回受理状态：

```json
{
  "status": "ACCEPTED",
  "stage": "STOCK_RESERVED",
  "message": "库存已锁定，订单生成中",
  "requestId": "REQ123"
}
```

前端拿到 `requestId` 后查询结果：

```http
GET /seckill/order/result?requestId=REQ123
```

处理中：

```json
{
  "status": "PROCESSING",
  "message": "排队处理中"
}
```

成功：

```json
{
  "status": "SUCCESS",
  "orderId": "O202607040001",
  "message": "下单成功，请支付"
}
```

失败：

```json
{
  "status": "FAILED",
  "reason": "订单创建失败，库存已释放"
}
```

**结果存储**

为了让前端快速查询，异步结果一般写 Redis，设置短 TTL：

```text
seckill:result:{requestId} = PROCESSING
seckill:result:{requestId} = SUCCESS:{orderId}
seckill:result:{requestId} = FAILED:{reason}
```

订单服务消费 MQ 后：

```text
创建成功 -> 写 SUCCESS + orderId
创建失败 -> 写 FAILED + reason -> 回补库存
```

**实时通知方式怎么选？**

| 方式 | 特点 | 适合场景 |
| :--- | :--- | :--- |
| 短轮询 | 实现最简单，前端每 500ms / 1s 查询一次 | 大多数秒杀场景 |
| 长轮询 | 服务端等待一小段时间，有结果再返回 | 想减少空查询，但实现稍复杂 |
| SSE | 服务端单向推送，浏览器支持较好 | Web 页面实时通知 |
| WebSocket | 双向通信，连接成本更高 | App、交易、直播、强实时互动 |

大多数互联网秒杀场景，短轮询就够了。因为结果通常几秒内出来，前端轮询 5 到 10 次即可。SSE / WebSocket 能让通知更实时，但不会让 MQ、订单落库这些业务动作本身变快。

**如果产品要求立刻看到订单号怎么办？**

可以改成“同步创建轻量订单 + 异步处理非核心动作”：

```text
Redis Lua 预扣成功
-> 同步创建订单主记录，状态 = 待支付 / 创建中
-> 立即返回 orderId
-> 库存 DB 扣减、营销、积分、通知异步处理
-> 异常时通过状态机、补偿任务和对账修复
```

这种方案用户体验更实时，但订单库要承受瞬时写入压力，需要配套：

- 订单号提前生成。
- 用户维度幂等。
- 订单表分库分表或写入削峰。
- 支付前校验订单状态。
- 库存回补、状态补偿和对账。

**关键细节**

- 前端不能把“Redis 预扣成功”展示成“订单已成功”，只能展示“库存已锁定 / 订单生成中”。
- 前端超时不等于失败，最终结果要以订单中心状态为准。
- 后端必须保证结果可查询，不能只依赖 MQ 消费日志。
- 失败场景要回补库存，并写失败原因，避免用户一直卡在处理中。
- 实时推送只是用户体验优化，不替代后端幂等、补偿和对账。

**面试表达**

> 高并发库存异步下单时，我会把用户体验和后端一致性拆开。接口同步完成库存预扣和资格校验，成功后返回 `requestId` 和“库存已锁定、订单生成中”，前端通过短轮询、SSE 或 WebSocket 获取最终结果。后端通过 MQ 异步创建订单，结果写入 Redis 或订单表；成功后前端跳转支付页，失败则提示并由后端回补库存。如果业务强要求立刻看到订单号，可以同步创建一条轻量订单，状态为待支付或创建中，库存 DB 扣减、营销和通知异步处理。核心是用户能实时看到状态，但系统仍保留削峰、幂等、补偿和最终一致能力。

## 2.3. 高并发下单后想立刻看到订单，应该怎么设计？

**常见问法**

- 如果用户下单完就要在订单列表里看到订单，高并发场景怎么做？
- 秒杀场景能不能同步返回 `orderId`？
- 同步创建订单会不会把订单库打垮？怎么兜底？

**一句话回答**

如果业务要求“下单后立刻看到订单”，就不要采用完全异步创建订单，而是采用 **Redis 预扣库存 + 同步创建轻量订单 + 非核心动作异步处理** 的方案。

**完整流程**

```text
用户点击抢购
-> 网关限流 / 风控 / 防刷
-> Redis Lua 原子判断库存 + 一人一单 + 预扣库存
-> 预扣成功
-> 同步创建轻量订单，状态 = WAIT_PAY 或 CREATING
-> 返回 orderId
-> 前端立即跳转订单详情页 / 支付页
-> 后端发送 MQ
-> 库存 DB 扣减、营销、积分、通知异步处理
-> 异步失败时通过状态机、补偿任务、对账修复
```

用户端看到的是：

```text
订单已生成，待支付
```

或者：

```text
订单已生成，正在确认，请稍候
```

**同步链路只做核心动作**

同步创建的订单要尽量“轻”，只落订单主表核心字段：

| 字段 | 说明 |
| :--- | :--- |
| `order_id` | 提前生成的全局唯一订单号 |
| `user_id` | 用户 ID |
| `sku_id` / `activity_id` | 商品和活动维度 |
| `amount` | 订单金额 |
| `status` | `WAIT_PAY` 或 `CREATING` |
| `expire_time` | 支付截止时间 |
| `request_id` | 幂等请求号 |
| `created_time` | 创建时间 |

同步链路不要做太多重活，例如：

- 不同步扣减库存 DB 的复杂流水。
- 不同步发积分、通知、站内信。
- 不同步做复杂营销核销。
- 不同步等待多个下游系统都成功。

这些动作统一通过 MQ 异步处理。

**为什么高并发下还能同步建订单？**

关键是：不是所有请求都进入订单库，只有 Redis 预扣成功的请求才能创建订单。

```text
入口流量 100 万
-> 网关限流、防刷过滤一批
-> Redis Lua 库存预扣过滤一批
-> 真正进入订单库的请求 <= 可售库存数量
```

也就是说，订单库承受的是“抢到资格的人”的写入，而不是全部抢购流量。

**抗高并发关键设计**

| 设计点 | 作用 |
| :--- | :--- |
| 网关限流、防刷 | 先把恶意和无效流量挡在外面 |
| Redis Lua 预扣 | 库存不足直接失败，避免打订单库 |
| 本地售罄标记 | 售罄后 JVM 本地快速失败，减少 Redis 压力 |
| 全局订单号预生成 | 避免创建订单时依赖数据库自增瓶颈 |
| 订单表分库分表 | 分散同步写入压力 |
| 精简订单主表索引 | 降低高并发写入成本 |
| `request_id` 唯一索引 | 防止接口重试导致重复建单 |
| `user_id + activity_id` 唯一索引 | 防止一人多单 |
| MQ 异步处理非核心动作 | 保留削峰能力 |

幂等索引示例：

```sql
create unique index uk_request_id on order_main(request_id);
create unique index uk_user_activity on order_main(user_id, activity_id);
```

同一个 `requestId` 重试时，直接返回已有 `orderId`，不要重复创建订单。

**订单状态怎么设计？**

| 状态 | 含义 | 前端展示 |
| :--- | :--- | :--- |
| `CREATING` | 订单主记录已创建，异步确认中 | 订单已生成，正在确认 |
| `WAIT_PAY` | 待支付 | 下单成功，请支付 |
| `CREATE_FAILED` | 异步确认失败 | 下单失败，库存已释放 |
| `CLOSED` | 超时未支付关闭 | 订单已关闭 |
| `PAID` | 已支付 | 支付成功 |

如果同步链路已经满足支付前置条件，可以直接返回 `WAIT_PAY`。如果还要等库存 DB、营销核销等异步确认，可以先返回 `CREATING`，等异步确认完成后再流转到 `WAIT_PAY`。

**支付前必须校验状态**

支付接口不能只看订单存在，还要看订单状态：

```text
CREATING -> 不允许支付，提示订单确认中
WAIT_PAY -> 允许支付
CREATE_FAILED -> 不允许支付，提示下单失败
CLOSED -> 不允许支付，提示订单已关闭
```

否则会出现订单异步确认失败，但用户已经支付的问题。

**异步失败怎么处理？**

如果 Redis 预扣成功、轻量订单也创建成功，但后续库存 DB 扣减或营销核销失败，需要做补偿：

```text
订单状态改为 CREATE_FAILED / CLOSED
回补 Redis 库存
库存流水标记失败
发送失败事件或站内通知
定时任务对账兜底
必要时人工修复
```

**和完全异步排队方案的区别**

| 方案 | 前端看到 | 后端核心链路 | 适合场景 |
| :--- | :--- | :--- | :--- |
| 排队型异步下单 | 库存已锁定，订单生成中 | 返回 `requestId`，订单异步创建 | 极端秒杀、订单库压力很大 |
| 订单型同步建单 | 订单已生成，待支付 / 确认中 | 同步创建轻量订单，返回 `orderId` | 希望用户立刻看到订单，流量可控 |

**面试表达**

> 如果业务要求下单后立刻在订单列表看到订单，我会把方案从完全异步创建订单调整为同步创建轻量订单。具体是先通过网关限流和 Redis Lua 预扣库存过滤大部分请求，只有预扣成功的请求才进入订单库；然后同步创建订单主记录，状态可以是 `WAIT_PAY` 或 `CREATING`，并立即返回 `orderId`。同步链路只做核心建单和幂等校验，库存 DB 扣减、营销、积分、通知等通过 MQ 异步处理。为了扛高并发，需要订单号预生成、订单表分库分表、`request_id` 或 `user_id + activity_id` 唯一索引、防重复建单、支付前状态校验，以及异步失败后的库存回补、状态补偿和对账。这样用户能立刻看到订单，同时系统仍然保留削峰和最终一致能力。

## 2.4. 订单已创建但异步确认失败，用户体验很差怎么办？

**常见问法**

- 如果订单已经展示给用户了，后面异步库存扣减失败怎么办？
- 用户看到“下单成功”后订单又失败，体验很差，怎么避免？
- 高并发场景下哪些动作必须同步确认，哪些动作可以异步？

**一句话回答**

不要把最终一致里的中间态包装成成功态。用户看到的状态必须和后端确定性匹配：如果核心交易还没确认，就展示 `CREATING`；只有库存、订单等核心动作确认后，才能展示 `WAIT_PAY` 或“下单成功”。

**方案一：创建 `CREATING` 订单，不允许立即支付**

适用于：用户要马上在订单列表看到记录，但库存 DB、营销资格等核心动作仍需要异步确认。

```text
Redis Lua 预扣成功
-> 同步创建轻量订单，状态 = CREATING
-> 返回 orderId
-> 前端展示：订单已生成，正在确认
-> 异步扣减 DB 库存 / 核销营销 / 确认资格
-> 成功后订单状态改为 WAIT_PAY
-> 前端展示：下单成功，请支付
```

如果异步失败：

```text
订单状态改为 CREATE_FAILED / CLOSED
回补 Redis 库存
库存流水标记失败
前端展示：下单失败，库存已释放
```

支付接口必须校验订单状态：

```text
CREATING -> 不允许支付，提示订单确认中
WAIT_PAY -> 允许支付
CREATE_FAILED -> 不允许支付，提示下单失败
CLOSED -> 不允许支付，提示订单已关闭
```

这个方案的关键点是：**用户能看到订单，但看到的是“确认中订单”，不是“已成功订单”**。

**方案二：核心动作同步，非核心动作异步**

适用于：产品强要求用户点击后立即看到“待支付订单”，并且不接受后续大概率失败。

这时就不能把库存确认这种核心动作完全异步，而要把核心交易链路同步完成：

```text
Redis Lua 预扣
-> 同步创建订单主记录
-> 同步写库存扣减流水 / 同步扣 DB 库存
-> 成功后返回 WAIT_PAY
-> 营销、积分、通知、统计、推荐异步处理
```

也就是说：

| 动作类型           | 是否适合同步   | 原因          |
| :------------- | :------- | :---------- |
| Redis 预扣库存     | 同步       | 控制是否有资格进入建单 |
| 订单主记录创建        | 同步       | 用户要立即看到订单   |
| 库存流水 / DB 库存确认 | 建议同步或强确认 | 失败会影响订单是否成立 |
| 支付前状态校验        | 同步       | 防止异常订单进入支付  |
| 积分、通知、统计       | 异步       | 失败不影响订单成立   |
| 推荐、运营埋点        | 异步       | 可延迟、可补偿     |

这个方案压力会更大，但 Redis 已经过滤掉大量失败请求，真正进入订单库和库存库的请求数量通常不会超过可售库存数量。

**两种方案怎么选？**

| 方案 | 用户看到 | 优点 | 代价 |
| :--- | :--- | :--- | :--- |
| `CREATING` 确认中 | 订单已生成，正在确认 | 用户能看到订单，同时避免过早承诺成功 | 前端和状态机更复杂 |
| 核心同步确认 | 下单成功，请支付 | 用户体验最好，状态确定性强 | 订单库和库存库同步压力更高 |

**关键原则**

- 不要让用户看到“下单成功”后又因为核心异步失败变成失败。
- 如果后续失败会影响交易成立，就不要过早进入 `WAIT_PAY`。
- 能异步的是通知、积分、统计、推荐等非核心动作。
- 库存确认、订单主记录、支付前状态校验属于核心动作，要么同步完成，要么用 `CREATING` 明确告诉用户仍在确认。
- 补偿不是用来掩盖错误承诺的，补偿是异常恢复手段，不应该成为用户体验主路径。

**面试表达**

> 如果订单已经展示给用户，但后续异步库存扣减失败，体验会很差。所以我不会把中间态包装成成功态。更成熟的做法有两种：第一种是同步创建轻量订单，但状态是 `CREATING`，前端展示“订单已生成，正在确认”，支付接口不允许支付；等库存 DB、营销资格等核心确认完成后再流转到 `WAIT_PAY`。第二种是如果业务强要求立即待支付，那就把订单主记录、库存流水或库存 DB 确认放到同步核心链路，只有核心动作都成功才返回 `WAIT_PAY`，积分、通知、统计等非核心动作再异步。核心原则是用户看到的状态必须和后端确定性一致，不能用最终一致掩盖一个不确定的成功态。

## 3. 如何设计接口幂等，防止重复下单或重复提交？

**常见问法**

- 用户连续点击提交按钮，怎么防止生成多笔订单？
- MQ 重复消费或支付回调重复通知，怎么保证只处理一次？

**解决方案**

幂等要根据场景选择不同手段：

- 前端防重复点击：按钮置灰、提交中状态，只是体验优化，不能作为最终保证。
- 唯一业务号：下单请求携带 `requestId` 或 `orderNo`，服务端根据唯一索引防重复。
- Redis `SETNX`：短时间防重，例如 `idempotent:submit:{userId}:{requestId}`。
- 数据库唯一索引：最终兜底，例如 `user_id + activity_id`、`payment_no`、`biz_no`。
- 状态机判断：只允许订单从合法状态流转到下一个状态。

### 3.1. requestId 如何具体实施防重？

**工作流设计**：

1. **第一阶段（获取 Token）**：用户进入确认页，前端自动调用后端接口获取一个全局唯一的 `requestId`（如 Snowflake ID 或带有业务前缀的 UUID），Redis 缓存此 ID 并设置 TTL（如 30 分钟）。
2. **第二阶段（携带提交）**：用户点击提交，请求头（如 `X-Idempotency-Key`）或请求体携带 `requestId` 发送给后端。
3. **第三阶段（双层校验）**：
   - **第一防线：Redis `SETNX` (防并发并发提交)**
     ```text
     SET idempotent:order:{requestId} "PROCESSING" NX PX 10000
     ```
     如果返回成功，说明是首次请求，执行业务；如果返回失败，说明是并发重复请求，直接拦截并报错。
     业务成功则将 Value 更新为 `"SUCCESS"`（TTL 适当延长，比如 1 小时）；若业务发生“可修改后重试”的异常（如库存不足、参数错误），则主动删除该 Key，允许重新提交。
   - **第二防线：MySQL 唯一索引 (最终兜底防穿透)**
     订单表将 `request_id` 设为 `UNIQUE` 索引。若 Redis 锁因网络超时或异常过期失效，MySQL 唯一索引将抛出 `DuplicateKeyException` 并触发事务回滚，以此保证绝对不会产生多条数据。

**Java 核心实现示例**：

```java
// 尝试加锁 (SETNX)，设置 10 秒过期时间防死锁
Boolean success = redisTemplate.opsForValue().setIfAbsent(
    "idempotent:order:" + requestId, "PROCESSING", 10, TimeUnit.SECONDS
);

if (Boolean.FALSE.equals(success)) {
    return ResponseEntity.status(HttpStatus.CONFLICT).body("请勿重复提交订单");
}

try {
    // 执行业务落库，且 DB 订单表有 UNIQUE KEY(request_id) 兜底
    OrderResult result = orderService.createOrder(orderDTO, requestId);
    
    // 业务成功，更新状态为 SUCCESS 并延长有效期
    redisTemplate.opsForValue().set("idempotent:order:" + requestId, "SUCCESS", 1, TimeUnit.HOURS);
    return ResponseEntity.ok(result);
} catch (BizException e) {
    // 业务可重试异常（如库存不足），删除 key 允许用户修改后重新提交
    redisTemplate.delete("idempotent:order:" + requestId);
    return ResponseEntity.badRequest().body(e.getMessage());
} catch (Exception e) {
    // 系统未知异常，可保留 PROCESSING 状态待 TTL 过期，或捕获后由重试机制处理
    throw e;
}
```

**关键细节**

- 幂等 key 必须来自稳定业务语义，不能只用时间戳。
- Redis 防重需要设置过期时间，但最终仍建议有 DB 唯一索引兜底。
- 幂等不是简单返回成功，要能返回第一次处理结果或当前业务状态。

**面试表达**

> 我会把幂等分为入口防重、业务唯一键和数据库兜底三层。像下单可以用请求号或用户活动唯一索引，支付回调用支付流水号唯一索引，状态变更再通过状态机限制非法重复流转。

## 4. 支付回调如何保证可靠处理？

**常见问法**

- 支付平台回调多次怎么办？
- 支付成功后订单更新失败怎么办？

**解决方案**

支付回调链路一般这样设计：

1. 校验签名和回调来源。
2. 根据支付流水号查询本地支付单。
3. 判断当前支付单状态，已成功则直接返回成功。
4. 在本地事务中更新支付单、订单状态、写支付流水或事件表。
5. 发送 MQ 通知履约、积分、优惠券、财务等下游。
6. 如果 MQ 发送失败，通过本地消息表或定时补偿任务重试。

**关键细节**

- 支付回调一定要幂等，支付流水号要有唯一索引。
- 订单状态只能从待支付变成已支付，不能被旧回调覆盖。
- 返回给支付平台的成功响应，要在本地核心状态处理成功之后。
- 支付金额、商户号、订单号必须和本地单据校验一致。

**面试表达**

> 支付回调我会先验签，再用支付流水号做幂等。核心状态更新放在本地事务里，后续下游动作通过 MQ 异步通知。如果 MQ 或下游失败，通过本地消息表和补偿任务保证最终一致。

## 5. 订单超时未支付如何自动关闭？

**常见问法**

- 30 分钟未支付自动取消订单怎么实现？
- 延迟消息丢了怎么办？

**解决方案**

常见实现有三种：

- RocketMQ 延迟消息：创建订单后发送延迟消息，到期检查订单状态。
- Redis 过期监听：设置订单过期 key，监听过期事件。
- 定时任务扫表：定期扫描超过支付时间且仍未支付的订单。

实际生产更推荐“延迟消息 + 定时任务兜底”：

1. 创建订单时发送延迟关闭消息。
2. 消费延迟消息时查询订单当前状态。
3. 如果仍是待支付，则关闭订单、释放库存、回退优惠券。
4. 如果已经支付，则忽略消息。
5. 定时任务补偿扫描漏处理订单。

**关键细节**

- 不能收到延迟消息就直接关单，必须查当前订单状态。
- 关闭订单、释放库存、退券要保证幂等。
- 延迟消息不是绝对准时，业务要允许一定时间误差。

**面试表达**

> 我会用延迟消息做主要触发，用定时任务做兜底。关单时先查订单状态，只有待支付才能关闭，然后幂等释放库存和优惠券，避免支付成功订单被误关。

## 5.1. Spring 本地事务失效的场景有哪些？同一个类调用为什么失效？

**常见问法**

- Spring 哪些场景下会导致事务失效？
- 同一个类里的事务方法互相调用，事务会生效吗？为什么？怎么解决？
- 事务方法里捕获了异常不抛出，事务会回滚吗？

**解决方案**

同一个类内的事务方法互相调用，**事务会失效（内部调用失效）**。
- **失效原因**：Spring 事务是基于 **AOP 动态代理** 实现的。外部调用时，Spring 会调用该类的**代理对象**，从而织入事务开启与提交/回滚的增强逻辑；而类内部方法调用（相当于 `this.methodB()`）是通过**原对象**直接调用，不经过代理对象拦截，因此事务注解完全被忽略。
- **常用解法**：
  1. **重构拆分类（推荐）**：将内部调用的方法抽取到另一个 Service/Helper 中，并在当前类中注入该 Bean 调用，保证调用走代理逻辑。
  2. **注入自身代理**：在 Service 内部通过 `@Autowired` 注入自己，然后通过注入的实例调用内部方法。
  3. **AopContext 获取当前代理**：使用 `((YourService) AopContext.currentProxy()).methodB()` 调用（需开启 `@EnableAspectJAutoProxy(exposeProxy = true)`）。

Spring 本地事务失效的其他主要场景：

1. **方法修饰符非 `public`**：Spring 默认只拦截 public 方法，private、protected 或 package-private 方法上的 `@Transactional` 会被忽略。
2. **异常被内部 `try-catch` 吞掉**：业务代码捕获了异常未重新抛出，代理对象感知不到异常，导致事务照常提交。
3. **回滚异常配置错误**：Spring 事务默认只对 `RuntimeException` 和 `Error` 进行回滚。若发生受检异常（Checked Exception，如 `IOException`），默认不回滚。需要配置为 `@Transactional(rollbackFor = Exception.class)`。
4. **多线程异步调用**：Spring 事务的数据库连接通过 `ThreadLocal` 绑定在当前线程。若开启子线程执行数据库操作，子线程无法共享主线程的连接与事务上下文。
5. **Bean 未被 Spring 管理**：类上漏写了 `@Service` 或 `@Component`，或者类是手动 `new` 出来的，Spring 无法为其生成 AOP 代理。
6. **数据库引擎不支持**：底层表采用 MyISAM 等不支持事务的引擎，Spring 声明事务也会失效。

**关键细节**

- **声明式事务的本质**：事务是借由数据库 Connection 进行控制的，Spring 主要是通过 AOP 控制 Connection 的 `commit()` 与 `rollback()`。
- **自定义异常回滚**：建议在项目开发中对所有 `@Transactional` 显式指定 `rollbackFor = Exception.class` 或自定义业务基类异常，避免受检异常漏回滚。
- **事务与异步多线程**：在 `@Transactional` 方法中调用外部 `@Async` 方法时，异步方法内的数据库操作与当前方法不在同一个数据库事务内，应警惕一致性风险。

**面试表达**

> Spring 事务失效最常见的是内部调用，因为 `this` 调用绕过了 AOP 代理对象。对此我们可以通过重构拆分类、或者自己注入自己的方式解决。另外，非 public 修饰符、内部捕获未抛出异常、未指定 `rollbackFor = Exception.class` 回滚受检异常、以及多线程异步调用等场景，都会由于代理拦截跳过、无异常触发或连接绑定在不同线程而导致事务失效。

## 6. 分布式事务如何处理？

**常见问法**

- 下单涉及订单、库存、优惠券、支付多个系统，怎么保证一致？
- 你会用 2PC、TCC 还是最终一致？

**解决方案**

互联网业务大多优先选择最终一致，而不是强一致分布式事务。

常见方案：

- 本地事务 + MQ：本服务先提交本地事务，再通过消息驱动下游。
- 本地消息表：业务数据和消息表在同一个事务中落库，由任务异步投递。
- 事务消息：例如 RocketMQ 事务消息，先发半消息，再根据本地事务结果提交或回滚。
- XA / 2PC：资源管理器参与两阶段提交，强调强一致，但锁资源时间长、性能和可用性成本高。
- Seata AT：业务代码侵入较低，基于全局事务、undo log 和全局锁自动回滚，适合简单关系型数据库场景。
- TCC：Try 预留资源、Confirm 确认、Cancel 取消，适合强业务约束场景。
- Saga：长事务拆成多个本地事务，每一步都有补偿动作。

**方案对比**

| 方案            | 一句话理解                                | 一致性   | 优点               | 代价 / 风险                           | 适合场景                   |
| :------------ | :----------------------------------- | :---- | :--------------- | :-------------------------------- | :--------------------- |
| XA / 2PC      | 多个资源一起 prepare，再统一 commit / rollback | 强一致   | 标准化、数据库层支持       | 锁持有时间长、阻塞、性能差、可用性低                | 金融核心、强一致小流量链路          |
| Seata AT      | 框架代理 SQL，记录 undo log，异常时自动反向回滚       | 偏强一致  | 业务侵入低，改造成本相对低    | 依赖数据库和框架代理，复杂 SQL、跨库场景要谨慎，全局锁影响性能 | 简单 CRUD、关系型数据库、希望低侵入改造 |
| TCC           | Try 预留、Confirm 确认、Cancel 回滚          | 业务强一致 | 可控性强，资源预留清晰      | 业务侵入大，每个接口都要写三套逻辑，空回滚、悬挂、幂等复杂     | 库存预占、资金冻结、优惠券锁定        |
| Saga          | 多个本地事务串联，失败后逐步补偿                     | 最终一致  | 适合长流程，服务解耦       | 中间状态对用户可见，补偿逻辑复杂                  | 履约、审批、订单长链路            |
| 本地消息表         | 业务表和消息表同事务落库，后台任务投递 MQ               | 最终一致  | 通用、可审计、可补偿、可人工处理 | 需要消息表、扫描任务、状态治理，有投递延迟             | 核心业务事件、需要可追踪和人工兜底      |
| RocketMQ 事务消息 | 半消息 + 本地事务 + 提交/回滚 + 回查              | 最终一致  | MQ 原生支持生产端一致性    | 只解决本地事务和消息发送一致，下游仍要幂等补偿           | 本地事务成功后必须可靠发一条消息       |
| 本地事务 + 普通 MQ  | 本地成功后发事件，下游异步处理                      | 最终一致  | 简单、吞吐高、服务解耦      | 本地成功但消息发送失败需要兜底                   | 非核心通知、可通过补偿修复的业务       |

**怎么保证最终一致性**

最终一致不是只靠一个组件，而是一整套闭环：

1. **本地事务先守住边界**：每个服务只保证自己的数据库事务，例如订单服务只保证订单创建和本地事件记录一致。
2. **事件驱动下游**：通过 MQ、事务消息或本地消息表把状态变化通知库存、优惠券、履约等服务。
3. **消费端必须幂等**：按订单号、流水号、业务事件 ID 建唯一索引或消费记录，重复消息不能重复扣减。
4. **状态机防乱序**：状态更新带当前状态条件，防止晚到消息把状态回退。
5. **失败自动重试**：临时异常走 MQ 重试、任务重试或退避重试。
6. **死信和补偿兜底**：重试多次仍失败，进入死信队列、补偿任务或人工处理。
7. **定时对账校准**：定期比对订单、库存、优惠券、支付、履约状态，发现不一致后自动修复或告警。

**选型建议**

- **能最终一致就不要强上 XA**：互联网高并发链路通常更关注吞吐、可用性和可恢复能力。
- **简单异步事件**：优先本地事务 + MQ，补本地消息表或补偿任务兜底。
- **本地事务和消息发送必须绑定**：优先 RocketMQ 事务消息或本地消息表。
- **需要审计、人工补偿、多次投递治理**：本地消息表更可控。
- **资源预占类强业务约束**：考虑 TCC，例如库存预占、资金冻结、优惠券锁定。
- **长流程、多步骤业务**：考虑 Saga，但要接受中间状态和补偿复杂度。
- **低侵入改造单体式数据库事务到分布式事务**：可以了解 Seata AT，但要评估 SQL 支持、全局锁和性能影响。

**关键细节**

- 能最终一致就不要轻易上强一致，复杂度和可用性成本很高。
- MQ 消费端必须幂等。
- 每个步骤都要有状态记录，方便补偿和对账。
- 对用户可见的核心状态要有明确主数据源，例如订单状态以订单系统为准。

**面试表达**

> 我会根据业务对一致性、实时性和吞吐的要求选方案。互联网业务多数不直接上 XA，而是用本地事务加 MQ 做最终一致：每个服务守住自己的本地事务，通过事务消息或本地消息表可靠发事件，下游消费端做幂等、状态机和重试，失败后进入死信、补偿任务和对账。资源预占这类强约束场景可以用 TCC，长流程可以用 Saga，低侵入数据库场景可以评估 Seata AT，但最终都要配套状态记录、监控告警和人工兜底。

## 7. MQ 消息如何保证不丢、不重复、可追踪？

**常见问法**

- MQ 发送失败怎么办？
- 消息重复消费怎么办？
- 消费成功但业务处理失败怎么办？

**解决方案**

消息可靠性要分别看生产端、Broker、消费端：

- 生产端：发送失败要重试，关键消息可以落本地消息表。
- Broker：开启持久化、副本和确认机制。
- 消费端：业务处理成功后再 ack。
- 重复消费：用业务唯一键、消费记录表或状态机保证幂等。
- 消费失败：重试、死信队列、告警和人工补偿。

**关键细节**

- MQ 通常保证至少一次投递，所以重复消费是正常情况。
- 消费端不要依赖“消息只来一次”。
- 消息体要带业务主键、事件类型、版本、traceId，方便排查。
- 下游状态变更要校验当前状态，避免乱序消息覆盖新状态。

**面试表达**

> 我会按生产端、Broker、消费端三段保证可靠性。生产端用本地消息表或事务消息兜底，消费端通过幂等表和状态机防重复，失败消息进入重试或死信队列，并配合告警和补偿任务。

## 8. 缓存和数据库如何保证一致性？

**常见问法**

- 缓存和数据库怎么保证一致性？更新和删除是在 DB 前还是后？为什么？
- 为什么是删除缓存，而不是更新缓存？
- 先更新数据库再删除缓存，如果第二步删除失败了怎么办？

**解决方案**

主流大厂通常采用 **Cache Aside Pattern（旁路缓存模式）**：
* **读请求**：先查缓存，命中则返回；未命中则查 DB，再将 DB 的值回填到缓存。
* **写请求**：**先更新数据库，再删除缓存**。

#### 1. 为什么是“删除”缓存，而不是“更新”缓存？
* **防止并发脏数据**：并发写场景下，如果更新缓存，可能因网络延迟导致“DB后写入的请求，缓存先更新；DB先写入的请求，缓存后更新”，使缓存沦为旧脏数据且永久不一致。而“删除”则是幂等的，下一次读请求会自动从 DB 加载最新数据回填。
* **避免无用计算与空间浪费**：缓存数据往往是多张表聚合计算的结果，每次 DB 变更都去更新缓存的计算开销极大。采用“删除”属于 Lazy 懒加载，只有数据被访问时才进行计算回填。

#### 2. 为什么选择“先更新数据库，再删除缓存”而不是“先删缓存，再更新数据库”？
* **先删缓存，再更新数据库（不推荐）**：
  在并发下极易产生脏数据。线程 A（写）删除缓存后，在更新 DB 成功前，线程 B（读）进来发现缓存未命中，去 DB 读取了旧数据并写回缓存，随后线程 A 更新 DB。此时 DB 是新值，缓存是旧值，且永久不一致。
* **先更新数据库，再删除缓存（推荐）**：
  只有在“缓存刚好失效，且读请求读取 DB 慢于写请求更新 DB，且读请求回写缓存晚于写请求删除缓存”这一极端场景下才会产生脏数据。由于写 DB 操作（涉及磁盘 IO、行锁）耗时远大于写缓存，读请求几乎必然在写请求删除缓存前完成回写，因此该方案在并发下发生不一致的概率极低。

**关键细节**

- **删除缓存失败的兜底方案**：
  - **MQ 异步重试**：删除失败时，将删除失败的 Key 写入消息队列，由消费者不断重试删除，直到成功。
  - **Canal 订阅 Binlog 异步删除（推荐）**：业务代码只管更新 DB。通过 Canal 伪装成 MySQL 备库解析 binlog，将数据变更推送到 MQ，由专门 the 清理服务消费 MQ 异步删除缓存，实现业务和缓存清理的彻底解耦。
- **防止并发回填的延迟双删**：
  若因历史原因必须采用“先删缓存再写 DB”方案，写操作最后须采用**延迟双删**：即删除缓存 -> 更新 DB -> 模拟休眠几百毫秒（确保其他读线程执行完毕并回填完毕） -> 再次删除缓存。

**面试表达**

> 我们项目使用的是旁路缓存（Cache Aside），采用“先更新数据库，再删除缓存”的策略。之所以删除而不是更新，是为了避免并发写导致的乱序脏数据，并降低冷数据的计算开销。之所以先 DB 后缓存，是因为并发读写下该顺序产生脏数据的概率最低。对于第二步删除失败的问题，我们使用 Canal 订阅 MySQL Binlog 发送到消息队列，通过 MQ 的重试机制来保证最终一致性。

## 9. 缓存穿透、击穿、雪崩怎么处理？

**常见问法**

- 大量请求查不存在的数据怎么办？
- 热点 key 过期瞬间打爆数据库怎么办？
- 大量缓存同时失效怎么办？

**解决方案**

- 缓存穿透：查询不存在的数据。可以用布隆过滤器、空值缓存、参数校验解决。
- 缓存击穿：热点 key 失效后大量请求打到 DB。可以用互斥锁、逻辑过期、热点永不过期解决。
- 缓存雪崩：大量 key 同时失效或 Redis 故障。可以给 TTL 加随机值、热点预热、多级缓存、限流降级。

**关键细节**

- 空值缓存 TTL 要短，避免长期缓存错误结果。
- 布隆过滤器适合判断一定不存在，但存在误判。
- 热点 key 可以提前识别并续期。
- Redis 故障时要有降级策略，不能让所有流量直接冲垮 DB。

**面试表达**

> 穿透、击穿、雪崩分别对应不存在数据、热点 key 失效、大面积缓存失效。处理思路分别是布隆过滤器或空值缓存、互斥锁或逻辑过期、TTL 随机化和限流降级。

## 10. 分布式锁怎么用才安全？

**常见问法**

- Redis 分布式锁怎么实现？`setnx` 加锁有什么问题？
- 锁过期了业务还没执行完怎么办？什么是看门狗机制？它怎么防死锁？
- Redis 分布式锁和 ZooKeeper 分布式锁有什么区别？如何选择？

**解决方案**

分布式锁有以下三种主流实现方式：

#### 1. Redis 自研简单版（单节点）
* **加锁**：必须使用单条原子命令，同时设置互斥与过期时间：
  ```text
  SET lock:key unique_request_id NX PX 30000
  ```
  * `NX` 保证互斥，`PX 30000` 设置 30s 生存时间防死锁，`unique_request_id`（如 UUID）标识持有者。
* **解锁**：必须使用 **Lua 脚本**。先判断锁是否属于自己，再执行删除，防止“线程 A 锁超时释放后，误删线程 B 新加的锁”：
  ```lua
  if redis.call('GET', KEYS[1]) == ARGV[1] then
      return redis.call('DEL', KEYS[1])
  else
      return 0
  end
  ```

#### 2. Redisson 框架版（生产首选）
自研版无法优雅解决“锁重入”、“阻塞等待”和“业务超时锁过期”问题，Redisson 提供了以下核心机制：
* **可重入锁**：底层使用 Redis **Hash** 结构存储锁。Key 为锁名，Field 为 `UUID + 线程ID`，Value 为重入次数。每次加锁 Value+1，释放时 Value-1。
* **阻塞等待**：未获取锁的线程不会使用 `while(true)` 忙等自旋，而是利用 Redis 的 **Pub/Sub（发布订阅）** 订阅锁释放通知，收到通知后再尝试获取锁，节省 CPU。
* **看门狗机制（Watchdog）**：
  * **目的**：防止业务未执行完，锁自动过期失效，导致并发互斥失效。
  * **原理**：当加锁未手动指定超时时间时，Redisson 默认给锁设置 30s 生存时间。同时在后台启动一个看门狗定时任务，每隔 **10 秒**（`lockWatchdogTimeout / 3`）检查持有锁的线程是否存活，若存活则发送 Lua 脚本将锁的 TTL 重新重置为 30s。
  * **防死锁**：若持有锁的服务突然宕机，看门狗随之死亡，锁到达 30s 生存时间后会自动被 Redis 清除，确保其他节点可正常获取锁。

#### 2.1. Spring Boot 项目中 Redisson 如何实操？

**1. 引入依赖**

Spring Boot 项目一般直接引入 Redisson 官方 starter，启动后可以注入 `RedissonClient`：

```xml
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
    <version>${redisson.version}</version>
</dependency>
```

如果项目使用 Spring Boot 3.x，要注意 Redisson 与 Spring Data Redis 版本兼容；老项目升级时尤其要检查 `redisson-spring-data-*` 依赖是否匹配。

**2. 配置 Redis 连接**

简单项目可以直接在 `application.yml` 指定 Redisson 配置文件：

```yaml
spring:
  redis:
    redisson:
      file: classpath:redisson.yaml
```

`redisson.yaml` 示例：

```yaml
singleServerConfig:
  address: "redis://127.0.0.1:6379"
  database: 0
lockWatchdogTimeout: 30000
```

生产环境通常会按实际架构配置单机、哨兵、集群或主从模式。`lockWatchdogTimeout` 默认是 30 秒，只有在未指定 `leaseTime` 的加锁方式下，看门狗才会自动续期。

**3. 业务代码标准写法**

推荐优先使用 `tryLock(waitTime, leaseTime, TimeUnit)`，避免请求一直阻塞：

```java
@Service
public class WorkOrderLockService {

    private final RedissonClient redissonClient;

    public WorkOrderLockService(RedissonClient redissonClient) {
        this.redissonClient = redissonClient;
    }

    public void recalculateAppoint(Long serviceWorkOrderId) {
        String lockKey = "lock:workorder:appoint:" + serviceWorkOrderId;
        RLock lock = redissonClient.getLock(lockKey);
        boolean locked = false;
        try {
            locked = lock.tryLock(3, 10, TimeUnit.SECONDS);
            if (!locked) {
                throw new BusinessException("工单正在处理中，请稍后重试");
            }

            // 临界区：重新读取状态、校验、计算分配结果、回写关键字段。
            doRecalculate(serviceWorkOrderId);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BusinessException("获取分布式锁被中断");
        } finally {
            if (locked && lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

参数含义：

| 参数 | 含义 | 面试表达 |
| --- | --- | --- |
| `waitTime` | 最多等多久拿锁 | 防止请求一直卡住，拿不到锁可以快速失败或返回处理中 |
| `leaseTime` | 拿到锁后多久自动释放 | 适合能预估耗时的短业务，避免服务异常导致死锁 |
| 不传 `leaseTime` 的 `lock()` / `tryLock(waitTime, unit)` | 依赖看门狗续期 | 适合耗时不稳定但必须执行完的短临界区，但仍要避免慢 RPC 和大事务 |

**4. 常见错误写法**

```java
RLock lock = redissonClient.getLock("lock:order:" + orderId);
lock.lock();
try {
    // 慢 RPC、长事务、批量循环都塞进来，风险很高
} finally {
    lock.unlock();
}
```

问题在于：`lock.lock()` 可能让线程长期等待；临界区太长会放大锁竞争；如果锁过期或被其他线程接管后直接 `unlock()`，还可能抛出 `IllegalMonitorStateException`。更稳的写法是 `tryLock` 控制等待时间、缩短临界区，并在 finally 中判断 `isHeldByCurrentThread()` 后再释放。

#### 3. ZooKeeper 分布式锁（高安全首选）
* **原理**：所有请求在锁目录下创建**临时顺序节点**（如 `/lock/000001`）。序号最小的节点成功获取锁；其余节点对自身序号的前一个节点注册 Watcher 监听。前一个节点释放/断开连接被删除时，后一个节点被唤醒。
* **优势**：由于是临时节点，客户端一旦宕机断开 Session，ZK 就会**自动删除节点释放锁**，天然防止死锁，不需要看门狗机制续期，一致性极高。
* **劣势**：性能略低于 Redis，因为节点创建与删除需经过 ZK 的强一致性集群同步（Paxos/ZAB 协议）。

**关键细节**

- **锁失效的安全边界**：即使有看门狗，也无法防范 Redis 节点主备切换时，由于主从异步复制导致的锁丢失（红锁 Redlock 复杂度高且仍有争议，通常不建议强推）。对于涉及资金安全的强一致场景，应由底层数据库唯一键/版本号乐观锁做终极拦截。
- **避免长事务持锁**：分布式锁只用于防并发保护，绝对不要把耗时的大事务、第三方 RPC 调用放在分布式锁内，锁的粒度应尽可能细。
- **未持锁误释放**：在调用 `unlock()` 时必须进行异常捕获，确保即使锁已被意外释放或过期，也不会因为 `IllegalMonitorStateException` 导致业务主线程中断。

**面试表达**

> 我们在生产中首选 Redisson 实现分布式锁，其底层使用 Hash 结构实现锁重入，并基于发布订阅避免无谓自旋。针对业务超时锁过期问题，Redisson 的看门狗机制每 10 秒会检查一次线程存活并自动续期。如果需要更高安全级别的锁，我们会选用 ZooKeeper，利用其临时顺序节点与 Watcher 监听机制，在客户端宕机时由 ZK 自动删节点释放锁，天生防死锁，安全级别最高但性能稍低。

### 10.1. 结合项目说说分布式锁有哪些使用场景？

**常见问法**

- 你项目里哪些地方用过或适合用分布式锁？
- 分布式锁在工单、库存、人员供给场景里解决什么问题？
- 分布式锁和幂等、状态机、数据库行锁是什么关系？

**业务场景**

分布式锁不要只讲成“防止并发”，更准确的说法是：同一个业务资源可能被多个服务实例、MQ 消费、XXL-Job、人工入口同时修改时，用锁把关键临界区串行化，避免同一资源被重复占用、重复流转或重复补偿。

| 项目 | 资源维度 | 典型并发入口 | 锁要解决的问题 |
| --- | --- | --- | --- |
| `singularity` | `reservationTokenId`、`servicePartId + timeSlot`、`groupId + date` | 下单预占、空预占重试、释放回退、调度抢人、人工重预占 | 防止同一服务人员同一时间段被多个工单重复预占，避免 token 状态从 `EMPTY`、`RESERVED`、`USED` 之间被并发改乱 |
| `mp-workorder` | `serviceWorkOrderId` | 工单首次分配、每日重算、黑名单触发、手动重算、修改时间/地址 | 防止同一个工单并发重算站点和库存组，导致 `site_id`、`inventory_group_id`、预约 token 结果不一致 |
| `mp-staff` | `moveId`、`staffId + moveType` | 工作流回调、人工重试、XXL-Job 补偿、MQ 后置事件 | 防止同一笔人员异动重复完成，避免重复更新服务时间、职业技能、停单停售状态或重复发送异动完成事件 |
| 补偿任务 | `bizType + bizId`、`cityCode + date` | 多实例定时任务、失败重试、人工补偿 | 防止多个任务实例重复处理同一批数据，同时允许不同城市、不同日期分片并行 |

**典型方案**

1. **库存预约锁资源**：在 `singularity` 里，锁粒度可以围绕 token 或具体服务时间段，临界区只覆盖“重新读取状态 -> 校验是否仍可预占 -> 占用资源 -> 写 token 状态和日志”。锁外再通过状态机、数据库行锁、幂等 key 兜底。
2. **工单分配锁工单**：在 `mp-workorder` 里，对同一个 `serviceWorkOrderId` 做短锁，保护分配规则计算和关键字段回写，避免两个入口同时更新站点、库存组和预约结果。
3. **人员异动锁业务单**：在 `mp-staff` 里，对同一个 `moveId` 或员工异动维度加锁，避免工作流回调和补偿任务重复执行异动完成逻辑；后续 MQ、重试任务仍要按业务单号幂等。
4. **任务补偿锁分片**：对 XXL-Job 或补偿任务，不建议全局一把大锁，可以按城市、日期、业务类型分片加锁，提高并发度。

**关键细节**

- 锁粒度要贴业务资源：订单锁订单，工单锁工单，库存锁 token 或时间段，人员异动锁异动单。
- 锁只包短临界区，不要把长事务、慢 RPC、批量循环全部放进锁里。
- 分布式锁不是最终防线，还要有状态机、唯一索引、幂等记录、数据库行锁、MQ 消费幂等和补偿对账。
- 锁失败时要有明确语义：直接返回“处理中”、稍后重试，或者进入补偿任务，不能静默吞掉。

**面试表达**

> 我在项目里理解分布式锁主要不是为了炫技，而是解决“同一个业务资源被多入口并发修改”的问题。比如 `singularity` 里库存 token 预占，要防止同一服务人员同一时间段被重复占用；`mp-workorder` 里工单重算、改约、改地址，要防止同一个工单的站点、库存组、预约 token 被并发写乱；`mp-staff` 里人员异动完成，要防止工作流回调、人工重试、补偿任务重复完成同一笔异动。但我不会只靠锁，锁一般只包住短临界区，底层还要配合唯一业务键、状态机、幂等注解、数据库行锁、MQ 消费幂等和补偿任务。这样即使锁超时、任务重试或 MQ 重复投递，数据最终也能收敛。

## 11. 订单列表分页与深分页怎么优化？

**常见问法**

- 千万级订单表如何优化分页查询？
- C端买家和B端商家的订单列表分页分别怎么设计？
- 为什么游标分页可以防止高并发下的数据错乱（重复或遗漏）？
- 分库分表之后，订单分页查询怎么做？

**解决方案**

优化订单列表分页需从“业务分流”和“技术选型”双向出发：

#### 1. 区分 C 端与 B 端查询痛点
* **C 端用户（买家）**：只查自己的订单，默认展示近 3 个月数据（小表体量），极少进行深翻页。
  - **SQL 延迟关联**：利用覆盖索引避免深翻页的高回表开销。
  - **游标分页（滚动加载）**：前端抛弃传统页码，采用临界值 `id < lastOrderId` 滚动加载。
  - **冷热数据分离**：将历史订单归档至冷表，C端默认只查热表，查冷表时限制检索深度。
* **B 端商家/管理员**：大体量数据且伴随多维条件过滤（时间、手机号、状态等），需要深分页及导出。
  - **Elasticsearch 读写分离**：利用 Canal 实时同步订单 binlog 到 ES，B 端的复杂分页检索全部走 ES，彻底解放 MySQL。

#### 2. 游标分页防数据错乱原理
* **传统 `LIMIT offset, size`**：在并发写/删时极易由于数据位置平移产生脏数据。例如：用户浏览第一页（`LIMIT 0, 2`）时新增了一条订单，第二页（`LIMIT 2, 2`）就会把原本在第一页尾部的数据重复读取一次。
* **游标分页（锚点定位）**：通过限定 `WHERE id < #{lastId} LIMIT 2`，以物理 ID 作为绝对闸门将已读取区间锁在门外，无论在此期间有多少写/删事件，均不会产生数据重叠或遗漏。

#### 3. 分库分表下的分页解法
* **买家端**：按 `user_id` 分片，分页请求携带 `user_id` 可以直接路由到单库单表，性能不受影响。
* **商家端/后台**：没有 `user_id` 无法进行分片路由，若直接向所有库广播查询再到内存合并，随着页码越深（如第 100 页），内存和 CPU 开销将成倍放大。
  - **解法一（推荐）**：使用 Canal 将分库分表的数据实时汇聚并同步到 **Elasticsearch**，商家列表搜索和分页完全走 ES。
  - **解法二（商家冗余表）**：设计一张单独以 `merchant_id` 为分片键的商家订单表（通过双写或 Canal 冗余），商家端分页请求路由到对应商家的分片上单表分页。

**关键细节**

- **延迟关联核心 SQL**：
  ```sql
  SELECT o.* FROM orders o
  JOIN (
      SELECT id FROM orders 
      WHERE user_id = #{userId} 
      ORDER BY create_time DESC 
      LIMIT 100000, 20
  ) temp ON o.id = temp.id;
  ```
  先走子查询利用索引覆盖（`(user_id, create_time, id)`）找到目标主键 ID，再回表拉取 20 条整行数据，有效缩短回表范围。
- **冷热分离策略**：通常通过定时任务或 Canal 把超过 3 个月已完成的订单移动到冷库归档，保证热库表体积处于较小级别。
- **后台导出保护**：禁止后台系统同步发起深分页大查询导出，必须走异步离线导出任务，由后台线程池低频率分批拉取写入 Excel 供用户下载。

**面试表达**

> 订单列表分页要分 C 端和 B 端分别优化。C 端由于是买家单维度查询且数据量小，我们使用延迟关联与游标分页防重复读取，并结合冷热分离保证热表体量；B 端多维查询和深分页则通过 Canal 异步同步到 Elasticsearch 进行读写分离。如果系统做了分库分表，商家端查询由于无法路由，我们也是利用同步至 Elasticsearch 的方案，直接在 ES 中进行高并发复杂分页，避免向 MySQL 所有分片进行无脑广播和内存合并排序。

## 12. 分库分表后如何处理唯一 ID 和查询问题？

**常见问法**

- 订单表太大了怎么分库分表？
- 分库分表后如何生成全局唯一订单号？
- 分片后跨用户查询怎么办？

**解决方案**

订单类数据常按 `user_id` 或 `order_id` 分片：

- 按 `user_id` 分片：用户维度查询方便，但按订单号查询需要路由表或订单号带分片信息。
- 按 `order_id` 分片：订单号查询方便，但用户订单列表可能需要额外索引表。

全局 ID 可以用：

- 雪花算法。
- 数据库号段模式。
- Redis 自增加业务前缀。

查询问题可以通过：

- 分片键路由。
- 冗余查询索引表。
- 搜索引擎承接复杂查询。
- 数据汇总到数仓或报表库。

**关键细节**

- 分库分表前要先确认单表瓶颈，不要过早拆分。
- 分片键一旦选错，后续迁移成本很高。
- 跨分片事务和跨分片查询要尽量避免。
- 订单号最好携带时间、业务类型或分片信息，方便排查和路由。

**面试表达**

> 分库分表核心是选好分片键。订单系统通常围绕用户和订单两个查询维度设计，主链路按分片键路由，复杂查询通过索引表或搜索引擎解决，全局 ID 用雪花算法或号段模式。

## 13. 优惠券领取和使用如何防止超发、重复使用？

**常见问法**

- 优惠券库存怎么扣？
- 用户重复领券怎么办？
- 下单失败后优惠券怎么退？

**解决方案**

领券阶段：

- 活动券库存用 Redis + Lua 预扣。
- 用户领券记录用 `user_id + coupon_template_id` 唯一索引防重复。
- 领券成功后异步落库，失败则回补库存。

用券阶段：

- 下单锁定优惠券，将券状态从未使用改成锁定。
- 支付成功后改成已使用。
- 订单取消或超时未支付后解锁或退回优惠券。

**关键细节**

- 券库存和券实例状态都要有流水。
- 优惠券状态流转要单向，避免已使用券被退回。
- 优惠计算要防篡改，不能完全相信前端传入金额。
- 高并发领券要限制单用户、单设备、单 IP 频率。

**面试表达**

> 优惠券要分领券库存和券实例状态两部分。领券用 Redis 原子扣库存和唯一索引防重复，用券时通过状态机从未使用到锁定、已使用或退回，并通过流水保证可追溯。

## 14. 订单状态机怎么设计？

**常见问法**

- 订单状态很多，如何避免状态乱跳？
- 支付、取消、发货、退款并发发生怎么办？

**解决方案**

订单状态机要定义清楚：

- 状态：待支付、已支付、已取消、待发货、已发货、已完成、退款中、已退款。
- 事件：支付成功、用户取消、超时关闭、发货、确认收货、申请退款。
- 转移规则：只有合法事件才能推动合法状态变化。

实现上可以：

- 用状态机框架。
- 或用数据库条件更新控制状态：

```sql
UPDATE orders
SET status = 'PAID'
WHERE order_id = #{orderId}
  AND status = 'WAIT_PAY';
```

**关键细节**

- 状态变更要记录操作日志和事件流水。
- 并发状态变更通过条件更新保证只有一个成功。
- 复杂业务可以拆主状态和子状态，避免一个状态字段承载过多语义。
- 状态事件要能驱动 MQ，下游按事件处理。

**面试表达**

> 订单状态不能靠 if else 随意改，要用状态机定义状态、事件和合法流转。落库时用条件更新保证并发安全，并记录状态流水，后续通过事件驱动下游系统。

## 15. 如何设计异步任务和补偿机制？

**常见问法**

- 下游接口偶发失败怎么办？
- 定时任务重复执行会不会有问题？
- 线上异常数据如何修复？

**解决方案**

异步任务要有可重试、可幂等、可观测能力：

- 任务表记录业务主键、任务类型、状态、重试次数、下次执行时间、失败原因。
- 执行前加状态判断或分布式锁，避免多实例重复执行。
- 执行成功后标记成功。
- 执行失败后按退避策略重试。
- 超过最大次数进入失败表或人工处理队列。

**关键细节**

- 补偿任务必须幂等，重复执行不能产生副作用。
- 任务要按业务主键分片，避免全表扫描。
- 失败原因要记录清楚，方便线上排查。
- 对账任务要区分“发现异常”和“自动修复”，高风险修复要人工确认。

**面试表达**

> 我会把补偿任务设计成有状态的任务，而不是简单定时扫表。任务需要记录执行状态、重试次数和失败原因，执行逻辑必须幂等，并配合告警和人工修复入口。

## 16. 文件上传和大文件处理怎么设计？

**常见问法**

- 大文件上传失败后怎么断点续传？
- 用户上传图片、视频后如何处理？

**解决方案**

常见设计：

- 小文件可以直接上传到应用服务，再转存对象存储。
- 大文件使用分片上传，每个分片带文件 MD5、分片序号和上传任务 ID。
- 服务端记录每个分片上传状态。
- 所有分片上传完成后合并文件，校验整体 MD5。
- 图片、视频处理通过异步任务完成，例如压缩、转码、鉴黄、生成缩略图。

**关键细节**

- 文件元数据和对象存储 key 要落库。
- 上传接口要限制大小、类型、频率。
- 文件访问最好用临时签名 URL，避免公开泄露。
- 异步处理失败要能重试。

**面试表达**

> 大文件上传我会采用分片上传和断点续传，服务端记录分片状态，合并后校验 MD5。后续转码、压缩、审核等耗时动作异步化，文件本体放对象存储，数据库只保存元数据。

## 17. 用户登录和 Token 如何设计？

**常见问法**

- 单点登录怎么做？
- JWT 和 Session 有什么区别？
- Token 失效和踢人怎么实现？

**解决方案**

常见方案：

- 单体或小系统可以用 Session + Redis，服务端集中存登录态。
- 前后端分离或开放 API 可以用 JWT，但要注意失效控制。
- 需要强控制登录态时，可以用 access token + refresh token，并在 Redis 维护登录状态。

**关键细节**

- JWT 无状态，但签发后在过期前不容易主动失效。
- 如果要支持踢人、改密下线、设备管理，仍需要服务端保存 token 版本或黑名单。
- 登录、刷新、退出都要记录设备和 IP 信息。
- 敏感操作需要二次校验或短期授权。

**面试表达**

> 如果系统需要强登录态控制，我不会只用纯 JWT，而会结合 Redis 保存 token 状态、设备信息和版本号。这样既能支持分布式鉴权，也能实现主动失效、踢人和改密下线。

## 18. 如何做限流、降级和熔断？

**常见问法**

- 高峰流量来了怎么保护系统？
- 下游服务变慢，如何避免拖垮自己？

**解决方案**

限流可以分层做：

- 网关限流：按 IP、用户、接口、租户限流。
- 应用限流：令牌桶、漏桶、滑动窗口。
- 资源限流：线程池隔离、连接池限制、队列长度限制。

降级和熔断：

- 下游超时或错误率升高时熔断，快速失败。
- 非核心能力返回默认值或降级结果。
- 核心链路优先保障，例如下单优先于推荐、营销、统计。

**关键细节**

- 限流阈值要结合压测数据，不是拍脑袋。
- 熔断后要有半开探测，不能永久不可用。
- 降级结果要符合业务预期，不能返回误导性成功。
- 线程池要按业务隔离，避免非核心任务耗尽核心资源。

**面试表达**

> 我会从网关、应用和资源三个层面限流，再通过熔断和降级保护核心链路。核心原则是保主流程、控非核心、快速失败，并通过监控和压测数据设定阈值。

## 19. 如何设计一个排行榜或计数系统？

**常见问法**

- 点赞数、阅读数、排行榜怎么实现？
- 计数实时更新会不会打爆数据库？

**解决方案**

常见方案：

- 实时计数写 Redis，例如 `INCR`。
- 排行榜用 Redis ZSet，分数为热度值、销量、积分等。
- 定时把 Redis 聚合结果刷回 DB。
- 对高频计数可以先写本地队列或 MQ，再批量聚合。

**关键细节**

- 计数类数据通常允许短暂不一致。
- 热点计数要避免每次都写 DB。
- 排行榜要区分实时榜、日榜、周榜、总榜。
- 刷库任务要幂等，避免重复累计。

**面试表达**

> 计数和排行榜我会优先放 Redis，利用 `INCR` 和 ZSet 支撑高频更新，再通过定时任务或 MQ 批量落库。业务上接受短暂不一致，换取更高吞吐。

## 20. 如何排查线上慢接口？

**常见问法**

- 一个接口突然变慢，你怎么定位？
- P99 升高可能是什么原因？

**解决方案**

排查顺序：

1. 看监控：QPS、RT、错误率、CPU、内存、GC、线程池、连接池。
2. 看链路追踪：定位慢在网关、应用、DB、Redis、MQ 还是下游 RPC。
3. 看日志：按 traceId 找关键耗时点。
4. 看数据库：慢 SQL、执行计划、锁等待、索引命中。
5. 看缓存：命中率、热点 key、大 key、Redis 延迟。
6. 看发布和配置变更：是否刚上线或改了开关。

**关键细节**

- 不要只盯代码，先确定慢在哪一段。
- P99 升高通常和少量慢请求、锁竞争、下游超时、线程池耗尽有关。
- 修复后要补监控、日志或告警，避免下次重复排查困难。

**面试表达**

> 我会先用监控和 traceId 定位耗时边界，再深入到慢 SQL、缓存、RPC、线程池和锁等待。排查不是凭感觉改代码，而是先确认瓶颈在哪一层。

## 21. Spring Bean 的生命周期与高频面试题

**常见问法**

- 请详细描述 Spring Bean 的生命周期。
- Spring 中 AOP 代理是在哪一步生成的？
- 什么是三级缓存？它是如何解决循环依赖的？
- BeanFactory 和 FactoryBean 有什么区别？

**解决方案**

### 21.1. Bean 生命周期核心流程

Spring 中的 Bean 生命周期可以归纳为以下四大核心阶段及八个具体步骤：

1. **实例化 (Instantiation)**：Spring 容器根据 `BeanDefinition`，通过反射调用构造方法创建 Bean 的原始实例（此时属性皆为默认值）。
2. **属性填充 (Populate Properties)**：Spring 对 Bean 进行依赖注入（解析 `@Autowired`、`@Value` 等注解），将依赖的其它 Bean 或配置值填充到属性中。
3. **Aware 接口回调**：若 Bean 实现了 Aware 系列接口，Spring 会将对应容器级对象注入 Bean 中，如：
   - `BeanNameAware` -> 注入 Bean 名称。
   - `BeanFactoryAware` -> 注入 `BeanFactory` 容器本身。
   - `ApplicationContextAware` -> 注入 `ApplicationContext` 上下文。
4. **BeanPostProcessor 前置处理**：调用所有已注册 `BeanPostProcessor` 的 `postProcessBeforeInitialization` 方法。
   - **注**：`@PostConstruct` 注解的方法就是在此步骤由 `CommonAnnotationBeanPostProcessor` 拦截并调用的。
5. **初始化 (Initialization)**：执行 Bean 自身的初始化逻辑，顺序为：
   - 实现了 `InitializingBean` 接口，调用 `afterPropertiesSet()`。
   - 配置了自定义初始化方法，调用 `init-method`（如 `@Bean(initMethod = "...")`）。
6. **BeanPostProcessor 后置处理**：调用所有已注册 `BeanPostProcessor` 的 `postProcessAfterInitialization` 方法。
   - **注**：**AOP 代理对象在此处生成**（由 `AbstractAutoProxyCreator` 拦截并返回代理对象）。
7. **就绪使用 (Bean Ready)**：完全初始化好的单例 Bean 被放入 Spring 容器的单例池（`singletonObjects`）中，供业务代码使用。
8. **销毁 (Destruction)**：当容器关闭时，执行销毁逻辑，顺序为：
   - 调用 `@PreDestroy` 标注的方法。
   - 调用 `DisposableBean#destroy()`。
   - 调用配置的自定义销毁方法 `destroy-method`。

---

### 21.2. 高频面试追问与深度解析

#### 1. AOP 代理对象是在生命周期的哪一步创建的？
* **常规情况**：在初始化后置处理阶段（第六步），即 `BeanPostProcessor#postProcessAfterInitialization` 中。通过 `AbstractAutoProxyCreator` 扫描 Bean 匹配的切面，通过 JDK 动态代理或 CGLIB 代理技术生成代理对象并返回。
* **循环依赖情况**：如果该 Bean 参与了循环依赖，那么它的 AOP 代理会被**提前创建**。在属性填充（第二步）发生循环依赖时，通过三级缓存中的 `ObjectFactory#getObject()` 提前触发 `getEarlyBeanReference` 方法生成代理对象，以确保注入给对方的是代理对象而非原始对象。

#### 2. Spring 如何通过三级缓存解决循环依赖？
Spring 内部通过三级缓存（均为 Map 结构）解决了单例 Bean 之间的属性/Setter 循环依赖（构造器循环依赖无法解决）：
* **一级缓存 (`singletonObjects`)**：存放完全初始化好的单例 Bean（成品）。
* **二级缓存 (`earlySingletonObjects`)**：存放提前暴露的原始 Bean 或提前创建的代理对象（半成品）。其主要作用是保证在多重循环依赖时，拿到的都是同一个半成品 Bean，避免重复创建。
* **三级缓存 (`singletonFactories`)**：存放 `ObjectFactory`（生成 Bean 的工厂）。当 Bean 实例化后，会将其包装为 ObjectFactory 存入三级缓存。
* **解决流程（以 A 依赖 B，B 依赖 A 为例）**：
  1. 实例化 A，将 A 的 `ObjectFactory` 放入三级缓存。
  2. 填充 A 的属性时，发现需要 B，去容器中查找 B。
  3. B 不存在，开始创建 B：实例化 B，将 B 的 `ObjectFactory` 放入三级缓存。
  4. 填充 B 的属性，发现需要 A，去容器中查找 A。
  5. 依次查找一级、二级缓存皆未命中，在三级缓存命中 A 的 `ObjectFactory`。通过 `ObjectFactory.getObject()`（内部触发 AOP 提前创建）获取 A 的早期引用（若有 AOP 则是代理后的 A），将其放入二级缓存，并清除三级缓存中的 A。
  6. B 完成属性填充，执行初始化，最终成为成品放入一级缓存。
  7. 返回到 A 的属性填充，此时 B 已经是成品，A 成功获取到 B。A 继续完成初始化，最后将 A 的代理/成品放入一级缓存。

#### 3. BeanFactory 和 FactoryBean 有什么区别？
* **`BeanFactory`**：是 Spring 框架的顶层核心容器接口，负责定义 Bean 的注册、获取、生命周期维护及依赖注入。它是“管理 Bean 的工厂”。
* **`FactoryBean`**：是一个特殊的 Bean 接口。实现了该接口的类本身也是容器中的一个 Bean，但它的主要作用是**自定义复杂 Bean 的创建逻辑**。
  - 当通过 `context.getBean("myBean")` 获取时，返回的是 `FactoryBean#getObject()` 方法返回的实际对象，而非 FactoryBean 实例本身。
  - 如果想要获取 FactoryBean 实例本身，需要在 beanName 前加 `&` 前缀，即 `context.getBean("&myBean")`。
  - 常用于第三方框架的集成（如 MyBatis 的 `SqlSessionFactoryBean`）。

#### 4. Spring 本地事务是如何与 Bean 生命周期和 AOP 结合的？
Spring 事务（`@Transactional`）本质上也是通过 AOP 动态代理实现的。
在 Bean 生命周期的后置处理阶段，`InfrastructureAdvisorAutoProxyCreator`（一个特殊的 `BeanPostProcessor`）会扫描带有 `@Transactional` 注解的方法或类，为其织入事务增强逻辑（在方法执行前开启 Connection 事务，执行后提交或回滚）。
因此，事务的生效极度依赖 AOP 代理对象。如果在类内部方法直接调用 `this.method()`，由于绕过了代理对象，直接调用了原始对象，事务必然失效。

## 22. 百万级数据如何实现高效导入与导出？

> 本题是业务场景题，重点讲流式处理、异步化和资源隔离。若追问“已经 OOM 了怎么查”，跳到 [[Java JVM高频面试题与线上排障指南#6. OOM 类型与排查|JVM 主文档：OOM 排查链路]]。

**常见问法**

- 百万级数据的 Excel 导入，怎么设计才能不 OOM？
- 大数据量导出，接口超时、内存溢出怎么处理？

**解决方案**

- **高效导入（读操作）**：
  - **流式解析**：严禁使用 `Apache POI` 的一次性全量加载（DOM 模式），应使用 `Alibaba EasyExcel` 或 `POI SAX` 模式。它们是基于流式事件驱动（Event-Driven）解析，内存占用极小。
  - **分批处理与并发校验**：在 `ReadListener` 中缓存数据（如每 1000 条为一个 Batch），使用线程池（线程安全的 Executor）并发进行数据校验（如正则校验、DB 查重）。
  - **批量插入**：校验通过的数据利用 MyBatis Batch 机制或 `SqlSession` 的批量执行模式，或单次拼接 `INSERT INTO ... VALUES (x), (y)...` 批量落库，单次插入 1000~2000 条，减少网络交互次数。
- **高效导出（写操作）**：
  - **流式/游标查询**：避免一次性将百万级数据加载到 JVM 内存。使用 MyBatis 的 `Cursor`（游标）或 JDBC `fetchSize` 机制分批从数据库拉取数据。
  - **分批写入文件**：使用 EasyExcel 的流式写入（`ExcelWriter`），分批将数据写入磁盘，当单 Sheet 超过 104 万行限制时，自动切换到下一个 Sheet。
  - **异步离线导出**：用户点击导出后，后台立即生成一个“导出任务记录”并返回“导出中”状态。通过 MQ 异步分发，由专门的导出服务从数据库拉取数据、生成 Excel 并压缩为 `.zip`，上传至对象存储（OSS），最后通过站内信/邮件通知用户下载。

**关键细节**

- **内存防线**：Excel 解析时限制最大行数（如防上传 1000 万行的恶意文件打爆磁盘），导入失败时返回带有错误标记行的“失败明细 Excel”给用户下载。
- **防止导出打爆连接池**：游标查询期间，数据库连接（Connection）会被一直占用。需要为导出业务分配独立的低优先级数据库连接池，避免阻塞正常交易流量。

**面试表达**

> 百万级导入我会使用 EasyExcel 走事件驱动流式解析防 OOM，分批校验后使用 MyBatis 批量落库。导出则绝对不走同步接口，而是采用 MQ 异步离线导出任务，通过 MyBatis Cursor 游标流式查库回写磁盘，压缩后上传 OSS 并通知用户下载，从而实现数据库与内存的双向保护。

## 23. 分布式 ID 生成器怎么设计？如何解决时钟回拨？

> 本题中的“号段缓存在 JVM 内存”是业务性能优化点，不展开 JVM 原理。JVM 内存区域和对象分配统一看 [[Java JVM高频面试题与线上排障指南#1. JVM 运行时内存区域|JVM 主文档：运行时内存区域]]。

**常见问法**

- 订单号、分布式 ID 怎么生成？
- 雪花算法的结构是怎样的？如何解决时钟回拨问题？
- 数据库号段模式（如 Leaf/Tinyid）的原理是什么？

**解决方案**

- **雪花算法（Snowflake）**：
  - **结构**：64 位 Long 型 ID，通常拆成 `1 位符号位（不用） -> 41 位时间戳（可支撑约 69 年） -> 10 位工作机器 ID（5 位数据中心 + 5 位机器） -> 12 位序列号（每毫秒最多 4096 个）`。
  - **时钟回拨解法**：如果当前时间小于上一次生成 ID 的时间，说明发生了时钟回拨。
    - **小幅度回拨（如 < 5ms）**：线程主动挂起等待 `sleep(offset)` 直至时间追回。
    - **历史最大时间戳记录**：在内存中维护上次生成 ID 的最大时间戳。如果回拨幅度小，可以用历史最大时间戳继续自增（借用时间戳），直到系统时间追上。
    - **备份 WorkerId**：如果回拨幅度大（如机器同步 NTP 发生几分钟回拨），可以自动向 ZK/Redis 申请一个新的临时 `workerId`，用新 WorkerId 配合回拨后的时间继续生成，防止冲突。
- **号段模式（Segment Pattern - 推荐）**：
  - **原理**：不直接每次从 DB 拿自增 ID，而是每次向 DB 申请一个号段（如 `[1000, 2000]`）缓存在 JVM 内存中。本地线程通过 `AtomicLong` 原子自增分配，用完后再去 DB 申请下一个号段。
  - **双号段缓存优化（极重要）**：为了防止当前号段用完、去 DB 申请新号段期间的瞬时线程阻塞，内存中缓存两个号段。当第一个号段消费到 10%~20% 时，后台线程**异步**去 DB 申请下一个号段并装载好，实现无缝切换。

**关键细节**

- **信息安全**：纯粹递增的 ID 会暴露商业机密（如日订单量），所以订单号通常是“业务前缀 + 分布式ID + 用户ID后四位（分表路由键） + 混淆随机数”。

**面试表达**

> 我们一般采用雪花算法或 Leaf 号段模式。针对雪花算法的时钟回拨，短时间回拨通过 sleep 等待或利用历史最大时间戳递增解决，长时间回拨则通过动态分配备份 workerId 处理。如果并发度极高，推荐使用带有异步“双号段缓存”的号段模式，避免申请号段时阻塞正常业务线程。

## 24. 敏感数据（手机号/身份证）加密后如何实现模糊检索？

**常见问法**

- 用户的手机号、身份证在数据库中是加密存储的，怎么实现模糊查询（如输入手机后四位查找订单）？
- 密文检索怎么设计？

**解决方案**

- **精确查询（等值查询）**：
  - 对明文数据进行哈希计算（如 `SHA256(phone)`），在数据库中额外存一列“哈希指纹列”（加上唯一索引）。查询时直接对输入值计算 SHA256 去匹配指纹列。
- **更简单的模糊查询方案（推荐先选这个）**：
  - 先和产品约定查询能力，不支持任意 `like '%xxx%'`，只支持手机号后 4 位、后 6 位、身份证后 4 位这类固定规则。
  - 入库时除密文字段外，额外存几列检索指纹，例如 `phone_hash`、`phone_last4_hash`、`id_card_last4_hash`。
  - 查询时把用户输入的后 4 位做同样的 HMAC / 加盐哈希，再匹配对应指纹列，查到候选记录后在应用层解密做二次校验。
  - 如果后台还有时间、城市、订单状态等过滤条件，可以先用这些普通条件缩小范围，再解密少量候选数据做内存匹配。
- **复杂模糊查询兜底**：
  - 如果业务真的要求任意片段搜索，再考虑 N-Gram 切片索引或 Elasticsearch 影子索引。
  - 这类方案实现和安全治理成本更高，要配权限、审计、脱敏展示、访问日志和数据同步校验，不建议一开始就上。

**关键细节**

- **加密算法选择**：明文加密必须使用对称加密（如 `AES-256` 算法，配合 KMS 密钥管理系统定期轮转密钥），不可使用不可逆哈希。
- **检索指纹也算敏感数据**：手机号后 4 位空间太小，不能直接裸 `SHA256(last4)`，更推荐用服务端密钥做 `HMAC` 或加盐哈希，并控制后台查询权限。
- **不要承诺完全等价于明文模糊查询**：加密字段天然不适合随意 `LIKE`，更合理的是收敛查询规则，用固定后缀检索覆盖大多数客服和后台场景。

**面试表达**

> 对加密的敏感字段，我不会直接说还能像明文一样随便 `LIKE`。更简单可控的方案是限制查询规则，比如后台只支持手机号后四位、身份证后四位检索。入库时保存密文，同时额外保存 `phone_hash`、`phone_last4_hash` 这类 HMAC 指纹列；查询时对输入做同样计算，先命中候选记录，再在应用层解密二次确认。如果业务确实要求任意片段模糊搜索，再考虑 N-Gram 切片索引或隔离的 ES 影子索引，但那属于复杂兜底方案，要配权限、审计和脱敏治理。

## 25. 社交系统中的“关注与粉丝”及“Feed 流”怎么设计？

**常见问法**

- 微博/抖音的关注和粉丝功能怎么设计？高并发下怎么知道两个人是不是共同关注？
- Feed 流（朋友圈/主页信息流）怎么设计？推模式、拉模式有什么区别？

**解决方案**

- **关注与粉丝存储**：
  - **Redis 实现**：使用 **Set** 或 **ZSet** 数据结构。
    - 用户 A 的关注列表：`following:A` -> `[B, C, D]`
    - 用户 B 的粉丝列表：`followers:B` -> `[A, E]`
    - **共同关注（求交集）**：`SINTER following:A following:B`
    - **我的关注中谁也关注了对方**：`SISMEMBER following:X following:Y`
  - **DB 备份**：数据库设计一张 `follow_relation` 关系表，建 `(user_id, follow_user_id)` 联合唯一索引。
- **Feed 流架构设计（推拉选择）**：
  - **推模式（Push / 写扩散）**：
    - **原理**：大V发了帖子，系统把帖子 ID 推送到该大V**所有粉丝的收件箱（Inbox）**。
    - **适用**：普通用户（粉丝少），写扩散成本低。
    - **缺点**：大V粉丝上千万，发一个贴需要写几千万行，引发瞬间数据库/缓存暴涨（写放大）。
  - **拉模式（Pull / 读扩散）**：
    - **原理**：大V发贴只写入自己的发件箱（Outbox）。粉丝打开页面时，系统去查询**自己关注的所有人的发件箱**，并在内存中进行合并和按时间排序。
    - **适用**：超级大V。
    - **缺点**：如果用户关注了 2000 个活跃用户，每次看页面都要去 2000 个发件箱拉取并合并排序（读放大）。
  - **推拉结合模式（业界标配）**：
    - 对粉丝进行分类：**在线活跃用户** 和 **离线僵尸用户**。
    - 大V发布帖子时：只主动推送到**当前在线的活跃粉丝**收件箱；对于离线僵尸用户，等他们上线时再使用**拉模式**异步同步。
    - 针对超大V（如上千万粉丝），强制使用**拉模式**；普通用户（如几百粉丝），使用**推模式**。

**关键细节**

- **Redis 限制**：大V的粉丝列表可能是一个大 Key，需要对粉丝 ZSet 进行分片存储（分桶），或者限制最多展示前 1000 个粉丝，其余走 DB 查。

**面试表达**

> 关注关系我们使用 Redis 的 ZSet 结合 MySQL 关系表实现，利用 `SINTER` 交集指令秒级计算共同关注。对于 Feed 流，为了防范大V发布导致的“写扩散灾难”和高关注度用户读取时的“读扩散延迟”，我们采用“推拉结合”方案，只向在线活跃粉丝推送，僵尸用户上线时再拉取，大V发布强制拉取，从而平衡读写性能。

## 高频追问速记

| 追问 | 回答重点 |
| --- | --- |
| Redis 扣库存成功但订单创建失败怎么办？ | 订单失败要回补库存，最好有库存流水和补偿任务。 |
| MQ 重复消费怎么办？ | 消费端按业务唯一键幂等，状态机防重复流转。 |
| 延迟消息一定准时吗？ | 不一定，所以只能作为触发器，到期后还要查订单状态。 |
| 为什么不用分布式事务强一致？ | 复杂度高、可用性差，多数互联网业务可接受最终一致。 |
| 缓存为什么删除而不是更新？ | 缓存可能是聚合结果，删除更简单可靠，由读请求回填。 |
| 分布式锁过期怎么办？ | 控制锁粒度和业务耗时，必要时用看门狗续期，但更要避免长时间持锁。 |
| Redisson `tryLock` 参数怎么理解？ | `waitTime` 控制最多等多久，`leaseTime` 控制自动释放时间；不指定 `leaseTime` 时才依赖看门狗续期。 |
| 项目里分布式锁用在哪些场景？ | `singularity` 锁库存 token/时间段，`mp-workorder` 锁同一工单重算，`mp-staff` 锁同一笔异动完成；最终还要靠幂等、状态机和补偿兜底。 |
| 如何防止状态乱跳？ | 状态机 + 条件更新 + 状态流水。 |
| 怎么证明补偿任务可靠？ | 有任务状态、重试次数、失败原因、幂等逻辑和人工处理入口。 |
| AOP 代理在哪一步生成？ | 通常在初始化后置处理中生成；若有循环依赖，则在三级缓存获取时提前生成。 |
| Spring 怎么解决循环依赖？ | 三级缓存机制。三级缓存存工厂，二级缓存存半成品（防重复触发并保单例），一级缓存存完全体。 |
| BeanFactory 与 FactoryBean 的区别？ | BeanFactory 是容器和工厂；FactoryBean 是特殊 Bean，用于自定义复杂 Bean 的创建过程。 |
| 大文件导入导出如何防止 OOM？ | 使用 EasyExcel 流式事件解析导入，并使用异步离线导出任务拉数据上传至 OSS。 |
| 雪花算法如何解决时钟回拨？ | 小幅度回拨 sleep 等待或利用历史最大时间戳，大幅度回拨则动态向 ZK/Redis 申请临时 workerId。 |
| 密文数据怎么做模糊搜索？ | 简单方案是限制为后四位/后六位检索，存 HMAC 指纹列命中候选后再解密校验；任意模糊再考虑 N-Gram 或 ES。 |
| Feed 流如何应对大V发布？ | 推拉结合模式。只推给在线活跃用户，离线僵尸用户上线再拉，超大V发帖走拉模式。 |

## 面试答题模板

回答业务场景题时可以套这个结构：

1. **先说目标**：这个场景首先要保证什么，例如不超卖、不重复、不丢消息、用户体验可接受。
2. **再说核心矛盾**：高并发、跨系统、异步一致、热点数据、状态复杂。
3. **讲主方案**：数据库条件更新、Redis Lua、MQ 削峰、状态机、本地消息表、延迟任务等。
4. **补关键细节**：幂等、补偿、对账、监控、告警、限流。
5. **说取舍**：为什么不用更重的方案，当前方案适合什么业务量和一致性要求。

可以收束成一句话：

> 我设计这类业务时不会只看功能能不能跑通，而是会同时考虑并发安全、幂等、最终一致、可补偿和可观测，保证线上出问题时能定位、能重试、能恢复。
