## 4. Day 1：理解 LLM、Token 和一次生成过程

### 4.1. 当天目标

1. 知道 LLM 不是数据库，也不是搜索引擎。
2. 理解模型生成答案的基本过程。
3. 讲清 Token 为什么影响成本、上下文和延迟。

### 4.2. 必学内容

#### LLM 是什么？

可以先把 LLM 理解成：**根据已有上下文，反复预测下一个 Token 的概率模型。**

简化过程：

```text
输入 Prompt
  -> 文本切成 Token
  -> 模型计算下一个 Token 的概率
  -> 通过采样参数选出一个 Token
  -> 把新 Token 加入上下文
  -> 重复生成，直到结束
```

因此模型的本质是生成，不是查询事实。它可以给出非常像真的答案，但不天然保证事实正确。

#### Token 是什么？

Token 是模型处理文本的基本单位，可以是一个字、词或词的一部分，不能简单等同于字符数。

Token 影响三个方面：

```text
Token 越多
  -> 输入输出成本越高
  -> 推理耗时通常越长
  -> 越容易接近上下文窗口上限
```

工程上要控制无效历史消息、重复资料和过长 Prompt。详细定义可复习 [[AI应用开发面试指南#5. Token 是什么？|Token 是什么]]。

### 4.3. 动手练习

选择一个模型聊天页面，连续执行下面 3 组实验：

1. 同一个问题分别要求“20 字回答”和“500 字回答”，比较响应时间和用量。
2. 提供一段短材料，要求模型只能基于材料回答一个材料外问题。
3. 开启新会话后追问“我刚才说了什么”，观察模型是否还能知道旧会话内容。

记录观察结果，不需要追求精确 Token 换算。

### 4.4. 当天产出

写出下面这段自己的版本：

> LLM 本质上根据上下文预测下一个 Token，并连续生成答案。它擅长理解和生成，但不等于数据库或事实引擎。Token 会影响上下文长度、调用成本和响应速度，所以工程上要控制输入长度，并对模型输出进行检索、校验和兜底。

### 4.5. 验收问题

- [ ] 为什么 LLM 会生成看起来合理但实际错误的答案？
- [ ] Token 和字符有什么区别？

- [ ] 为什么同一个问题的答案可能不完全相同？

## 5. Day 2：上下文、消息角色和采样参数

### 5.1. 当天目标

1. 理解上下文窗口不是永久记忆。
2. 区分 system、user、assistant 消息。
3. 能解释 temperature、top-p、top-k 的关系。

### 5.2. 必学内容

#### 上下文窗口

上下文窗口是模型一次请求能够处理的 Token 总量，通常包含：

```text
System Prompt
+ 历史对话
+ 当前问题
+ RAG 检索资料
+ 工具调用结果
+ 模型本次输出
```

窗口再大也不能当永久记忆。长上下文会增加成本和延迟，还可能因为无关信息太多导致模型抓错重点。详细内容复习 [[AI应用开发面试指南#6. 上下文窗口是什么？|上下文窗口是什么]]。

#### 消息角色

| 角色 | 作用 | 示例 |
|---|---|---|
| system | 定义角色、边界和全局规则 | 你是订单客服，只能根据查询结果回答 |
| user | 用户当前提出的需求 | 查询订单 1001 为什么还没发货 |
| assistant | 模型之前的回答 | 已查询到订单状态为待出库 |
| tool | 工具执行结果，部分协议使用该角色 | 订单 1001，仓库处理中 |

#### temperature、top-p、top-k

三者都影响“从候选 Token 中怎么选”：

1. `temperature` 调整概率分布的平滑程度。越低越稳定，越高越发散。
2. `top-k` 只保留概率最高的 K 个候选 Token。
3. `top-p` 从高概率开始累加，只保留累计概率达到 p 的候选集合，候选数量动态变化。

简化顺序：

```text
模型计算候选概率
  -> temperature 调整分布
  -> top-k / top-p 缩小候选范围
  -> 从剩余候选中采样
```

面试时不要背死某个参数值。工具调用、结构化抽取偏稳定；创意文案可以适度提高随机性。完整口径见 [[AI Agent岗位面试准备计划#Q2：temperature、top-p、top-k 分别是什么？它们有什么关系？|采样参数面试回答]]。

### 5.3. 动手练习

```text

```

| 观察项 | 低随机性 | 高随机性 |
|---|---|---|
| 多次答案是否相似 | | |
| 格式是否稳定 | | |
| 文案是否更丰富 | | |
| 是否出现额外编造 | | |

## 6. Day 3：幻觉、结构化输出和工程约束

### 6.1. 当天目标

1. 知道幻觉不可能只靠一句 Prompt 彻底消除。
2. 能让模型输出 JSON。
3. 理解“模型生成意图，后端校验并执行”的边界。

### 6.2. 必学内容

#### 幻觉

幻觉是模型生成了缺少依据、与事实不符或根本不存在的内容。

常见原因：

1. 模型目标是生成合理文本，而不是保证事实正确。
2. Prompt 信息不足或存在歧义。
3. 上下文材料错误、过时或相互冲突。
4. 采样随机性过高。
5. 模型能力边界不足，却被要求必须回答。

工程控制手段：

```text
可信数据 / RAG
  + 引用来源
  + 没有依据时允许拒答
  + JSON Schema 和业务规则校验
  + 高风险动作人工确认
  + 测试集和线上观测
```

#### 结构化输出

结构化输出不是“提示模型尽量返回 JSON”就结束，还需要：

1. 定义字段类型、必填项和枚举范围。
2. 后端解析并校验。
3. 解析失败时有限重试或修复。
4. 业务字段继续做真实性和权限校验。

练习 Schema：

```json
{
  "orderId": "1001",
  "intent": "QUERY_LOGISTICS",
  "riskLevel": "LOW",
  "needHumanConfirm": false,
  "reply": "订单正在仓库出库中"
}
```

即使 JSON 格式正确，`orderId` 和订单状态也可能是编造的，因此必须由后端查询真实系统。

### 6.3. 动手练习

写一个“客服意图识别”Prompt，输入 10 条用户问题，要求输出：

```text
intent: QUERY_ORDER / QUERY_LOGISTICS / APPLY_REFUND / OTHER
orderId: string 或 null
needHumanConfirm: boolean
reply: string
```

重点测试：

1. 用户没提供订单号。
2. 用户提供错误格式订单号。
3. 用户要求退款。
4. 用户同时问物流和退款。
5. 用户输入与订单无关的问题。

### 6.4. 当天验收

- [ ] 能说出至少 4 种幻觉控制手段。
- [ ] 能区分“JSON 格式正确”和“业务结果正确”。
- [ ] 能解释为什么退款、改价、取消订单必须人工确认或后端规则控制。
- [ ] 完成 10 条结构化输出测试并记录失败样例。

## 7. Day 4：看懂模型 API，并完成最小调用

### 7.1. 当天目标

1. 看懂一次模型 HTTP 请求的 URL、Header、Body 和响应。
2. 先用 `curl` 调通，再写 Java 或 Python。
3. 能排查 401、429、超时和 JSON 解析错误。

### 7.2. 必学请求结构

不同厂商字段会有差异，但 OpenAI-compatible Chat API 通常类似：

```http
POST /v1/chat/completions
Authorization: Bearer ${LLM_API_KEY}
Content-Type: application/json
```

```json
{
  "model": "模型名称",
  "messages": [
    {"role": "system", "content": "你是 Java 面试助手"},
    {"role": "user", "content": "解释什么是 Token"}
  ],
  "temperature": 0.2,
  "stream": false
}
```

需要识别的响应信息：

1. 模型生成的文本。
2. 请求 ID，方便排查。
3. 输入和输出 Token 用量。
4. finish reason，判断正常结束还是被截断。
5. 错误码和错误信息。

### 7.3. Python 最低要求

只需要能读懂以下概念：

1. `os.getenv` 读取环境变量。
2. `requests.post` 或客户端 SDK 发请求。
3. `dict/list` 对应 JSON 对象和数组。
4. `try/except` 处理网络和解析异常。
5. `response.json()` 读取响应。

这一阶段不要求学习装饰器、异步框架、数据分析和模型训练。

### 7.4. 错误排查表

| 现象 | 优先检查 |
|---|---|
| 401 / 403 | API Key、Header、权限、Base URL |
| 404 | 请求路径、模型服务版本、Base URL 是否重复拼接 |
| 429 | 限流、余额、并发数、重试间隔 |
| 连接超时 | 网络、代理、域名、连接超时配置 |
| 读取超时 | 模型响应慢、输出过长、read timeout 太短 |
| JSON 解析失败 | 返回是否为错误页、字段结构是否变化、流式数据是否被当普通 JSON |

### 7.5. 当天验收

- [ ] 使用 `curl` 成功获得一次模型回答。
- [ ] 能找到响应里的模型文本和 Token 用量。
- [ ] 故意使用错误 Key，观察并记录错误响应。
- [ ] 写出一个最小 Python 调用脚本，或至少能不看资料解释每一行作用。

## 8. Day 5：用 Spring Boot 完成普通聊天接口

### 8.1. 当天目标

实现：

```text
POST /api/chat
  -> 参数校验
  -> 组装 messages
  -> WebClient 调模型
  -> 提取回答
  -> 返回统一结果
```

### 8.2. 推荐模块

| 类 | 职责 |
|---|---|
| `LlmProperties` | 管理 Base URL、Key、模型和超时 |
| `ChatController` | 接收请求、参数校验 |
| `ChatService` | 组装 Prompt 和编排调用 |
| `LlmClient` | 封装模型 HTTP 协议 |
| `ChatRequest/ChatResponse` | 定义业务入参和出参 |
| `GlobalExceptionHandler` | 转换超时、限流和模型服务异常 |

第一周不需要为了“通用”设计多层抽象，但要把 Controller、业务编排和外部模型调用分开。

### 8.3. 必做工程约束

1. `message` 不能为空，并限制最大长度。
2. API Key 不记录到日志。
3. 设置连接超时和读取超时。
4. 模型服务失败时返回明确错误，不把完整第三方响应暴露给前端。
5. 日志记录 requestId、模型、耗时和结果状态。
6. 不要对 4xx 错误盲目重试。

### 8.4. 测试用例

| 用例 | 预期结果 |
|---|---|
| 正常问题 | 返回模型回答 |
| 空问题 | 本地参数校验失败，不调用模型 |
| 超长问题 | 返回明确参数错误 |
| 错误 API Key | 转换为模型鉴权失败 |
| 模型超时 | 返回超时提示，并记录耗时 |
| 模型返回空内容 | 返回可识别的服务异常 |

### 8.5. 当天验收

- [ ] `POST /api/chat` 可以正常使用。
- [ ] Controller 中没有直接拼完整 HTTP 请求。
- [ ] 配置中没有明文密钥。
- [ ] 至少验证正常、空参数、错误 Key 三个场景。

## 9. Day 6：实现 SSE 流式输出

### 9.1. 当天目标

实现：

```text
GET /api/chat/stream?message=...
Content-Type: text/event-stream
```

模型生成一个片段，后端就向客户端推送一个片段，而不是等待完整答案生成后一次返回。

### 9.2. 必学内容

#### 为什么要流式输出？

流式输出主要改善感知速度：

```text
普通响应：等待完整答案 -> 一次返回
流式响应：首个片段到达 -> 持续展示 -> 完成
```

它不会让模型总计算时间凭空消失，但能降低用户感知等待时间。

#### SSE 和 WebSocket 怎么选？

| 对比项 | SSE | WebSocket |
|---|---|---|
| 通信方向 | 服务端向客户端单向推送 | 双向通信 |
| 协议复杂度 | 低，基于 HTTP | 较高，需要连接协议 |
| 聊天生成 | 很适合 | 可以，但通常不是必需 |
| 自动重连 | 浏览器原生有一定支持 | 需要自行设计 |

模型回答主要是服务端持续向前端推送，因此第一周使用 SSE 足够。

### 9.3. 流式处理要点

1. 请求模型时设置 `stream=true`。
2. 正确解析 `data:` 事件，识别结束标记。
3. 不要等待完整响应后再伪装成分片返回。
4. 客户端断开后取消上游订阅，避免继续消耗 Token。
5. 处理模型中途报错和连接超时。
6. 不要在每个 Token 上打印大量 INFO 日志。

### 9.4. 测试场景

1. 正常生成 300 字回答，观察是否逐步返回。
2. 生成过程中主动关闭客户端，观察后端是否停止处理。
3. 把读取超时调短，确认中途错误能够结束连接。
4. 连续请求 3 次，观察连接是否互相影响。

### 9.5. 当天验收

- [ ] 客户端能看到逐步输出，而不是最后一次性出现。
- [ ] 正常完成时连接能够结束。
- [ ] 客户端取消后不会留下无限运行的请求。
- [ ] 能解释 SSE 为什么适合模型流式回答。

### 10.2. 下午：准备 5 个口述答案

#### Q1：LLM 是什么？

> LLM 可以理解为根据上下文预测下一个 Token 并连续生成内容的概率模型。它擅长理解和生成，但不天然保证事实正确，所以业务系统要增加检索、结构化校验、权限和人工兜底。

#### Q2：Token 为什么重要？

> Token 是模型处理文本的基本单位，输入和输出 Token 会影响上下文容量、调用成本和响应时间。工程上要控制历史对话、检索材料和 Prompt 长度，只传真正需要的信息。

#### Q3：上下文窗口是什么？

> 上下文窗口是模型一次请求能够处理的 Token 总量，包括系统提示、历史对话、当前问题、检索资料、工具结果和输出。它不是永久记忆，窗口太长还会增加成本、延迟和信息干扰。

#### Q4：什么是幻觉，怎么控制？

> 幻觉是模型生成了缺少事实依据的内容。不能只靠 Prompt 解决，我会结合可信数据或 RAG、引用来源、无依据拒答、结构化和业务规则校验、高风险人工确认，以及评估集和线上观测共同控制。

#### Q5：什么是结构化输出？

> 结构化输出是让模型按照约定 Schema 返回 JSON 等固定格式，方便后端解析和进入业务流程。但格式正确不代表业务正确，后端仍要做字段、权限、状态和真实性校验。

## 11. 本周自测题

先口述再看提示：

1. **模型为什么不是数据库？**
   提示：概率生成、事实保证、数据时效。
2. **Token 过多会产生什么问题？**
   提示：窗口、成本、延迟、噪声。
3. **上下文窗口和记忆有什么区别？**
   提示：一次请求、外部持久化、按需召回。
4. **temperature 降到很低是否就不会幻觉？**
   提示：只能降低随机性，不能补充事实依据。
5. **top-p 和 top-k 有什么区别？**
   提示：累计概率、固定候选数量。
6. **JSON 返回成功为什么仍不能直接执行退款？**
   提示：格式正确不等于业务正确，权限和状态校验。
7. **流式输出解决了什么问题？**
   提示：感知延迟，不等于总耗时消失。
8. **什么时候不应该重试模型请求？**
   提示：鉴权失败、参数错误、可能导致重复动作。
9. **API Key 为什么不能写进配置文件提交？**
   提示：泄露、滥用、费用和数据安全。
10. **第一周 Demo 距离生产系统还缺什么？**
    提示：鉴权限流、配额、持久化、评估、监控、降级、内容安全。

## 12. 学习边界：本周暂时不要做什么

为了保证第一周真正完成，不要同时展开：

1. 不学习模型训练、微调和 Transformer 数学推导。
2. 不搭建向量数据库和完整 RAG。
3. 不研究 Multi-Agent、AutoGen 和复杂工作流。
4. 不同时对比五六个模型框架。
5. 不花大量时间制作聊天前端，curl 或简单页面足够演示。
6. 不为了架构完整引入数据库、Redis、MQ 和微服务拆分。

第一周的合格线不是“看了多少资料”，而是：**能讲清基础、能调通模型、能展示流式接口、能说明工程风险。**
