## 1. MySQL 库表与索引设计最佳实践

### 1.1. 为什么建表时不建议使用 NULL 作为默认值？
在 MySQL 官方及各大厂的《数据库设计规约》中，普遍要求：**“尽可能将列定义为 `NOT NULL`，并给定合适的默认值，避免使用 `NULL`。”** 其背后原因主要包含存储开销、索引维护以及 SQL 执行陷阱：

#### 1. 物理存储与空间开销
* **行记录头（Record Header）的 Null 值列表**：
  在 InnoDB 存储引擎中，每一行物理记录头都需要开辟额外的空间来维护一个 **Null 值列表（Null Bitmap）**。列表中用 1 个 `bit` 位（向上取整到字节）来标记该行中哪些允许为 NULL 的列实际存储了 NULL 值。这意味着即便一列没有存入 NULL，只要它被声明为“允许为 NULL”，这一位 bit 的记录头空间也必须保留。
* **二级索引空间**：
  在 B+ 树的二级索引中，`NULL` 值需要占用额外的索引空间来存储 NULL 的标识，以便进行物理定位和排序，从而增加了索引文件的物理开销。

#### 2. 查询优化器的估算瓶颈
* **基数计算（Cardinality）不准确**：
  优化器依赖索引的统计信息（基数）来评估执行计划。大量 `NULL` 值的存在会极大地干扰索引基数的统计精度，可能导致优化器做出错误的判断，甚至放弃索引选择全表扫描。

#### 3. 业务逻辑与 SQL 计算陷阱（极易引发 Bug）
* **聚合函数忽略 NULL**：
  `COUNT(*)` 会统计包含 NULL 在内的总行数，但 **`COUNT(column)` 只会统计值不为 NULL 的行数**。类似地，`SUM`、`AVG`、`MIN`/`MAX` 等聚合函数均会自动把 NULL 值过滤掉，可能导致数据计算失真。
* **三值逻辑判定**：
  在 SQL 中，任何值与 `NULL` 的比较（如 `=`, `!=`, `<`, `>`）结果都不是 `TRUE` 或 `FALSE`，而是 `UNKNOWN`。
  * *示例*：查询 `WHERE age != 18` 时，`age` 为 `NULL` 的行是**不会**被查出来的。查询 NULL 必须使用特定的 `IS NULL` 或 `IS NOT NULL`。
* **`NOT IN` 遇上 NULL 结果恒为空**：
  如果 `NOT IN` 的集合中包含了 `NULL` 值（例如 `WHERE id NOT IN (1, 2, NULL)`），该条件等价于 `id != 1 AND id != 2 AND id != NULL`。因为任何值与 `NULL` 进行不等比较结果都是 `UNKNOWN`，所以整个查询**永远返回空结果集**。
* **字符串拼接抹杀**：
  使用 `CONCAT('hello ', NULL)` 时，拼接出来的最终结果直接是 `NULL`，这违背了常规的逻辑认知。

### 1.2. tinyint(1) 和 tinyint(4) 有什么区别？
在很多人的误区中，以为括号内的数字代表最大长度。其实在 MySQL 中：

#### 1. 存储与表示范围：完全相同
无论是 `tinyint(1)` 还是 `tinyint(4)`，它们底层都是普通的 `TINYINT` 类型：
* **占用空间**：都占用 **1 个字节（1 Byte）**。
* **数值范围**：有符号（Signed）为 `-128 ~ 127`；无符号（Unsigned）为 `0 ~ 255`。
* **说明**：你可以向 `tinyint(1)` 中正常存入 `100` 或 `200`，不会发生任何数据的截断。

#### 2. 显示宽度的差异
* 括号内的 `1` 或 `4` 代表**显示宽度（Display Width）**。它只有在配合 **`ZEROFILL`（零填充）** 关键字时才有视觉效果。
  * *示例*：定义为 `tinyint(4) ZEROFILL` 时，存入数字 `5`，在命令行客户端查询显示为 `0005`。
* **注意**：从 MySQL 8.0.17 开始，官方已经废弃了整数类型的显示宽度（包括 `int(11)`、`tinyint(4)` 等括号语法）以及 `ZEROFILL`，未来版本将只保留纯粹的类型定义。

#### 3. JDBC 驱动（Java）中的隐式类型转换机制（开发防坑重点）
这是日常开发中两者最关键的区别：
* **`tinyint(1)`**：MySQL JDBC 驱动（Connector/J）默认会将其映射和转换为 Java 中的 **`java.lang.Boolean`** 类型（`0` 对应 `false`，非 `0` 对应 `true`）。
  * *最佳实践*：表达“是否删除”（`is_deleted`）、“是否启用”（`is_enabled`）等布尔逻辑的字段，建表时推荐使用 `tinyint(1)`，Java 实体类中对应声明为 `Boolean`。如果需要禁用这一行为，可在 JDBC 连接 URL 中配置 `tinyInt1isBit=false`。
* **`tinyint(4)`**：JDBC 驱动会将其识别为普通的微型数值类型，Java 中通常映射为 **`java.lang.Byte`** 或 **`java.lang.Integer`**，不会做布尔转换。

## 2. MySQL 索引底层原理与优化机制

### 2.1. 什么是 B+ 树？其底层物理结构与高度计算

B+ 树是一种专为磁盘等外存储设备设计的多路平衡查找树（Multi-way Search Tree）。在 MySQL InnoDB 引擎中，它是索引的最底层承载结构。

#### 1. B+ 树的底层物理结构特征
1. **多路分叉（高扇出）**：B+ 树的节点可以有数个甚至上千个子节点。这能极大地降低树的高度，减少寻址所需的磁盘 I/O。
2. **非叶子与叶子节点职责分离**：
   * **非叶子节点（内节点）**：**只存储键值（Key）和子节点指针（Pointer）**，绝对不存储实际的数据行（Data）。
   * **叶子节点**：存储了**所有的键值以及与其对应的实际数据行**（在主键索引中是整行物理数据，在二级索引中是主键值）。
3. **叶子节点双向链表**：所有叶子节点都处于同一层，且它们之间通过**双向链表**前后相连，这使得对索引进行范围扫描和排序遍历非常高效，无需在树的层级之间来回跳跃。

#### 2. B+ 树的高度计算与磁盘 I/O（大厂面试杀手锏）
MySQL 每次磁盘 I/O 读取数据的基本物理单位是**页（Page，默认大小 16KB）**。我们可以由此估算出 B+ 树能承载的数据量和树高关系：

* **非叶子节点页（Page）**：
  * 假设表的主键是 `BIGINT` 类型（占 8 字节），子节点指针占 6 字节。
  * 那么一个 16KB 的非叶子节点数据页可以存放的键值对个数大约为：
    `16KB / (8B + 6B) = 16384 / 14 ≈ 1170` 个键值对
* **叶子节点页（Page）**：
  * 假设一行的业务记录（包含所有列数据）平均大小是 1KB。
  * 那么一个 16KB 的叶子节点数据页可以存放的行数大约为：
    `16KB / 1KB = 16` 行数据
* **三层 B+ 树的承载容量**：
  * 第一层（根节点）：1 个节点，可容纳 1170 个分支指针。
  * 第二层（非叶子节点）：1170 个节点，可容纳 `1170 * 1170 ≈ 1,368,900` 个分支指针。
  * 第三层（叶子节点）：约 1,368,900 个数据页，共可存储 `1,368,900 * 16 ≈ 21,902,400` 行数据（**两千多万行**）。
  
> [!TIP]
> **高度与磁盘 I/O 的关系**：
> 因为树的高度仅为 3，查询任何一条记录最多只需要 3 次随机磁盘 I/O。实际上，根节点和非叶子节点通常会被常驻缓存在 InnoDB 的 Buffer Pool 内存中，**真正读取数据行可能只需要 1 次随机物理磁盘 I/O**。

### 2.2. 索引底层数据结构选型剖析
大厂最常问：“为什么 MySQL 选择 B+ 树作为索引结构，而不是 B 树、Hash 表、二叉搜索树或红黑树？”

#### 1. B+ 树 vs B 树：
* **磁盘 I/O 效率更高**：B+ 树非叶子节点不存数据，一个页中就能容纳多得多的键值，分叉极多，树高度显著低于 B 树，磁盘 I/O 极少。
* **范围查询性能极佳**：B+ 树的叶子节点全部通过双向链表连接。进行范围查询时，只需在树中二分查到边界点，然后顺着链表水平遍历即可。而 B 树需要不断在层级间进行中序遍历，引发大量的随机磁盘 I/O。
* **查询效率更稳定**：B+ 树的所有数据都存在叶子节点，时间复杂度稳定为 `O(log N)`；而 B 树在非叶子节点就可能返回数据，性能容易产生波动。

#### 2. B+ 树 vs Hash 索引：
* Hash 索引基于哈希表，仅支持等值查询，**不支持范围查询、排序、模糊查询、联合索引的最左匹配**。

#### 3. B+ 树 vs 二叉平衡树/红黑树：
* 二叉分叉数最多为 2，在千万级数据下树高会突破 20 层，寻址所需的磁盘 I/O 太多，性能雪崩。

### 2.3. 聚簇索引 vs 二级索引（非聚簇索引）
大厂核心基础考点：“主键索引和二级索引是否都使用 B+ 树？它们的叶子节点分别存了什么？为什么二级索引不存整行数据？”

* **结构共性**：在 InnoDB 中，**主键索引和二级索引在底层物理存储上都使用 B+ 树结构**。
* **聚簇索引（Clustered Index，即主键索引）**：
  * **叶子节点内容**：直接存储了**整行完整的物理数据**（这被称为“索引即数据，数据即索引”）。
  * **数量限制**：一张表**有且仅能有一个**聚簇索引。
  * **选型规则**：InnoDB 会默认将主键作为聚簇索引；若无主键，则选择第一个唯一非空索引；若连唯一索引都没有，InnoDB 会在后台自动生成一个隐藏的 6 字节的 `row_id` 作为聚簇索引。
* **二级索引（Secondary Index，即辅助索引 / 非聚簇索引）**：
  * **叶子节点内容**：不存储整行数据，**只存储二级索引列的值 + 对应记录的主键（id）值**。
  * **为什么二级索引不存整行数据？**
    1. **极大节省磁盘空间**：如果一张表有 5 个二级索引，每个二级索引的叶子节点都存一份整行数据，那么这行数据就会被物理复制 5 遍。对于海量数据表，这会造成极大的磁盘开销。
    2. **保证数据一致性，提高写入性能**：如果每个二级索引都存行数据，每次执行 `UPDATE` 修改某列值时，数据库必须同步修改 5 个索引树上对应的叶子节点，这会大幅降低写操作的并发吞吐量，并大大增加并发修改下数据一致性（ACID）的保证难度。

#### 1. 聚簇索引与二级索引的物理结构对比图：

##### 聚簇索引（以主键 ID 为索引）的 B+ 树：
```text
                  [ 根节点 ]
                 /          \
            [ 内节点 ]      [ 内节点 ]
             /      \        /      \
      [叶子节点]  [叶子节点] [叶子节点] [叶子节点] 
      ┌────────┐  ┌────────┐
      │id=1    │  │id=3    │
      │name=张 │  │name=李 │  <--- 存放了完整的物理行数据！
      │age=18  │  │age=25  │
      └────────┘  └────────┘
```

##### 二级索引（以 age 字段为索引）的 B+ 树：
```text
                  [ 根节点 ]
                 /          \
            [ 内节点 ]      [ 内节点 ]
             /      \        /      \
      [叶子节点]  [叶子节点] [叶子节点] [叶子节点] 
      ┌────────┐  ┌────────┐
      │age=18  │  │age=25  │
      │id=1    │  │id=3    │  <--- 叶子节点存放“二级索引列的值 + 对应记录的主键 ID”！
      └────────┘  └────────┘
```

### 2.4. 二级索引非叶子节点能不能直接返回数据？

大厂追问里经常会问：“二级索引的非叶子节点有没有存二级索引列的值？如果覆盖索引需要的字段在非叶子节点就看到了，能不能直接返回？”

#### 1. 二级索引的非叶子节点存不存索引列的值？

**会存。**二级索引整棵 B+ 树都是按二级索引键排序和导航的，所以非叶子节点必须存放二级索引列的值，否则无法决定下一层应该走哪个子页。

以二级索引 `KEY idx_age(age)` 为例：

```text
非叶子节点：age 索引键值 + 子页指针
叶子节点：age 索引键值 + 主键 id
```

如果是联合索引 `KEY idx_city_age(city, age)`：

```text
非叶子节点：(city, age) 联合索引键值 + 子页指针
叶子节点：(city, age) 联合索引键值 + 主键 id
```

这里要区分两件事：

* **非叶子节点会存索引列值**，因为它要做导航和分支判断。
* **非叶子节点不存完整行数据，也不承担返回完整匹配记录的职责**。

#### 2. 二级索引的非叶子节点有没有主键值？

常规面试可以这样回答：**二级索引叶子节点一定存主键值，用于回表；非叶子节点主要存索引键和子页指针，用于导航。**

如果面试官继续追底层细节，需要更严谨一点：InnoDB 的二级索引记录会把主键列作为隐藏后缀参与排序和唯一定位，尤其是非唯一二级索引中，主键会用于区分重复索引值。因此在内部节点中，也可能出现主键列参与比较和定位。

但不要把它理解成：

```text
非叶子节点也存一份完整主键用于回表
```

更准确的说法是：

```text
非叶子节点的核心职责是导航；
叶子节点才保存完整二级索引记录，并通过主键值完成回表。
```

例如：

```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    age INT,
    name VARCHAR(32),
    KEY idx_age(age)
);
```

`idx_age` 逻辑上可以理解为按 `(age, id)` 排序，因为 `age` 可能重复：

```text
age = 18, id = 1
age = 18, id = 5
age = 18, id = 9
```

主键 `id` 参与排序定位，是为了让二级索引记录可唯一比较；真正回表还是发生在叶子节点读取到主键之后。

#### 3. 覆盖索引能不能在非叶子节点直接返回？

**不能。**覆盖索引不是“在非叶子节点返回数据”，而是“走到二级索引叶子节点后，发现查询所需字段都已经在叶子节点里，因此不用再回主键索引查整行数据”。

例如：

```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    age INT,
    name VARCHAR(32),
    city VARCHAR(32),
    KEY idx_age_name(age, name)
);
```

查询：

```sql
SELECT name
FROM user
WHERE age = 18;
```

执行路径是：

```text
1. 从 idx_age_name 的根节点开始查
2. 经过非叶子节点定位到叶子节点
3. 在叶子节点找到 age = 18 的索引记录
4. 叶子节点中已经包含 name
5. 直接返回 name，不需要回表
```

覆盖索引省掉的是：

```text
二级索引叶子节点 -> 主键索引叶子节点
```

不是省掉：

```text
根节点 -> 非叶子节点 -> 叶子节点
```

#### 4. 为什么即使非叶子节点看到了目标值，也必须走到叶子节点？

因为 InnoDB 使用的是 B+ 树。B+ 树和 B 树的一个关键区别是：

```text
B 树：非叶子节点也可能存真实数据，命中后可能直接返回。
B+ 树：真实索引记录集中在叶子节点，非叶子节点主要用于导航。
```

非叶子节点里的值是**分隔键 / 导航键**，不是完整匹配记录。它只告诉你下一层应该走哪个页。

例如：

```text
             [18] [30]          非叶子节点：导航边界
              |    |
              v    v
[18,id=1] [18,id=5] [18,id=9]   叶子节点：真实索引记录
[20,id=2] [25,id=3]
```

非叶子节点里的 `18` 只是说：

```text
从这里往下，有一批 key >= 18 的记录
```

它不代表：

```text
这就是一条 age = 18 的完整记录
```

也不代表：

```text
所有 age = 18 的记录都在这里
```

必须走到叶子节点的原因：

1. **非叶子节点不是完整数据集合**：它只保存用于导航的分隔键和子页指针。
2. **二级索引值可能大量重复**：如 `age = 18` 可能有几万条，非叶子节点里的一个 `18` 不能代表所有匹配记录。
3. **范围查询依赖叶子链表**：`WHERE age BETWEEN 18 AND 30` 必须先定位到叶子节点，再沿叶子节点链表顺序扫描。
4. **覆盖索引也依赖叶子记录**：覆盖索引的查询字段存在于二级索引叶子记录中，而不是靠非叶子节点返回。

#### 5. 面试表达

> InnoDB 使用的是 B+ 树，非叶子节点是目录页，存索引键和子页指针，用于导航；叶子节点才存完整索引记录。二级索引的非叶子节点当然会存二级索引列的值，否则无法导航；对于非唯一二级索引，主键还可能作为隐藏后缀参与排序和定位。但即使在非叶子节点看到了目标值，也不能直接返回，因为它只是分隔键，不代表完整匹配结果。覆盖索引省掉的是回表过程，不是省掉从根节点走到叶子节点的过程。

### 2.5. 深度解析：什么是“回表”及其性能危害？

在 MySQL 的二级索引（非聚簇索引）查询中，“回表”是一个直接关乎系统性能的重要概念。

#### 1. 什么是回表？
当使用二级索引进行查询时，如果查询的列**超出了该二级索引所覆盖的字段范围**，数据库就不得不先去二级索引的 B+ 树中检索查到主键 ID，再拿着主键 ID 去主键索引（聚簇索引）的 B+ 树中重新检索查找，直到获取到整行数据的物理过程。

一句话说：**先走二级索引，再拿主键去主键索引查询整行数据，这个“回到主键索引查”的动作才叫回表**。

要注意一个面试中很容易混淆的点：**不是所有“去主键查询”都叫回表**。如果 SQL 一开始就是 `WHERE id = ?`，直接走主键索引查询，这叫主键索引查询；只有前面先走了普通二级索引，又因为字段不够，需要再根据二级索引叶子节点里的主键值去聚簇索引查数据，才叫回表。

| 查询场景 | 是否回表 | 原因 |
| --- | --- | --- |
| `SELECT name FROM users WHERE age = 18;`，只有 `idx_age(age)` | 是 | 二级索引叶子节点有 `age + id`，但没有 `name`，需要拿 `id` 去主键索引查整行 |
| `SELECT id FROM users WHERE age = 18;`，只有 `idx_age(age)` | 否 | 二级索引叶子节点已经存了主键 `id`，查询列被索引覆盖 |
| `SELECT name FROM users WHERE age = 18;`，有 `idx_age_name(age, name)` | 否 | 联合索引叶子节点已经有 `age + name + id`，属于覆盖索引 |
| `SELECT * FROM users WHERE id = 1;` | 否 | 这是直接走主键索引，不是先走二级索引再回主键索引 |
| `SELECT name FROM users WHERE phone = '138xxx';`，`phone` 没有索引 | 否 | 通常是全表扫描，直接扫描聚簇索引叶子节点，叶子节点本身就是整行数据 |

#### 2. 没有命中索引时会回表吗？
严格说：**没有命中二级索引，一般不叫回表**。

比如 `phone` 字段没有索引，执行：

```sql
SELECT name FROM users WHERE phone = '138xxx';
```

优化器通常会选择全表扫描。对于 InnoDB 来说，全表扫描扫描的是**聚簇索引的叶子节点**，而聚簇索引叶子节点存放的就是完整行记录，所以它不是“先查二级索引，再拿主键回到主键索引查整行”的过程，因此不叫回表。

但有一个细节：如果某条 SQL 没有命中你预期的索引，优化器却选择了另一个普通二级索引，并且查询字段不被这个二级索引覆盖，那么仍然可能发生回表。

可以通过 `EXPLAIN` 快速判断：

| EXPLAIN 现象 | 含义 | 是否通常回表 |
| --- | --- | --- |
| `key = NULL`，`type = ALL` | 没有使用索引，走全表扫描 | 否，通常不叫回表 |
| `key = 某个二级索引`，且 `Extra` 没有 `Using index` | 使用二级索引但不是覆盖索引 | 可能回表 |
| `key = 某个二级索引`，`Extra = Using index` | 覆盖索引 | 否 |

#### 3. 回表的物理寻址过程
以 `SELECT name, age FROM users WHERE age = 18;` 为例（`age` 是二级索引，`name` 没被该索引覆盖）：

```text
  1. 扫描 age 索引树 ──> 找到 age = 18 ──> 取得主键 id = 1
                                                │
  2. 扫描主键索引树 (回表) <── 重新寻址定位到物理数据页 <──┘
```

1. **第一步（扫描二级索引）**：从 `age` 索引树的根节点出发，通过内节点定位，最终检索到叶子节点，匹配 `age = 18` 记录，从中读取出对应的主键 ID（假设是 `id = 1`）。
2. **第二步（回表，扫描主键索引）**：拿着 `id = 1`，从主键索引树的根节点重新开始检索，在叶子节点的数据页中定位到 `id = 1` 的物理行数据，并读取出 `name` 字段的值。

#### 4. 回表对数据库性能的致命伤害
* **大量引发随机 I/O（Random I/O）**：
  在二级索引中，数据是按照索引列顺序存放的，但在二级索引中连续的记录，其对应的主键 ID 却可能是**完全离散、随机分布**在主键索引树的各个物理数据页中的。
  因此，回表操作本质上是在主键索引树中执行大量的**随机寻址**。对于磁盘（尤其是机械硬盘，即使是 SSD 也存在随机寻址损耗）来说，随机 I/O 的速度要比顺序读取慢几个数量级。频繁的回表会使数据库 I/O 负载骤增，成为性能瓶颈。

#### 5. 如何规避回表？
1. **建立覆盖索引（Covering Index）**：
   将查询所需的列全部建在同一个联合索引中。
   * *优化方案*：如果经常执行 `SELECT name FROM users WHERE age = 18;`，建立联合索引 `KEY(age, name)`，此时二级索引的叶子节点同时包含了 `age` 和 `name`，即可直接返回数据，**免除回表**。
2. **杜绝 `SELECT *`**：
   在编写 SQL 时，只查询业务真正需要的字段。多余的字段（即使只是多一个字段）由于不在二级索引中，都可能强迫优化器去执行回表。

#### 6. 面试表达
> 回表不是泛指所有主键查询，也不是泛指所有慢查询，而是指先通过二级索引定位到主键值，再拿这个主键值回到聚簇索引中查完整行数据。比如 `age` 是普通索引，查询 `name` 时，如果 `name` 不在索引里，就必须回表；但如果只查 `id`，或者建立了 `(age, name)` 覆盖索引，就可以直接从二级索引叶子节点返回，不需要回表。如果 SQL 根本没有命中索引，`EXPLAIN` 显示 `type = ALL`、`key = NULL`，那通常是全表扫描，扫描聚簇索引叶子节点拿整行数据，不叫回表。

### 2.6. 覆盖索引与索引下推（ICP）
为了消除或减少回表带来的 I/O 开销，MySQL 提供了以下优化手段：

#### 1. 覆盖索引（Covering Index）
* **原理**：如果一个索引（通常是联合索引）中已经**包含了查询语句中需要的所有字段**，那么 MySQL 就不需要再去扫描聚簇索引，而是直接在二级索引的叶子节点中返回数据。
* *示例*：在 `(age, name)` 上建有联合索引。执行 `SELECT name FROM users WHERE age = 18;`，因为 `name` 本身就存在于该联合索引的叶子节点中，此时直接从二级索引返回，**无需回表**。
* *应用*：在编写高并发、高性能查询 SQL 时，应当尽量避免使用 `SELECT *`，只查询必要的字段，以此来尽量命中覆盖索引。

#### 2. 索引下推（Index Condition Pushdown - ICP）
* **原理**：在 MySQL 5.6 引入的优化机制。在联合索引 `(name, age)` 中，执行 `SELECT * FROM users WHERE name LIKE '张%' AND age = 18;`：
  * **无索引下推（5.6之前）**：存储引擎根据最左前缀匹配找到所有匹配 `name LIKE '张%'` 的主键 ID（此时 age 条件在存储引擎层无法被直接利用），然后将这些 ID 全部**回表**，将完整的行记录返回给 MySQL Server 层。由 Server 层去过滤 `age = 18` 的数据。
  * **有索引下推（5.6及以后）**：存储引擎在扫描二级索引 `(name, age)` 时，虽然遇到了模糊匹配，但由于二级索引中存有 `age` 的值，存储引擎会**在回表之前，先根据 `age = 18` 条件在二级索引内进行第一轮过滤**，过滤掉不符合条件的行。最终，只把符合双重条件的数据 ID 进行**回表**。
  * **效果**：极大减少了回表次数和 Server 层与存储引擎层的数据传输量。

### 2.7. 最左前缀匹配原则与索引失效场景

#### 1. 最左前缀匹配原则
对于联合索引 `(a, b, c)`，MySQL 是按照从左到右的顺序进行排序 and 查找的：
* **生效情况**：查询条件中包含 `a`（如 `a`、`a AND b`、`a AND b AND c`）时，索引能够生效。如果条件顺序写成 `b AND a`，MySQL 优化器会自动将其调整为 `a AND b` 以利用索引。
* **失效情况**：如果查询条件不包含最左前缀 `a`（如单独查 `b`、`c` 或 `b AND c`），则无法利用索引。
* **范围查询切断**：如果在联合索引中，前导列遇到了范围查询（如 `WHERE a > 10 AND b = 2`），那么 `a` 列能够用上索引，但由于 `a` 是范围查询，其后面的 `b` 字段在全局上不能继续帮助 B+ 树缩小扫描区间。更准确地说：**范围查询切断的是后续字段的索引定位能力，不一定切断后续字段的索引层过滤能力**。
* **MySQL 8.0 索引跳跃扫描（Index Skip Scan）**：
  若联合索引前导列 `a` 的基数非常小（比如只有男女两个值），而 `b` 的基数很大。执行 `SELECT * FROM table WHERE b = 100` 时，即使没有使用最左列 `a`，MySQL 8.0 依然能够通过内部自动构造类似 `WHERE a = '男' AND b = 100 UNION WHERE a = '女' AND b = 100` 的逻辑来跳过前导列，使用上索引。

#### 2. 索引跳跃扫描（Index Skip Scan）
索引跳跃扫描可以理解为：**联合索引没有命中最左列时，MySQL 8.0 优化器尝试枚举最左列的少量不同值，从而间接使用后面的索引列**。

比如有联合索引：

```sql
KEY idx_gender_age(gender, age)
```

正常按照最左前缀原则，下面 SQL 没有 `gender` 条件，不能直接完整利用 `(gender, age)`：

```sql
SELECT * FROM users WHERE age = 30;
```

但如果 `gender` 的取值非常少，比如只有 `男`、`女`，优化器可能在内部把查询转换成类似下面的多次小范围扫描：

```sql
SELECT * FROM users WHERE gender = '男' AND age = 30
UNION ALL
SELECT * FROM users WHERE gender = '女' AND age = 30;
```

这样每一次扫描都满足 `(gender, age)` 的最左前缀，相当于优化器帮你“补上”了最左列，再去使用后面的 `age` 字段过滤，所以叫跳跃扫描。

**适用场景**：

1. 联合索引前导列基数很低，例如性别、状态、类型。
2. 后续列过滤性较好，例如 `age = 30`、`created_time` 范围较小、某个业务编号过滤性强。
3. 优化器评估后认为多次小范围索引扫描成本低于全表扫描。

**不适合场景**：

1. 前导列基数很高，例如用户 ID、手机号、订单号。
2. 后续列过滤性也不强，即使跳跃扫描仍要扫描大量记录。
3. 统计信息不准确，优化器评估成本后没有选择 Skip Scan。

看 `EXPLAIN` 时，可以关注 `Extra` 中是否出现类似 `Using index for skip scan` 的信息。需要注意，Skip Scan 是优化器基于成本的选择，不是只要不满足最左前缀就一定触发。

**面试表达**：
> 索引跳跃扫描是 MySQL 8.0 对最左前缀原则的一个优化，但不是推翻最左前缀原则。它适合联合索引前导列基数很低、后续列过滤性较好的场景。比如 `(gender, age)` 索引查询 `WHERE age = 30`，虽然没有 `gender` 条件，但如果 `gender` 只有少量取值，优化器可以枚举 `gender` 的不同值，把查询拆成多次满足最左前缀的小范围索引扫描。不过它依赖成本评估，不一定会走。

#### 3. 范围查询后面的字段还能走索引下推吗？
可能会。这里要区分两个概念：

1. **索引定位能力**：用于确定 B+ 树扫描的起止范围，减少需要扫描的索引记录数。
2. **索引层过滤能力**：扫描索引记录时，在存储引擎层先判断条件，过滤掉不满足条件的记录，减少回表次数。

以联合索引 `idx_a_b(a, b)` 为例：

```sql
SELECT * FROM t WHERE a > 10 AND b = 2;
```

`a > 10` 可以用于范围定位，MySQL 能定位到 `a > 10` 的起始位置并向后扫描。但因为 `a` 是范围条件，`b = 2` 不能继续把扫描范围缩小成一个连续区间，所以它不能继续用于索引寻址。

但是，`b` 本身在联合索引 `(a, b)` 里。扫描 `a > 10` 范围内的二级索引记录时，存储引擎可以直接读取索引记录中的 `b` 值，先判断 `b = 2`，把不满足条件的记录过滤掉，再决定是否回表。这就是索引下推（ICP）的价值。

| 能力 | `WHERE a > 10 AND b = 2` 中的 `b` 是否可用 | 说明 |
| --- | --- | --- |
| 缩小 B+ 树扫描范围 | 通常不能 | `a` 是范围查询，`b` 不能继续形成精确连续区间 |
| 索引下推过滤 | 可能可以 | `b` 在联合索引中，可以在回表前先过滤 |
| 覆盖索引直接返回 | 取决于查询列 | 如果查询列都在 `(a, b)` 和主键中，则可能无需回表 |

看 `EXPLAIN` 时可以关注：

* `key`：是否选择了联合索引 `idx_a_b`。
* `key_len`：通常反映真正用于索引范围定位的字段长度。
* `Extra = Using index condition`：说明使用了索引下推。
* `Extra = Using index`：说明覆盖索引，不需要回表，通常不强调 ICP。

**面试表达**：
> 范围查询切断后续字段，并不是说后续字段完全没用了，而是说后续字段通常不能继续参与索引寻址，不能继续缩小 B+ 树扫描范围。比如 `(a, b)` 索引里 `WHERE a > 10 AND b = 2`，`a` 用来做范围扫描，`b` 不能继续定位范围；但如果开启 ICP，存储引擎扫描二级索引时可以先判断 `b = 2`，过滤掉不满足条件的记录，从而减少回表。所以它失效的是索引定位能力，不一定失效的是索引下推过滤能力。

#### 4. 大厂常考：索引失效的 6 大黄金场景
在开发中，即使列上建有索引，如果 SQL 编写不当，也会导致索引失效进而全表扫描：
1. **对索引列进行函数操作或表达式计算**：
   * *失效*：`WHERE SUBSTR(name, 1, 3) = 'abc'` 或 `WHERE age + 1 = 18`。
   * *原因*：B+ 树的索引是针对原始字段值排序构建的，经过函数或算术运算后的值在树中无序。
2. **隐式类型转换（类型不匹配）**：
   * *失效*：字段 `phone` 是 `VARCHAR` 类型，执行 `WHERE phone = 13800000000`。
   * *原因*：当字符串列与数字比较时，MySQL 会隐式调用函数将字符串转换为数字进行比对，等同于在索引列上加了函数，导致失效。
3. **模糊查询以通配符 `%` 开头**：
   * *失效*：`WHERE name LIKE '%张'`。
   * *原因*：最左前缀无法确定，B+ 树索引无法进行二分定位查找，只能全表扫描。
4. **使用 `OR` 连接且存在无索引列**：
   * *失效*：`WHERE a = 1 OR b = 2`（其中 `a` 有索引，`b` 没有）。
   * *原因*：即使通过 `a` 快速找到了记录，依然必须为了 `b` 去全表扫描，因此优化器直接放弃索引。
5. **在联合索引中违背最左前缀匹配**：
   * （详见最左匹配部分）。
6. **优化器评估回表成本过高**：
   * *失效*：当查询返回的数据量占整张表的比例过高（通常大于 20% ~ 30%）且需要回表时，优化器认为全表扫描（顺序 I/O）的成本低于不断回表（随机 I/O）的成本，从而主动放弃索引。

## 3. 事务与多版本并发控制（MVCC）深度解析

### 2.0. 什么是 ACID？
ACID 是数据库事务可靠性的四个核心特性，用来保证一组数据库操作在并发、异常和宕机场景下仍然是可信的。

1. **Atomicity（原子性）**：一个事务里的操作要么全部成功，要么全部失败回滚，不能只执行一半。典型例子是转账：A 扣款和 B 加款必须同时成功或同时失败。InnoDB 主要通过 `undo log` 支持回滚。
2. **Consistency（一致性）**：事务执行前后，数据都要满足数据库约束和业务规则，不能破坏主键、唯一键、外键、状态流转、余额不为负等约束。一致性是目标，原子性、隔离性、持久性共同支撑它。
3. **Isolation（隔离性）**：多个事务并发执行时，彼此之间不能随意干扰。数据库通过隔离级别、锁和 MVCC 控制脏读、不可重复读、幻读等问题。
4. **Durability（持久性）**：事务一旦提交成功，数据修改就应该持久保存，即使数据库宕机也不能丢。InnoDB 主要依赖 `redo log` 和刷盘策略保证崩溃恢复。

**面试表达**

> ACID 分别是原子性、一致性、隔离性和持久性。原子性保证事务要么全成功要么全失败；一致性保证事务前后数据符合约束和业务规则；隔离性解决并发事务互相影响的问题；持久性保证事务提交后即使宕机也不会丢。MySQL InnoDB 中，`undo log` 支撑回滚和原子性，`redo log` 支撑提交后的持久性，锁和 MVCC 支撑隔离性，最终共同保证数据一致。

---

### 2.1. 并发事务引发的三个经典问题
在多个事务并发运行的情况下，由于隔离级别不够，会导致以下三类数据不一致的问题：
1. **脏读（Dirty Read）**：事务 A 读取了事务 B **尚未提交**的修改数据。随后事务 B 发生回滚，事务 A 读到的是无效数据。
2. **不可重复读（Non-repeatable Read）**：事务 A 在一个事务中多次读取同一行数据。在多次读取之间，事务 B **修改或删除了该行数据并提交**，导致事务 A 读到的结果不一致或记录直接消失了（侧重于已存在数据的**修改/删除**）。
3. **幻读（Phantom Read）**：事务 A 根据相同的查询条件多次读取数据集。在多次读取之间，事务 B **插入了符合该条件的新数据行并提交**，导致事务 A 后续查询看到了之前没有的数据行（侧重于数据行数的**增加**）。


---

### 2.2. MVCC（多版本并发控制）的底层实现原理
MVCC 是一种通过**不加锁**的方式实现“读写不冲突（读不加锁，写不阻塞读）”的并发控制技术。在 MySQL InnoDB 存储引擎中，它主要依赖于以下三个核心机制：

#### 1. 隐藏字段
InnoDB 为每行记录自动生成三个隐藏列：
* `DB_TRX_ID`：最近一次插入或修改该行记录的事务 ID。
* `DB_ROLL_PTR`：回滚指针，指向该行数据在 `undo log` 中的上一个版本。
* `DB_ROW_ID`：隐藏的单调自增主键（若表无显式主键则自动生成）。

#### 2. Undo Log（回滚日志）与版本链
每次对数据行进行修改时，旧版本的数据会被记录到 `undo log` 中。物理行记录通过 `DB_ROLL_PTR` 指针指向历史版本，从而将同一行数据的各个旧版本连接起来，形成一条 **版本链（Version Chain）**。

#### 3. Read View（一致性视图）
当事务执行**快照读（普通的 `SELECT`）**时，会生成一个一致性视图 `Read View`。该视图用来判定版本链中的哪个版本对当前事务是可见的。
Read View 包含以下四个核心字段：
* `m_ids`：在生成 Read View 时，当前系统里活跃（未提交）的事务 ID 列表。
* `min_trx_id`：当前活跃事务 ID 列表中的最小值。
* `max_trx_id`：系统准备分配给下一个事务的 ID 值（即最大活跃事务 ID + 1）。
* `creator_trx_id`：创建这个 Read View 的事务 ID。

#### 4. 可见性判定算法（重点，面试必背）
当读取版本链中的某条记录时，将该记录的 `DB_TRX_ID` 与当前事务的 `Read View` 进行对比：
1. 若 `DB_TRX_ID == creator_trx_id`：说明该版本数据是当前事务自己修改的，**可见**。
2. 若 `DB_TRX_ID < min_trx_id`：说明修改该版本的事务在生成 Read View 前就已经提交，**可见**。
3. 若 `DB_TRX_ID >= max_trx_id`：说明修改该版本的事务在生成 Read View 后才开启，**不可见**。
4. 若 `min_trx_id <= DB_TRX_ID < max_trx_id`：
   * 若 `DB_TRX_ID` **在** `m_ids` 列表中：说明修改该版本的事务仍处于活跃未提交状态，**不可见**。
   * 若 `DB_TRX_ID` **不在** `m_ids` 列表中：说明修改该版本的事务已经提交，**可见**。

> [!NOTE]
> 如果当前版本的记录被判定为**不可见**，则沿着 `DB_ROLL_PTR` 回滚指针在 `undo log` 中寻找上一个版本继续判定，直到找到可见版本；若全不可见，则说明该记录对当前事务不可见。

---

### 2.3. RC 与 RR 级别下 MVCC 的本质差异
* **RC（Read Committed，读已提交）**：**每次执行 SELECT 语句时都会重新生成一个新的 Read View**。这导致了同一事务中多次查询，会看到其他并发事务已提交的最新修改，从而产生“不可重复读”。
* **RR（Repeatable Read，可重复读）**：**仅在事务第一次执行 SELECT 语句时生成一个 Read View**，后续整个事务中的 SELECT 都会复用这同一个 Read View。因此即使其他事务修改并提交了数据，当前事务读取到的依旧是最初版本，实现了“可重复读”。

## 4. 锁机制与防范幻读

### 3.1. 行锁的三种类型
1. **Record Lock（记录锁）**：单条索引记录上的锁。它只锁住存在的索引记录本身。
2. **Gap Lock（间隙锁）**：锁住索引记录之间的间隙，或者第一条索引记录之前、最后一条索引记录之后的间隙。其唯一职责是**阻止其他事务在间隙中插入数据（防止幻读）**。间隙锁之间是不互斥的，多个事务可以同时持有同一个间隙的 Gap Lock。
3. **Next-Key Lock（临键锁）**：**Record Lock + Gap Lock 的组合**。它既锁住索引记录本身，又锁住记录左侧的间隙，区间表现为**左开右闭**，形如 `(x, y]`。

---

### 3.2. MySQL RR 级别下是如何防范幻读的？
MySQL InnoDB 引擎在 RR（可重复读）级别下，是通过 **MVCC（快照读）** 与 **Next-Key Locks（当前读）** 两种机制共同对抗幻读的：

#### 1. 快照读（普通的无锁 `SELECT`）
* **快照读可以通过 MVCC 避免幻读**。因为整个事务复用同一个 `Read View`，其他事务新插入（`INSERT`）的行，其 `DB_TRX_ID` 必然大于当前 `Read View` 的最大事务 ID，判定为不可见。

#### 2. 当前读（带锁查询或写操作如 `SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`）
* **当前读必须通过 Next-Key Locks 避免幻读**。执行当前读时，InnoDB 会对匹配的记录区间加 **Next-Key Locks（临键锁，包含记录锁与间隙锁）**，锁死该区间的间隙。其他并发事务此时如果试图在这个间隙中执行插入（`INSERT`）操作，会被直接阻塞，直到当前读的事务提交释放锁。

---

### 3.3. Next-Key Locks 临键锁的加锁与退化规则
在 `REPEATABLE READ` 隔离级别下，InnoDB 的加锁基本单位是 Next-Key Lock `(x, y]`，为了在安全的同时最大化并发性能，建立了一套锁退化规则：

* **唯一索引等值查询**：
  * 若记录**存在**：临键锁会退化为 **Record Lock（记录锁）**，只锁住该条记录。
  * 若记录**不存在**：临键锁会退化为 **Gap Lock（间隙锁）**。例如，表中仅有 `id = 5` 和 `id = 10`。查询 `id = 7 FOR UPDATE` 时，锁定的范围 `(5, 10]` 会退化为间隙锁，锁定范围为 `(5, 10)`，防止插入 `id = 7`。
* **非唯一索引等值查询**：
  * 会对匹配到的非唯一索引行加 Next-Key Lock `(x, y]`。
  * 此外，为了防止右侧幻读插入，还会对该记录右侧相邻的下一个索引键值加一个 **Gap Lock（间隙锁）** `(y, z)`。
* **唯一索引范围查询**：
  * 会对每一个匹配到的区间加 Next-Key Lock，只有在边界等值且记录存在时才会降级为记录锁或间隙锁。

---

### 3.4. 混合读写下导致幻读的 2 种边界失效场景（大厂加分点）
在以下特定读写混合的操作顺序下，InnoDB 的 RR 级别依然会产生幻读：

* **边界情况 1：先快照读，后当前读**
  1. 事务 A 先执行普通的 `SELECT`（快照读）发现 `id = 5` 不存在。
  2. 此时，事务 B 插入了 `id = 5` 并提交。
  3. 事务 A 接着执行 `SELECT * FROM test WHERE id = 5 FOR UPDATE`（当前读）。因为当前读必须读取最新数据版本，它会越过 Read View 查到事务 B 新增并提交的 `id = 5` 记录，发生幻读。
* **边界情况 2：快照读后，更新了并发事务新插入的行（版本篡改）**
  1. 事务 A 先 `SELECT` 发现 `id = 5` 不存在。
  2. 事务 B 插入 `id = 5` 并提交。
  3. 事务 A 执行更新：`UPDATE test SET name = "xxx" WHERE id = 5`。因为 `UPDATE` 属于当前读，所以能更新成功，且该行的 `DB_TRX_ID` 被修改为**事务 A 的事务 ID**。
  4. 事务 A 再次执行快照读 `SELECT`，此时因为 `DB_TRX_ID` 变为了事务 A 自己，符合可见性规则，事务 A 突然读出了这条新记录，产生幻读。

## 5. 模拟面试连环对答通关话术（DB 篇）

### Q1：面试官问：“为什么 MySQL 建表时不建议使用 NULL 作为默认值？”
* **通关话术**：
  > “不建议使用 `NULL` 主要是出于**物理存储、查询优化和 SQL 业务逻辑陷阱**三方面的考虑。
  > 
  > 首先，从物理存储来看，InnoDB 行记录的头信息里需要使用额外的 **Null 值列表（Null Bitmap）** 来标记哪些列存了 NULL 值，即使列值不是 NULL，只要定义允许为 NULL 也会有记录头的 bit 位空间开销。二级索引存 NULL 也会有额外标识开销。此外，NULL 会使索引的基数计算变难，导致优化器做错执行计划。
  > 
  > 其次，从 SQL 运行陷阱来看，`NULL` 会引发很多违背直觉的 Bug。比如 `COUNT(column)` 会自动忽略 NULL 行导致聚合结果不准；使用 `CONCAT` 拼接含有 NULL 的字段会直接返回整个 NULL。
  > 最危险的是 **`NOT IN` 子查询**，如果子查询结果集里含有 NULL 值，那么整个外层查询会因为三值逻辑判定（任何值与 NULL 比较结果都是 UNKNOWN）永远返回空结果。所以，大厂的规范通常要求定义列为 `NOT NULL`，并给定数字 `0` 或空串 `""` 作为默认值。”

### Q2：面试官问：“tinyint(1) 和 tinyint(4) 有什么区别？建表时表达是否状态应该用哪个？”
* **通关话术**：
  > “在底层的存储空间和数值表示范围上，`tinyint(1)` 和 `tinyint(4)` 是**完全没有区别**的，它们都占用 1 个字节，支持 `-128 到 127` 的整数范围。括号内的数字代表显示宽度，只有配合 `ZEROFILL` 且数字位数不足时会在前面自动补 0，但该语法已经在 MySQL 8.0.17 之后被官方废弃了。
  > 
  > 它们最关键的区别在于 **Java JDBC 驱动的行为**。MySQL 的 JDBC 驱动默认会把 `tinyint(1)` 隐式转换为 Java 的 `Boolean` 类型（`0` 对应 `false`，非 `0` 对应 `true`），而将 `tinyint(4)` 识别为普通的数字（Byte/Integer）。
  > 
  > 所以在实际开发中，如果是表达**是否逻辑**（如是否删除 `is_deleted`），推荐在数据库中使用 `tinyint(1)`，在 Java 中声明为 `Boolean`；如果是需要存储微型数字的场景，则推荐使用普通的 `TINYINT` 或 `tinyint(4)`。”

### Q3：面试官问：“什么是 MVCC？它能完全避免幻读吗？”
* **通关话术**：
  > “**MVCC** 即多版本并发控制，是通过**版本链（Undo Log）**和**一致性视图（Read View）**来实现并发读写不冲突、读不加锁的机制。在 RR（可重复读）级别下，由于整个事务只在第一次查询时生成 Read View，后续查询复用此 View，从而保证了可重复读。
  > 
  > 关于**能否完全解决幻读**，答案是：**快照读（普通 SELECT）可以避免幻读**，因为 Read View 规则会判定新插入数据的事务 ID 大于 View 的最大值，因而是不可见的；**当前读（如带有 FOR UPDATE）可以通过 Next-Key Locks 临键锁（记录锁 + 间隙锁）加锁阻塞其他事务在该间隙插入，从而也能够避免幻读**。
  > 
  > 但是，**混合读写下依然无法完全避免**。典型的失效边界情况是：事务 A 先执行快照读发现记录不存在，此时并发事务 B 插入记录并提交，随后事务 A 对这一行新纪录执行了更新（UPDATE）或者当前读（FOR UPDATE），此时因为当前读或更新需要读取最新版本，它会越过 Read View 的限制查出新记录（若执行了更新还会将该记录的隐藏事务 ID 改为当前事务），使得后续的快照读也读出了该记录，从而产生幻读。”

### Q4：面试官问：“什么是 Next-Key Locks？它的加锁和退化规则是怎样的？”
* **通关话术**：
  > “**Next-Key Lock（临键锁）** 是 InnoDB 行锁的默认加锁单位，它是记录锁（Record Lock）与间隙锁（Gap Lock）的组合，锁定区间表现为左开右闭 `(x, y]`。其核心目的是解决当前读场景下的幻读问题。
  > 
  > 为了兼顾并发性能，InnoDB 制定了锁退化规则：
  > 第一，在**唯一索引等值查询**时：如果记录存在，临键锁会退化为**记录锁**只锁住该行；如果记录不存在，临键锁会退化为**间隙锁**锁住所在的间隙。
  > 第二，在**非唯一索引等值查询**时：除了对匹配行加临键锁，还会对其右侧相邻的下一个索引值加一个**间隙锁**，以防止插入同名记录。
  > 第三，在**范围查询**时：会针对每一个匹配到的区间加临键锁，只有在唯一索引等值边界且记录存在时才退化为记录锁或间隙锁。”

### Q5：面试官问：“MySQL 索引为什么用 B+ 树？索引在什么情况下会失效？”
* **通关话术**：
  > “MySQL 使用 B+ 树作为索引结构，主要是因为 B+ 树在**磁盘 I/O 效率、范围查询和稳定性**上具有绝对优势。
  > 
  > 第一，B+ 树的非叶子节点不存行数据，只存键值和指针，这使得一页能容纳更多索引，分叉极多，千万级数据树高通常也只有 3 到 4 层，极大地减少了查询时的随机磁盘 I/O。
  > 第二，B+ 树的叶子节点全部相连并组成双向有序链表，非常利于执行范围区间查询；而像 B 树、Hash 表或红黑树，在层级、范围遍历和等值以外的查询场景中都存在明显的局限。
  > 
  > 至于索引失效，我们在实际开发中要注意规避 6 个场景：一是在索引列上做函数或算术运算；二是类型不匹配引发隐式转换，比如字符串列传了数字去查；三是 LIKE 模糊查询以 `%` 开头；四是联合索引查询违背了最左前缀原则；五是 OR 连接的条件里存在没有索引的列；六是查询返回的数据比例过高，导致优化器评估回表的随机 I/O 成本高于全表顺序 I/O 从而主动放弃索引。”

## 6. 互联网大厂 MySQL 高频补充面试题

这一节偏生产实践和高阶追问，适合补充回答“你怎么排查数据库问题”“线上慢 SQL 怎么处理”“MySQL 为什么能保证 crash-safe”“主从延迟怎么办”这类问题。回答时建议按“原理 -> 风险 -> 解决方案 -> 项目表达”的顺序讲，不要只背概念。

### Q6：Buffer Pool 是什么？MySQL 为什么不能每次都直接读磁盘？

**常见问法**

- InnoDB Buffer Pool 是什么？
- MySQL 查询一行数据一定会发生磁盘 I/O 吗？
- 脏页、刷盘、LRU 链表怎么理解？

**答案**

Buffer Pool 是 InnoDB 的核心内存缓存区域，用来缓存数据页、索引页、Undo 页、自适应哈希索引等内容。MySQL 的数据在磁盘上以页为单位存储，InnoDB 默认页大小是 16KB。查询数据时，InnoDB 会先看目标页是否已经在 Buffer Pool 中：

1. 如果页在 Buffer Pool 中，直接从内存读取。
2. 如果不在，就从磁盘读取页到 Buffer Pool，再返回数据。
3. 修改数据时，先修改 Buffer Pool 中的数据页，并将其标记为脏页。
4. 后续由后台线程按照策略把脏页刷回磁盘。

Buffer Pool 的意义是把随机磁盘 I/O 尽量转成内存访问和后台批量刷盘，显著提升查询和写入性能。

**关键细节**

- InnoDB 不是按行读取磁盘，而是按页读取。
- 脏页是内存中已经被修改、但还没有刷回磁盘的数据页。
- Buffer Pool 使用改进版 LRU 管理冷热页，避免全表扫描把热点页全部淘汰。
- 数据页即使还没刷盘，也可以依靠 redo log 在崩溃后恢复。

**面试表达**

> Buffer Pool 是 InnoDB 的核心缓存层。查询时先查 Buffer Pool，命中就不需要读磁盘；未命中才从磁盘加载 16KB 数据页。更新时也是先改内存页并标记脏页，后续异步刷盘。真正保证崩溃后不丢数据的是 redo log，而不是每次更新都立刻把数据页刷回磁盘。

---

### Q7：redo log、undo log、binlog 分别解决什么问题？

**常见问法**

- redo log、undo log、binlog 有什么区别？
- 为什么有了 binlog 还需要 redo log？
- MySQL 如何保证 crash-safe？

**答案**

三类日志职责不同：

| 日志 | 所属层级 | 主要作用 |
| --- | --- | --- |
| redo log | InnoDB 引擎层 | 保证事务提交后的修改具备 crash-safe 能力 |
| undo log | InnoDB 引擎层 | 支持事务回滚和 MVCC 版本链 |
| binlog | MySQL Server 层 | 记录逻辑变更，用于主从复制、数据恢复、审计 |

**redo log**

redo log 是物理日志，记录某个数据页做了什么修改。事务提交时，只要 redo log 持久化成功，就算数据页还没刷盘，MySQL 崩溃后也能根据 redo log 重放恢复，所以 redo log 解决的是“已提交事务不能丢”的问题。

**undo log**

undo log 记录数据修改前的旧版本，用于事务回滚，同时也构成 MVCC 的版本链。事务执行失败或主动回滚时，可以通过 undo log 把数据恢复到修改前。

**binlog**

binlog 是 Server 层逻辑日志，记录 SQL 或行级变更，主要用于主从复制和按时间点恢复。它不能替代 redo log，因为 binlog 不负责 InnoDB 的页级崩溃恢复。

**关键细节**

- redo log 偏物理恢复，binlog 偏逻辑复制和归档。
- undo log 不只是回滚，也支撑 MVCC 快照读。
- redo log 是循环写，binlog 是追加写。
- crash-safe 依赖 redo log 和两阶段提交。

**面试表达**

> 我会把 redo、undo、binlog 分开讲。redo log 保证事务提交后崩溃可恢复；undo log 保证事务可以回滚，同时支持 MVCC；binlog 用于主从复制和数据恢复。binlog 不能替代 redo log，因为它不负责 InnoDB 数据页级别的崩溃恢复。

---

### Q8：MySQL 两阶段提交是什么？为什么需要它？

**常见问法**

- redo log 和 binlog 如何保证一致？
- MySQL 提交事务时发生宕机怎么办？
- 什么是两阶段提交？

**答案**

MySQL 更新一条数据时，既要写 InnoDB 的 redo log，也要写 Server 层 binlog。如果二者不一致，会导致主库恢复和从库复制结果不一致。

两阶段提交大致流程：

1. 执行业务更新，修改 Buffer Pool 中的数据页。
2. 写 redo log，并标记为 `prepare` 状态。
3. 写 binlog。
4. 提交事务，把 redo log 标记为 `commit` 状态。

崩溃恢复时：

- 如果 redo log 没有 prepare，说明事务未提交，回滚。
- 如果 redo log 是 prepare，但没有对应 binlog，说明 binlog 没写成功，回滚。
- 如果 redo log 是 prepare，并且有完整 binlog，说明事务可以提交，恢复为已提交状态。

**关键细节**

- 两阶段提交解决的是 redo log 和 binlog 的一致性问题。
- 没有两阶段提交，可能出现主库恢复有数据、从库没数据，或者反过来。
- binlog 写成功与 redo commit 之间宕机，恢复时可以根据 binlog 判断是否提交。

**面试表达**

> 两阶段提交是为了保证 redo log 和 binlog 一致。事务提交时先把 redo 写成 prepare，再写 binlog，最后把 redo 改为 commit。崩溃恢复时，如果 redo prepare 且 binlog 完整，就提交；如果 binlog 不完整，就回滚。这样可以避免主库 crash recovery 和从库复制出现不一致。

---

### Q9：MySQL 主从复制原理是什么？主从延迟怎么排查？

**常见问法**

- MySQL 主从复制流程是什么？
- 主从延迟怎么产生？
- 读写分离下读到旧数据怎么办？

**答案**

MySQL 主从复制核心流程：

1. 主库提交事务后写 binlog。
2. 从库 I/O 线程连接主库，拉取 binlog。
3. 从库把拉到的日志写入 relay log。
4. 从库 SQL 线程读取 relay log，在从库重放变更。

主从延迟常见原因：

- 主库写入压力大，从库重放速度跟不上。
- 从库单线程或并行度不足。
- 大事务、大批量更新导致从库长时间执行。
- 从库机器配置弱，磁盘 I/O 或 CPU 不足。
- 从库还承担复杂查询，影响复制线程。
- 网络抖动导致 binlog 拉取变慢。

解决思路：

- 避免大事务，批量更新拆小批。
- 开启并调优并行复制。
- 从库不要跑重查询，复杂分析走独立报表库或数仓。
- 核心读写链路必要时读主库。
- 对强一致读使用“写后读主”或基于 GTID/位点等待从库追上。
- 建立复制延迟监控和告警。

**关键细节**

- 读写分离不是强一致方案，天然可能读到旧数据。
- 用户刚写完马上查自己的数据，常见做法是短时间读主库。
- 大事务会显著放大主从延迟。
- 延迟不是只看秒数，还要结合业务是否读到了旧状态。

**面试表达**

> 主从复制本质是主库写 binlog，从库拉取后写 relay log，再重放执行。主从延迟通常来自写入高峰、大事务、从库重放慢、从库承担读压力或网络问题。业务上不能假设读写分离强一致，像支付后查订单、下单后查状态这类写后读场景，我会短时间读主库或等待从库位点追平。

### Q10：MySQL 死锁是怎么产生的？线上怎么排查和避免？

**常见问法**

- 什么是死锁？
- InnoDB 死锁后会怎么处理？
- 线上死锁怎么排查？

**答案**

死锁是多个事务互相持有对方需要的锁，形成循环等待。例如：

```text
事务 A 持有订单 1 的锁，等待订单 2 的锁
事务 B 持有订单 2 的锁，等待订单 1 的锁
```

InnoDB 有死锁检测机制，发现死锁后会选择一个事务回滚，让另一个事务继续执行。应用侧通常会收到类似 `Deadlock found when trying to get lock` 的异常。

排查方式：

- 执行 `SHOW ENGINE INNODB STATUS` 查看最近一次死锁信息。
- 看事务持有哪些锁、等待哪些锁、执行了哪些 SQL。
- 结合业务日志、traceId、SQL 参数定位代码路径。
- 看是否存在不同事务更新顺序不一致。
- 看 SQL 是否没走索引导致锁范围扩大。

避免方式：

- 保证多个事务按相同顺序更新资源。
- 尽量缩短事务时间，事务里不要做慢 RPC 或复杂计算。
- SQL 必须命中索引，避免行锁退化成大范围锁。
- 批量更新拆小批，减少一次事务持锁数量。
- 捕获死锁异常后，可以做有限次数重试。

**关键细节**

- 死锁不是 MySQL bug，而是并发写入中常见现象。
- 没有索引或索引选择差，会导致锁范围扩大，死锁概率上升。
- 自动重试要有次数上限，避免持续重试压垮数据库。

**面试表达**

> 死锁本质是事务之间循环等待锁。线上排查我会先看 `SHOW ENGINE INNODB STATUS` 的最近死锁信息，确认两边事务执行的 SQL、锁等待对象和业务路径。治理上会统一资源更新顺序、缩短事务、确保 SQL 命中索引，并对死锁异常做有限重试。

### Q11：线上慢 SQL 怎么排查？

**常见问法**

- 线上接口突然变慢，怀疑 MySQL，怎么排查？
- EXPLAIN 重点看哪些字段？
- 慢 SQL 怎么优化？

**答案**

排查顺序：

1. 看监控：QPS、RT、慢 SQL 数量、连接数、CPU、磁盘 I/O、Buffer Pool 命中率。
2. 看慢日志：确认具体 SQL、执行时间、扫描行数、返回行数。
3. 用 `EXPLAIN` 看执行计划。
4. 确认索引是否命中，是否回表、排序、临时表、全表扫描。
5. 看 SQL 是否存在隐式转换、函数计算、`LIKE '%xx'`、`OR`、大分页等问题。
6. 结合业务场景决定是改 SQL、加索引、拆查询、缓存、归档还是分库分表。

`EXPLAIN` 重点字段：

| 字段 | 关注点 |
| --- | --- |
| type | 访问类型，常见从好到差：const、ref、range、index、ALL |
| key | 实际使用的索引 |
| rows | 预估扫描行数 |
| Extra | 是否 Using index、Using filesort、Using temporary |
| filtered | 条件过滤比例 |

**关键细节**

- 不要看到慢 SQL 就立刻加索引，要先判断选择性、查询频率和写入成本。
- `Using filesort` 不一定绝对坏，小数据量可接受；大数据排序要警惕。
- `rows` 预估不准时，要关注统计信息是否过旧。
- 慢 SQL 优化要结合业务可接受的一致性和实时性。

**面试表达**

> 我排查慢 SQL 会先从监控和慢日志定位具体 SQL，再用 EXPLAIN 看 type、key、rows、Extra。优化时先判断是否索引失效、回表过多、排序临时表、大分页或返回数据过多。方案不一定只是加索引，也可能是改 SQL、覆盖索引、延迟关联、冷热归档、缓存或搜索引擎承接复杂查询。

### Q12：COUNT(*)、COUNT(1)、COUNT(column) 有什么区别？

**常见问法**

- `COUNT(*)` 和 `COUNT(1)` 哪个更快？
- 为什么 `COUNT(column)` 结果可能不一样？
- 大表统计总数怎么优化？

**答案**

语义区别：

- `COUNT(*)`：统计结果集行数，不关心列值是否为 NULL。
- `COUNT(1)`：统计常量 1 的行数，语义上和 `COUNT(*)` 基本等价。
- `COUNT(column)`：只统计该列不为 NULL 的行。

在 InnoDB 中，`COUNT(*)` 不会像 MyISAM 那样直接存表总行数，因为 InnoDB 支持 MVCC，不同事务视图下能看到的行数可能不同。优化器通常会选择成本较低的索引来扫描计数。

大表总数优化：

- 不强实时的总数用缓存或统计表。
- 后台异步维护计数。
- 分页不一定展示精确总页数。
- 管理后台复杂条件统计可以走 ES、数仓或离线报表。

**关键细节**

- `COUNT(column)` 会忽略 NULL。
- 大表 `COUNT(*)` 可能很慢，因为仍要扫描可见记录。
- 高并发计数不要每次实时扫大表。

**面试表达**

> `COUNT(*)` 和 `COUNT(1)` 都是统计行数，差异通常不用过度纠结；`COUNT(column)` 会忽略 NULL，语义不同。InnoDB 不能简单保存一个全局行数，因为 MVCC 下不同事务可见行不同。大表计数如果不要求强实时，我会用统计表、缓存或异步任务维护。

### Q13：MySQL 深分页为什么慢？怎么优化？

**常见问法**

- `LIMIT 1000000, 20` 为什么慢？
- 深分页怎么优化？
- 游标分页和延迟关联怎么选？

**答案**

深分页慢的原因是：MySQL 需要先找到并丢弃前面大量数据，再返回目标页。例如：

```sql
SELECT *
FROM order_info
WHERE user_id = 100
ORDER BY id DESC
LIMIT 1000000, 20;
```

即使走索引，也要扫描大量索引记录，并可能发生大量回表。

常见优化：

1. **游标分页**：用上一页最后一条记录的排序键继续查。

```sql
SELECT *
FROM order_info
WHERE user_id = 100
  AND id < #{lastId}
ORDER BY id DESC
LIMIT 20;
```

2. **延迟关联**：先通过覆盖索引查出主键，再回表取完整数据。

```sql
SELECT o.*
FROM order_info o
JOIN (
    SELECT id
    FROM order_info
    WHERE user_id = 100
    ORDER BY id DESC
    LIMIT 1000000, 20
) t ON o.id = t.id;
```

游标分页和延迟关联的选择：

| 方案 | 适合场景 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 游标分页 | C 端订单列表、消息列表、评论列表、动态流、日志流水 | 从根上避免大 `offset`，性能稳定，越深越明显 | 不适合直接跳到第 N 页，依赖稳定排序字段 |
| 延迟关联 | 后台管理系统、运营列表、必须保留 `pageNo/pageSize` 的页码分页 | 兼容传统页码分页，先用覆盖索引查主键，减少大量无效回表 | `offset` 扫描成本仍然存在，只是减少回表成本 |
| ES/搜索系统 | 复杂检索、多条件排序、深翻页 | 更适合检索和排序场景 | 架构复杂，要处理数据同步一致性 |

所以默认选择是：**能用游标分页就优先用游标分页；必须支持任意页跳转时，再考虑延迟关联**。

延迟关联要想有效，最好让子查询命中覆盖索引。例如：

```sql
KEY idx_user_id_id(user_id, id)
```

这样子查询 `SELECT id FROM order_info WHERE user_id = 100 ORDER BY id DESC LIMIT 1000000, 20` 只扫描二级索引，不会一开始就对大量被丢弃的数据回表。外层再根据最终 20 个 `id` 回表取完整数据。

3. **限制最大翻页深度**：C 端产品通常不允许无限翻页。
4. **搜索引擎承接**：复杂多条件检索、排序、深翻页交给 ES 等系统。
5. **冷热归档**：历史数据归档，减少热表体量。

**关键细节**

- 游标分页性能最好，但不适合任意跳页。
- 延迟关联适合必须支持页码翻页，但仍要控制深度。
- 游标分页是从根上消除大 `offset`；延迟关联是减少深分页过程中的无效回表，并没有消除 `offset` 扫描。
- 深分页本质上是产品和数据模型问题，不是简单加索引就能完全解决。

**面试表达**

> 深分页慢是因为数据库要扫描并丢弃大量前置记录，还可能伴随大量回表。游标分页和延迟关联解决的问题不完全一样：游标分页是从根上避免大 `offset`，适合 C 端连续翻页、信息流、订单流水这类场景；延迟关联适合必须支持页码跳转的后台列表，它通过先查覆盖索引中的主键，再关联回表查详情，减少无效回表，但 `offset` 扫描成本还在。所以我会优先选择游标分页；如果产品必须支持任意页跳转，再用延迟关联、覆盖索引、限制最大页数等方式兜底。

### Q14：分库分表后会带来哪些问题？

**常见问法**

- 什么情况下需要分库分表？
- 分库分表后有哪些复杂问题？
- 分片键怎么选？

**答案**

分库分表主要解决单表数据量过大、单库写入压力过高、索引膨胀、DDL 成本高等问题。但它会引入一系列复杂度：

- **分片键选择**：分片键必须高频出现在查询条件中，否则会广播查询。
- **跨分片查询**：按非分片键查询需要扫多个分片，性能差。
- **全局唯一 ID**：不能依赖单表自增，需要雪花算法、号段模式等方案。
- **分布式事务**：跨分片写入要尽量避免，必要时用最终一致、TCC 或补偿。
- **分页排序聚合**：跨分片 `ORDER BY`、`GROUP BY`、分页需要多路归并，成本高。
- **扩容迁移**：后续扩分片会涉及数据迁移和路由切换。
- **热点分片**：分片键不均匀会导致某些库表压力过高。

分片键选择原则：

- 查询频率高，绝大多数请求都带这个字段。
- 离散度高，数据分布均匀。
- 尽量避免频繁变更。
- 尽量能把单个业务对象的读写聚合到同一分片。

**关键细节**

- 分库分表是最后手段，优先考虑索引优化、冷热归档、读写分离、缓存、ES、垂直拆分。
- 分片后最怕没有分片键的查询。
- 跨分片分页和统计通常需要异构索引或离线分析系统。

**面试表达**

> 分库分表能解决单库单表容量和写入瓶颈，但会引入分片键、跨分片查询、全局 ID、分布式事务、扩容迁移和跨分片分页聚合问题。我一般不会一上来就分库分表，而是先做索引、冷热归档、缓存和读写分离；确实到瓶颈时，优先选择查询最常带、离散度高、能聚合业务读写的字段作为分片键。

### Q15：大表加字段或加索引怎么做更安全？

**常见问法**

- 大表直接 `ALTER TABLE` 有什么风险？
- Online DDL 是不是完全无锁？
- 上亿表加索引怎么做？

**答案**

大表 DDL 风险主要包括：

- 长时间占用元数据锁（MDL），阻塞读写或阻塞后续 DDL。
- 复制延迟放大，从库重放 DDL 时间长。
- 临时空间暴涨。
- DDL 失败回滚时间长。
- 加索引期间影响 Buffer Pool、磁盘 I/O 和线上查询。

安全做法：

1. 评估表大小、QPS、从库延迟、磁盘空间。
2. 低峰期执行，设置变更窗口。
3. 优先使用 Online DDL 能力，但不能误以为完全无锁。
4. 超大表使用 `pt-online-schema-change` 或 `gh-ost` 这类在线变更工具。
5. 先在从库或影子表验证耗时和风险。
6. 分批灰度，监控主从延迟、锁等待、错误率。
7. 准备回滚和降级方案。

**关键细节**

- Online DDL 也会涉及元数据锁，只是尽量缩短阻塞时间。
- 长事务会阻塞 DDL 获取 MDL。
- DDL 前要排查是否有长事务和慢查询。
- 对核心业务表，DDL 是变更流程，不是随手执行一条 SQL。

**面试表达**

> 大表 DDL 最大风险是元数据锁、主从延迟、临时空间和线上 I/O 抖动。我会先评估表大小和流量，低峰期执行，提前排查长事务；超大表优先用 gh-ost 或 pt-osc 做在线变更，并监控主从延迟、锁等待和错误率。Online DDL 不是完全无锁，核心表变更一定要有回滚和降级预案。

### Q16：为什么不建议在事务里做 RPC 或耗时操作？

**常见问法**

- 事务为什么要尽量短？
- 事务里调用外部接口有什么问题？
- 长事务会带来哪些风险？

**答案**

事务持续时间越长，持有锁、占用连接和保留版本链的时间就越长。事务里做 RPC、文件处理、复杂计算，会导致数据库资源被无意义占用。

长事务风险：

- 行锁持有时间变长，其他事务阻塞。
- 死锁概率上升。
- Undo Log 版本链变长，影响 MVCC 读取和 purge。
- 数据库连接占用时间变长，连接池耗尽。
- 主从复制延迟放大。
- 回滚成本高。

推荐做法：

- 事务内只做必要的数据库读写。
- 外部 RPC 放在事务前做校验，或事务提交后通过 MQ 异步通知。
- 状态变更使用本地事务保证，副作用用 MQ、任务补偿处理。
- 批处理拆小事务。

**关键细节**

- 事务不是包得越大越安全，边界越大风险越高。
- 微服务里不要试图用一个本地事务包住远程调用。
- 事务提交后再发 MQ 也有一致性风险，需要本地消息表或事务消息兜底。

**面试表达**

> 我会尽量让事务短而清晰，事务里只做必要的数据库修改，不放慢 RPC 或复杂计算。长事务会拉长锁持有时间，增加死锁、连接池耗尽、Undo 版本堆积和主从延迟风险。跨服务副作用更适合通过 MQ、补偿任务或本地消息表做最终一致。

### Q17：如何设计高并发下的库存扣减 SQL？

**常见问法**

- MySQL 如何防止库存扣成负数？
- 先查库存再扣库存有什么问题？
- 高并发扣减库存怎么写 SQL？

**答案**

不要先查库存再更新：

```text
SELECT stock FROM sku_stock WHERE sku_id = 1;
if stock >= num:
    UPDATE sku_stock SET stock = stock - num WHERE sku_id = 1;
```

这种 read-modify-write 在并发下容易超卖。更好的方式是用条件更新，让校验和扣减在一条 SQL 中完成：

```sql
UPDATE sku_stock
SET available_stock = available_stock - #{num}
WHERE sku_id = #{skuId}
  AND available_stock >= #{num};
```

根据影响行数判断是否扣减成功：

- 影响行数为 1：扣减成功。
- 影响行数为 0：库存不足或记录不存在。

**关键细节**

- `sku_id` 必须有索引，最好是主键或唯一索引，避免锁范围扩大。
- 扣减流水表要用业务单号做唯一索引，防止重复扣减。
- 热点库存行会有行锁竞争，秒杀场景更适合 Redis 预扣 + MQ 异步落库。
- 回补库存也要幂等，不能重复回补。

**面试表达**

> 普通交易场景我会用 MySQL 条件更新防止超卖，把库存校验和扣减合并到一条 SQL 中，通过影响行数判断结果。消费端或下游还要用订单号做扣减流水唯一索引防重复。对于秒杀这种热点库存，单行 MySQL 锁竞争会很严重，更适合 Redis Lua 预扣、MQ 削峰和异步落库。

### Q18：MySQL 高频追问速答表

| 问题 | 快速回答 |
| --- | --- |
| ACID 是什么？ | 事务的原子性、一致性、隔离性、持久性；InnoDB 通过 undo log、redo log、锁和 MVCC 等机制支撑。 |
| Buffer Pool 是什么？ | InnoDB 的内存缓存层，缓存数据页和索引页，查询先读内存，修改先改内存脏页，再异步刷盘。 |
| redo log 和 binlog 区别？ | redo 是 InnoDB 物理日志，保证 crash-safe；binlog 是 Server 层逻辑日志，负责复制和恢复。 |
| undo log 有什么用？ | 支持事务回滚，同时保存历史版本，支撑 MVCC 快照读。 |
| 两阶段提交解决什么？ | 保证 redo log 和 binlog 一致，避免主库崩溃恢复和从库复制结果不一致。 |
| 主从延迟怎么处理？ | 避免大事务，调并行复制，从库隔离重查询，强一致读走主库或等待位点追平。 |
| 死锁怎么排查？ | 用 `SHOW ENGINE INNODB STATUS` 看最近死锁，结合 SQL、锁等待和业务路径定位。 |
| 慢 SQL 怎么优化？ | 慢日志定位，EXPLAIN 看 type/key/rows/Extra，再判断索引、回表、排序、临时表、大分页。 |
| COUNT(*) 慢怎么办？ | 不强实时用统计表、缓存或异步任务维护；大表不要每次实时扫表。 |
| 深分页怎么优化？ | 优先游标分页；必须页码翻页可用延迟关联；复杂检索交给 ES 或限制最大翻页。 |
| 分库分表最大问题？ | 分片键、跨分片查询、全局 ID、分布式事务、扩容迁移、跨分片分页聚合。 |
| 大表 DDL 怎么做？ | 低峰期、查长事务、监控主从延迟；超大表用 gh-ost/pt-osc 等在线变更工具。 |
| 事务里能不能做 RPC？ | 不建议。会拉长锁持有时间，增加死锁、连接池耗尽、Undo 堆积和主从延迟风险。 |
