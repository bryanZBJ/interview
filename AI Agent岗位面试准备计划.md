# AI Agent 岗位面试准备计划

本文面向有 Java 后端经验、想面试 **AI Agent 工程师 / 智能体编排工程师 / 大模型应用工程师** 的候选人。重点不是从零训练模型，而是把大模型、RAG、工具调用、多 Agent 协作和业务系统集成起来，做成可上线、可评估、可观测、可控风险的业务智能体。

---

## 1. 两个岗位到底在招什么人？

从 JD 看，两个岗位都不是纯算法岗，也不是传统 Java CRUD 岗，而是更偏 **AI 应用落地 + 后端工程化 + 业务流程自动化**。

### 1.1. 共同关键词

| 关键词                             | 岗位真正关注点                             | Java 后端怎么理解               |
| ------------------------------- | ----------------------------------- | ------------------------- |
| LLM / 大模型                       | 理解模型能力边界、上下文窗口、幻觉、成本和延迟             | 把模型当成“不稳定但很强的推理服务”，需要工程约束 |
| Agent                           | 让模型围绕目标规划步骤、选择工具、观察结果并继续执行          | 类似“流程编排 + 策略决策 + 工具执行”的组合 |
| Function Calling / Tool Calling | 模型生成结构化调用意图，后端执行真实业务动作              | 后端负责接口封装、权限、参数校验、幂等和审计    |
| RAG                             | 用企业知识库、政策库、商品库、日志等材料增强回答            | 类似搜索系统 + Prompt 组装 + 引用来源 |
| 工作流编排                           | 把多步骤任务拆解、调度、汇总和校验                   | 类似状态机、任务流、异步任务、补偿机制       |
| 多 Agent 协作                      | Planner / Executor / Reviewer 等角色协同 | 类似把复杂任务拆成多个职责清晰的服务或模块     |
| 评估与观测                           | 看准确率、工具调用成功率、延迟、成本、失败率              | 类似线上服务的监控、告警、日志、回归测试      |
| 业务场景落地                          | 客服、运营、供应链、财务、自动化系统                  | 后端优势在理解业务对象、流程和系统集成       |

**一句话定位**

> 这类岗位要的是能把 LLM 做成生产系统的人：既懂 Prompt、RAG、Agent，又懂后端接口、数据库、缓存、异步任务、权限、安全、监控和业务流程。

### 1.2. 两个岗位的差异

| 岗位 | 更偏方向 | 面试重点 |
|---|---|---|
| 智能体编排工程师 | 通用大模型应用、自动化流程、多模态/数据分析 | Agent 规划、工作流编排、任务拆解、评估体系、Coze / LangChain / AutoGen |
| AI Agent 工程师（跨境电商品牌） | 跨境电商业务 Agent 落地 | ERP / WMS / CRM / 物流 / 支付 API 对接，商品、客服、供应链、运营、财务自动化 |

如果你是 Java 后端，第二类岗位更容易讲出优势，因为它强依赖业务系统集成、API 设计、数据一致性、权限和生产稳定性。

## 3. 必须掌握的技能地图

### 3.1. 第一层：大模型应用基础

必须会讲：

1. LLM 的输入输出机制、上下文窗口、token 成本。
2. temperature、top-p、top-k 的含义、关系及其对稳定性的影响。
3. 幻觉产生原因和工程控制手段。
4. 结构化输出：JSON schema、字段校验、失败重试。
5. 流式输出：提升交互体验，但后端仍要处理超时和中断。

面试高频问题：

> 大模型为什么不能直接连数据库执行操作？

回答：

> 因为模型本身不可信、不可审计，也不能绕过权限体系。正确做法是让模型只生成结构化意图和参数，后端工具服务负责权限校验、参数校验、幂等控制、真实执行和审计日志。

### 3.2. 第二层：Prompt 工程

必须会做：

1. 系统提示词：角色、目标、边界。
2. 任务提示词：输入、输出、步骤、约束。
3. few-shot：给模型示例，稳定输出风格。
4. 结构化输出：让模型输出 JSON，而不是自由文本。
5. Prompt 版本管理：不同场景、不同模型可回滚。

模板：

```text
你是一个跨境电商运营助手。
任务：根据商品信息、库存、评论和物流时效，生成运营建议。
约束：
1. 只能基于提供的数据回答。
2. 如果数据不足，明确说明缺少哪些信息。
3. 高风险动作只输出建议，不直接执行。
输出格式：
{
  "summary": "...",
  "risk": "...",
  "actions": [...]
}
```

### 3.3. 第三层：RAG 知识库

必须会讲完整链路：

```text
文档采集
  -> 解析清洗
  -> chunk 切分
  -> embedding 向量化
  -> 向量库 / 搜索引擎入库
  -> 用户问题向量化
  -> 召回相关片段
  -> 重排序
  -> Prompt 组装
  -> 大模型回答
  -> 引用来源 / 反馈 / 评估
```

关键考点：

| 考点 | 面试要点 |
|---|---|
| chunk 切分 | 不能过大也不能过小，要保留语义完整性 |
| 混合检索 | 向量检索适合语义，关键词检索适合编码、SKU、订单号、专有名词 |
| rerank | 对召回结果重新排序，提高上下文质量 |
| 权限过滤 | 检索前或检索后都要按用户权限过滤文档 |
| 引用来源 | 降低幻觉，方便用户追溯 |
| 评估集 | 用标准问题集评估召回率、准确率和无证据拒答 |

面试表达：

> RAG 不只是接一个向量库，而是文档解析、切分、向量化、召回、重排、上下文拼接、引用来源、权限过滤和效果评估的一整条链路。真正上线时我会重点关注检索命中率、答案准确率、幻觉率、延迟和 token 成本。

### 3.4. 第四层：Tool Calling / Function Calling

必须会设计工具：

```json
{
  "name": "query_inventory",
  "description": "查询指定 SKU 在指定仓库的可售库存",
  "parameters": {
    "type": "object",
    "properties": {
      "sku": { "type": "string" },
      "warehouseCode": { "type": "string" }
    },
    "required": ["sku", "warehouseCode"]
  }
}
```

后端执行流程：

```text
模型生成工具调用参数
  -> 参数 schema 校验
  -> 用户权限校验
  -> 业务幂等校验
  -> 调用 ERP / WMS / CRM / 物流 / 支付 API
  -> 记录审计日志
  -> 返回结构化结果
  -> 模型汇总解释
```

必须能讲的风险：

1. 模型可能选错工具。
2. 模型可能生成错误参数。
3. 工具可能超时、失败或返回脏数据。
4. 高风险动作不能自动执行，例如退款、改价、取消订单。
5. 所有真实动作要有权限、幂等、审计和人工确认。

### 3.5. 第五层：Agent 与工作流编排

Agent 基本循环：

```text
Goal
  -> Plan
  -> Act
  -> Observe
  -> Reflect / Validate
  -> Final Answer
```

生产系统常见架构：

| 架构 | 适用场景 | 优点 | 风险 |
|---|---|---|---|
| Workflow | 流程固定、风险高 | 可控、可测、可审计 | 灵活性低 |
| Single Agent | 路径不固定、工具较少 | 实现快、交互自然 | 稳定性不如工作流 |
| Multi-Agent | 复杂任务拆解 | 职责清晰，适合 Planner / Executor / Reviewer | 成本和不可控性更高 |

面试表达：

> 我不会为了 Agent 而 Agent。固定流程优先用 workflow，例如售后退款审批、报表生成、对账检查；路径不固定、需要动态选择工具时再用 Agent。高风险动作一定要加规则校验和人工确认。

### 3.6. 第六层：评估、观测和稳定性

必须会设计指标：

| 指标 | 说明 |
|---|---|
| 任务完成率 | Agent 是否完成用户目标 |
| 工具选择准确率 | 是否选对工具 |
| 参数正确率 | 工具入参是否完整、合法 |
| RAG 召回率 | 是否找到正确材料 |
| 答案准确率 | 输出是否符合事实和业务规则 |
| 幻觉率 | 是否编造不存在的信息 |
| 延迟 | 首 token 延迟、总耗时、工具调用耗时 |
| 成本 | token 消耗、模型调用次数、工具调用次数 |
| 失败率 | 模型失败、工具失败、解析失败、超时失败 |

线上日志至少记录：

1. 用户问题。
2. Prompt 版本。
3. 模型名称。
4. 检索片段和来源。
5. 工具调用名称、参数、结果和耗时。
6. token 消耗。
7. 最终输出。
8. 用户反馈。

## 4. 跨境电商 Agent 需要补的业务知识

如果面试跨境电商品牌岗位，建议至少能讲清这些业务对象：

| 业务域 | 需要理解什么 | 可做的 Agent 场景 |
|---|---|---|
| 商品 / Listing | SKU、SPU、标题、卖点、类目、价格、图片、库存 | Listing 生成、标题优化、卖点抽取 |
| 订单 | 下单、支付、发货、退款、售后 | 订单查询、异常订单解释 |
| 库存 / WMS | 可售库存、锁定库存、在途库存、仓库 | 库存预测、缺货预警、补货建议 |
| 物流 | 轨迹、时效、承运商、异常件 | 物流追踪、异常解释 |
| 客服 | 售前咨询、售后退款、评论分析 | 多语言客服、评论情绪分析 |
| 财务 | 对账、收款、退款、费用 | 对账自动化、异常账单识别 |
| 运营 | 广告、转化率、评论、竞品 | 评论总结、运营建议、报表解读 |

面试表达：

> 跨境电商 Agent 的难点不是只生成文案，而是要能接 ERP、WMS、CRM、物流和支付系统，把商品、订单、库存、客服、财务数据串起来。Agent 可以负责理解意图和生成建议，真正执行仍然由受控后端工具完成。

## 5. 8 周准备计划

### 第 1 周：补齐大模型基础

目标：

1. 理解 LLM、token、上下文窗口、temperature、top-p、top-k、幻觉。
2. 熟悉 OpenAI / Claude / 通义 / DeepSeek 等模型 API 的基本调用方式。
3. 能用 Java 或 Python 写一个最小聊天 Demo。

输出物：

1. 一个支持流式输出的聊天接口。
2. 一页笔记：大模型为什么需要工程约束。
3. 准备 5 个面试回答：LLM、token、上下文、幻觉、结构化输出。

### 第 2 周：Prompt 与结构化输出

目标：

1. 掌握 Prompt 模板、few-shot、输出格式约束。
2. 学会让模型输出 JSON，并做后端校验。
3. 学会失败重试和输出修复。

输出物：

1. 一个“商品 Listing 生成”Prompt。
2. 一个“客服回复生成”Prompt。
3. 一个 JSON schema 校验 Demo。

### 第 3 周：RAG 知识库

目标：

1. 跑通文档上传、切分、embedding、向量检索、回答生成。
2. 理解 chunk、topK、rerank、混合检索。
3. 能讲清 RAG 怎么降低幻觉。

输出物：

1. 一个企业知识库问答 Demo。
2. 一组 20 条测试问题和标准答案。
3. 一页 RAG 架构图和面试表达。

### 第 4 周：Tool Calling 与业务系统集成

目标：

1. 把订单、库存、物流、客服工单等接口封装成工具。
2. 实现工具 schema、参数校验、权限校验、审计日志。
3. 明确高风险动作的人工确认机制。

输出物：

1. 3 到 5 个业务工具，例如 `queryOrder`、`queryInventory`、`trackShipment`。
2. 一条完整调用链日志。
3. 一个“模型只生成意图，后端负责执行”的面试故事。

### 第 5 周：Agent 和工作流编排

目标：

1. 理解 Plan-Act-Observe。
2. 掌握 workflow 和 Agent 的取舍。
3. 重点掌握 Dify，能用 Coze 搭建可演示 Agent，了解 LangChain / LlamaIndex / AutoGen 的定位。

输出物：

1. 一个订单异常排查 Agent。
2. 一个固定流程 workflow，例如售后工单摘要和处理建议。
3. 一份 Agent vs Workflow 对比表。

### 第 6 周：多 Agent、记忆和上下文管理

目标：

1. 理解 Planner / Executor / Reviewer 分工。
2. 掌握短期上下文、长期记忆、用户画像、会话摘要。
3. 避免“无限记忆”导致污染和越权。

输出物：

1. 一个 Planner 拆解任务、Executor 调工具、Reviewer 校验结果的 Demo。
2. 一份记忆系统设计：存什么、不存什么、怎么过期、怎么审计。
3. 一组失败案例：记忆污染、上下文过长、工具误调用。

### 第 7 周：评估、观测和生产稳定性

目标：

1. 建立评估集，覆盖 RAG、工具调用、任务完成率。
2. 记录 token、延迟、成本、失败原因。
3. 设计降级、重试、限流、人工兜底。

输出物：

1. 一个评估脚本或评估表。
2. 一个监控指标清单。
3. 一套线上故障排查口径：模型慢、检索不准、工具失败、成本暴涨。

### 第 8 周：项目包装和模拟面试

目标：

1. 把 Demo 包装成可写简历、可讲架构、可演示的项目。
2. 准备跨境电商业务 Agent 方案。
3. 做 2 到 3 轮模拟面试。

输出物：

1. 简历项目描述。
2. 项目架构图。
3. 15 个高频问答。
4. 3 分钟项目介绍稿。

## 6. 建议准备的两个项目

### 项目 1：跨境电商运营 Agent

项目描述：

> 基于 Spring Boot + LLM + Tool Calling 构建跨境电商运营助手，支持自然语言查询商品、库存、订单、物流和评论数据，并生成 Listing 优化、库存预警、异常订单解释和运营建议。

核心模块：

1. Agent 网关：接收用户问题，做权限和租户校验。
2. Prompt 编排：按场景拼接系统提示词和业务上下文。
3. 工具服务：封装商品、订单、库存、物流、评论接口。
4. 执行控制：参数校验、幂等、超时、重试、人工确认。
5. 观测审计：记录工具调用、token、延迟、结果和反馈。

可讲亮点：

1. 模型只做意图识别和结果解释，真实业务操作由后端工具执行。
2. 工具调用前做权限、参数、幂等和风险等级校验。
3. 对退款、改价、取消订单等高风险动作只生成建议，必须人工确认。
4. 通过调用链日志定位模型选错工具、参数错误或业务接口失败。
5. 用测试集评估任务完成率和工具调用准确率。

### 项目 2：企业知识库 RAG + 智能客服

项目描述：

> 基于 Spring Boot + 向量数据库 + 大模型 API 构建企业知识库问答和智能客服系统，支持政策文档、商品说明、售后规则和历史工单检索，回答时返回引用来源，并对无依据问题拒答。

核心模块：

1. 文档入库：解析 PDF / Markdown / Word，切分 chunk。
2. 向量检索：embedding 入库，支持语义召回。
3. 混合检索：结合关键词检索解决 SKU、订单号、政策编号问题。
4. RAG 生成：拼接上下文，要求引用来源和无证据拒答。
5. 反馈评估：记录用户反馈，维护标准问题集。

可讲亮点：

1. chunk 切分保留标题层级，减少语义断裂。
2. 对专业术语、SKU、订单号使用关键词 + 向量混合检索。
3. Prompt 中强制基于检索材料回答，没有证据则拒答。
4. 对不同部门文档做权限过滤，防止越权检索。
5. 通过评估集持续观察召回率、准确率和幻觉率。

## 7. 高频面试问答

### Q1：Java 后端为什么适合转 AI Agent 工程师？

> 因为 Agent 落地不是只调模型 API，而是要接业务系统、设计工具接口、处理权限、幂等、异步任务、异常补偿、审计和监控。Java 后端在这些工程化能力上有优势，只需要补齐 LLM、RAG、Prompt、Tool Calling 和 Agent 编排能力。

### Q2：temperature、top-p、top-k 分别是什么？它们有什么关系？

> 这三个都是模型生成下一个 token 时的采样参数。temperature 控制概率分布的随机程度，值越低输出越稳定，值越高更有发散性；top-k 固定只保留概率最高的 K 个候选 token；top-p 则从高到低累计概率，只保留累计概率达到 p 的候选 token，因此候选数量是动态的。

可以把链路理解为：模型先计算下一个 token 的概率，temperature 先调整概率分布，再通过 top-k 或 top-p 缩小候选集合，最后从候选集合中采样生成结果。

**参数怎么用**

1. 知识库问答、代码生成、工具调用等需要稳定输出的场景，temperature 通常设为 `0` 到 `0.3`，并以较保守的 top-p 为主。
2. 文案生成、运营创意、头脑风暴等需要多样性的场景，可以适当提高 temperature 或 top-p。
3. 不建议把 temperature、top-p、top-k 都调得很激进，否则会放大随机性，导致回答不稳定、工具参数漂移。
4. Agent 调用工具时，除了降低随机性，还要使用 JSON Schema / Structured Output、参数校验和重试兜底，不能只依赖采样参数。

### Q3：Agent 和普通工作流有什么区别？

> 工作流适合步骤固定、规则明确、风险较高的场景，可控、可测、可审计；Agent 适合路径不固定、需要动态选择工具和持续观察结果的场景。生产系统里通常优先工作流，只有流程不确定且收益明显时才引入受控 Agent。

### Q4：Function Calling 的本质是什么？

> Function Calling 的本质不是让模型直接执行函数，而是让模型按 schema 生成工具调用意图和参数。后端拿到参数后做权限、参数、幂等、风控和审计校验，再调用真实业务系统。

### Q5：怎么避免 Agent 乱调用工具？

> 第一，工具描述要清晰，参数 schema 要严格；第二，后端必须做权限和参数校验；第三，高风险工具要人工确认；第四，限制工具范围和调用次数；第五，记录工具调用链路，用评估集持续测试工具选择准确率。

### Q6：RAG 为什么不是把文档直接塞给模型？

> 因为上下文窗口有限，全部塞进去成本高、延迟高，还会引入无关信息。RAG 的核心是先检索最相关的材料，再把少量高质量上下文交给模型回答，并返回引用来源。

### Q7：怎么评估一个 Agent 是否可上线？

> 不能只看 demo，要看任务完成率、工具选择准确率、参数正确率、RAG 召回率、答案准确率、幻觉率、延迟、成本、失败率和用户反馈。上线前要有评估集，线上要有日志、监控、告警和人工兜底。

### Q8：跨境电商 Agent 的难点是什么？

> 难点在于多系统、多语言、多业务对象和高风险动作。它要接 ERP、WMS、CRM、物流、支付等系统，还要理解商品、订单、库存、客服、财务和运营数据。模型可以负责理解和生成建议，但真实执行必须由后端受控工具完成。

### Q9：多 Agent 怎么设计？

> 我会先判断是否真的需要多 Agent。简单任务用单 Agent 或 workflow 即可。复杂任务可以拆成 Planner、Executor、Reviewer：Planner 负责任务拆解，Executor 负责工具调用，Reviewer 负责结果校验和风险检查。这样职责清晰，但也要控制调用成本和不可控性。

## 10. Coze / Dify 是否需要学习？

结论：**需要学，但学习深度不同。Dify 建议重点掌握，Coze 建议做到会用、会演示、会讲清适用场景。**

这两个岗位都提到 Coze、LangChain、AutoGen、工作流编排、RAG、Function Calling。面试官真正想确认的不是你会不会拖拽节点，而是你能不能把 Agent 从 Demo 做成业务系统：能接知识库、能调工具、能编排流程、能观测日志、能处理权限和异常。

### 10.1. 为什么建议优先学 Dify？

Dify 更适合作为你准备面试的主平台，因为它覆盖了 AI Agent 岗位里最常见的几类能力：

| 能力 | Dify 对应练习 | 面试价值 |
|---|---|---|
| Workflow / Chatflow | 编排多步骤任务、条件分支、变量传递 | 能讲清固定流程和 Agent 的取舍 |
| Knowledge / RAG | 上传业务文档、构建知识库、测试检索效果 | 能讲清 RAG 召回、引用、幻觉控制 |
| Tools | 接外部 API、Swagger、MCP 或自定义工具 | 能讲清 Function Calling 和业务系统集成 |
| Logs / Feedback | 查看对话、错误、耗时、token 和用户反馈 | 能讲清 Agent 评估、观测和持续优化 |
| API 发布 | 将应用接入自己的后端或前端 | 能体现 Java 后端集成能力 |

**学习目标**

1. 会创建一个 Chatflow / Workflow。
2. 会接一个知识库，并测试检索效果。
3. 会封装 2 到 3 个业务工具，例如查订单、查库存、查物流。
4. 会查看日志、错误、token、耗时和用户反馈。
5. 会把 Dify 应用通过 API 接入 Java 后端。

**面试表达**

> Dify 我会作为 AI Agent 原型和流程编排平台来使用。它适合快速验证 RAG、工具调用、工作流和日志观测。但真正生产落地时，我会把订单、库存、物流、工单等能力封装成后端工具服务，Agent 负责意图识别和流程编排，真实执行仍然经过权限、参数校验、幂等、审计和监控。

### 10.2. Coze 学到什么程度？

Coze 更适合做轻量 Agent Demo，尤其适合快速搭一个能展示的 Bot / Agent：

1. 会创建一个 Agent。
2. 会配置角色 Prompt。
3. 会接知识库。
4. 会接插件或工具。
5. 会搭一个简单工作流。
6. 会演示一个完整业务场景。

对你来说，Coze 不需要一开始深挖到底层实现。你要能用它做出一个“跨境电商运营助手”或“客服问答助手”，并讲清楚它适合快速验证想法，但生产系统仍然要回到后端工程化。

**面试表达**

> Coze 我会用来快速搭建 Agent Demo，比如商品 Listing 生成、评论分析、客服问答和运营建议。它的优势是上手快，适合验证业务流程；但如果涉及真实订单、库存、退款、改价等高风险动作，我不会让平台直接执行，而是通过后端工具服务做权限、风控、幂等和审计。

### 10.3. Coze、Dify、代码框架怎么取舍？

| 方案 | 适合场景 | 你在面试中怎么讲 |
|---|---|---|
| Coze | 快速做 Agent / Bot Demo，验证 Prompt、知识库和简单流程 | 适合原型验证和业务演示 |
| Dify | 做 RAG、Workflow、工具调用、API 集成和日志观测 | 适合面试项目主平台 |
| LangChain / LlamaIndex | 代码级 RAG、工具调用、链路编排 | 适合深入理解底层实现 |
| Spring AI | Java / Spring 生态接入模型、RAG 和工具调用 | 适合你后续做 Java 版生产 Demo |
| AutoGen / CrewAI | 多 Agent 协作实验 | 适合了解 Planner / Executor / Reviewer 思路 |

**一句话取舍**

> Coze 用来快速搭 Demo，Dify 用来系统化准备 RAG、Workflow、Tools 和日志观测，Spring AI / LangChain 用来理解代码级落地。面试时不要只说“我会某个平台”，而要强调平台背后的工程能力：工具封装、权限控制、幂等审计、日志观测、评估集和人工兜底。

### 10.6. 面试时最重要的边界感

3. 对 Java 开发来说，最值得争取的岗位不是纯算法岗，而是 `Java + AI 项目`、`AI 智能体平台后端`、`AI 应用开发`、`AI Agent 集成开发` 这类岗位。
4. Dify / Coze 更像“让面试官相信你真的做过 Agent 应用”的作品载体。它们能提高面试打开率，但不能替代 RAG、工具调用、接口集成、权限、幂等、审计、日志、评估这些工程能力。
5. 只会一点 Python 语法，直接面中高薪 AI Agent 开发岗难度较大；但如果目标是 AI 应用开发、AI Agent 集成开发、Java 后端 AI 方向岗位，可以通过 4 到 6 周补齐一个可演示项目来争取面试机会。

**对你的学习优先级调整**

1. 第一优先级：Python Web 基础，至少掌握 FastAPI 或 Flask，能写接口、调模型、接数据库。
2. 第二优先级：RAG 实战，包括文档清洗、分块、Embedding、向量库、召回、重排和答案引用。
3. 第三优先级：Function Calling / Tool Calling，把订单、库存、物流、工单等能力封装成工具。
4. 第四优先级：工作流编排，理解固定 Workflow、Agent 自主规划、多 Agent 协作的边界。
5. 第五优先级：Dify 重点学，Coze 会用即可；用它们做 Demo，但面试时要讲清楚底层工程方案。

**面试中可以这样说**

> 我看过厦门本地 AI Agent 岗位，企业真正需要的是把大模型接入业务系统：例如 ERP、WMS、CRM、MES、OA、物流、支付等。Coze 和 Dify 我会作为快速原型和流程编排工具来用，但我不会把它们当成全部能力。我的优势是 Java 后端和业务系统经验，我会把订单、库存、工单、服务履约这些能力封装成 Tool，由 Agent 做意图识别、RAG 检索和流程编排，最终执行仍然经过后端服务的权限、幂等、审计、日志和监控。
