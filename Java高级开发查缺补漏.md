## 2. Redis 底层与生产问题

### 必会知识点

- Redis 快的原因：内存操作、单线程避免锁竞争、IO 多路复用、简单高效的数据结构。
- 核心数据类型：String、Hash、List、Set、ZSet。
- 底层结构：SDS、dict、quicklist/listpack、skiplist。
- 过期策略：惰性删除、定期删除。
- 淘汰策略：LRU、LFU、TTL、随机淘汰、noeviction。
- 持久化：RDB、AOF、混合持久化。
- 高可用：主从复制、哨兵、Cluster。
- 生产问题：BigKey、HotKey、缓存击穿、缓存雪崩、缓存穿透、缓存一致性。

### 大厂高频面试题

#### Q1：Redis 为什么快？

**答案**

Redis 快主要因为：

1. 数据存储在内存中，避免磁盘 IO。
2. 单线程执行命令，避免多线程锁竞争和上下文切换。
3. 使用 IO 多路复用，可以一个线程处理多个连接。
4. 数据结构设计高效，例如 SDS、哈希表、跳表。
5. 网络协议简单，命令执行路径短。

**注意**

Redis 单线程主要指命令执行单线程。Redis 6 以后引入了多线程 IO，用于网络读写，但核心命令执行仍然保持单线程模型。

#### Q2：Redis 各个数据类型的底层实现是什么？

**一句话回答**

Redis 的数据类型是逻辑类型，底层会根据元素数量、元素大小和数据特征选择不同编码。面试里要区分 `type` 和 `encoding`：`type` 是 String、List、Hash、Set、ZSet 这类对外类型，`encoding` 才是内部实现，例如 `int`、`embstr`、`raw`、`listpack`、`hashtable`、`quicklist`、`intset`、`skiplist`。

**核心类型底层实现**

| 数据类型 | 常见底层编码 / 结构 | 适合场景 | 关键理解 |
| --- | --- | --- | --- |
| String | `int`、`embstr`、`raw`，底层核心是 SDS | 缓存值、计数器、分布式锁、库存数 | SDS 二进制安全、记录长度、支持动态扩容；小整数可用 `int` 编码，小字符串可用 `embstr`，大字符串用 `raw`。 |
| List | `quicklist`，节点内部通常是 listpack | 队列、最近列表、简单消息缓冲 | quicklist 可以理解成“双向链表 + 紧凑列表”，兼顾链表插入删除和 listpack 内存紧凑。 |
| Hash | 小对象用 listpack，大对象转 hashtable | 对象属性、用户信息、配置字段 | field/value 较少且较小时用紧凑结构省内存；字段多或值大时转哈希表，提高读写效率。 |
| Set | 整数小集合用 intset，否则用 hashtable | 去重、标签、交并差 | 元素全是整数且数量少时用 intset 省内存；出现非整数或元素变多后转哈希表。 |
| ZSet | 小集合用 listpack，大集合用 dict + skiplist | 排行榜、延迟队列、权重排序 | dict 用于按 member O(1) 查分数，skiplist 用于按 score 排序和范围查询。 |
| Bitmap | 本质是 String / SDS 的位操作 | 签到、在线状态、布尔标记 | 不是独立复杂结构，而是在 String 的字节数组上按 bit 读写。 |
| HyperLogLog | 稀疏 / 密集结构，密集结构约 12KB | UV 估算、去重计数 | 用概率算法估算基数，牺牲少量精度换固定小内存，不保存原始元素。 |
| Geo | 底层基于 ZSet | 附近的人、门店距离 | 把经纬度编码成 geohash 后作为 ZSet score，利用有序集合做范围查询。 |
| Stream | radix tree + listpack，消费组维护 PEL | 轻量消息队列、事件流 | 消息按 ID 追加，底层用 rax 索引消息块，listpack 紧凑保存消息内容。 |
| Pub/Sub | 订阅关系字典，不持久化消息 | 在线通知、广播 | Pub/Sub 不是持久化数据类型，消费者离线会丢消息，不能当可靠 MQ。 |

**版本口径**

- Redis 早期很多小对象用 `ziplist`，新版本更多使用 `listpack` 替代。
- 面试中可以说“老版本是 ziplist，新版本是 listpack”，避免被追问版本差异时答死。
- `OBJECT ENCODING key` 可以查看某个 key 当前实际编码。

**为什么要有多种编码**

Redis 不是一种类型固定一种结构，而是按数据规模自动切换：

1. 小数据优先节省内存，例如 listpack、intset。
2. 大数据优先保证查询和更新效率，例如 hashtable、skiplist。
3. 编码转换通常是单向的，例如小 Hash 转成 hashtable 后，不会因为字段减少再自动转回 listpack。

**典型追问**

**1. ZSet 为什么要同时用 dict 和 skiplist？**

- 只用 skiplist：可以按分数排序和范围查询，但按 member 查 score 不够快。
- 只用 dict：可以快速查 member，但无法高效做区间排序。
- 所以 Redis 大 ZSet 同时维护 dict 和 skiplist，写入时两边都更新，用空间换查询能力。

**2. List 为什么从 linkedlist / ziplist 演进到 quicklist？**

- linkedlist 每个节点都有指针，内存碎片和指针开销大。
- ziplist/listpack 内存紧凑，但连续内存太大时插入删除成本高。
- quicklist 把多个 listpack 串成链表，单个节点内部紧凑，整体又能高效头尾操作。

**3. Hash 什么时候会从 listpack 转 hashtable？**

当字段数量或单个 field/value 长度超过配置阈值时，会从 listpack 转为 hashtable。核心原因是：小对象用紧凑结构省内存，大对象用哈希表保证读写性能。

**面试表达**

> Redis 的数据类型和底层编码不是一一固定的。String 底层核心是 SDS，小整数可能用 int，小字符串用 embstr，大字符串用 raw；List 现在主要是 quicklist，节点里用 listpack；Hash 小的时候用 listpack，大了转 hashtable；Set 小整数集合用 intset，否则用 hashtable；ZSet 小的时候用 listpack，大了用 dict + skiplist。Bitmap 本质是 String 的位操作，Geo 底层是 ZSet，Stream 底层是 radix tree 加 listpack。Redis 这样设计是为了在小数据时省内存，大数据时保证查询和更新效率。

#### Q3：什么是 BigKey？有什么危害？怎么处理？

**答案**

BigKey 指 value 特别大或元素数量特别多的 key，例如一个 Hash 有几十万个 field，一个 String 有几十 MB。

危害：

- 网络传输慢，阻塞 Redis。
- 删除耗时长，可能造成主线程卡顿。
- 集群下造成数据倾斜。
- 迁移、持久化、主从同步成本高。

处理方式：

- 拆分 key，例如按业务 ID、日期、分桶拆分。
- 删除时使用 `UNLINK` 异步删除，避免 `DEL` 阻塞。
- 用 `SCAN` 渐进式扫描，不用 `KEYS`。
- 对 ZSet、Hash、List 做元素数量上限控制。

**追问：为什么删除大 key 推荐 UNLINK，而不是 DEL？**

`DEL` 是同步删除。它不仅会把 key 从 Redis 字典中删除，还会在 Redis 主线程里释放 value 占用的内存。如果 key 很大，例如大 Hash、大 List、大 Set、大 ZSet，释放内存可能耗时，导致 Redis 主线程卡顿。

`UNLINK` 是异步删除：

```text
1. 主线程先把 key 从 keyspace 中摘掉
2. 客户端立刻看不到这个 key
3. value 内存交给后台 lazy free 线程慢慢释放
```

所以 `UNLINK` 适合删除 BigKey，能降低大对象释放内存时对 Redis 主线程的阻塞。

命令示例：

```bash
UNLINK user:cache:1001
UNLINK key1 key2 key3
```

`UNLINK` 返回成功解绑的 key 数量。注意：执行后 key 对业务来说已经不存在，只是内存释放异步进行。

**DEL 和 UNLINK 区别**

| 命令 | 删除方式 | 是否可能阻塞主线程 | 适合场景 |
| --- | --- | --- | --- |
| `DEL` | 同步删除并释放内存 | 大 key 可能阻塞 | 小 key、普通删除 |
| `UNLINK` | 先摘除 key，后台释放内存 | 主线程阻塞更少 | BigKey、大集合、批量清理 |

**线上批量删除建议**

不要用 `KEYS prefix:*` 在线上查大范围 key，因为 `KEYS` 会全量扫描，可能阻塞 Redis。

推荐：

```text
SCAN 分批扫描 + UNLINK 分批删除
```

例如按前缀分批清理：

```bash
SCAN cursor MATCH user:cache:* COUNT 1000
UNLINK key1 key2 key3
```

Java / Spring Data Redis 中通常可以用：

```java
redisTemplate.unlink("user:cache:1001");
redisTemplate.unlink(keys);
```

Redis 也有 lazyfree 相关配置，例如：

```conf
lazyfree-lazy-user-del yes
```

开启后，用户执行 `DEL` 时也可以走类似 `UNLINK` 的异步释放逻辑。不过线上语义更清晰的做法是：删除大 key 时明确使用 `UNLINK`。

**面试表达**

> 删除 Redis 大 key 时不建议直接用 `DEL`，因为 `DEL` 会在主线程同步释放内存，大 Hash、大 List、大 ZSet 可能造成 Redis 卡顿。更推荐用 `UNLINK`，它会先把 key 从 keyspace 中摘掉，业务侧马上看不到这个 key，然后把内存释放交给后台 lazy free 线程处理。线上批量清理也不要用 `KEYS`，而是用 `SCAN` 分批扫描，再分批 `UNLINK`。

#### Q4：什么是 HotKey？怎么解决？

**答案**

HotKey 是被高频访问的 key，例如秒杀库存、热门商品、热门配置。

解决方式：

- 本地缓存，如 Caffeine，降低 Redis 压力。
- 热点 key 分片，例如 `stock:{skuId}:{bucket}`。
- 提前预热，避免冷启动瞬间打爆 Redis。
- 限流和降级，库存售罄后本地标记快速失败。
- 监控 Redis 节点 QPS 和命令耗时，发现热点后动态处理。

**结合项目表达**

> 秒杀或库存场景里，我不会把所有压力都压到一个 Redis key 上。可以用库存分桶、本地售罄标记、限流和预热一起做，核心目标是让热点流量在应用层和 Redis 层逐步被削掉。

#### Q5：RDB 和 AOF 有什么区别？

**答案**

RDB 和 AOF 都是 Redis 的持久化机制，用来在 Redis 重启后恢复数据。

一句话区别：

```text
RDB 保存某一时刻的数据快照
AOF 保存每一条写命令日志
```

**RDB 是什么？**

RDB（Redis Database）是快照持久化。它会在某个时间点，把 Redis 内存中的数据生成一个二进制快照文件：

```text
dump.rdb
```

可以理解成某一刻 Redis 全量数据的照片。

常见触发方式：

- `save`：主线程同步生成 RDB，会阻塞 Redis，不建议在线上手动使用。
- `bgsave`：fork 子进程生成 RDB，主线程继续处理请求，线上更常见。
- 配置规则自动触发，例如：

```conf
save 900 1
save 300 10
save 60 10000
```

含义：

```text
900 秒内至少 1 次修改 -> 触发 RDB
300 秒内至少 10 次修改 -> 触发 RDB
60 秒内至少 10000 次修改 -> 触发 RDB
```

RDB 优点：

- 文件紧凑，适合备份。
- 恢复速度快。
- 对主线程影响相对小，通常由 `bgsave` 子进程完成。
- 适合全量备份和灾难恢复。

RDB 缺点：

- 两次快照之间 Redis 宕机，可能丢数据。
- `bgsave` 需要 fork 子进程，大内存实例 fork 成本较高。
- fork 后如果写入频繁，会有 Copy-On-Write 内存开销。

**AOF 是什么？**

AOF（Append Only File）是追加日志持久化。它会把 Redis 执行过的写命令追加到 AOF 文件中：

```text
appendonly.aof
```

例如执行过：

```bash
SET name zhangsan
INCR counter
DEL cache:1
```

AOF 会记录对应的写命令。Redis 重启时，通过重放 AOF 中的写命令恢复数据。

**AOF 刷盘策略**

```conf
appendfsync always
appendfsync everysec
appendfsync no
```

| 策略 | 含义 | 数据安全 | 性能 |
| --- | --- | --- | --- |
| `always` | 每条写命令都 fsync | 最安全 | 最慢 |
| `everysec` | 每秒 fsync 一次 | 最多丢约 1 秒数据 | 线上常用 |
| `no` | 交给操作系统决定 | 可能丢更多 | 最快 |

线上最常见是：

```conf
appendfsync everysec
```

**AOF rewrite**

AOF 会不断追加，文件会越来越大，所以 Redis 有 AOF rewrite。rewrite 不是把旧 AOF 逐行压缩，而是根据当前内存数据生成一份新的最小命令集合。

例如原来 AOF 中有：

```bash
SET count 1
INCR count
INCR count
INCR count
```

重写后可以变成：

```bash
SET count 4
```

这样文件更小，恢复更快。

**RDB 和 AOF 对比**

| 对比 | RDB | AOF |
| --- | --- | --- |
| 持久化内容 | 某一时刻的数据快照 | 每条写命令日志 |
| 文件 | `dump.rdb` | `appendonly.aof` |
| 数据安全 | 两次快照之间可能丢数据 | `everysec` 最多丢约 1 秒 |
| 文件大小 | 较小 | 通常较大 |
| 恢复速度 | 快 | 相对慢，需要重放命令 |
| 适合场景 | 备份、全量恢复 | 更高数据安全要求 |
| 性能影响 | fork + COW 开销 | 持续追加写 + fsync 开销 |

**混合持久化**

Redis 4.0 后支持 AOF 混合持久化。AOF 重写后的文件可以分成两部分：

```text
前半部分：RDB 格式快照
后半部分：AOF 增量命令
```

好处是：

- RDB 部分让恢复更快。
- AOF 增量部分让数据更完整。

线上常见组合：

```text
开启 AOF everysec
开启 AOF rewrite
开启混合持久化
RDB 用于备份和灾难恢复
```

**面试表达**

> RDB 是快照持久化，保存某个时间点 Redis 的全量数据，文件小、恢复快，适合备份，但两次快照之间宕机会丢数据。AOF 是追加日志持久化，记录每条写命令，重启时重放命令恢复数据，数据安全性更好，常用 `everysec`，最多丢约 1 秒数据，但文件更大，恢复相对慢，需要 AOF rewrite 控制文件大小。线上一般会开启 AOF everysec 和混合持久化，RDB 更多用于备份和灾难恢复。

#### Q6：Redis 过期 key 是怎么删除的？

**答案**

Redis 采用惰性删除 + 定期删除。

- **惰性删除**：访问 key 时发现过期，再删除。
- **定期删除**：后台周期性抽样检查部分过期 key 并删除。

只靠惰性删除会导致过期 key 长期占用内存；只靠定期删除会消耗大量 CPU，所以 Redis 采用组合策略。

#### Q7：Redis 主从、哨兵、Cluster 有什么区别？

**答案**

- **主从复制**：主节点写，从节点读，提高读能力和容灾能力，但主节点故障需要额外机制切换。
- **哨兵**：负责监控主从节点，主节点故障时自动选主和故障转移。
- **Cluster**：通过 16384 个槽位做数据分片，支持水平扩展，每个分片内部可以有主从。

**面试表达**

> 主从解决读扩展和备份，哨兵解决自动故障转移，Cluster 解决容量和吞吐水平扩展。真正生产选择要看数据量、QPS、可用性要求和运维复杂度。

#### Q8：Redis 和数据库怎么保证一致性？

**答案**

主流是 Cache Aside：

1. 读请求先查缓存，未命中查 DB，再回填缓存。
2. 写请求先更新 DB，再删除缓存。
3. 删除缓存失败时，通过重试、MQ、Canal 订阅 Binlog 兜底。

不建议更新缓存，因为缓存可能是复杂聚合结果，并发写时容易乱序覆盖。

#### Q9：缓存穿透、缓存击穿、缓存雪崩分别是什么？怎么处理？

**答案**

- **缓存穿透**：查询的数据在 DB 中也不存在，导致请求每次都穿过缓存打到数据库。常见于恶意请求不存在的 ID。
- **缓存击穿**：某个热点 key 过期瞬间，大量并发请求同时打到 DB。
- **缓存雪崩**：大量 key 同时失效，或者 Redis 集群故障，导致大量流量直接压到 DB。

处理方案：

- **穿透**：参数校验、布隆过滤器、空值缓存。
- **击穿**：互斥锁、逻辑过期、热点 key 永不过期、异步刷新。
- **雪崩**：TTL 加随机值、缓存预热、多级缓存、限流降级、Redis 高可用。

**关键细节**

- 空值缓存 TTL 要短，避免缓存错误结果。
- 布隆过滤器有误判，但不会漏判，适合挡住大量不存在请求。
- 互斥锁要设置超时时间，避免锁泄漏。
- Redis 故障时要有降级策略，不能让所有流量直接冲垮 DB。

**面试表达**

> 穿透、击穿、雪崩分别对应“不存在数据反复查”“热点 key 失效”“大量 key 同时失效或 Redis 故障”。我的处理思路是：穿透用布隆过滤器和空值缓存，击穿用互斥锁或逻辑过期，雪崩用 TTL 随机化、预热、多级缓存和限流降级。

#### Q10：Redis 内存淘汰策略有哪些？线上怎么选？

**答案**

当 Redis 达到 `maxmemory` 限制时，会根据配置的淘汰策略处理写入。常见策略：

| 策略 | 含义 |
| --- | --- |
| `noeviction` | 不淘汰，写入新数据时报错 |
| `allkeys-lru` | 从所有 key 中淘汰最近最少使用的 key |
| `volatile-lru` | 只从设置了过期时间的 key 中淘汰最近最少使用的 key |
| `allkeys-lfu` | 从所有 key 中淘汰最少使用的 key |
| `volatile-lfu` | 只从设置了过期时间的 key 中淘汰最少使用的 key |
| `allkeys-random` | 从所有 key 中随机淘汰 |
| `volatile-random` | 只从设置了过期时间的 key 中随机淘汰 |
| `volatile-ttl` | 淘汰设置了过期时间且剩余 TTL 最短的 key |

线上选择：

- 纯缓存场景：常用 `allkeys-lru` 或 `allkeys-lfu`。
- 只允许淘汰带 TTL 的缓存：用 `volatile-lru`、`volatile-lfu`。
- 不能接受自动淘汰核心 key：使用 `noeviction`，但要做好内存告警和扩容。

**关键细节**

- 淘汰策略不是数据安全方案，被淘汰的数据不会自动恢复。
- 核心业务数据不要只放 Redis，必须有 DB 或其他持久化兜底。
- `allkeys-lfu` 对长期热点识别更好，`allkeys-lru` 对最近访问更敏感。
- 设置 `maxmemory` 后必须配套内存、淘汰次数、命令失败监控。

**面试表达**

> Redis 内存达到上限后会按淘汰策略释放 key。纯缓存场景我会倾向 `allkeys-lru` 或 `allkeys-lfu`；如果只有带 TTL 的数据允许被淘汰，就用 `volatile` 系列；如果核心 key 不能被淘汰，就用 `noeviction` 并配合容量告警。淘汰策略只能缓解内存压力，不能替代持久化和容量规划。

#### Q11：Redis 分布式锁怎么实现才安全？

**答案**

简单 Redis 锁通常用 `SET key value NX PX ttl`：

```text
SET lock:order:{orderId} requestId NX PX 30000
```

含义：

- `NX`：只有 key 不存在时才能设置成功。
- `PX`：设置过期时间，避免服务宕机后锁永久不释放。
- `value`：必须是唯一请求标识，释放锁时校验是不是自己持有的锁。

释放锁必须用 Lua 脚本保证“判断 value + 删除 key”的原子性：

```lua
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
```

生产中更常用 Redisson：

- 支持可重入锁。
- 支持看门狗自动续期。
- 支持阻塞等待，避免业务自己写自旋。
- 释放锁也由框架保证原子性。

**Java 项目中怎么使用 Redisson？**

Spring Boot 项目常用 starter：

```xml
<dependency>
    <groupId>org.redisson</groupId>
    <artifactId>redisson-spring-boot-starter</artifactId>
    <version>按项目 Spring Boot / Redisson 版本选择</version>
</dependency>
```

简单 Redis 配置可以走 Spring Boot 自动配置：

```yaml
spring:
  data:
    redis:
      host: 127.0.0.1
      port: 6379
```

也可以单独使用 `redisson.yaml`：

```yaml
singleServerConfig:
  address: "redis://127.0.0.1:6379"
  database: 0
lockWatchdogTimeout: 30000
```

再在 `application.yml` 中引用：

```yaml
spring:
  redisson:
    config: classpath:redisson.yaml
```

基础用法：

```java
@Service
public class OrderService {

    @Resource
    private RedissonClient redissonClient;

    public void createOrder(Long userId, Long skuId) {
        String lockKey = "order:create:" + userId + ":" + skuId;
        RLock lock = redissonClient.getLock(lockKey);

        lock.lock();
        try {
            // 1. 校验是否重复下单
            // 2. 校验库存
            // 3. 创建订单
            // 4. 扣减库存
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

`lock.lock()` 不指定过期时间时，Redisson 会启用看门狗自动续期。只要当前 JVM 进程还活着、业务线程还没有释放锁，Redisson 会自动延长锁 TTL，避免业务执行时间超过锁过期时间导致锁提前释放。

生产里更常用 `tryLock` 控制等待时间，避免线程一直阻塞：

```java
public void createOrder(Long userId, Long skuId) throws InterruptedException {
    String lockKey = "order:create:" + userId + ":" + skuId;
    RLock lock = redissonClient.getLock(lockKey);

    boolean locked = lock.tryLock(3, 10, TimeUnit.SECONDS);
    if (!locked) {
        throw new RuntimeException("系统繁忙，请稍后重试");
    }

    try {
        // 业务逻辑
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

这里：

```text
tryLock(3, 10, TimeUnit.SECONDS)
```

表示最多等待 3 秒获取锁，获取成功后锁 10 秒自动释放。

注意：指定 `leaseTime` 后，锁会按固定时间释放，通常不会依赖无限看门狗续期。如果业务耗时不确定，更适合用不指定 `leaseTime` 的 `lock()`，或按项目 Redisson 版本选择只指定等待时间的 `tryLock` 用法；如果业务耗时可控，用固定 `leaseTime` 更直接。

Redisson 可重入示例：

```java
RLock lock = redissonClient.getLock("test:lock");

lock.lock();
try {
    lock.lock();
    try {
        // 同一线程可以再次获得同一把锁
    } finally {
        lock.unlock();
    }
} finally {
    lock.unlock();
}
```

同一线程加了几次锁，就要释放几次。底层会记录线程标识和重入次数。

**关键细节**

- 加锁必须有过期时间。
- 解锁必须校验 value，不能误删别人的锁。
- 使用 Redisson 时，`unlock()` 必须放在 `finally`，解锁前建议判断 `isHeldByCurrentThread()`。
- 锁粒度要小，锁内不要放慢 RPC 或长事务。
- 锁只是并发控制，最终一致仍要靠数据库唯一索引、状态机或乐观锁兜底。
- Redis 主从异步复制下，主节点故障可能导致锁丢失，强一致场景要谨慎。

**面试表达**

> Redis 分布式锁基础写法是 `SET key value NX PX ttl`，释放时用 Lua 校验 value 后再删除，防止误删别人的锁。生产里我更倾向使用 Redisson，Java 项目中注入 `RedissonClient`，通过 `getLock(lockKey)` 获取 `RLock`，用 `tryLock` 控制等待时间，业务执行完在 `finally` 中 `unlock`。Redisson 支持可重入、看门狗续期和阻塞等待，但分布式锁不能替代最终一致保障，涉及库存、订单、支付这类场景，最后仍要有数据库唯一索引、状态机、幂等和补偿兜底。

#### Q12：Redis Lua 脚本有什么用？为什么库存扣减常用 Lua？

**答案**

Lua 脚本可以把多条 Redis 命令合并成一个原子操作执行。Redis 执行脚本时，不会被其他命令插入打断，因此适合处理“读取 -> 判断 -> 修改”的并发场景。

秒杀预扣通常不是单个 `DECR` 就够了，而是一组操作：

1. 查询库存是否大于 0。
2. 判断用户是否已经抢过，避免一人多单。
3. 扣减库存。
4. 记录用户抢购标记或预扣流水。

如果这些步骤放在 Java 里连续调用 Redis，例如先 `GET stock`、再 `SISMEMBER user`、再 `DECR stock`、最后 `SADD user`，多条命令之间可能被其他请求插入。高并发下，多个请求可能同时读到库存充足，然后一起扣减，或者重复下单校验还没写入就被第二个请求绕过。

Lua 的价值是把这组判断和修改放到 Redis 内部一次执行完，利用 Redis 单线程执行脚本的特性，保证脚本执行期间不会被其他命令打断。

库存扣减 + 一人一单示例：

```lua
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock <= 0 then
    return -1
end

if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
    return -2
end

redis.call('DECR', KEYS[1])
redis.call('SADD', KEYS[2], ARGV[1])
return 1
```

返回值可以约定为：`1` 表示预扣成功，`-1` 表示库存不足，`-2` 表示重复抢购。应用层只需要根据返回码决定后续是否发送 MQ 或提示失败。

**关键细节**

- Lua 解决的是 Redis 层“多命令组合”的原子性，不是分布式事务。
- 直接调用 Redis 命令适合单步原子操作，例如单纯 `DECR`；一旦涉及“先判断再修改”“多个 key 一起更新”，就更适合 Lua。
- Lua 可以减少多次网络往返，把判断逻辑放到 Redis 端完成，秒杀入口性能更稳定。
- Lua 脚本要尽量短，不能执行耗时逻辑，否则会阻塞 Redis 主线程。
- Cluster 模式下，脚本访问的多个 key 最好在同一个 hash slot，可用 `{}` hash tag 控制。
- Lua 保证 Redis 预扣原子，不保证订单最终创建成功；后续订单落库、支付超时、MQ 投递失败仍要靠库存流水、异步下单、补偿任务和对账兜底。

**面试表达**

> 高并发秒杀里用 Lua，不是因为 `DECR` 本身不原子，而是因为秒杀预扣通常包含库存判断、一人一单校验、扣库存和写抢购标记。把这些命令放在 Java 里分开调，中间可能被其他请求插入；放到 Lua 里可以在 Redis 内部一次执行完，避免并发穿插导致超卖或重复抢购。但 Lua 只保证 Redis 预扣这一段原子，后续订单落库、MQ 异步下单和库存回补还要通过幂等、补偿和对账保证最终一致。

#### Q13：Pipeline、事务和 Lua 有什么区别？

**答案**

| 机制 | 核心作用 | 是否原子 | 典型场景 |
| --- | --- | --- | --- |
| Pipeline | 批量发送命令，减少网络往返 | 否 | 批量写缓存、批量查询 |
| MULTI/EXEC 事务 | 顺序执行一组命令 | 单个 EXEC 执行期间不被插入，但不支持回滚语义 | 简单批量原子执行 |
| Lua 脚本 | 服务端执行脚本，支持判断和修改 | 是 | 库存扣减、限流、分布式锁释放 |

Pipeline 只是减少网络往返，不保证命令之间的业务原子性。Redis 事务没有数据库事务那种失败回滚能力，一条命令执行失败并不会自动回滚前面已执行的命令。Lua 可以包含条件判断，更适合复杂原子逻辑。

**关键细节**

- 批量提速用 Pipeline。
- 需要原子判断和修改，用 Lua。
- Redis 事务不是 MySQL 事务，不要误以为支持 ACID 回滚。
- 大 Pipeline 会占用内存和网络缓冲区，要控制批次大小。

**面试表达**

> Pipeline 解决的是性能问题，减少网络 RTT；事务解决的是一组命令顺序执行问题，但没有数据库事务那种回滚能力；Lua 更适合库存扣减、限流、解锁这类需要判断加修改的原子逻辑。面试时不要把 Pipeline 说成事务，也不要把 Redis 事务说成 MySQL 事务。

#### Q14：Redis Cluster 为什么是 16384 个槽？请求如何路由？

**答案**

Redis Cluster 把整个 key 空间划分为 16384 个 hash slot。每个 key 通过 CRC16 计算后对 16384 取模，得到所属槽位：

```text
slot = CRC16(key) % 16384
```

每个主节点负责一部分槽位。客户端访问 key 时，会根据 key 计算槽位并路由到对应节点。如果访问错节点，Redis 会返回 `MOVED` 或 `ASK` 重定向，客户端更新路由缓存后再次访问。

**关键细节**

- Cluster 支持水平扩容，扩容本质是迁移部分槽位到新节点。
- 多 key 操作要求 key 在同一个槽位，否则会报跨槽错误。
- 可以用 hash tag 控制槽位，例如 `order:{1001}:detail` 和 `stock:{1001}:lock` 都按 `{1001}` 计算槽位。
- Cluster 每个分片内部仍然可以做主从复制和故障转移。

**面试表达**

> Redis Cluster 通过 16384 个槽位做数据分片，key 根据 CRC16 取模映射到槽位，槽位再分配到不同主节点。客户端通常会缓存槽位路由，遇到 MOVED/ASK 重定向再更新。多 key 操作要注意同槽问题，必要时用 hash tag 把相关 key 放到同一个槽位。

#### Q15：Redis Stream 可以用来做什么？和 List、Pub/Sub 有什么区别？

**答案**

Redis Stream 是 Redis 提供的日志型消息结构，支持消息 ID、消费组、消费者、Pending 列表和 ack 机制。它可以用于轻量级消息队列、异步任务、小规模事件流。

对比：

| 结构 | 特点 | 局限 |
| --- | --- | --- |
| List | 简单队列，`LPUSH/BRPOP` 易用 | 不适合多消费者组和消息确认 |
| Pub/Sub | 发布订阅实时推送 | 消费者不在线会丢消息，没有持久消费进度 |
| Stream | 支持消费组、ack、Pending、历史消息 | 不是专业 MQ，复杂可靠性和堆积治理弱于 RocketMQ/Kafka |

**关键细节**

- Stream 可以用 `XADD` 写入消息。
- 消费组可以用 `XGROUP CREATE` 创建。
- 消费后需要 `XACK` 确认。
- 未确认消息会留在 Pending 中，可以做失败恢复。
- 大规模消息队列、跨服务核心事件仍建议用 RocketMQ/Kafka。

**面试表达**

> Redis Stream 比 List 更像消息队列，因为它支持消费组、ack 和 pending 消息；比 Pub/Sub 更可靠，因为消费者离线后仍能从历史消息继续消费。但它不适合替代专业 MQ，核心订单、支付、履约事件我还是会优先用 RocketMQ 或 Kafka。

#### Q16：如何用 Redis 做限流？

**答案**

常见限流模型：

- **固定窗口计数器**：每个时间窗口一个计数 key，超过阈值拒绝。
- **滑动窗口**：用 ZSet 记录请求时间戳，按时间范围统计。
- **令牌桶**：按固定速率生成令牌，请求消耗令牌。
- **漏桶**：请求按固定速率流出，削平突发流量。

简单固定窗口示例：

```text
INCR rate:user:{userId}:202606061200
EXPIRE rate:user:{userId}:202606061200 60
```

但 `INCR` 和 `EXPIRE` 分开执行可能出现并发或异常问题，更推荐用 Lua 保证原子性。

**关键细节**

- 固定窗口实现简单，但窗口边界可能有突刺。
- 滑动窗口更平滑，但 ZSet 成本更高。
- 限流 key 要设置 TTL，避免无限增长。
- 对高并发接口，限流要放在网关、应用和 Redis 多层配合。

**面试表达**

> Redis 限流可以用计数器、滑动窗口、令牌桶等方案。简单接口可以用 `INCR + EXPIRE`，但生产上最好用 Lua 保证自增和过期设置原子。固定窗口容易有边界突刺，滑动窗口更平滑但成本更高。核心接口通常会网关限流、应用限流和 Redis 限流组合使用。

#### Q17：Redis ZSet 常见业务场景有哪些？

**答案**

ZSet 同时具备 member 唯一和 score 排序能力，适合：

- 排行榜：销量榜、积分榜、热度榜。
- 延迟队列：score 存执行时间戳。
- 时间线：score 存发布时间。
- TopN 查询：按 score 取前 N。
- 范围查询：按分数区间统计。

排行榜示例：

```text
ZINCRBY product:rank:sales 1 skuId
ZREVRANGE product:rank:sales 0 9 WITHSCORES
```

延迟队列示例：

```text
ZADD delay:order-close 1893456000000 orderNo
ZRANGEBYSCORE delay:order-close 0 now LIMIT 0 100
```

**关键细节**

- ZSet 大 key 要控制 member 数量，可以按业务、时间、分桶拆分。
- 延迟队列轮询要避免多实例重复抢任务，可以用 Lua 或分布式锁。
- 关键延迟任务建议用 MQ 延迟消息或任务系统兜底，Redis ZSet 更适合轻量场景。

**面试表达**

> ZSet 的核心优势是唯一元素加分数排序，所以常用于排行榜、时间线和轻量延迟队列。排行榜可以用 `ZINCRBY` 更新分数，用 `ZREVRANGE` 查 TopN；延迟队列可以把执行时间作为 score。需要注意大 key、重复消费和可靠性，核心任务不能只依赖 Redis ZSet。

#### Q18：Redis 线上 P99 突然升高怎么排查？

**答案**

排查思路：

1. 看 Redis 实例 CPU、内存、网络、连接数。
2. 看慢命令日志 `SLOWLOG`，确认是否有大范围 `HGETALL`、`SMEMBERS`、`ZRANGE`、`KEYS`。
3. 查 BigKey、HotKey。
4. 看是否发生 AOF rewrite、RDB bgsave、主从全量同步。
5. 看是否有大量 key 同时过期。
6. 看客户端连接池是否耗尽或超时配置不合理。
7. Cluster 场景看是否有槽位迁移、节点故障或数据倾斜。

**关键细节**

- `KEYS`、大集合全量读取、BigKey 删除都可能阻塞主线程。
- 持久化和主从同步会带来额外 CPU、内存和 I/O 压力。
- 客户端连接池满也会表现为 Redis 慢。
- P99 问题要结合应用 trace，看慢在 Redis 服务端还是客户端等待。

**面试表达**

> Redis P99 升高我会先看实例 CPU、内存、网络和连接数，再查慢命令、BigKey、HotKey、持久化、主从同步和大量过期。很多时候不是 Redis 命令本身慢，而是客户端连接池耗尽、BigKey 阻塞、热点 key 打爆单节点或后台 rewrite 导致抖动。

#### Q19：Redis 高频追问速答表

| 问题 | 快速回答 |
| --- | --- |
| 缓存穿透怎么处理？ | 参数校验、布隆过滤器、空值缓存，空值 TTL 要短。 |
| 缓存击穿怎么处理？ | 热点 key 用互斥锁、逻辑过期、异步刷新或热点永不过期。 |
| 缓存雪崩怎么处理？ | TTL 随机化、预热、多级缓存、限流降级和 Redis 高可用。 |
| Redis 内存满了怎么办？ | 配置 `maxmemory` 和淘汰策略，监控内存和淘汰次数，必要时拆分和扩容。 |
| 分布式锁怎么安全释放？ | value 设唯一标识，Lua 校验 value 后删除，避免误删别人的锁。 |
| Lua 有什么用？ | 把多条 Redis 命令封装成原子操作，常用于库存扣减、限流和解锁。 |
| Pipeline 是事务吗？ | 不是。Pipeline 只是减少网络往返，不保证业务原子性。 |
| Redis 事务能回滚吗？ | 不像 MySQL 那样自动回滚，命令执行失败通常不会回滚已执行命令。 |
| Cluster 为什么有槽？ | 通过 16384 个 hash slot 做数据分片，key 根据 CRC16 映射到槽位。 |
| 多 key 跨槽怎么办？ | 尽量避免；必要时用 `{}` hash tag 让相关 key 落到同一槽。 |
| Stream 和 Pub/Sub 区别？ | Stream 支持消费组、ack 和历史消息；Pub/Sub 消费者离线会丢消息。 |
| Redis 做限流注意什么？ | 用 Lua 保证计数和过期原子，固定窗口简单但有边界突刺。 |
| ZSet 能做什么？ | 排行榜、TopN、时间线、轻量延迟队列。 |
| P99 升高怎么查？ | 看慢命令、BigKey、HotKey、持久化、主从同步、连接池和 Cluster 数据倾斜。 |

## 3. RocketMQ / Kafka 消息队列

### 必会知识点

- MQ 价值：削峰、异步、解耦、最终一致性。
- 消息可靠性：生产端、Broker、消费端三段保证。
- 消费语义：至少一次、最多一次、恰好一次。
- RocketMQ：事务消息、延迟消息、顺序消息、重试、死信队列。
- Kafka：Topic、Partition、Consumer Group、Offset、Replica、ISR、ACK、Rebalance。
- 生产问题：消息丢失、重复消费、消息乱序、消息积压。

### 大厂高频面试题

#### Q1：如何保证消息不丢？

**答案**

要分三段看：

1. **生产端**：同步发送或可靠异步发送，失败重试；关键消息落本地消息表或使用事务消息。
2. **Broker**：开启刷盘、主从复制、副本机制，避免单点故障。
3. **消费端**：业务处理成功后再 ACK；失败进入重试或死信；消费逻辑必须幂等。

**面试表达**

> 消息可靠性不能只说 MQ 本身可靠，要从发送、存储、消费三段保证。我们项目里的状态 MQ 或补偿消息，如果是关键事件，通常要有业务主键、traceId、重试和人工补偿入口。

#### Q2：消息重复消费怎么处理？

**答案**

MQ 通常只能保证至少一次投递，所以重复消费是正常现象。消费端必须幂等。

常见方案：

- 消费记录表：以 `messageId` 或业务主键建立唯一索引。
- 业务唯一键：例如订单号、支付流水号、工单 ID + 事件类型。
- 状态机约束：只允许从合法状态流转到下一个状态。
- 幂等写 SQL：例如 `UPDATE ... WHERE status = ?`。

#### Q3：消息乱序怎么处理？

**答案**

先判断是否真的需要全局有序。大多数业务只需要同一个业务主键内有序。

方案：

- 同一订单或同一工单的消息发送到同一个队列/分区。
- 消费端单线程或按 key 串行消费。
- 消息体带版本号或状态，消费时校验当前状态。
- 对乱序消息可以丢弃、延迟重试或进入补偿。

**结合项目表达**

> 工单状态事件不一定需要全局有序，但同一个工单内部必须防止状态倒退。我会用状态机和条件更新做最终保护，即使 MQ 乱序到达，也不能覆盖更晚的状态。

#### Q4：RocketMQ 事务消息原理是什么？

**答案**

RocketMQ 事务消息分三步：

1. 生产者先发送半消息，Broker 保存但不投递给消费者。
2. 生产者执行本地事务。
3. 根据本地事务结果提交或回滚半消息。

如果 Broker 长时间没有收到提交/回滚结果，会回查生产者本地事务状态。

适用于“本地事务成功后必须发消息”的场景，例如订单创建成功后必须通知库存或履约。

#### Q5：延迟消息一定准时吗？

**答案**

不一定。延迟消息只能作为触发器，不能作为绝对定时器。Broker 积压、消费端慢、系统时间误差都可能导致延迟。

所以订单关单类场景要在消费延迟消息时再次查询订单状态，只有仍为待支付才允许关闭。还要有定时任务兜底扫描，防止延迟消息丢失或消费失败。

#### Q6：Kafka 为什么吞吐高？

**答案**

Kafka 吞吐高主要因为：

- 顺序写磁盘，减少随机 IO。
- Page Cache 利用操作系统缓存。
- 零拷贝降低数据从内核态到用户态的拷贝成本。
- 分区机制支持并行读写。
- 批量发送和压缩降低网络开销。

#### Q7：Kafka Rebalance 有什么问题？

**答案**

Rebalance 是消费者组内分区重新分配。当消费者数量变化、订阅 Topic 变化、心跳超时等情况发生时会触发。

问题：

- Rebalance 期间部分分区会暂停消费。
- 频繁 Rebalance 会导致消费抖动和积压。
- 如果消费耗时过长，可能被误判为失活。

优化：

- 控制单条消息处理耗时。
- 合理配置 `max.poll.interval.ms`、`session.timeout.ms`。
- 避免频繁扩缩容消费者。
- 消费逻辑和业务处理解耦，先拉取再异步处理时要谨慎管理 offset。

#### Q8：消息积压怎么处理？

**答案**

先定位是生产突然放量、消费者异常、下游慢，还是单条消息处理过慢。

处理方式：

- 临时扩容消费者实例。
- 增加 Topic 队列/分区数量，提升并行度。
- 慢逻辑异步化或批量化。
- 对失败消息隔离，不要让少量异常消息阻塞整个队列。
- 如果积压极大，可以临时转储到新 Topic 或离线处理。

## 4. 微服务治理与可观测性

### 必会知识点

- 注册发现：服务提供者注册，消费者订阅，注册中心推送变更。
- RPC 核心：序列化、网络通信、代理调用、负载均衡、超时重试。
- 服务治理：限流、熔断、降级、隔离、灰度、配置中心。
- 可观测性：日志、指标、链路追踪。
- 常见指标：QPS、RT、P95/P99、错误率、CPU、内存、GC、线程池、连接池。
- 线上排查：traceId 定位链路，日志看业务分支，指标看瓶颈层，dump 看 JVM 状态。

### 大厂高频面试题

#### Q1：一次 RPC 调用大概经历哪些过程？

**答案**

一次 RPC 调用通常包括：

1. 客户端通过代理对象发起调用。
2. 根据服务名从注册中心获取可用 provider 列表。
3. 通过负载均衡选择一个 provider。
4. 将请求参数序列化。
5. 通过网络连接发送请求。
6. 服务端反序列化并调用真实业务方法。
7. 服务端序列化响应并返回。
8. 客户端反序列化结果，返回给调用方。

过程中还会涉及超时、重试、过滤器、限流、熔断、链路追踪等能力。

#### Q2：RPC 重试有什么风险？

**答案**

重试可以提升可用性，但也有风险：

- 非幂等接口重试可能导致重复写，例如重复下单、重复扣库存。
- 下游已经慢了，重试会放大流量，造成雪崩。
- 超时不代表下游没有执行成功，可能只是响应没回来。

所以写接口要谨慎重试，必须有幂等保护。查询接口可以适当重试，但也要控制次数和超时时间。

#### Q3：限流、熔断、降级有什么区别？

**答案**

- **限流**：控制进入系统的流量，防止超过承载能力。
- **熔断**：下游错误率或超时升高时，快速失败，避免调用方持续等待。
- **降级**：非核心能力返回默认值或简化结果，保护核心链路。

**面试表达**

> 限流是入口控量，熔断是故障隔离，降级是业务取舍。比如下单链路里，库存和工单创建是核心，推荐、统计、部分运营标签可以降级或异步处理。

#### Q4：如何设计一个可观测的后端系统？

**答案**

需要三类信息：

1. **日志**：关键业务节点、入参摘要、结果、异常、traceId。
2. **指标**：QPS、RT、P99、错误率、线程池、连接池、队列长度、MQ 积压。
3. **链路追踪**：跨服务调用耗时，定位慢在网关、应用、DB、Redis、MQ 还是下游 RPC。

日志要避免打印敏感信息，指标要有告警阈值，traceId 要贯穿整个请求链路。

#### Q5：P99 突然升高怎么排查？

**答案**

P99 升高说明少量慢请求变多，排查顺序：

1. 看是否有流量突增或热点请求。
2. 看 DB 慢 SQL、锁等待、连接池耗尽。
3. 看 Redis 慢命令、HotKey、BigKey。
4. 看 RPC 下游超时、重试、熔断情况。
5. 看线程池队列、拒绝策略、任务积压。
6. 看 GC 是否频繁，CPU 是否飙高。

**结合项目表达**

> 我会先通过监控和 traceId 找到慢在哪一层，再进入具体层排查。比如工单分配慢，要区分是规则计算慢、库存 RPC 慢、DB 查询慢，还是异步任务线程池积压。

#### Q6：配置中心 Apollo 在生产中有什么价值？

**答案**

配置中心的价值：

- 动态开关：紧急关闭某个功能或任务。
- 灰度发布：按环境、集群、机器或业务维度逐步放量。
- 参数治理：限流阈值、线程池参数、任务开关不需要重新发版。
- 故障恢复：线上异常时快速回滚配置。

注意配置变更要有审计记录，关键参数要有范围校验，避免配置错误直接打爆系统。

## 5. MySQL SQL 优化与日志体系

### 必会知识点

- `EXPLAIN`：type、key、rows、Extra 是高频重点。
- SQL 优化：索引设计、覆盖索引、避免回表、减少排序、延迟关联、分页优化。
- InnoDB 日志：Redo Log、Undo Log、Binlog。
- 事务提交：Redo Log 和 Binlog 两阶段提交。
- 主从复制：Binlog dump、relay log、SQL thread；关注主从延迟。
- 死锁排查：锁等待、事务持锁顺序、索引缺失导致锁范围扩大。

### 大厂高频面试题

#### Q1：`EXPLAIN` 主要看哪些字段？

**答案**

重点看：

- **type**：访问类型，常见从好到坏为 `const`、`eq_ref`、`ref`、`range`、`index`、`ALL`。
- **key**：实际使用的索引。
- **rows**：预估扫描行数。
- **Extra**：是否出现 `Using filesort`、`Using temporary`、`Using index` 等。

如果 type 是 `ALL`，rows 很大，Extra 里还有 filesort 或 temporary，通常说明 SQL 需要重点优化。

#### Q2：慢 SQL 怎么优化？

**答案**

常见步骤：

1. 用慢日志定位 SQL。
2. 用 `EXPLAIN` 看执行计划。
3. 判断是否命中合适索引。
4. 看是否回表过多，能否用覆盖索引。
5. 看是否有排序、临时表、函数计算、隐式转换。
6. 大分页场景用游标分页、延迟关联或搜索引擎。
7. 优化后用真实数据量验证，不只看小数据集。

#### Q3：Redo Log、Undo Log、Binlog 分别做什么？

**答案**

- **Redo Log**：InnoDB 的重做日志，保证事务提交后的持久性，崩溃后可以恢复已提交数据。
- **Undo Log**：回滚日志，用于事务回滚和 MVCC 版本链。
- **Binlog**：MySQL Server 层日志，用于主从复制和数据恢复。

Redo Log 偏向保证崩溃恢复，Undo Log 偏向保证回滚和快照读，Binlog 偏向复制和归档。

#### Q4：为什么需要两阶段提交？

**答案**

事务提交时需要保证 Redo Log 和 Binlog 一致，否则会出现主库恢复数据和从库复制数据不一致。

两阶段提交大致过程：

1. Redo Log 写入 prepare 状态。
2. 写 Binlog。
3. Redo Log 写入 commit 状态。

崩溃恢复时，可以根据 Redo Log 和 Binlog 的状态判断事务是否应该提交。

#### Q5：主从延迟怎么产生？怎么解决？

**答案**

主从延迟常见原因：

- 主库写入压力大，从库回放慢。
- 大事务或大批量更新。
- 从库 SQL 单线程回放瓶颈。
- 从库机器性能差。
- 网络延迟。

解决方式：

- 避免大事务，拆成小批次。
- 开启并行复制。
- 读请求按业务容忍度选择主库或从库。
- 对强一致读使用主库读或读写一致性路由。
- 监控 seconds behind master 或复制延迟指标。

#### Q6：死锁怎么排查和避免？

**答案**

排查：

- 查看 `show engine innodb status` 中的最近死锁信息。
- 找出两个事务分别持有哪些锁、等待哪些锁。
- 分析 SQL 是否缺索引导致锁范围扩大。

避免：

- 多个事务按固定顺序访问资源。
- 尽量使用合适索引，减少锁范围。
- 控制事务大小和持续时间。
- 热点更新可考虑队列化、分桶、乐观锁。

## 6. 设计模式与项目表达

### 必会知识点

高级岗问设计模式，通常不是让你背定义，而是看你能否把设计模式落到项目复杂度里。

- 策略模式：不同业务规则可替换。
- 责任链模式：多个规则按顺序执行，命中即返回或继续传递。
- 工厂模式：根据场景创建不同策略或规则。
- 模板方法：固定流程 + 可变步骤。
- 状态机：限制状态合法流转。
- 观察者/发布订阅：状态变化后通知下游。
- 装饰器/代理：增强原有能力，例如日志、事务、鉴权。

### 大厂高频面试题

#### Q1：你项目中用过哪些设计模式？

**答案**

可以结合你的履约项目这样回答：

- 工单分配规则适合讲策略模式和责任链模式。
- `AppointCalculateRuleFactory` 适合讲工厂模式。
- 服务工单和库存 token 状态流转适合讲状态机。
- XXL-Job 补偿任务适合讲模板方法：扫描、过滤、执行、记录结果、失败重试。
- MQ 状态事件适合讲观察者或发布订阅。

**面试表达**

> 我不会为了套设计模式而设计模式，而是在规则多、状态多、扩展多的地方使用。比如工单分配有固定关系、库存优先、当前站点、兜底分发等规则，如果都写在一个大 if else 里，很难维护。所以我会通过规则工厂和规则链把规则拆开，让每个规则只关心自己的命中条件和输出结果。

#### Q2：责任链和策略模式有什么区别？

**答案**

- **策略模式**：多个算法或规则中选择一个执行，重点是“可替换”。
- **责任链模式**：多个处理器按顺序传递，某个处理器可以处理、跳过或继续传递，重点是“链式处理”。

在工单分配中，一个具体规则可以是策略；多个规则按顺序执行组成责任链。

#### Q3：状态机比 if else 好在哪里？

**答案**

状态机的优势：

- 状态、事件、流转关系清晰。
- 可以防止非法状态跳转。
- 方便记录状态流水。
- 适合和 MQ 事件、补偿任务结合。
- 后续新增状态或事件时，影响范围更可控。

**结合项目表达**

> 像服务工单、订单、库存 token 这类对象，不只是一个状态字段，而是一套状态流转规则。状态机可以把“当前状态 + 事件 -> 下一个状态”显式定义出来，再配合 DB 条件更新，能同时保证业务清晰和并发安全。

#### Q4：如何避免一个复杂业务方法越来越长？

**答案**

可以从四个方向拆：

1. **按业务阶段拆**：参数校验、上下文构建、规则计算、结果持久化、事件发送。
2. **按规则拆**：每个规则一个类，只负责自己的判断。
3. **按副作用拆**：DB 更新、RPC 调用、MQ 发送、日志记录分层处理。
4. **按异常恢复拆**：主流程和补偿流程不要混在一起。

高级岗表达重点是“拆完之后更容易测试、排查、扩展”，不是简单把代码分散到多个类。

## 7. 项目量化与追问准备

### 必会准备项

面试高级 Java，项目讲得再复杂，也要准备量化结果。建议你把以下数据补齐：

- 系统支撑的城市数、站点数、服务人员规模。
- 日均工单量、峰值工单量、核心接口 QPS。
- 工单分配成功率、库存预占成功率。
- 关键接口平均耗时、P95、P99。
- 定时任务处理数据量、耗时变化。
- 补偿任务减少了多少人工排查或修复成本。
- 某个线上问题修复前后的影响对比。

### 大厂高频面试题

#### Q1：你负责的项目有什么技术难点？

**答案模板**

> 我负责的难点主要不是单个接口开发，而是跨服务链路闭环。以到家服务履约为例，一次工单分配会涉及人员能力、服务区域、服务时间、站点、库存组、库存 token、工单主表和预约 JSON。难点在于接口返回成功不代表最终一致，必须继续确认库存 token 真实落库、工单站点回写、状态事件和补偿任务是否都闭环。

#### Q2：你做过最有价值的一次线上问题排查是什么？

**答案模板**

> 我会按 traceId 先定位请求链路，再看规则分支、下游 RPC、DB 落库和 MQ 事件。比如“有库存但没有命中库存规则”这类问题，不能只看库存接口返回，还要看候选站点是否在城市上下文里、库存组 ID 是否正确透传、规则过滤后是否被兜底逻辑覆盖。最后修复时不仅改代码，还要补日志、补监控或补偿入口。

#### Q3：如何证明你的优化有效？

**答案模板**

> 我一般会从三个维度证明：第一是业务指标，例如分配成功率、预占成功率、异常单量下降；第二是性能指标，例如接口 P99、任务耗时、SQL 扫描行数下降；第三是运维指标，例如补偿任务可追踪、人工修复耗时下降、问题复现和定位时间变短。没有数据的优化很难说服面试官。

#### Q4：如果让你重新设计这个系统，你会改哪里？

**答案模板**

> 我会优先增强三点：第一是关键链路的可观测性，把规则命中、库存请求、token 结果、工单回写都串到同一个 traceId；第二是状态和补偿任务的标准化，让异常恢复有统一任务状态、重试次数和失败原因；第三是关键字段语义治理，例如 `siteId/groupId/inventoryGroupId` 这种跨服务字段，统一文档和校验，减少误用。

## 9. 高频追问速记

| 问题 | 快速回答 |
| --- | --- |
| JVM CPU 飙高怎么查？ | `top` 找进程，`top -Hp` 找线程，线程 ID 转 16 进制，`jstack` 定位代码栈。 |
| 频繁 Full GC 怎么查？ | 看 GC 日志、堆使用趋势、dump 分析对象引用链，重点排查泄漏、大对象、晋升过快。 |
| Redis 为什么快？ | 内存操作、单线程命令执行、IO 多路复用、高效数据结构。 |
| BigKey 怎么处理？ | 拆 key、限制元素数、`UNLINK` 删除、`SCAN` 渐进扫描、监控预警。 |
| HotKey 怎么处理？ | 本地缓存、key 分片、预热、限流、售罄标记、降级。 |
| MQ 重复消费怎么办？ | 消费端幂等，唯一索引、消费记录表、状态机和条件更新。 |
| 消息乱序怎么办？ | 按业务 key 保序，状态机防状态倒退，乱序消息延迟或补偿。 |
| RPC 重试有什么风险？ | 非幂等写接口可能重复执行，下游慢时会放大流量。 |
| P99 升高怎么查？ | 先看 trace 和监控，定位 DB、Redis、RPC、线程池、GC 哪一层慢。 |
| 慢 SQL 怎么优化？ | 慢日志定位，`EXPLAIN` 分析，索引、覆盖索引、分页、排序、临时表逐项处理。 |
| 两阶段提交解决什么？ | 保证 Redo Log 和 Binlog 一致，避免崩溃恢复和主从复制数据不一致。 |
| 项目怎么体现高级能力？ | 讲跨服务闭环、幂等补偿、可观测性、性能指标、线上问题治理和量化结果。 |
