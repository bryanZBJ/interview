# 微服务治理面试指南

这份材料补齐 Java 后端大厂面试里常见的微服务治理专题，重点覆盖 Sentinel、OpenFeign/Feign、Hystrix、Resilience4j、Spring Cloud LoadBalancer、Spring Cloud Gateway、超时重试、限流熔断降级、线程隔离、开放网关、灰度发布和线上排障。回答时不要只背组件名，要讲清楚“为什么需要治理 -> 治理什么风险 -> 组件如何落地 -> 生产如何兜底”。

## 0. 高频知识地图

| 主题           | 面试常问点                                 | 回答关键词                                                        |

## 0.1. 常见治理组件怎么使用？

这一节按“没实际用过也能快速建立手感”的方式整理。面试里不用背完整配置，但要能说清楚：加什么依赖、写什么注解/配置、保护什么资源、失败后怎么降级、生产规则怎么管理。

### 0.1.1. OpenFeign 怎么用？

**解决什么问题**：把 HTTP 调用写成 Java 接口，减少手写 `RestTemplate` / `WebClient` 请求代码。

**基本用法**：

```java
@SpringBootApplication
@EnableFeignClients
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}

@FeignClient(name = "inventory-service", path = "/inventory")
public interface InventoryClient {

    @GetMapping("/stock/{skuId}")
    StockDTO queryStock(@PathVariable("skuId") Long skuId);
}
```

**常用配置**：

```yaml
spring:
  cloud:
    openfeign:
      client:
        config:
          inventory-service:
            connectTimeout: 1000
            readTimeout: 3000
            loggerLevel: basic
```

**生产注意**：

- `name` 通常对应注册中心里的服务名，配合 Spring Cloud LoadBalancer 选实例。
- 连接超时和读取超时必须配置，不能用默认值硬扛生产流量。
- 写接口谨慎重试，必须先有幂等键、唯一索引或状态机兜底。
- 透传 `traceId`、`tenantId`、用户态、灰度标识通常用 `RequestInterceptor`。

**面试表达**：

> OpenFeign 的用法就是启动类加 `@EnableFeignClients`，远程服务定义成 `@FeignClient` 接口，方法上写 Spring MVC 注解。真正生产落地时，重点不是会写接口，而是要配好超时、日志、连接池、鉴权头透传和降级策略。

### 0.1.2. Spring Cloud LoadBalancer 怎么用？

**解决什么问题**：客户端拿到服务实例列表后，在调用方进程内选择一个实例发请求。

**常见使用方式**：

- 和 OpenFeign 配合：`@FeignClient(name = "inventory-service")` 里的服务名会走负载均衡。
- 和 `RestTemplate` 配合：给 `RestTemplate` Bean 加 `@LoadBalanced`。
- 和 `WebClient` 配合：给 `WebClient.Builder` Bean 加 `@LoadBalanced`。

```java
@Bean
@LoadBalanced
public RestTemplate restTemplate() {
    return new RestTemplate();
}

// 调用时用服务名，而不是写死 IP
restTemplate.getForObject(
    "http://inventory-service/inventory/stock/1001",
    StockDTO.class
);
```

**生产注意**：

- 负载均衡只解决“选哪台实例”，不解决下游慢、超时、熔断、重试风暴。
- 灰度发布时要结合版本标签、Header 或元数据路由。
- 慢实例即使没宕机，也会拖高 P99，需要摘流、降权或熔断配合。

**面试表达**：

> LoadBalancer 是客户端负载均衡。调用方从注册中心拿到实例列表，在本地按轮询、随机或自定义策略选实例。它解决的是选实例问题，不能替代超时、熔断和限流。

### 0.1.3. Sentinel 怎么用？

**解决什么问题**：对接口、方法、热点参数等资源做限流、熔断降级、系统保护，防止流量突刺或慢下游拖垮服务。

**最小使用方式**：

```java
Entry entry = null;
try {
    entry = SphU.entry("queryStock");
    // 被保护的业务逻辑
    return stockService.queryStock(skuId);
} catch (BlockException ex) {
    // 被限流或熔断后的处理
    return StockDTO.empty();
} finally {
    if (entry != null) {
        entry.exit();
    }
}
```

**注解方式**：

```java
@SentinelResource(
    value = "queryStock",
    blockHandler = "queryStockBlocked",
    fallback = "queryStockFallback"
)
public StockDTO queryStock(Long skuId) {
    return stockService.queryStock(skuId);
}
```

**规则怎么来**：

- 本地代码加载规则：适合 demo 或简单验证。
- Sentinel Dashboard 配规则：适合观察和临时调整。
- Nacos/Apollo 等配置中心持久化规则：适合生产，有版本、灰度、回滚和审计。

**生产注意**：

- `blockHandler` 处理限流、熔断这类 Sentinel 拦截；`fallback` 处理业务异常降级。
- 资源名要稳定，规则要可审计，不能只在 Dashboard 临时改完就丢。
- 阈值来自压测和线上指标，不是拍脑袋。

**面试表达**：

> Sentinel 的使用思路是先定义资源，再配置规则。资源可以是接口、方法或热点参数，规则可以按 QPS、线程数、慢调用比例、异常比例等触发。生产上规则最好接配置中心持久化，配合监控和告警，而不是只在控制台临时改。

### 0.1.4. Hystrix 怎么用？

**解决什么问题**：老 Spring Cloud Netflix 体系里，用 Command 模型做依赖隔离、熔断和 Fallback。

**典型老项目写法**：

```java
@HystrixCommand(
    fallbackMethod = "queryStockFallback",
    commandProperties = {
        @HystrixProperty(name = "execution.isolation.thread.timeoutInMilliseconds", value = "2000")
    }
)
public StockDTO queryStock(Long skuId) {
    return inventoryClient.queryStock(skuId);
}

public StockDTO queryStockFallback(Long skuId) {
    return StockDTO.empty();
}
```

**生产注意**：

- Hystrix 已经是老组件，新项目通常不主动引入。
- 看到老项目里有 Hystrix，要重点理解线程池隔离、信号量隔离、熔断窗口、Fallback。
- Fallback 只能做业务可接受的降级，不能给支付、扣库存这类核心写链路返回假成功。

**面试表达**：

> Hystrix 现在更多是理解老项目。它把远程依赖包装成 Command，通过线程池或信号量隔离，失败或超时后进入 Fallback。新项目我不会优先选 Hystrix，但它的隔离、熔断、降级思想仍然要懂。

### 0.1.5. Resilience4j 怎么用？

**解决什么问题**：轻量级容错组件，可以组合熔断、限流、隔离、重试和超时控制。

**注解用法**：

```java
@CircuitBreaker(name = "inventory", fallbackMethod = "fallback")
@Retry(name = "inventory")
@RateLimiter(name = "inventory")
public StockDTO queryStock(Long skuId) {
    return inventoryClient.queryStock(skuId);
}

private StockDTO fallback(Long skuId, Exception ex) {
    return StockDTO.empty();
}
```

**常见配置**：

```yaml
resilience4j:
  circuitbreaker:
    instances:
      inventory:
        slidingWindowSize: 100
        minimumNumberOfCalls: 20
        failureRateThreshold: 50
        waitDurationInOpenState: 10s
  retry:
    instances:
      inventory:
        maxAttempts: 3
        waitDuration: 200ms
  ratelimiter:
    instances:
      inventory:
        limitForPeriod: 100
        limitRefreshPeriod: 1s
        timeoutDuration: 0
```

**生产注意**：

- 熔断、重试、限流不要乱叠。下游已经慢时，重试可能放大故障。
- Fallback 方法通常和原方法参数一致，再额外加一个异常参数。
- 配合 Actuator / Micrometer 暴露指标，观察熔断状态、失败率、拒绝数。

**面试表达**：

> Resilience4j 更像一组可组合的治理能力。常见是用 `@CircuitBreaker` 做熔断，用 `@Retry` 控制重试，用 `@RateLimiter` 限流，用 `@Bulkhead` 做隔离。面试里我会强调组合顺序和业务语义，尤其是写接口不能无脑重试和降级。

### 0.1.6. Spring Cloud Gateway 怎么用？

**解决什么问题**：做外部流量统一入口，负责路由、鉴权、签名、限流、灰度、审计和统一错误处理。

**基本路由配置**：

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order-route
          uri: lb://order-service
          predicates:
            - Path=/api/order/**
          filters:
            - StripPrefix=1
```

**自定义过滤器思路**：

```java
@Component
public class AuthGlobalFilter implements GlobalFilter {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        ServerHttpRequest request = exchange.getRequest();
        // 校验 token、appKey、sign、nonce、timestamp
        return chain.filter(exchange);
    }
}
```

**生产注意**：

- Gateway 是入口统一治理，不等于服务内可以不鉴权。
- 开放接口要做 `appKey + sign + timestamp + nonce`，防伪造、防篡改、防重放。
- 网关限流、重试、熔断要和后端 Feign/Sentinel 策略统一，避免多层重试放大流量。

**面试表达**：

> Gateway 的使用核心是 Route、Predicate、Filter。Route 决定转发到哪个服务，Predicate 决定什么请求匹配，Filter 做鉴权、签名、限流、灰度和审计。服务内仍要做业务权限和数据权限兜底。

### 0.1.7. 如果没用过，面试怎么说？

**建议说法**：

> 我实际项目里主要接触的是 Feign/Dubbo 调用、超时配置、限流降级思想和配置灰度。Sentinel、Hystrix、Resilience4j 这类组件我了解它们解决的问题和基本使用方式：都是围绕远程调用稳定性做限流、熔断、隔离、降级，只是生态和模型不同。Sentinel 偏流量治理，Hystrix 是老项目里的 Command + 隔离模型，Resilience4j 更轻量模块化。真正落地时我会先从业务资源、超时、幂等、监控和降级语义设计，而不是为了用组件而用组件。

## 1. 什么是微服务治理？为什么大厂面试常问？

**一句话回答**：微服务治理是围绕服务调用链路做稳定性、流量、配置、路由和可观测性管理，解决服务拆分后带来的调用复杂、故障扩散、流量突刺、发布风险和定位困难问题。

**核心能力**：

- 服务发现：服务实例动态注册、订阅和健康检查。
- 负载均衡：从多个实例中选择一个合适目标。
- 超时重试：避免调用方无限等待，同时提升瞬时故障成功率。
- 熔断降级：下游异常时快速失败，保护调用方线程。
- 限流隔离：控制流量和并发，避免资源被打满。
- 灰度发布：小流量验证变更，异常时快速回滚。
- 可观测性：通过日志、指标、Trace 找到链路问题。

**三点总结**：

1. 微服务治理不是一个单独组件，而是一组工程能力。
2. 注册中心解决“找谁”，负载均衡解决“选谁”，限流熔断解决“坏了怎么办”，可观测性解决“怎么定位”。
3. 面试里要能结合项目讲：下游慢、流量大、发布风险、配置错误、链路排障。

**面试表达**：

> 微服务治理本质上是把服务拆分后的不确定性管起来。服务拆多以后，调用链变长、实例动态上下线、下游可能超时，单点问题容易扩散成雪崩。所以我会从注册发现、负载均衡、超时重试、限流熔断降级、灰度发布和可观测性几个层面来设计。

## 2. OpenFeign 是什么？底层怎么工作？

**一句话回答**：OpenFeign 是声明式 HTTP 客户端，在 Spring Cloud 中通过接口和注解定义远程调用，启动时为接口创建代理对象，调用方法时把参数编码成 HTTP 请求，再通过负载均衡和 HTTP Client 发给目标服务。

**核心流程**：

1. 使用 `@EnableFeignClients` 扫描 Feign 接口。
2. `@FeignClient(name = "order-service")` 声明目标服务。
3. Spring 为接口生成代理对象。
4. 方法调用时解析 Spring MVC 注解、路径、参数和请求体。
5. 通过 Encoder / Decoder 做序列化和反序列化。
6. 结合 LoadBalancer 从注册中心实例列表中选择目标实例。
7. 通过底层 HTTP Client 发起请求。

**常见追问**：

- Feign 是 RPC 吗？严格说不是，它是声明式 HTTP 客户端；Dubbo 这类才是典型 RPC 框架。
- Feign 和 RestTemplate 区别？Feign 面向接口声明，RestTemplate 是手写请求代码。
- Feign 为什么能负载均衡？Spring Cloud OpenFeign 可以结合 Spring Cloud LoadBalancer，根据服务名解析实例。

**面试表达**：

> OpenFeign 不是神秘 RPC，它本质是接口代理加 HTTP 调用。开发时写接口，运行时由 Spring 生成代理，解析注解后拼出 HTTP 请求，再结合 LoadBalancer 和注册中心找到具体实例。它提升的是调用代码可读性，但超时、重试、熔断、日志和连接池仍然要配置好。

## 2.1. HTTP 和 RPC 有什么区别？

**一句话回答**：HTTP 是应用层通信协议，RPC 是远程调用模型；RPC 底层可以用 HTTP，也可以用 TCP、自定义协议或 HTTP/2。

| 对比项 | HTTP / REST | RPC |
| --- | --- | --- |
| 本质 | 标准通信协议和 API 风格 | 像调用本地方法一样调用远程服务 |
| 调用表达 | `GET /users/1` | `userService.getById(1)` |
| 关注点 | URL、Method、Header、Status Code、Body | 接口、方法、参数、返回值、序列化、治理 |
| 数据格式 | JSON、XML、表单、文件等 | Protobuf、Hessian、Kryo、JSON 等 |
| 底层协议 | HTTP/1.1、HTTP/2、HTTP/3 | TCP、自定义协议、HTTP/2、HTTP 等 |
| 典型框架 | Spring MVC、OpenFeign、RestTemplate、WebClient | Dubbo、gRPC、Thrift |
| 适合场景 | 对外接口、前后端通信、跨语言系统集成 | 内部服务高频调用、强契约接口调用 |

**容易踩坑**

- 不要说“RPC 一定不是 HTTP”。gRPC 就是基于 HTTP/2。
- 不要说“HTTP 一定不是 RPC”。OpenFeign 是 HTTP 调用，但开发体验有明显 RPC 化封装。
- 更准确的说法是：HTTP 是协议，RPC 是调用模型。

**面试表达**

> HTTP 和 RPC 不是同一层概念。HTTP 是通信协议，RPC 是远程调用模型。HTTP 更强调资源和请求响应，RPC 更强调接口方法调用体验。RPC 底层可以用 TCP、自定义二进制协议，也可以像 gRPC 一样基于 HTTP/2。

## 2.2. 为什么 Spring Cloud 默认更多使用 HTTP，而不是 Dubbo 这类 RPC？

**一句话回答**：Spring Cloud 更偏微服务生态和云原生通用治理，HTTP 和 Spring MVC、Gateway、Ingress、Service Mesh、跨语言、调试工具天然兼容。

**核心原因**

| 原因 | 说明 |
| --- | --- |
| 通用性强 | HTTP 是标准协议，浏览器、网关、代理、负载均衡、监控工具都支持 |
| 跨语言友好 | Java、Go、Python、Node、前端、第三方系统都容易接入 |
| Spring MVC 生态一致 | `@RestController`、`@RequestMapping`、Feign 都能复用 Spring Web 注解模型 |
| 网关兼容好 | Spring Cloud Gateway、Nginx、Ingress、API Gateway 天然处理 HTTP |
| 调试方便 | curl、Postman、浏览器、日志抓包都能排查 |
| 云原生适配好 | Kubernetes Ingress、Service Mesh、链路追踪、指标采集对 HTTP 支持成熟 |
| 对外接口统一 | 微服务接口经常既要内部调用，也要对外开放，HTTP 更统一 |

**但不是 RPC 不好**

- Dubbo 更适合 Java 内部高性能服务调用。
- gRPC 更适合强契约、跨语言、高性能内部调用。
- Spring Cloud OpenFeign 是把 HTTP 封装成类似 RPC 的接口调用体验。

**面试表达**

> Spring Cloud 偏 HTTP，不是因为 RPC 不好，而是因为它更重视微服务生态的通用性。HTTP 和 Spring MVC、Gateway、Kubernetes、Ingress、Service Mesh、链路追踪和调试工具都天然兼容。OpenFeign 让开发体验像 RPC，但底层还是 HTTP。内部高性能、强契约场景可以用 Dubbo 或 gRPC。

## 2.3. HTTP 和 RPC 各自有什么优劣势？

**一句话回答**：HTTP 的优势是标准、通用、跨语言、易调试；RPC 的优势是调用体验好、强契约、治理能力和性能通常更强。

| 对比项 | HTTP / REST | RPC |
| --- | --- | --- |
| 通用性 | 很强，几乎所有系统都支持 | 依赖具体框架和协议 |
| 跨语言 | 很友好 | 看框架，gRPC 好，Dubbo 早期偏 Java |
| 调试 | curl、Postman、浏览器直接测 | 通常需要 SDK、IDL 或专用工具 |
| 可读性 | JSON + URL 可读性强 | 二进制协议可读性弱 |
| 性能 | JSON 和 HTTP 头开销较大 | 二进制序列化、长连接通常更高效 |
| 契约 | 相对弱，字段变更容易运行期暴露 | 相对强，IDL 或接口约束更明显 |
| 服务治理 | 需要配合网关、注册中心、熔断限流组件 | 框架通常内置注册、负载均衡、超时、重试等 |
| 对外开放 | 很适合 | 接入成本较高 |
| 内部调用 | 可以用，尤其 Spring Cloud 体系 | 很适合高频低延迟调用 |

**面试表达**

> HTTP 更像通用公路，标准、开放、易调试，适合对外 API、前后端通信和异构系统集成；RPC 更像内部专用通道，调用体验像本地方法，序列化和协议更紧凑，适合内部高频服务调用。选型不是谁绝对更好，而是看接口边界、性能要求、团队技术栈和治理体系。

## 2.4. HTTP 一定比 RPC 慢吗？

**一句话回答**：不一定。传统 JSON over HTTP/1.1 通常比二进制 RPC 开销大，但性能还受连接复用、序列化、线程模型、网络、业务耗时和治理配置影响。

**分析思路**

- 如果是 JSON + HTTP/1.1 短连接，开销确实可能更大。
- 如果开启连接池、Keep-Alive、HTTP/2、多路复用，HTTP 性能会好很多。
- gRPC 基于 HTTP/2 + Protobuf，本身就是高性能 RPC。
- 真实接口耗时经常主要来自 DB、Redis、外部接口、锁等待，而不是协议本身。

**面试表达**

> 不能简单说 HTTP 一定慢。协议和序列化会影响性能，但真实线上接口更多时候慢在数据库、缓存、下游服务、线程池或锁等待。只有在高频低延迟内部调用里，二进制 RPC 的优势才会更明显。

## 2.5. 什么场景适合 HTTP？什么场景适合 RPC？

**一句话回答**：对外、跨语言、开放接口优先 HTTP；内部高频、强契约、低延迟服务调用可以考虑 RPC。

| 场景 | 更推荐 | 原因 |
| --- | --- | --- |
| 前端调用后端 | HTTP | 浏览器天然支持 |
| App / 小程序 / 第三方开放接口 | HTTP | 标准、易接入、易调试 |
| 企业间系统集成 | HTTP | 异构系统兼容更好 |
| Java 内部核心链路高频调用 | RPC / Dubbo | 调用体验好、治理能力强 |
| 跨语言高性能内部调用 | gRPC | IDL 强契约、HTTP/2、Protobuf |
| 网关统一入口 | HTTP | 网关、Ingress、代理链路天然支持 |
| 简单中小型 Spring Cloud 服务 | HTTP + OpenFeign | 技术栈统一，维护成本低 |

**面试表达**

> 我一般按边界选型。系统边界和开放接口优先 HTTP，因为接入简单、标准通用；内部高频强依赖调用，如果团队有 Dubbo/gRPC 体系，可以选 RPC。Spring Cloud 体系里，OpenFeign 已经能覆盖大量内部服务调用场景。

## 2.6. OpenFeign 算 RPC 吗？

**一句话回答**：严格说 OpenFeign 不是典型 RPC 框架，它是声明式 HTTP 客户端；但它把 HTTP 调用封装成了类似 RPC 的接口调用体验。

**为什么像 RPC**

```java
@FeignClient(name = "order-service")
public interface OrderClient {
    @GetMapping("/orders/{id}")
    OrderDTO getById(@PathVariable Long id);
}
```

业务代码里像本地方法：

```java
OrderDTO order = orderClient.getById(1L);
```

但底层仍然是：

```text
接口代理 -> 拼 HTTP 请求 -> 服务发现/负载均衡 -> HTTP Client 调用
```

**面试表达**

> OpenFeign 更准确叫声明式 HTTP 客户端。它的开发体验像 RPC，但底层是 HTTP，不是 Dubbo 那类私有协议或典型 RPC 框架。面试里我会说它是“HTTP 调用的 RPC 化封装”。

## 2.7. 大厂为什么常同时使用 HTTP 和 RPC？

**一句话回答**：因为系统边界不同。对外要通用、易接入，内部要高性能、强治理，所以经常是外部 HTTP，内部 RPC 或 HTTP + Feign 混用。

**典型架构**

```text
外部用户 / 第三方
  -> API Gateway / Nginx / Ingress
  -> HTTP REST API
  -> 内部服务
  -> RPC / Feign / MQ
  -> 下游核心服务
```

**常见组合**

- 对外：HTTP REST / OpenAPI。
- Java 内部：Dubbo、Spring Cloud OpenFeign。
- 跨语言内部：gRPC。
- 异步解耦：MQ。
- 网关入口：HTTP。

**面试表达**

> 大厂通常不是二选一，而是分层使用。对外接口用 HTTP，因为标准、可治理、第三方接入成本低；内部高频核心链路可能用 Dubbo 或 gRPC；异步副作用用 MQ。关键是边界清晰，而不是全公司只允许一种协议。

## 2.8. 如果面试官问“RPC 比 HTTP 好，为什么不用 RPC 全部替代 HTTP？”

**一句话回答**：因为 RPC 不是万能的，它更适合内部服务；对外开放、浏览器访问、网关治理、跨系统集成仍然是 HTTP 更合适。

**回答要点**

- 浏览器和前端天然走 HTTP。
- 第三方系统更容易接 REST API，不一定愿意引入你的 RPC SDK。
- API Gateway、Ingress、WAF、CDN、审计、限流等基础设施多数围绕 HTTP 成熟建设。
- RPC 强契约和框架依赖会增加接入门槛。
- 对外接口更看重稳定契约、兼容性、可观测、权限控制，不只是性能。

**面试表达**

> RPC 在内部高频调用上有优势，但对外开放接口不能只看性能。HTTP 更标准、更容易被第三方和网关基础设施接入，也更方便调试和审计。所以真实系统一般是对外 HTTP，内部按场景选择 HTTP/Feign、Dubbo、gRPC 或 MQ。

## 3. OpenFeign 的超时参数怎么理解？

**一句话回答**：OpenFeign 常见超时包括连接超时和读取超时，连接超时控制建立连接阶段，读取超时控制连接建立后等待响应数据的时间。

| 参数 | 含义 | 典型问题 |
| --- | --- | --- |
| connectTimeout | 建立 TCP 连接、DNS 等连接阶段超时 | 目标服务不可达、网络不通、连接池异常 |
| readTimeout | 连接建立后等待响应数据超时 | 下游接口慢、DB 慢、线程池阻塞 |

**关键细节**：

- 超时时间不能只按平均 RT 配，要看 P95/P99 和业务可接受时间。
- 上游超时要小于入口网关超时，避免请求已经被网关断开，下游还在执行。
- 写接口超时后不能简单重试，可能下游已经执行成功但响应丢了。
- 超时要和线程池、连接池、熔断、重试一起看。

**面试表达**：

> Feign 超时我会分连接超时和读取超时。连接超时偏网络和目标实例可达性，读取超时偏下游处理慢。生产上不能把超时设得过长，否则调用线程会被拖住；也不能过短，否则正常慢请求被误判失败。写接口超时后尤其要注意幂等，因为超时不代表下游没执行。

### 3.1. P95 和 P99 指的是什么？

**一句话回答**：P95、P99 是接口耗时的百分位指标，P95 表示 95% 的请求耗时小于等于这个值，P99 表示 99% 的请求耗时小于等于这个值，用来观察尾部慢请求。

假设某接口 1000 次请求按耗时从小到大排序：

```text
第 950 个请求的耗时 = P95
第 990 个请求的耗时 = P99
```

例如：

```text
平均 RT = 80ms
P95 = 200ms
P99 = 1200ms
```

这说明大部分请求还可以，但有少量请求非常慢。平均值容易掩盖尾部问题，所以微服务治理里配置超时、判断慢调用、定位抖动时，不能只看平均 RT，还要看 P95/P99。

**P99 升高常见原因**

- 少量实例变慢或发生抖动。
- 下游 RPC、DB、Redis 偶发慢调用。
- 线程池或连接池排队。
- GC 停顿、锁竞争或热点资源争用。
- 重试、限流、熔断规则配置不合理。

**面试表达**：

> P95/P99 是接口耗时的百分位指标。P95 表示 95% 的请求都能在这个时间内完成，P99 表示 99% 的请求都能在这个时间内完成。相比平均 RT，它更能反映尾部慢请求。微服务里如果 P99 升高，通常说明少量请求被慢实例、下游依赖、线程池排队、慢 SQL、GC 或锁竞争拖慢，需要结合 Trace、线程池、连接池和下游指标一起排查。

## 4. OpenFeign 的重试有什么风险？

**一句话回答**：重试能提高瞬时网络抖动下的成功率，但也可能放大流量、造成重试风暴，并让非幂等写接口重复执行。

**主要风险**：

- 下游已经慢了，上游重试会继续放大压力。
- 超时不代表失败，可能业务已经执行成功。
- 非幂等接口重试会重复下单、重复扣库存、重复发券。
- 多层都重试会形成指数级放大，例如网关、Feign、MQ、业务代码都重试。

**生产建议**：

- 查询接口可以少量重试，写接口默认谨慎重试。
- 必须设置最大次数、总耗时上限和退避策略。
- 配合熔断限流，避免下游故障时持续重试。
- 关键写接口必须有业务幂等键、唯一索引或状态机兜底。

**面试表达**：

> 我不会把重试当成万能补偿。重试适合瞬时网络抖动和幂等查询，但如果下游已经慢了，重试会加剧雪崩。写接口即使要重试，也必须先有 requestId、唯一索引、状态机或幂等表兜底，并控制重试次数和退避。

## 5. Feign 调用失败如何做 Fallback？

**一句话回答**：Fallback 是远程调用失败、超时、熔断或限流后的降级逻辑，用来给调用方返回可接受的兜底结果，但不能掩盖核心业务失败。

**常见降级结果**：

- 非核心信息返回空列表、默认值或缓存值。
- 推荐、标签、画像、统计类能力可以降级。
- 核心交易、支付、库存扣减不能伪造成功。
- 需要记录日志、指标和告警，避免静默失败。

**Fallback 设计原则**：

1. 降级结果必须符合业务语义。
2. 核心链路宁可明确失败，也不要返回假成功。
3. Fallback 内部逻辑要轻量，不要再调用一串不稳定下游。
4. 要区分异常类型：超时、限流、熔断、业务异常不能混为一谈。

**面试表达**：

> Fallback 不是把错误吞掉，而是给非核心能力准备兜底。比如推荐服务失败可以返回默认推荐，画像服务失败可以返回空标签；但库存扣减、支付成功这类核心结果不能靠 Fallback 伪造成功，否则数据会乱。

## 6. Feign 如何做日志、鉴权和链路追踪透传？

**一句话回答**：Feign 通常通过 `RequestInterceptor` 统一添加请求头，例如 token、租户、traceId、灰度标识，再配合日志级别和可观测组件记录请求链路。

**常见透传字段**：

- `traceId` / `spanId`：链路追踪。
- `Authorization`：用户或服务鉴权。
- `tenantId`：多租户隔离。
- `grayTag` / `version`：灰度路由。
- `requestId`：接口幂等和排障关联。

**注意点**：

- 不要在日志里打印敏感 token、手机号、身份证。
- 异步线程要显式透传上下文，不能只依赖 `ThreadLocal`。
- Feign full 日志适合排查，线上长期全量打开会有性能和隐私风险。

**面试表达**：

> Feign 透传我通常会用拦截器统一处理，把 traceId、用户态、租户、灰度标签放到请求头里。这样下游日志和链路追踪能串起来。线上要注意脱敏和日志级别，不能为了排查把所有请求响应体长期打满。

## 7. Ribbon 和 Spring Cloud LoadBalancer 有什么区别？

**一句话回答**：Ribbon 是 Netflix 体系里的客户端负载均衡组件，已经逐步退出主流维护；Spring Cloud LoadBalancer 是 Spring Cloud 体系推荐的客户端负载均衡方案，用来替代 Ribbon。

| 对比项   | Ribbon            | Spring Cloud LoadBalancer |
| ----- | ----------------- | ------------------------- |
| 生态    | Netflix OSS 老体系   | Spring Cloud 当前体系         |
| 模式    | 客户端负载均衡           | 客户端负载均衡                   |
| 常见算法  | 轮询、随机、权重等         | 轮询、随机，可扩展                 |
| 服务发现  | 结合 Eureka/Nacos 等 | 结合 DiscoveryClient        |
| 新项目建议 | 不优先选              | 更推荐                       |

**面试表达**：

> Ribbon 和 Spring Cloud LoadBalancer 解决的问题类似，都是客户端侧从实例列表里选一个服务实例。区别在于 Ribbon 属于 Netflix 老体系，新项目更倾向 Spring Cloud LoadBalancer。面试时我会重点讲客户端负载均衡原理，而不是纠结组件名。

## 8. 客户端负载均衡和服务端负载均衡有什么区别？

**一句话回答**：服务端负载均衡由 Nginx/网关等统一接收请求再转发，客户端负载均衡由调用方拿到实例列表后自己选择目标实例。

| 对比项 | 服务端负载均衡 | 客户端负载均衡 |
| --- | --- | --- |
| 代表 | Nginx、SLB、网关 | Ribbon、Spring Cloud LoadBalancer |
| 实例选择位置 | 网关/代理层 | 调用方进程内 |
| 优点 | 调用方简单、统一入口 | 少一跳、策略可按客户端定制 |
| 缺点 | 代理层可能成为瓶颈 | 客户端要维护实例列表和策略 |

**面试表达**：

> 服务端负载均衡是请求先到统一代理，由代理转发；客户端负载均衡是调用方自己订阅服务实例并选择目标。Spring Cloud 体系内部服务调用常用客户端负载均衡，对外入口仍常用网关或 Nginx。

## 9. 负载均衡算法有哪些？线上怎么选？

**一句话回答**：常见算法包括轮询、随机、权重、最少连接、一致性哈希，线上要结合实例权重、健康状态、延迟和灰度标签选择。

| 算法 | 特点 | 适用场景 |
| --- | --- | --- |
| 轮询 | 简单平均分配 | 实例规格接近 |
| 随机 | 实现简单，整体趋于均匀 | 实例数量多 |
| 权重 | 高配机器多分流 | 新老机器规格不同 |
| 最少连接 | 优先分给连接少的实例 | 请求耗时差异大 |
| 一致性哈希 | 同一 key 尽量落同一实例 | 缓存、本地状态亲和 |

**注意点**：

- 要剔除不健康实例。
- 灰度发布时要按版本或标签路由。
- 慢实例不一定宕机，但会拖慢 P99，需要动态摘流。
- 负载均衡不是熔断，不能替代下游保护。

**面试表达**：

> 负载均衡不能只说轮询。生产里要看实例健康、权重、版本标签和实时延迟。比如灰度发布时，同一个用户或租户要路由到指定版本；某台机器 RT 飙高时，即使还活着，也应该降低权重或临时摘流。

## 10. Sentinel 是什么？和 Hystrix 的定位有什么区别？

**一句话回答**：Sentinel 是面向流量治理的组件，重点从流量入口、资源维度、运行指标出发做限流、熔断降级、系统自适应保护；Hystrix 更强调通过 Command 模型做隔离、熔断和 Fallback，属于 Netflix OSS 老体系。

| 对比项 | Sentinel | Hystrix |
| --- | --- | --- |
| 关注点 | 流量治理、限流、熔断、系统保护 | 熔断、隔离、降级 |
| 资源模型 | Resource + Rule | HystrixCommand |
| 隔离方式 | 线程数限制等轻量隔离为主 | 线程池隔离、信号量隔离 |
| 规则动态性 | 规则可动态变更 | 依赖动态属性配置 |
| 新项目趋势 | Spring Cloud Alibaba 常用 | 维护模式，不建议新项目重度引入 |

**面试表达**：

> Sentinel 更像流量治理平台，核心是把接口、方法、热点参数等抽象成资源，再基于 QPS、线程数、异常比例、慢调用比例做保护。Hystrix 是更早的熔断隔离组件，典型模型是 Command、线程池隔离和 Fallback。新项目我会优先考虑 Sentinel 或 Resilience4j。

## 11. Sentinel 的 Resource 和 Rule 怎么理解？

**一句话回答**：Resource 是 Sentinel 保护的资源，可以是接口、方法、代码块；Rule 是保护规则，用来描述这个资源在什么条件下限流、熔断或降级。

**常见资源**：

- HTTP 接口：`/api/order/create`
- Feign 调用：`inventory-service#reserve`
- 业务方法：`createOrder`
- 热点参数：`skuId`

**常见规则**：

- 流控规则：QPS、线程数、调用来源。
- 熔断规则：慢调用比例、异常比例、异常数。
- 热点规则：按参数值限制热点 key。
- 系统规则：按 CPU、load、入口 QPS、平均 RT、并发数保护系统。

**面试表达**：

> Sentinel 里先定义资源，再给资源配置规则。资源可以是接口，也可以是某个关键方法。规则描述什么时候拒绝请求，比如 QPS 超阈值、线程数过多、异常比例升高或慢调用比例升高。这样治理粒度比单纯网关限流更细。

## 12. Sentinel 限流有哪些维度？

**一句话回答**：Sentinel 常见限流维度包括 QPS、并发线程数、调用来源、调用链路、关联资源和热点参数。

| 维度    | 含义                 | 场景               |
| ----- | ------------------ | ---------------- |
| QPS   | 每秒通过请求数            | 控制接口入口流量         |
| 并发线程数 | 当前处理该资源的线程数        | 防止慢接口占满线程        |
| 调用来源  | 按 caller/origin 区分 | 给不同业务方不同额度       |
| 调用链路  | 按入口链路区分            | 同一资源在不同入口不同阈值    |
| 关联资源  | 一个资源影响另一个资源        | 写操作忙时限制读操作       |
| 热点参数  | 按参数值限流             | 秒杀 sku、热门城市、热点用户 |

**面试表达**：

> Sentinel 限流不只是简单接口 QPS。比如某个 SKU 特别热，就要做热点参数限流；某个下游慢导致线程堆积，就按线程数限流；不同业务方调用同一接口，还可以按来源设置不同阈值。

## 13. Sentinel 的流控效果有哪些？

**一句话回答**：常见流控效果包括直接拒绝、Warm Up 预热和匀速排队，分别适合快速保护、冷启动保护和削峰填谷。

| 流控效果 | 含义 | 适用场景 |
| --- | --- | --- |
| 直接拒绝 | 超过阈值立即拒绝 | 保护核心服务，快速失败 |
| Warm Up | 阈值逐步升高 | 冷启动、缓存未预热 |
| 匀速排队 | 按固定间隔放行 | 瞬时突刺，允许排队等待 |

**注意点**：

- 直接拒绝要给业务明确提示，例如“系统繁忙，请稍后再试”。
- Warm Up 适合刚启动的应用，避免瞬间打满连接池和缓存。
- 匀速排队要设置最大等待时间，避免请求堆积拖垮线程。

**面试表达**：

> Sentinel 流控不是只有拒绝。直接拒绝适合快速保护，Warm Up 适合应用刚启动或缓存未预热，匀速排队适合短时间突刺但业务允许等待的场景。线上我会结合业务语义选择，不能所有接口都一刀切。

## 14. Sentinel 熔断规则有哪些？

**一句话回答**：Sentinel 常见熔断策略包括慢调用比例、异常比例和异常数，用来在下游变慢或错误升高时快速切断调用，防止故障扩散。

| 策略 | 判断依据 | 适合场景 |
| --- | --- | --- |
| 慢调用比例 | 超过 RT 阈值的请求比例 | 下游变慢但不一定报错 |
| 异常比例 | 异常请求占比 | 错误率突然升高 |
| 异常数 | 时间窗口内异常个数 | 低流量但异常明显 |

**核心状态**：

- Closed：正常通过。
- Open：熔断打开，请求快速失败。
- Half-Open：探测恢复，少量请求通过。

**面试表达**：

> 熔断不是等下游完全挂掉才触发。很多事故是下游变慢导致线程被占满，所以慢调用比例很关键。熔断打开后请求快速失败，过一段时间进入半开状态，用少量请求探测恢复，恢复后再关闭熔断。

## 15. Sentinel 热点参数限流怎么讲？

**一句话回答**：热点参数限流是针对请求参数值做限流，比如某个 `skuId`、城市、用户或门店突然变热，只限制这个热点值，而不是把整个接口都限掉。

**适用场景**：

- 秒杀中某个 SKU 被集中抢购。
- 某个城市、站点、供应商流量突然暴涨。
- 某个用户或 IP 异常刷接口。
- 排行榜、商品详情、库存查询出现热点 key。

**面试表达**：

> 普通接口限流是把整个接口流量压下来，但热点参数限流更精细。比如只有某个 skuId 爆了，我希望限制这个 sku，而不是影响其他商品正常访问。它特别适合秒杀、库存、门店和城市维度的热点保护。

## 16. Sentinel 系统自适应保护是什么？

**一句话回答**：系统自适应保护是从整机维度保护应用，根据 CPU、load、入口 QPS、平均 RT、并发线程数等指标判断系统是否接近承载边界，必要时拒绝新请求。

**核心理解**：

- 它不是保护某个下游，而是保护当前应用整体。
- 入口流量才会触发系统规则。
- 目标是在系统负载可控前提下尽量提高吞吐。
- 不能代替接口级限流和熔断。

**面试表达**：

> 系统保护是最后一道整体保护，关注的是当前应用机器是否快被打满。比如 CPU、load、入口 QPS、平均 RT、并发数都异常时，可以拒绝部分入口流量，避免整个应用被压垮。但它不能替代具体接口的限流和下游熔断。

## 17. Sentinel 规则应该怎么动态配置？

**一句话回答**：生产中 Sentinel 规则不应该只写死在代码里，应该通过配置中心或规则管理平台动态下发，并记录版本、变更人、灰度范围和回滚能力。

**推荐做法**：

- 开发环境可以本地规则或 Dashboard 临时验证。
- 生产规则接入 Nacos/Apollo 等持久化配置。
- 每次变更记录版本、diff、发布人和时间。
- 先灰度少量机器，再逐步全量。
- 关键接口阈值来自压测、容量评估和历史指标。

**面试表达**：

> Sentinel 规则如果只在 Dashboard 里临时改，重启可能丢失，也不好审计。生产上我更倾向接配置中心，规则变更像配置发布一样有版本、灰度、回滚和审计。阈值不能拍脑袋，要来自压测和线上监控。

## 18. 限流、熔断、降级、隔离有什么区别？

**一句话回答**：限流是控制进入系统的流量，熔断是发现下游异常后快速失败，降级是业务上返回兜底结果，隔离是把不同资源的故障影响限制在局部。

| 概念 | 解决问题 | 例子 |
| --- | --- | --- |
| 限流 | 流量超过承载能力 | 秒杀接口每秒只放行 1000 |
| 熔断 | 下游慢或错导致故障扩散 | 库存服务错误率高，调用快速失败 |
| 降级 | 非核心能力不可用时保核心 | 推荐失败返回默认列表 |
| 隔离 | 一个资源不要拖垮全局 | 为不同下游设置独立线程池/并发数 |

**面试表达**：

> 限流是入口控量，熔断是故障切断，降级是业务取舍，隔离是资源边界。四者经常一起出现：先限流挡住过量流量，下游慢了熔断，非核心能力降级，关键下游用线程池或并发数隔离，避免拖垮主线程池。

## 19. Hystrix 的核心原理是什么？

**一句话回答**：Hystrix 把每个远程依赖封装成 Command，通过线程池或信号量隔离执行，监控调用成功率、失败率、超时和拒绝情况，异常时触发熔断和 Fallback，防止级联故障。

**核心组件**：

- `HystrixCommand`：封装一次依赖调用。
- 线程池隔离：每个依赖可配置独立线程池。
- 信号量隔离：轻量限制并发数。
- Circuit Breaker：根据错误率和请求量打开熔断。
- Fallback：失败后的降级逻辑。
- Metrics：监控成功、失败、超时、拒绝、熔断状态。

**面试表达**：

> Hystrix 的思想是把不可靠的远程依赖包在 Command 里，通过隔离和熔断避免拖垮调用方。比如某个下游慢了，只会打满它自己的线程池，不会把 Tomcat 业务线程全部占住。失败后进入 Fallback，返回业务可接受的降级结果。

## 20. Hystrix 线程池隔离和信号量隔离怎么选？

**一句话回答**：线程池隔离更彻底，能让调用方从慢依赖中脱身，但有线程切换成本；信号量隔离更轻量，适合可信、低延迟、不会长时间阻塞的本地或轻量调用。

| 隔离方式 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- |
| 线程池隔离 | 隔离彻底，可超时脱身 | 线程切换、队列、上下文传递成本 | 第三方接口、慢 RPC、不可信依赖 |
| 信号量隔离 | 轻量、开销低 | 无法真正中断慢调用 | 本地方法、快速缓存、可信依赖 |

**注意点**：

- 线程池隔离要配置核心线程数、队列长度、拒绝策略和超时。
- 信号量隔离必须保证依赖快速失败，否则会占住调用线程。
- `ThreadLocal`、事务上下文、MDC 在跨线程时要显式处理。

**面试表达**：

> 如果下游是第三方接口或慢 RPC，我更倾向线程池隔离，因为它能把风险限制在独立线程池里；如果只是本地缓存或非常快的内部逻辑，信号量隔离更轻。线程池隔离的代价是上下文传递和线程切换，尤其要注意事务和 ThreadLocal。

## 21. Hystrix 为什么逐渐被替代？

**一句话回答**：Hystrix 已处于维护模式，Netflix OSS 体系整体逐步淡出 Spring Cloud 主流；新项目通常更倾向 Sentinel、Resilience4j 或 Spring Cloud CircuitBreaker 体系。

**面试表达要点**：

- 老项目仍可能大量使用 Hystrix，要理解它的熔断隔离思想。
- 新项目不建议为了追老技术强行引入 Hystrix。
- Resilience4j 更轻量，适合 Java 8+ 函数式装饰。
- Sentinel 在 Spring Cloud Alibaba 生态中更常见，流量治理能力更强。

**面试表达**：

> Hystrix 的思想仍然重要，比如隔离、熔断、Fallback、指标监控。但它本身已经不是新项目首选。现在如果是 Spring Cloud Alibaba 体系，我会优先 Sentinel；如果是 Spring Cloud CircuitBreaker 体系，可以考虑 Resilience4j。

## 22. Resilience4j 和 Hystrix 有什么区别？

**一句话回答**：Resilience4j 是轻量级容错库，提供 CircuitBreaker、RateLimiter、Bulkhead、Retry、TimeLimiter 等模块化能力；Hystrix 更重，围绕 Command、线程池隔离和 Netflix 生态设计。

| 对比项 | Hystrix | Resilience4j |
| --- | --- | --- |
| 设计 | Command 模型 | 函数式装饰器 |
| 模块 | 熔断、隔离、Fallback 一体化 | 熔断、限流、隔离、重试、超时模块化 |
| 隔离 | 强调线程池隔离 | Bulkhead 支持信号量/线程池等模式 |
| 状态 | 维护模式 | 新项目更常见 |
| Spring 集成 | 老 Spring Cloud Netflix | Spring Cloud CircuitBreaker |

**面试表达**：

> Resilience4j 更像一组可组合的容错组件，你可以只用 CircuitBreaker，也可以叠加 RateLimiter、Retry、Bulkhead、TimeLimiter。Hystrix 更像老一代完整熔断隔离框架。面试重点不是背 API，而是能讲清楚熔断、隔离、重试和超时如何组合。

## 23. Sentinel 和 Resilience4j 怎么选？

**一句话回答**：如果项目是 Spring Cloud Alibaba 体系、需要控制台和流量治理能力，Sentinel 更顺手；如果项目更偏 Spring Cloud CircuitBreaker、希望轻量组合熔断限流重试，Resilience4j 更合适。

**选型维度**：

- 技术栈：Spring Cloud Alibaba vs 原生 Spring Cloud。
- 治理重点：流量入口治理 vs 客户端容错组合。
- 规则管理：是否需要 Dashboard、配置中心动态规则。
- 团队经验：运维、监控、配置发布能力是否匹配。
- 生态兼容：网关、Feign、注册中心和监控体系。

**面试表达**：

> 选型我会看生态和治理目标。Sentinel 在限流、热点参数、系统保护和流量治理上更强，适合 Spring Cloud Alibaba；Resilience4j 更轻量模块化，适合按调用方组合熔断、限流、隔离、重试。核心不是哪个更高级，而是团队能不能管好规则和监控。

## 24. 如何设计一个接口的限流方案？

**一句话回答**：接口限流要分层设计，从网关、应用、资源和业务维度一起做，阈值来自容量评估和压测，限流结果要符合业务语义。

**设计步骤**：

1. 明确保护对象：入口接口、下游服务、DB、Redis、线程池。
2. 明确限流维度：IP、用户、租户、接口、参数、来源服务。
3. 明确算法：固定窗口、滑动窗口、令牌桶、漏桶。
4. 明确部署层次：网关限流、应用限流、Redis 分布式限流、Sentinel 资源限流。
5. 明确降级结果：拒绝、排队、默认值、异步处理。
6. 明确观测指标：通过量、拒绝量、RT、错误率、业务成功率。

**面试表达**：

> 我做限流不会只在代码里写一个计数器。入口网关按 IP、用户、租户挡住异常流量；应用内用 Sentinel 按资源做 QPS 或线程数限流；热点参数单独保护；涉及全局额度时再用 Redis 或集群限流。阈值来自压测和线上指标，限流结果要让前端和业务能理解。

## 25. 如何设计熔断和降级方案？

**一句话回答**：熔断降级要先梳理核心链路和非核心链路，核心链路快速失败并告警，非核心链路返回可接受兜底结果，同时配合半开探测和恢复机制。

**设计步骤**：

1. 梳理调用链，标出核心和非核心依赖。
2. 为每个下游设置超时、并发上限和熔断规则。
3. 根据慢调用比例、异常比例、异常数触发熔断。
4. 非核心能力提供缓存、默认值、空结果或异步补偿。
5. 核心写链路明确失败，不返回假成功。
6. 熔断后半开探测，恢复后逐步放量。
7. 配置告警和仪表盘，避免长期降级无人发现。

**面试表达**：

> 熔断解决的是下游异常时不要拖垮上游，降级解决的是业务上怎么继续提供可接受结果。比如下单链路里推荐和标签可以降级，库存和支付不能伪造成功。熔断打开后还要有半开探测和恢复观察，不能一熔断就永远不可用。

## 26. 如何防止重试风暴？

**一句话回答**：防止重试风暴要控制重试层级、次数、总耗时和退避策略，并让重试和熔断、限流、幂等、队列削峰联动。

**治理手段**：

- 明确只有一层负责重试，避免网关、Feign、业务、MQ 多层叠加。
- 设置最大重试次数和请求总 deadline。
- 使用指数退避和随机抖动，避免同一时间重试。
- 下游错误率升高时触发熔断，停止继续重试。
- 写接口先做幂等，再考虑重试。
- 对可异步任务转 MQ 或任务表，避免同步重试压垮链路。

**面试表达**：

> 重试风暴通常不是一次重试造成的，而是多层重试叠加造成的。比如网关重试 2 次，Feign 重试 2 次，业务再重试 2 次，下游压力会被放大很多倍。我会统一重试策略，加总超时、退避和熔断，写接口必须先幂等。

## 27. 如何设计灰度发布？

**一句话回答**：灰度发布是把新版本按用户、租户、地区、机器、流量比例或业务标签逐步放量，边观察边扩大范围，异常时快速回滚。

**常见灰度维度**：

- 按用户 ID、手机号尾号、会员等级。
- 按租户、城市、站点、供应商。
- 按机器、集群、版本号。
- 按接口、业务开关、配置项。
- 按流量比例逐步放量。

**关键能力**：

- 标签透传：网关到服务间调用要传递灰度标签。
- 路由规则：注册中心或网关根据标签选择实例。
- 配置开关：功能可随时关闭。
- 指标观察：错误率、RT、业务转化、投诉、告警。
- 快速回滚：发布包、配置和数据兼容都要能退。

**面试表达**：

> 灰度发布不是发一台机器就完事。要有灰度标签透传、路由规则、配置开关和指标观察。比如先让内部用户或某个城市进入新版本，观察错误率和业务指标，没问题再扩大比例；一旦异常，能通过配置或发布系统快速回滚。

## 28. 什么是蓝绿发布、滚动发布、金丝雀发布？

**一句话回答**：蓝绿发布是新旧两套环境切流，滚动发布是分批替换实例，金丝雀发布是先让少量用户或流量验证新版本。

| 发布方式 | 特点 | 优点 | 风险 |
| --- | --- | --- | --- |
| 蓝绿发布 | 两套完整环境，一键切流 | 回滚快 | 资源成本高，数据兼容要求高 |
| 滚动发布 | 一批批替换实例 | 成本低，过程平滑 | 新旧版本共存时间长 |
| 金丝雀发布 | 少量流量先试 | 风险最小，适合验证 | 需要路由和观测能力 |

**面试表达**：

> 蓝绿强调两套环境快速切换，滚动强调分批替换实例，金丝雀强调少量流量验证。实际项目常组合使用，比如先金丝雀灰度，再滚动扩大发布，关键是数据结构和接口协议要兼容，否则回滚也可能出问题。

## 29. 接口突然大量熔断，怎么排查？

**排查路径**：

1. 看是否真实下游故障：错误率、RT、超时、连接池。
2. 看是否规则变更：熔断阈值、时间窗口、最小请求数。
3. 看是否流量突增：入口 QPS、热点参数、调用来源。
4. 看是否发布变更：新版本、配置、依赖、数据库变更。
5. 看线程池/连接池：是否耗尽、队列堆积、拒绝数升高。
6. 看调用链 Trace：慢在哪一段，是网关、应用、DB、Redis 还是 RPC。
7. 短期止血：摘流、扩容、回滚、调高合理阈值、关闭非核心入口。

**面试表达**：

> 大量熔断我不会第一时间怪 Sentinel 或 Hystrix。先确认下游是不是真的慢或错误，再看规则有没有刚改过，最后看流量和发布变更。短期可以摘流、扩容、回滚或降级非核心入口，长期要修复慢调用和容量模型。

## 30. Feign 调用一直超时，怎么排查？

**排查路径**：

1. 确认是连接超时还是读取超时。
2. 看注册中心实例是否健康，IP/端口是否可达。
3. 看负载均衡是否打到慢实例或下线实例。
4. 看下游接口自身 RT、线程池、DB、Redis、锁等待。
5. 看调用方连接池是否耗尽。
6. 看超时配置是否过短或多层超时不一致。
7. 看是否有重试放大流量。
8. 看是否缺少 Trace 和日志导致链路断点。

**面试表达**：

> Feign 超时要先分清 connect timeout 和 read timeout。连接超时更多看网络、服务发现、实例端口；读取超时更多看下游处理慢。然后检查负载均衡、连接池、线程池、DB 慢查询和重试策略，不能只把 readTimeout 调大。

## 31. 限流规则上线后误伤用户，怎么处理？

**处理步骤**：

1. 先确认误伤范围：接口、用户、租户、来源、参数。
2. 临时调高阈值或回滚规则，恢复核心业务。
3. 区分正常高峰和异常流量。
4. 对异常来源单独限流，不要全局一刀切。
5. 引入灰度规则和小流量验证。
6. 复盘阈值来源，补压测和容量数据。

**面试表达**：

> 限流误伤说明规则粒度或阈值有问题。止血时先回滚或调高阈值，然后看是不是某个来源、租户、热点参数导致的。如果只是某个 sku 或用户异常，不应该把整个接口限掉，应该做更细粒度的热点参数或来源限流。

## 32. 如何把微服务治理和项目经验结合？

**回答结构**：

1. **业务背景**：订单、库存、履约、支付、营销、调度。
2. **链路风险**：流量突刺、下游慢、重复调用、配置错误、发布风险。
3. **治理方案**：Feign 超时、Sentinel 限流熔断、配置灰度、负载均衡、链路追踪。
4. **兜底机制**：幂等、状态机、补偿任务、MQ 削峰、告警和人工修复。
5. **结果表达**：核心链路更稳、故障影响面更小、排障更快、发布风险更低。

**示例表达**：

> 在履约和库存链路里，我会把微服务治理讲成一套组合拳。服务间调用用 Feign/Dubbo，并配置合理超时和幂等；热点库存或规则接口用 Sentinel 做限流，防止突刺打满应用；下游库存或人员服务慢时触发熔断，非核心展示信息降级；灰度发布通过配置中心和路由标签逐步放量；最后用 traceId、指标和日志把跨服务链路串起来，出问题时能快速定位。

## 33. 开放网关 / API Gateway 如何设计？

**一句话回答**：开放网关是外部流量进入微服务体系的统一入口，负责路由转发、统一鉴权、签名验签、防重放、限流、灰度、日志审计和协议适配，把入口通用能力从业务服务中收拢出来。

**典型职责**：

| 能力 | 作用 | 常见做法 |
| --- | --- | --- |
| 路由转发 | 把外部 API 转到内部服务 | path/method/header 路由、服务发现、负载均衡 |
| 统一认证 | 判断调用方是谁 | JWT、OAuth2、appKey、AK/SK |
| 签名验签 | 防伪造、防篡改 | appKey + secret + timestamp + nonce + body 计算签名 |
| 防重放 | 防止合法请求被重复提交 | timestamp 时间窗口、nonce/requestId 去重、业务幂等 |
| 限流降级 | 保护入口和后端服务 | IP、用户、租户、appKey、接口维度限流 |
| 灰度路由 | 控制流量进入不同版本 | 用户/租户/版本标签、权重、配置中心 |
| 观测审计 | 方便排障和追责 | access log、traceId、调用方、耗时、状态码、错误码 |
| 协议适配 | 屏蔽内部服务细节 | 路径重写、Header 清洗、参数转换、响应包装 |

**开放网关和普通内部网关的区别**

- 内部网关主要面向内部系统入口，重点是路由、鉴权、限流、统一入口。
- 开放网关面向外部调用方、供应商、合作伙伴或开放平台，通常更强调 appKey 管理、接口签名、调用配额、防重放、版本管理、审计和 API 文档。

**三点总结**

1. 网关做统一入口能力，但不能把所有业务逻辑都塞进网关。
2. 网关鉴权只能做粗粒度安全，服务内仍要做资源归属、数据权限和幂等。
3. 开放接口要把签名、防重放、限流、审计、幂等一起设计，不能只靠一个 token。

**面试表达**

> 开放网关我会理解成外部流量进入微服务体系的统一入口。它负责路由、统一认证、签名验签、防重放、限流、灰度、日志审计和协议适配。比如供应商调用接口时，先通过 appKey 找到调用方，用 secret 校验签名，再检查 timestamp 和 nonce 防重放，最后按路由转发到内部服务。网关只做通用入口治理，订单归属、数据权限、幂等和状态校验仍然要放在业务服务里兜底。

## 34. Spring Cloud 是否有配套的开放网关？

**一句话回答**：有。Spring Cloud 体系里常用的是 **Spring Cloud Gateway**，它是 Spring 官方生态中的 API Gateway 项目，支持基于 Route、Predicate、Filter 的路由和横切处理；但它不是 Service Mesh。

**Spring Cloud Gateway 核心概念**

| 概念 | 作用 | 面试表达 |
| --- | --- | --- |
| Route | 路由定义 | 描述某类请求要转发到哪里 |
| Predicate | 路由匹配条件 | 根据 Path、Method、Header、Query、Host 等判断是否命中 |
| Filter | 过滤器 | 对请求或响应做处理，例如鉴权、限流、改 Header、重写路径 |
| GlobalFilter | 全局过滤器 | 对所有路由统一生效，例如日志、Trace、统一鉴权 |

**能做什么**

1. 路由转发：按路径、域名、Header、Method 转发到不同服务。
2. 负载均衡：结合服务发现，把请求转到某个服务实例。
3. 安全治理：统一鉴权、Token Relay、Header 清洗、HTTPS/TLS。
4. 流量治理：限流、熔断、重试、请求大小限制。
5. 协议和内容处理：路径重写、请求/响应 Header 修改、请求/响应体处理。
6. 可观测性：访问日志、指标、Trace、Actuator 端点。

**示例配置**

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order-api
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1
            - name: RequestRateLimiter
```

**注意点**

- Spring Cloud Gateway 是 API 网关，适合做入口治理，不是业务服务之间所有东西都必须经过它。
- WebFlux 版本底层是响应式模型，过滤器里不要写阻塞逻辑。
- 网关限流、重试、熔断要和后端 Feign、Sentinel、线程池策略统一，避免多层治理互相放大。
- 对外开放接口不要只依赖 Gateway 自带能力，还要结合 appKey、签名、防重放、配额、审计和服务内权限。

**面试表达**

> Spring Cloud 体系里配套的 API 网关是 Spring Cloud Gateway。它通过 Route 描述转发目标，通过 Predicate 判断请求是否命中路由，通过 Filter 做鉴权、限流、改写 Header、路径重写、熔断重试等横切逻辑。如果是开放平台场景，我会在 Gateway 上做统一入口、签名验签、防重放、限流和审计，但业务权限、数据归属和幂等仍然放在后端服务里。

## 35. API Gateway 和 Service Mesh 有什么区别？

**一句话回答**：API Gateway 管的是南北向流量，也就是外部请求进系统的入口；Service Mesh 管的是东西向流量，也就是服务和服务之间的调用治理。

| 对比项 | API Gateway / 开放网关 | Service Mesh / 服务网格 |
| --- | --- | --- |
| 流量方向 | 南北向，外部到内部 | 东西向，服务到服务 |
| 典型位置 | 系统入口、边界层 | 每个服务旁边的 Sidecar 或数据平面 |
| 关注点 | 路由、鉴权、签名、限流、CORS、审计、协议适配 | 服务间 mTLS、流量治理、熔断、重试、灰度、可观测 |
| 典型组件 | Spring Cloud Gateway、Kong、Nginx、APISIX | Istio、Envoy、Linkerd |
| 和业务关系 | 更贴近 API 暴露和开放平台 | 更贴近基础设施和服务间通信治理 |

**怎么选**

- 只需要统一入口、鉴权、限流、路由、开放接口治理：优先 API Gateway。
- 服务数量很多，跨语言明显，需要统一服务间 mTLS、流量镜像、熔断、灰度、观测：考虑 Service Mesh。
- Java/Spring Cloud 中小规模微服务，Gateway + Feign/Dubbo + Sentinel + Trace 往往已经够用。
- Service Mesh 会引入 Sidecar、控制面、治理规则和运维复杂度，不是所有团队都需要。

**面试表达**

> API Gateway 和 Service Mesh 不是替代关系。Gateway 主要管外部流量进来的统一入口，比如鉴权、签名、限流、路由和审计；Service Mesh 主要管服务间调用，比如 mTLS、服务间熔断重试、灰度和可观测。如果项目主要是 Java Spring Cloud 体系，Spring Cloud Gateway 加 Sentinel、Feign 和链路追踪就能覆盖大部分治理需求；只有服务规模、跨语言和基础设施治理复杂到一定程度，才会考虑 Istio/Envoy 这类 Service Mesh。

## 高频追问速记

| 问题 | 速答 |
| --- | --- |
| OpenFeign 是 RPC 吗？ | 不是典型 RPC，本质是声明式 HTTP 客户端，通过接口代理发 HTTP 请求。 |
| HTTP 和 RPC 有什么区别？ | HTTP 是通信协议，RPC 是远程调用模型；RPC 底层可以用 HTTP、HTTP/2、TCP 或自定义协议。 |
| 为什么 Spring Cloud 偏 HTTP？ | HTTP 通用、跨语言、易调试，和 Spring MVC、Gateway、Ingress、Service Mesh、云原生生态兼容好。 |
| HTTP 和 RPC 怎么选？ | 对外、前后端、跨系统集成优先 HTTP；内部高频、强契约、低延迟调用可以选 Dubbo/gRPC。 |
| HTTP 一定比 RPC 慢吗？ | 不一定，协议只是因素之一；真实耗时还要看连接复用、序列化、线程池、DB、缓存和下游。 |
| 为什么不用 RPC 替代所有 HTTP？ | RPC 更适合内部调用；对外开放、浏览器访问、网关治理和第三方接入仍然是 HTTP 更合适。 |
| Feign 超时有哪几类？ | 连接超时和读取超时；前者偏网络连接，后者偏下游处理慢。 |
| Feign 重试有什么风险？ | 非幂等写接口可能重复执行，下游慢时会放大流量形成重试风暴。 |
| Ribbon 和 LoadBalancer 区别？ | 都是客户端负载均衡，Ribbon 是老 Netflix 体系，新项目更推荐 Spring Cloud LoadBalancer。 |
| Sentinel 的核心概念？ | Resource 和 Rule，资源是保护对象，规则描述限流、熔断、系统保护条件。 |
| Sentinel 限流维度？ | QPS、线程数、调用来源、链路、关联资源、热点参数。 |
| Sentinel 熔断策略？ | 慢调用比例、异常比例、异常数。 |
| Hystrix 的核心思想？ | Command 封装依赖调用，通过隔离、熔断和 Fallback 防止级联故障。 |
| 线程池隔离和信号量隔离怎么选？ | 慢 RPC/第三方用线程池隔离，轻量可信调用可用信号量隔离。 |
| 限流和熔断区别？ | 限流控制入口流量，熔断切断异常下游调用。 |
| 降级最怕什么？ | 返回假成功，尤其核心交易链路会造成数据不一致。 |
| 灰度发布关键是什么？ | 标签透传、路由规则、指标观察、配置开关和快速回滚。 |
| Spring Cloud 有网关吗？ | 有，Spring Cloud Gateway 是 API Gateway，核心是 Route、Predicate、Filter。 |
| API Gateway 和 Service Mesh 区别？ | Gateway 管南北向入口流量，Service Mesh 管东西向服务间流量。 |
| 开放网关要做什么？ | 路由、鉴权、签名、防重放、限流、灰度、审计和服务内权限兜底。 |

## 面试答题模板

### 组件原理题模板

> 这个组件解决的是服务调用稳定性问题。以 Sentinel 为例，它先把接口、方法或参数抽象成资源，再给资源配置规则，基于 QPS、线程数、慢调用比例、异常比例等指标决定通过、限流或熔断。面试时我会结合具体业务资源讲，而不是只背概念。

### 选型题模板

> 如果是 Spring Cloud Alibaba 体系，限流熔断我优先考虑 Sentinel；如果是标准 Spring Cloud CircuitBreaker 体系，可以考虑 Resilience4j；老项目遇到 Hystrix 要理解线程池隔离、熔断和 Fallback，但新项目一般不再主动重度引入。

### 稳定性设计题模板

> 我会先给接口设置合理超时，再判断是否允许重试；对入口流量做限流，对慢下游做熔断，对非核心能力做降级，对不同下游做隔离；最后通过 traceId、指标、日志和告警保证问题能定位。核心写链路一定要有幂等和状态机兜底。

### 排障题模板

> 我会先判断是流量问题、下游问题、规则问题还是发布问题。看入口 QPS、错误率、P99、线程池、连接池、熔断状态、最近规则变更和发布记录；短期先摘流、扩容、回滚或降级，长期修复慢调用、阈值和容量模型。
