# Elasticsearch / OpenSearch 面试指南

这份材料面向 Java 后端搜索与日志检索面试题，重点覆盖倒排索引、mapping、分词、Query DSL、query/filter、分片副本、写入流程、搜索流程、深分页、聚合、集群治理，以及 Elasticsearch 和 OpenSearch 的关系与选型。

| 主题 | 高频问题 | 回答关键词 |
| --- | --- | --- |
| 原理 | 为什么搜索快 | 倒排索引、term dictionary、posting list、segment |

## 1. Elasticsearch 是什么？适合解决什么问题？

**一句话回答**：Elasticsearch 是基于 Lucene 的分布式搜索和分析引擎，适合全文检索、日志检索、聚合分析、近实时搜索和复杂条件过滤。

**适合场景**：

- 商品、订单、工单、文章全文搜索。
- 日志检索和故障排查。
- 多条件筛选、排序、聚合统计。
- 地理位置搜索和向量检索等扩展场景。

**不适合场景**：

- 强事务更新。
- 高频精确 OLTP 写入。
- 替代 MySQL 做主库。
- 大量复杂 join。

**面试表达**：

> ES 更适合做搜索和分析，不适合作为强事务主库。我的理解是，MySQL 保证业务事实和事务一致性，ES 提供搜索、筛选、聚合和日志检索能力，两者通过同步链路保持最终一致。

## 2. OpenSearch 和 Elasticsearch 有什么关系？

**一句话回答**：OpenSearch 是从 Elasticsearch / Kibana 早期开源版本分支出来的开源搜索与分析项目，核心能力和 ES 类似，但版本演进、许可、插件和生态存在差异。

| 对比项 | Elasticsearch | OpenSearch |
| --- | --- | --- |
| 背景 | Elastic 公司主导 | OpenSearch 项目社区/AWS 发起并维护 |
| 核心能力 | 搜索、聚合、日志、向量等 | 搜索、聚合、日志、可观测、安全插件等 |
| API | 很多基础 API 类似 | 与早期 ES 兼容度较高，后续版本有差异 |
| 选型关注 | Elastic Stack 生态 | 开源许可、AWS/OpenSearch 生态 |

**面试表达**：

> 面试里我会把 OpenSearch 理解成 ES 生态的开源分支，核心搜索原理仍然是倒排索引、分片副本、Query DSL 这套。但生产选型不能只看 API 像不像，还要看版本兼容、插件、安全、运维平台和云厂商支持。

## 3. 倒排索引是什么？

**一句话回答**：倒排索引是从词项到文档列表的映射，搜索时先找到关键词对应的文档集合，再计算相关性和排序。

**例子**：

| 文档 | 内容 |
| --- | --- |
| doc1 | Java 搜索 引擎 |
| doc2 | Java 面试 搜索 |

倒排后：

| 词项 | 文档列表 |
| --- | --- |
| Java | doc1, doc2 |
| 搜索 | doc1, doc2 |
| 引擎 | doc1 |
| 面试 | doc2 |

**面试表达**：

> MySQL B+Tree 更适合按字段值精确查找或范围扫描，倒排索引则适合从关键词反查文档。全文检索快，就是因为它不用逐行扫文本，而是先定位词项对应的文档集合。

## 4. text 和 keyword 有什么区别？

**一句话回答**：`text` 会分词，适合全文检索；`keyword` 不分词，适合精确匹配、排序、聚合和过滤。

| 类型 | 是否分词 | 场景 |
| --- | --- | --- |
| text | 是 | 商品标题、文章内容、备注 |
| keyword | 否 | 订单号、状态、用户 ID、标签、枚举 |

**面试表达**：

> 如果字段要全文搜索，用 text；如果字段要精确过滤、排序或聚合，用 keyword。比如商品标题可以 text + keyword 双字段，既支持搜索，也支持精确聚合或排序。

## 5. analyzer 分词器怎么理解？

**一句话回答**：分词器负责把文本拆成 token，并做小写化、停用词、同义词等处理；索引和查询时分词规则要匹配，否则搜索效果会不稳定。

**关键点**：

- 英文常见按单词拆分。
- 中文需要 IK、jieba、智能分词或业务词库。
- 同义词会影响召回。
- 分词变更通常需要重建索引。

**面试表达**：

> 中文搜索效果很大程度取决于分词和词库。比如“上门保洁服务”如果分词不合理，召回会很差。线上调整 analyzer 通常不是改个配置就完事，很多时候要重建索引。

## 6. match 和 term 有什么区别？

**一句话回答**：`match` 会对查询文本分词后检索，适合 text 全文检索；`term` 不分词，按精确词项匹配，适合 keyword、枚举、ID 等字段。

**面试表达**：

> 面试里最容易犯的错是拿 term 查 text 字段。match 会分析查询文本，term 是精确词项匹配。订单号、状态用 term；标题、内容搜索用 match。

## 7. query context 和 filter context 有什么区别？

**一句话回答**：query context 关注“匹配得多好”并计算相关性分数；filter context 只判断是否匹配，不计算分数，通常更快且可缓存。

| 对比项 | query | filter |
| --- | --- | --- |
| 是否算分 | 是 | 否 |
| 适用 | 全文检索、相关性排序 | 状态、时间、租户、范围过滤 |
| 性能 | 相对重 | 更轻，常可缓存 |

**面试表达**：

> 需要相关性排序的条件放 query，比如标题匹配；只是业务过滤的条件放 filter，比如租户、状态、时间范围。这样既能保证搜索效果，也能减少不必要的打分开销。

## 8. bool 查询怎么用？

**一句话回答**：bool 查询用 `must`、`should`、`filter`、`must_not` 组合多个条件，是 ES/OpenSearch 最常用的复合查询。

| 子句 | 含义 |
| --- | --- |
| must | 必须匹配，参与评分 |
| should | 可选匹配，可提高得分 |
| filter | 必须匹配，不参与评分 |
| must_not | 必须不匹配 |

**面试表达**：

> 实际项目里常见写法是：关键词放 must 或 should，状态、租户、时间范围放 filter，排除条件放 must_not。这样可读性好，性能也更稳。

## 9. ES 写入流程怎么理解？

**一句话回答**：写请求根据路由找到主分片，主分片写入内存 buffer 和 translog，再复制到副本；refresh 后生成可搜索的 segment，flush 后提交持久化。

**关键概念**：

- routing：决定文档进入哪个主分片。
- primary shard：先处理写入。
- replica shard：复制主分片数据。
- translog：故障恢复日志。
- refresh：让新文档变得可搜索。
- segment merge：合并小 segment，减少查询开销。

**面试表达**：

> ES 是近实时搜索，不是写完立刻一定能搜到。写入先到主分片，再复制副本，refresh 后才对搜索可见。调小 refresh interval 能更实时，但会增加 segment 和系统开销。

## 10. refresh、flush、merge 有什么区别？

| 概念 | 作用 | 面试理解 |
| --- | --- | --- |
| refresh | 生成可搜索 segment | 影响搜索可见性 |
| flush | 提交 Lucene commit，清理 translog | 影响恢复成本 |
| merge | 合并小 segment | 影响查询性能和 IO |

**面试表达**：

> refresh 解决“什么时候能搜到”，flush 解决“故障恢复日志别太大”，merge 解决“小 segment 太多影响查询”。写入高峰期如果频繁 refresh 和 merge，会对性能有明显影响。

## 11. shard 和 replica 怎么设计？

**一句话回答**：主分片决定数据切分和最大水平扩展边界，副本提高查询能力和容灾；分片数要按数据量、查询压力、节点规模和未来扩展设计。

**注意点**：

- 分片太少：扩展和并行能力不足。
- 分片太多：元数据、文件句柄、内存和调度开销大。
- 副本能提升读和容灾，但会增加写入复制成本。
- 大索引建议用时间维度滚动索引。

**面试表达**：

> 分片不是越多越好。我要看单索引数据量、查询 QPS、节点数量和保留周期。日志类索引更适合按天或按大小 rollover，避免一个索引无限膨胀。

## 12. ES 搜索流程是什么？

**一句话回答**：搜索一般分 query phase 和 fetch phase，协调节点先把查询发给相关分片，各分片返回 TopN 文档 ID 和分数，再由协调节点合并排序并拉取完整文档。

**面试表达**：

> ES 搜索是分布式 TopN。每个分片先查自己的 TopN，协调节点再做全局合并。深分页慢也和这个有关，因为每个分片都要返回更大的候选集合。

## 13. 深分页为什么慢？怎么优化？

**一句话回答**：`from + size` 深分页需要每个分片取大量候选结果再全局排序，页数越深，内存和 CPU 开销越大。

**优化方案**：

- 限制最大页数，只支持前几页。
- 使用 `search_after` 做游标翻页。
- 使用 PIT 保持分页期间视图一致。
- 批量导出用 scroll 或专门导出链路。
- 业务上用“上一页/下一页”替代任意跳页。

**面试表达**：

> 深分页慢不是网络慢，而是每个分片都要取很多数据再排序。用户搜索场景通常限制前几页；后台导出或全量扫描用 search_after、PIT 或 scroll，不能让用户随便查第 10000 页。

## 14. ES 如何做排序和聚合？

**一句话回答**：排序和聚合依赖字段的结构化存储，keyword、数值、日期字段更适合；text 字段不适合直接排序聚合。

**注意点**：

- 排序字段要有 doc_values。
- 高基数字段聚合可能占用大量内存。
- text 字段如需聚合，应使用 keyword 子字段。
- 聚合结果受分片、数据量和精度参数影响。

**面试表达**：

> 排序和聚合要在 mapping 阶段提前设计。比如状态、城市、供应商、站点适合 keyword；标题是 text，但可以加 keyword 子字段用于聚合。不能等线上聚合慢了才发现字段类型设计错了。

## 15. nested 和 object 有什么区别？

**一句话回答**：object 会把数组对象扁平化，可能导致字段交叉匹配；nested 会把数组中的每个对象作为独立隐藏文档，保证对象内部字段关系不乱。

**场景**：

- 商品多个规格：规格名和规格值要绑定，适合 nested。
- 订单多明细：明细内字段要成组匹配，适合 nested。
- 简单对象无数组关联要求，用 object 即可。

**面试表达**：

> 如果数组对象里字段必须成组匹配，就用 nested。否则 object 扁平化后可能出现“颜色=红色”和“尺码=XL”来自不同对象却被匹配到的问题。

## 16. ES 数据如何和 MySQL 保持一致？

**一句话回答**：MySQL 做主库，ES 做搜索索引，通过同步链路保证最终一致，常见方案有业务双写、MQ 异步、Binlog 同步和定时校验补偿。

**方案对比**：

| 方案 | 优点 | 风险 |
| --- | --- | --- |
| 业务双写 | 简单直接 | 一端成功一端失败难处理 |
| MQ 异步 | 解耦，便于重试 | 有延迟，需要幂等 |
| Binlog 同步 | 对业务侵入小 | 链路复杂，字段映射要治理 |
| 定时校验 | 可兜底修复 | 不实时 |

**面试表达**：

> 我不会让 ES 当事实主库。MySQL 更新成功后，通过 MQ 或 Binlog 同步 ES，失败进入重试和补偿。查询允许短暂延迟，关键业务判断仍回 MySQL。

## 17. ES 查询慢怎么排查？

**排查路径**：

1. 看查询 DSL：是否 wildcard、regexp、script、深分页。
2. 看 mapping：字段类型是否合理，是否误用 text/keyword。
3. 看分片：是否分片过多、热点分片、单分片过大。
4. 看聚合：是否高基数字段聚合。
5. 看资源：CPU、heap、GC、磁盘 IO、线程池队列。
6. 看数据：是否冷热数据混查，是否缺少时间过滤。
7. 用 profile / slow log 定位慢查询段。

**面试表达**：

> ES 慢查询我先看 DSL 和 mapping，再看分片和资源。很多问题不是机器不够，而是 wildcard、深分页、高基数聚合或字段类型设计不合理。

## 18. ES 写入慢怎么排查？

**排查路径**：

- bulk 大小是否合理。
- refresh interval 是否过短。
- replica 是否过多。
- segment merge 是否压力大。
- 磁盘 IO 是否打满。
- mapping 是否动态字段爆炸。
- 写入是否集中到热点分片。

**面试表达**：

> 写入慢要看 bulk、refresh、replica、merge 和热点分片。日志类场景通常批量写入，适当调大 refresh interval；但搜索实时性要求高的业务要权衡延迟和写入吞吐。

## 19. OpenSearch 选型要注意什么？

**一句话回答**：OpenSearch 核心搜索能力和 ES 类似，但生产选型要重点确认版本兼容、客户端兼容、插件能力、安全能力、运维工具和云服务支持。

**面试表达**：

> 如果公司用 OpenSearch，我不会把它简单说成 ES 改名。核心原理相似，但版本、插件和 API 兼容要逐项确认。尤其是客户端版本、监控、安全插件和索引模板迁移，不能默认完全兼容。

## 20. 如何结合项目讲搜索系统？

**回答结构**：

1. MySQL 是事实主库，ES/OpenSearch 是搜索索引。
2. 按业务设计 mapping：订单号 keyword、标题 text、状态 keyword、时间 date。
3. 写入链路通过 MQ/Binlog 同步。
4. 查询链路用 bool + filter 优化。
5. 深分页、聚合、慢查询有专门治理。

**面试表达**：

> 在订单或工单搜索里，我会让 MySQL 保存真实状态，ES/OpenSearch 支撑多条件搜索。订单号、状态、站点用 keyword，备注和标题用 text。数据通过 MQ 或 Binlog 同步，查询时把租户、状态、时间放 filter，关键词放 query。异常同步通过补偿任务修复。
