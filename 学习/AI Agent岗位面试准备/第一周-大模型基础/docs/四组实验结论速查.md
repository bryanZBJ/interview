# 四组实验结论速查

## 使用说明

这份文档用于快速掌握第一周四组实验的核心结论，适合面试前复习。文中是原理和工程上的预期结论，不是你已经完成的实测数据。面试时可以说“我理解的结论是”，没有实际执行时不要说“我实验证明了”。

## 一、Token 与上下文

### 核心结论

- Token 是 Tokenizer 切分出的模型处理单位，不等于字符数、汉字数或单词数。
- 要求模型生成更长的回答，主要增加 `completion_tokens`。
- 每次携带更多历史消息，主要增加 `prompt_tokens`。模型不是自动永久记住对话，而是客户端把历史 `messages` 再次发送。
- `total_tokens` 通常等于输入 Token 与输出 Token 之和，并受模型上下文窗口限制。
- Token 增多通常会提高成本、预处理时间和 KV Cache 占用，但端到端延迟还受网络、服务负载和生成速度影响，不能只凭一次请求归因。
- `max_tokens` 限制最大输出 Token 数。达到上限时，`finish_reason` 通常为 `length`。

### 生产建议

控制历史轮数，删除无关消息，对旧对话做摘要，并同时监控 `prompt_tokens`、`completion_tokens`、总耗时和调用成本。

### 面试口述

> Token 是模型分词后的处理单位，不等于字符数。回答变长主要增加输出 Token，携带更多历史主要增加输入 Token。生产中我会裁剪或摘要旧上下文，同时监控 Token、延迟和成本。

## 二、采样参数

### 核心结论

- `temperature` 调整概率分布的集中程度。低值通常更稳定，高值通常更多样。
- `top_p` 保留累计概率达到阈值的最小动态候选集。`top_p=0.3` 不是固定保留 30 个 Token。
- `top-k` 固定保留概率最高的 K 个候选 Token，与 `top_p` 的动态候选集不同。
- 调参时应固定 Prompt 和其他参数，一次只改一个变量，否则无法判断输出变化由谁引起。
- 采样存在随机性，单次输出不能当作稳定规律。
- 低 `temperature` 只能提高稳定性，不能保证事实正确，也不能消除幻觉。

### 生产建议

客服分类、参数抽取等需要稳定性的任务使用较低随机性；文案创作可适当提高多样性。事实正确性仍要依靠数据源和后端校验，不能依赖采样参数。

### 面试口述

> temperature 调整整体概率分布，top-p 根据累计概率动态截断候选集，top-k 则固定保留前 K 个候选。它们影响生成多样性，不保证事实正确。调参时我会使用控制变量法，通常一次只重点调整一个参数。

## 三、幻觉对照

### 核心结论

- 幻觉是模型生成了表达流畅、看似合理，但缺少事实依据或与事实冲突的内容。
- 没有提供事实材料时，模型更容易根据常识补全政策、时间或联系方式。
- 提供材料能降低幻觉，但模型仍可能增加材料中没有的细节。
- “只能根据材料回答 + 无依据拒答 + 标注引用”能进一步降低风险，但不是绝对保证。
- “有引用”不等于“引用正确”，后端仍要检查引用是否真正支持结论。
- Prompt 不是权限系统、规则引擎或事实数据库。模型不能自行承诺退款成功、订单存在等真实业务结果。

### 生产建议

使用可信数据源或 RAG 提供证据，要求无依据拒答和标注引用，然后由后端查询真实业务数据、校验规则和权限，高风险操作加入人工审核。

### 面试口述

> 降低幻觉需要可信材料、限定证据范围、无依据拒答、引用和后端复核的组合。Prompt 只能降低风险，不能把概率生成模型变成绝对可信的业务数据源。

## 四、结构化输出

### 核心结论

- JSON Output 主要解决“输出能否被 JSON 解析”，不自动保证字段、枚举、格式和业务语义正确。
- JSON Schema 解决结构约束，例如必填字段、枚举值、数值范围、正则格式和是否允许额外字段。
- 通过 Schema 校验只能证明数据“形状正确”，不能证明当前用户拥有该订单，也不能证明订单真实存在。
- 模型识别出“申请退款”不等于“退款已成功”。缺少订单号时应该补问或转人工，不能直接执行。
- 模型返回的 `confidence` 是自报值，可用于辅助路由，不能当作真实准确率或跳过审核的依据。
- 生产链路应该分层校验：JSON 语法 -> Schema -> 权限 -> 业务真实性 -> 决定执行或转人工。

### 生产建议

使用标准 JSON 解析器和 JSON Schema 校验器，不用正则表达式解析 JSON。退款、改地址等高风险动作必须查询真实系统并通过权限和状态校验，必要时要求人工确认。

### 面试口述

> 结构化输出只是把自然语言变成可解析协议。可靠链路应先做 JSON 语法和 Schema 校验，再校验当前用户权限，最后查询订单系统确认业务真实性。只有全部通过，才可能执行真实动作。

## 五、快速验收

不看上文，能用自己的话回答下面 8 题，即可进入第二周。

### Q1：Token 为什么不等于汉字数？

**参考回答：**Token 是模型的 Tokenizer 根据词表和分词算法切分出的处理单位。一个汉字可能被分成一个或多个 Token，常见词组也可能合并为一个 Token，标点、数字和空格也可能占用 Token。因此不能用字符串长度代替真实 Token 数，应使用对应模型的 Tokenizer 或 API 返回的 `usage` 统计。

### Q2：历史对话为什么会增加 `prompt_tokens`？

**参考回答：**模型通常不会自动永久记住上一次对话。为了让模型理解上下文，客户端会在当前请求的 `messages` 中重新携带历史消息。这些历史内容都属于本次请求的输入，所以会增加 `prompt_tokens`，并进一步增加成本、预处理时间和上下文窗口占用。

### Q3：`temperature`、`top_p`、`top-k` 有什么区别？

**参考回答：**`temperature` 缩放 logits，调整整体概率分布的集中程度；`top_p` 保留累计概率达到阈值的最小动态候选集；`top-k` 固定保留概率最高的 K 个候选。三者都会影响下一个 Token 的采样和输出多样性，但不能保证事实正确。工程调参时应固定其他条件，一次只重点调整一个参数。

### Q4：低 `temperature` 为什么不能消除幻觉？

**参考回答：**低 `temperature` 只是让模型更倾向选择高概率 Token，使输出相对稳定，它不会补齐缺失的知识，也不会验证训练数据、参考材料或业务事实。如果模型的高概率答案本身就是错的，低温度反而可能更稳定地输出错误结论。降低幻觉还需要可信数据、RAG、拒答规则和后端校验。

### Q5：提供参考材料后为什么仍需要无依据拒答？

**参考回答：**参考材料可能不完整、过期或与问题无关，模型也可能在材料之外自行补全细节。无依据拒答规则用来明确证据边界：材料没有支持的结论就回答“无法确认”，而不是根据常识猜测。生产系统还要检查材料时效性、来源权限和结论是否被证据支持。

### Q6：有引用为什么不等于引用正确？

**参考回答：**模型可能编造不存在的引用，也可能引用了真实片段，但该片段并不支持它的结论。即使引用内容正确，也可能已过期、版本不匹配或来自当前用户无权访问的文档。因此后端需要验证引用来源、权限、版本，并检查引用片段是否真正支持结论。

### Q7：合法 JSON 为什么不等于符合 JSON Schema？

**参考回答：**合法 JSON 只说明字符串符合 JSON 语法，能被标准解析器解析。JSON Schema 还会校验必填字段、数据类型、枚举范围、数值区间、字符串格式和是否允许额外字段。例如 `{"confidence":"high"}` 是合法 JSON，但如果 Schema 要求 `confidence` 是 0 到 1 的数字，它就不合格。

### Q8：通过 Schema 校验后，为什么还要校验权限和业务真实性？

**参考回答：**Schema 只能证明数据的结构和格式正确，不能证明订单真实存在、当前用户拥有该订单，也不能证明订单当前状态允许退款或改地址。因此必须再根据当前登录用户校验数据权限，并调用订单、库存或支付等真实业务系统查询最新状态。只有格式、权限、业务规则都通过，才能执行高风险动作。
