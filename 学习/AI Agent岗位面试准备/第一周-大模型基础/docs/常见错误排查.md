# 常见错误排查

## 1. 先做五分钟基线检查

在 `第一周-大模型基础/java-demo` 执行：

```bash
java -version
mvn -version
rg -n 'base-url|api-key|model|mock-enabled|connect-timeout|response-timeout' src/main/resources/application.yml
printf 'LLM_MOCK_ENABLED=%s\n' "${LLM_MOCK_ENABLED:-未设置，默认 true}"
test -n "$DEEPSEEK_API_KEY" && echo 'DEEPSEEK_API_KEY 已设置' || echo 'DEEPSEEK_API_KEY 未设置'
mvn test
```

不要执行 `echo "$DEEPSEEK_API_KEY"`，也不要把环境变量快照、完整 Authorization Header 或上游原始错误正文贴进文档和聊天记录。

排查前先分清模式：响应内容以“Mock 回答：”开头且 `usage` 为 0，说明走的是 `MockLlmClient`；只有 `LLM_MOCK_ENABLED=false` 才会装配 `DeepSeekLlmClient`。

## 2. HTTP 与网络错误

### 2.1 401/403：鉴权失败

**现象**

- DeepSeek 上游返回 401 或 403。
- `DeepSeekLlmClient` 映射为 `AUTHENTICATION`。
- 普通接口经 `GlobalExceptionHandler` 对外返回 502 和“大模型认证失败，请联系管理员”。

**检查清单**

- [ ] 已设置 `DEEPSEEK_API_KEY`，且值不是示例占位符。
- [ ] `LLM_MOCK_ENABLED=false`，确认确实在测真实模式。
- [ ] Key 前后没有引号残留、换行或多余空格。
- [ ] `WebClientConfiguration` 仍使用 `Authorization: Bearer <key>`。
- [ ] Key 未被撤销，账号权限与调用模型匹配。
- [ ] 日志和响应中没有泄漏 Key 或上游原始正文。

**本地验证**

```bash
mvn -Dtest=DeepSeekLlmClientTest#mapsAuthenticationFailures test
```

该测试只验证错误映射，不会验证你的真实 Key。

### 2.2 404：路径或 base URL 错误

**现象**

- 上游返回 404，当前代码会按其他 4xx 映射为 `UPSTREAM`，普通接口对外为 502。

**检查清单**

- [ ] 默认 base URL 是 `https://api.deepseek.com`。
- [ ] `DeepSeekLlmClient` 的相对路径是 `/chat/completions`。
- [ ] 自定义 `LLM_BASE_URL` 没有误带错误代理路径、重复版本段或指向网页地址。
- [ ] 如果公司网关改写路径，已核对网关实际转发规则。
- [ ] 不要只看本项目 502，要结合安全日志中的 `LlmErrorType.UPSTREAM` 和代理侧状态判断。

### 2.3 429：限流

**现象**

- 上游 429 被映射为 `RATE_LIMIT`，普通接口对外仍返回 429。

**检查清单**

- [ ] 降低并发和请求频率，核对 RPM/TPM 限制。
- [ ] 限制用户输入与 `maxTokens`，避免单请求消耗过高。
- [ ] 使用有上限的指数退避并加入随机抖动。
- [ ] 不要立即无限重试，否则会放大限流。
- [ ] 给调用方返回可理解的稍后重试提示。

**本地验证**

```bash
mvn -Dtest=DeepSeekLlmClientTest#mapsRateLimitFailure test
```

### 2.4 5xx：上游故障或过载

**现象**

- 上游 500/503 等被映射为 `UPSTREAM`，普通接口对外为 502。

**检查清单**

- [ ] 记录 requestId、模型、耗时和状态，不记录 Prompt 全文或 Key。
- [ ] 确认不是本地代理/DNS/TLS 故障伪装成上游错误。
- [ ] 对幂等聊天请求可做少量、有退避的重试；设置总重试时限。
- [ ] 设定熔断和降级策略，必要时切回明确标识的 Mock 仅用于开发。
- [ ] 持续故障时查看 DeepSeek 官方状态与错误码文档。

**本地验证**

```bash
mvn -Dtest=DeepSeekLlmClientTest#mapsServerFailure test
```

### 2.5 连接超时：连接没有建立

**现象**

- DNS、路由、代理、TLS 或目标不可达导致连接阶段失败。
- `ConnectTimeoutException` 映射为 `TIMEOUT`；其他 `WebClientRequestException` 可能映射为 `UPSTREAM` 和“无法连接模型服务”。

**检查清单**

- [ ] 核对 `LLM_BASE_URL` 域名、协议和端口。
- [ ] 检查 DNS、VPN、公司代理、防火墙和证书链。
- [ ] 检查 `LLM_CONNECT_TIMEOUT`，默认 3 秒是否适合当前网络。
- [ ] 区分“连接失败”和“已连接但等不到响应”。
- [ ] 不要通过无限增大连接超时掩盖路由问题。

### 2.6 响应超时：已发出请求但迟迟无响应

**现象**

- Reactor Netty 的 `responseTimeout` 到期，或测试服务器使用 `NO_RESPONSE`。
- 映射为 `TIMEOUT`，普通接口对外 504。

**检查清单**

- [ ] 检查 `LLM_RESPONSE_TIMEOUT`，默认 60 秒。
- [ ] 缩短 Prompt、上下文和 `maxTokens` 后重试。
- [ ] 对流式接口区分“首字节超时”和“分片中途停顿”。
- [ ] 查看同一 requestId 的开始、失败和耗时日志。
- [ ] 超时后确认订阅被取消，避免后台继续消耗。

**本地验证**

```bash
mvn -Dtest=DeepSeekLlmClientTest#mapsNetworkResponseTimeout test
mvn -Dtest=DeepSeekLlmClientTest#mapsStreamingTimeoutLikeRegularCall test
```

## 3. 响应协议错误

### 3.1 非法 JSON

**现象**

- 普通响应无法反序列化，或 SSE data 不是合法 JSON。
- `mapFailure`/`toStreamChunk` 最终映射为 `PROTOCOL`，普通接口对外 502。

**检查清单**

- [ ] 查看 Content-Type 是否与预期一致。
- [ ] 排除代理返回 HTML 登录页、WAF 页面或网关错误页。
- [ ] 检查是否把 SSE 的 `data:` 前缀或 `[DONE]` 当 JSON 解析。
- [ ] 保存脱敏后的最小失败样例用于 `MockWebServer` 回归测试。
- [ ] 不向调用方透传上游原始正文。

**本地验证**

```bash
mvn -Dtest=DeepSeekLlmClientTest#mapsEmptyOrInvalidSseDataToProtocolFailure test
```

### 3.2 空 choices 或必要字段缺失

**现象**

- 普通响应 `choices=[]`，或第一项缺少 `message.content`/`usage`。
- `DeepSeekLlmClient.toResult` 抛出 `PROTOCOL`。

**检查清单**

- [ ] 区分普通响应和流式 usage 分片：普通空 choices 是错误，流式 include_usage 分片空 choices 是合法的。
- [ ] 检查上游是否升级了响应 Schema。
- [ ] 检查内容安全、工具调用或推理模式是否改变了字段形态。
- [ ] 不要用空字符串假装成功；先确认业务是否允许无文本结果。
- [ ] 为新合法响应形态增加显式模型和测试。

**本地验证**

```bash
mvn -Dtest=DeepSeekLlmClientTest#mapsEmptyChoicesToProtocolFailure test
mvn -Dtest=DeepSeekLlmClientTest#suppressesEmptyDeltaForUsageChunk test
```

### 3.3 SSE 中断或缺少 `[DONE]`

**现象**

- 已收到部分 `delta`，连接却提前断开。
- 当前代码在没有收到 `[DONE]` 时追加 `PROTOCOL` 错误，不会发送应用侧 `done`。

**检查清单**

- [ ] 客户端是否主动关闭、页面跳转或触发 `AbortController`。
- [ ] 网关是否关闭长连接、开启响应缓冲或设置过短 idle timeout。
- [ ] 上游是否确实发送 `data: [DONE]`。
- [ ] 是否出现非法/空 SSE data。
- [ ] 日志是 `cancelled`、`TIMEOUT` 还是 `PROTOCOL`，三者不要混为一谈。
- [ ] 当前实现没有自定义 `event:error`，前端必须处理连接异常和“未收到 done”。
- [ ] 不要把已收到的部分文本自动标记为完整答案。

**本地验证**

现有测试覆盖正常 `[DONE]`、非法 data 和超时；若要稳定复现“合法 delta 后直接 EOF”，应在 `DeepSeekLlmClientTest` 增加一个不含 `[DONE]` 的 MockWebServer 用例。当前文档不虚构该测试已经存在。

## 4. 应用侧错误

### 4.1 400 VALIDATION_ERROR

检查 `message` 非空且不超过有效上限，`systemPrompt<=2000`，`temperature` 在 0 到 2，`topP` 在 0 到 1，`maxTokens` 在 1 到 4096。普通 POST 使用 Bean Validation，流式 GET 由 `ChatController.validateStreamParameters` 手工校验。

### 4.2 503 CONFIGURATION

真实模式缺少 API Key 时，`DeepSeekLlmClient.validateApiKey` 在发送请求前抛 `CONFIGURATION`。检查环境变量是否注入到启动该 Java 进程的同一个 Shell/IDE Run Configuration。Mock 模式允许 Key 为空。

### 4.3 端口占用或服务未启动

```bash
lsof -nP -iTCP:8080 -sTCP:LISTEN
curl -i 'http://localhost:8080/api/chat/stream?message=hello'
```

若 8080 被占用，可设置 `SERVER_PORT=8081` 后重启，并同步修改 curl 地址。

## 5. 完整排错覆盖清单

- [ ] 401/403：Key、Bearer Header、权限、无敏感信息泄漏。
- [ ] 404：base URL、`/chat/completions`、代理路径。
- [ ] 429：RPM/TPM、并发、退避、重试上限。
- [ ] 5xx：上游状态、有限重试、熔断降级。
- [ ] 连接超时：DNS、代理、TLS、路由、`connect-timeout`。
- [ ] 响应超时：Prompt/输出长度、`response-timeout`、取消传播。
- [ ] 非法 JSON：Content-Type、HTML/WAF、SSE data、脱敏样例。
- [ ] 空 choices：区分普通错误与流式 usage 合法分片。
- [ ] SSE 中断：取消、网关 idle timeout、`[DONE]`、部分答案状态。
- [ ] 参数校验：普通 POST 与流式 GET 的范围一致。
- [ ] 配置错误：真实模式缺 Key，Mock 模式不要求 Key。
- [ ] 日志安全：只记录 requestId、模型、流式标记、耗时和结果，不记录 Key 与上游原文。
