# 面试问答与三分钟讲稿

## 一、10 个面试问答

### Q1：LLM 是如何生成回答的？Token 又是什么？

**回答：** LLM 根据当前上下文计算下一个 Token 的概率分布，按采样策略选出一个 Token，再把它加入上下文继续预测，直到满足停止条件。Token 是 tokenizer 切分后的模型处理单位，不等于字符或单词。工程上要关注输入、输出 Token、上下文上限、耗时和 `finish_reason`。

### Q2：temperature、top-p、top-k 有什么区别？

**回答：** temperature 缩放 logits，改变概率分布的集中程度；top-p 保留累计概率达到阈值的最小动态候选集合；top-k 固定只保留概率最高的 K 个候选。三者都影响下一 Token 采样，但机制不同。我的 Demo 实际只发送 `temperature` 和 `top_p`，没有 `top_k`，实验时一次只改一个参数。

### Q3：上下文窗口和模型记忆是一回事吗？

**回答：** 不是。上下文窗口是单次推理可见的 Token 范围，模型不会天然记住另一条独立 HTTP 请求。要做多轮对话，应用需要保存必要历史并在下一次请求中重新发送，还要做裁剪、摘要或检索，防止超过上下文上限。当前 Demo 只发送可选 system 和本轮 user，是单轮实现。

### Q4：怎样降低大模型幻觉？

**回答：** 我会先限定可靠材料和回答边界，要求材料缺失时拒答并给出证据；关键场景再接检索、数据库或工具；结果返回后做事实核验、Schema 校验和人工确认。Prompt 只能降低风险，不能保证事实绝对正确。

### Q5：为什么要求模型输出 JSON 仍不够？

**回答：** 因为“像 JSON”、合法 JSON、符合业务 Schema 是三层不同保证。即使语法合法，字段也可能缺失、类型错误或出现非法枚举。应用侧仍需 Jackson 反序列化、Schema 或 Bean Validation、枚举白名单和长度限制。当前 Demo 尚未增加 `response_format` 和内容 Schema 校验，我不会把它说成已实现。

### Q6：Java 普通聊天的真实调用链是什么？

**回答：** `POST /api/chat` 进入 `ChatController.chat`，完成参数校验并构造 `ChatCommand`；`ChatService.chat` 生成 requestId、记录生命周期并调用 `LlmClient.chat`；Spring 根据配置注入 `MockLlmClient` 或 `DeepSeekLlmClient`。真实客户端用 WebClient 调 `/chat/completions`，解析为 `ChatResult`，最后由 `ChatResponse.from` 返回 JSON。

### Q7：为什么要设计 `LlmClient` 接口和 Mock 实现？

**回答：** `LlmClient` 把业务层与具体模型厂商协议隔离，普通和流式方法形成稳定契约。Mock 让没有 Key、断网或不想消耗额度时仍能验证 Controller、Service 和 SSE 流程。两种实现通过 `@ConditionalOnProperty` 互斥装配，业务代码无需判断模式。但 Mock 成功不能证明真实 DeepSeek 可用。

### Q8：SSE 流式链路如何保证顺序和正确结束？

**回答：** `DeepSeekLlmClient.stream` 先发 meta，再用 WebClient 的 `bodyToFlux(String.class)` 读取上游 SSE，并用 `concatMap` 顺序转换 `delta.content`。空 choices 的 usage 分片和空 delta 会被过滤；收到上游 `[DONE]` 后才生成应用侧 done。如果连接结束却没有 `[DONE]`，代码按协议错误处理，避免把部分答案标为完整。

### Q9：客户端关闭 SSE 后，服务端怎样避免继续消耗？

**回答：** WebFlux 下游断开会触发订阅取消，取消信号沿 Reactor 链传播到 WebClient 上游；`ChatService.stream` 用 `doOnCancel` 记录 requestId、模型和耗时。生产环境还要结合网关超时和连接指标验证。当前实现没有断点续传，也没有自定义 SSE error 事件。

### Q10：这个 Demo 做了哪些错误治理？还缺什么？

**回答：** 当前实现校验输入和采样参数，Key 通过环境变量注入，配置连接和响应超时；把 401/403、429、5xx、连接失败、超时、非法 JSON、空 choices 和 SSE 协议异常分类为 `LlmErrorType`，普通接口再返回不含上游敏感正文的统一错误。生产化还应补充受控重试、熔断、指标、流式 error 事件、流式 usage、会话管理、结构化输出校验和更完整的取消集成测试。

## 二、三分钟项目介绍

### 0:00-0:30：项目目标

> 我做了一个 AI Agent 岗位面试第一周的大模型基础学习 Demo，技术栈是 Java 17、Spring Boot 3、WebFlux 和 WebClient。目标不是直接上 Spring AI 或复杂 Agent 框架，而是先把 Token、上下文、采样、幻觉、DeepSeek Chat Completion 协议，以及普通和流式调用链真正弄清楚。

### 0:30-1:10：架构与普通链路

> 项目按 Controller、Service、Client 分层。ChatController 负责 HTTP 参数校验和 DTO 转换；ChatService 生成 UUID requestId，记录开始、成功、失败和耗时；LlmClient 定义普通 Mono 和流式 Flux 两个契约。Spring 根据 `llm.mock-enabled` 互斥装配 MockLlmClient 或 DeepSeekLlmClient，所以业务层不依赖具体厂商。普通请求最终由 WebClient 调用 `/chat/completions`，把第一项 choice、finish_reason 和 usage 映射为业务响应。

### 1:10-1:50：流式链路

> 流式接口使用 SSE。DeepSeekLlmClient 先生成 meta，再逐条解析上游 data 中的 delta.content，并通过 concatMap 保证顺序。因为请求设置了 `stream_options.include_usage=true`，上游可能在 `[DONE]` 前发送一个 choices 为空的 usage 分片，当前代码会过滤它。收到 `[DONE]` 后才产生应用侧 done；如果连接提前结束，则按协议错误处理。客户端断开会触发 Reactor 取消并向 WebClient 上游传播，ChatService 会记录取消日志。

### 1:50-2:30：工程治理

> 工程上我没有把 API Key 写入代码，而是通过环境变量绑定到 LlmProperties，只在 Authorization Bearer Header 中使用。输入长度、temperature、topP 和 maxTokens 在调用前校验；Reactor Netty 配置连接和响应超时。客户端把 401/403、429、5xx、超时、非法 JSON、空 choices 等情况转换成内部错误类型，GlobalExceptionHandler 再返回安全消息，不把上游正文、Key 或堆栈暴露给调用方。

### 2:30-3:00：测试、边界与收获

> 测试分为 WebTestClient、服务测试和 MockWebServer 协议测试，覆盖请求字段、Bearer Header、普通解析、SSE 顺序、`[DONE]`、鉴权、限流、5xx、超时和协议错误。这些测试不调用真实 DeepSeek；Mock 成功也不等于真实 API 成功，真实结果必须在本人配置 Key 后单独验证。当前边界是没有会话记忆、流式 usage 对外输出、结构化 Schema 校验和自定义 SSE error 事件。通过这个项目，我能从模型概念讲到 Java 异步调用和生产风险，而不只是会调一个 SDK。

## 三、讲稿使用方式

1. 第一遍照稿读，控制在 3 分钟左右。
2. 第二遍只看四个关键词：目标、分层、流式、治理。
3. 第三遍脱稿，并请对方从 10 个问题中随机追问。
4. 若被问到真实 API 效果，只陈述自己实际执行并记录过的结果；未执行就明确说“当前以 MockWebServer 和 Mock 模式完成协议与流程验证，真实调用待使用有效 Key 验证”。
