# 实验二：采样参数

## 目标

1. 使用同一个商品标题 Prompt，对比不同 `temperature` 下输出的稳定性和多样性。
2. 固定其他条件，只调整 `top_p`，观察候选范围收缩后的变化。
3. 掌握控制变量法：DeepSeek 请求通常一次只调整 `temperature` 或 `top_p`，不同时调整两者。
4. 能解释 `top-k` 的通用概念，但不把它伪造为本实验的 DeepSeek 请求字段。

## 准备

- 执行：`export DEEPSEEK_API_KEY='你的密钥'`。
- 安装 `curl` 和 `jq`。
- 使用同一模型、同一 Prompt、同一 `max_tokens`，每个参数值连续执行 3 次。
- DeepSeek 非思考模式才适合观察本实验参数；若当前模型或模式声明忽略采样参数，应先切换到支持采样的模式。
- 本实验的每个请求都显式携带 `thinking:{type:"disabled"}`，避免默认思考模式干扰采样对照。
- 官方接口说明通常建议修改 `temperature` 或 `top_p` 其中一个，不建议同时修改。本实验严格遵守这一点。

## 逐步操作

1. 运行温度组：请求体只发送 `temperature`，依次取 `0.1`、`0.7`、`1.3`，每个值执行 3 次。
2. 不要在温度组请求体中发送 `top_p`，由服务端使用默认值。
3. 运行 nucleus sampling 组：请求体只发送 `top_p`，依次取 `0.3`、`0.8`、`1.0`，每个值执行 3 次。
4. 不要在 `top_p` 组请求体中发送 `temperature`，由服务端使用默认值。
5. 对每次结果记录标题、是否重复、事实偏离、可读性、Token 和耗时。
6. 只根据多次样本描述趋势，不把单次结果当成确定规律。

## 固定 Prompt / 样例

所有请求固定使用以下 Prompt：

```text
你是跨境电商文案助手。请为下面商品生成1个中文标题，要求30字以内，包含核心品类、材质和一个卖点，不得编造未提供的信息，只输出标题。

商品资料：不锈钢保温杯，容量500ml，食品级304内胆，支持12小时保温，颜色为黑色。
```

事实边界固定为：

```text
允许出现：保温杯、不锈钢、500ml、304内胆、12小时保温、黑色。
视为编造：防漏、智能测温、316材质、24小时保温、赠品等资料中没有的信息。
```

## 可复制请求或调用方式

先定义公共变量：

```bash
export PROMPT='你是跨境电商文案助手。请为下面商品生成1个中文标题，要求30字以内，包含核心品类、材质和一个卖点，不得编造未提供的信息，只输出标题。

商品资料：不锈钢保温杯，容量500ml，食品级304内胆，支持12小时保温，颜色为黑色。'
```

### temperature 组：只发送 temperature

```bash
for TEMPERATURE in 0.1 0.7 1.3; do
  for RUN in 1 2 3; do
    echo "temperature=$TEMPERATURE run=$RUN"
    jq -n \
      --arg prompt "$PROMPT" \
      --argjson temperature "$TEMPERATURE" \
      '{model:"deepseek-v4-flash",thinking:{type:"disabled"},messages:[{role:"user",content:$prompt}],temperature:$temperature,max_tokens:100}' |
      curl -sS https://api.deepseek.com/chat/completions \
        -H "Authorization: Bearer $DEEPSEEK_API_KEY" \
        -H 'Content-Type: application/json' \
        -d @- |
      jq '{content:.choices[0].message.content,usage:.usage,finish_reason:.choices[0].finish_reason}'
  done
done
```

检查点：上面的 JSON 只有 `temperature`，没有 `top_p`，也没有 `top_k`。

### top_p 组：只发送 top_p

```bash
for TOP_P in 0.3 0.8 1.0; do
  for RUN in 1 2 3; do
    echo "top_p=$TOP_P run=$RUN"
    jq -n \
      --arg prompt "$PROMPT" \
      --argjson top_p "$TOP_P" \
      '{model:"deepseek-v4-flash",thinking:{type:"disabled"},messages:[{role:"user",content:$prompt}],top_p:$top_p,max_tokens:100}' |
      curl -sS https://api.deepseek.com/chat/completions \
        -H "Authorization: Bearer $DEEPSEEK_API_KEY" \
        -H 'Content-Type: application/json' \
        -d @- |
      jq '{content:.choices[0].message.content,usage:.usage,finish_reason:.choices[0].finish_reason}'
  done
done
```

检查点：上面的 JSON 只有 `top_p`，没有 `temperature`，也没有 `top_k`。

### top-k 只讲概念

`top-k` 是通用采样概念：每一步只保留概率最高的 K 个候选 Token，再从中采样。它和 `top_p` 的区别是，`top-k` 的候选数量固定，`top_p` 按累计概率动态决定候选数量。

本实验不向 DeepSeek 请求体发送 `top_k`。面试中可以讲原理，但不能因为其他模型支持该字段，就声称本实验所用 DeepSeek Chat 请求也支持它。

## 预期观察

- 较低 `temperature` 的输出通常更集中、更稳定；较高值通常增加表达多样性，也可能增加偏离事实约束的风险。
- 较低 `top_p` 会缩小参与采样的累计概率范围，输出通常更保守；较高值允许更宽的候选范围。
- “通常”不等于每次都如此。模型服务、Prompt 和候选概率分布都会影响结果，因此每组必须至少执行 3 次。
- 采样参数不能替代事实校验。即使 `temperature=0.1`，模型仍可能生成资料中没有的卖点。
- 同时修改 `temperature` 和 `top_p` 会让实验难以归因，也不符合 DeepSeek 官方通常只调整其中一个的建议。

## 结论

`temperature` 调整概率分布的随机程度，`top_p` 用累计概率质量限制候选范围。两者都影响生成多样性，但工程调参要使用控制变量法，通常一次只调一个。`top-k` 可以作为通用概念理解，本实验不把它放进 DeepSeek 请求字段。
