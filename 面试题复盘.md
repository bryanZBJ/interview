### 1. JVM 运行时内存区域怎么划分？

**你的回答**

> 堆、虚拟机栈、本地方法栈，程序计数器和元数据。其中程序计数器、虚拟机栈和本地方法栈是线程私有，堆和元数据是线程共享。

**点评**

**评分：8/10**

分类是对的，术语建议更标准：把“元数据”表达成“方法区 / 元空间”。JDK 8 以后永久代移除，类元信息主要放在本地内存的元空间中。

**标准回答**

JVM 运行时内存区域可以分为线程私有和线程共享两类：

| 分类 | 区域 | 说明 |
| --- | --- | --- |
| 线程私有 | 程序计数器 | 记录当前线程执行到哪条字节码指令 |
| 线程私有 | Java 虚拟机栈 | 方法调用产生栈帧 |
| 线程私有 | 本地方法栈 | Native 方法调用栈 |
| 线程共享 | Java 堆 | 存放对象实例和数组，是 GC 主要管理区域 |
| 线程共享 | 方法区 / 元空间 | 存放类元信息、方法元数据、运行时常量池等 |

**面试表达**

> JVM 内存区域可以按线程私有和线程共享来讲。线程私有包括程序计数器、Java 虚拟机栈和本地方法栈；线程共享包括 Java 堆和方法区，JDK 8 以后方法区的主要实现是元空间。线上排查 OOM 时不能只看 Java 堆，还要关注元空间、直接内存和线程数量。

### 2. 堆、栈、方法区/元空间分别存什么？

**你的回答**

> 堆存的是对象和数组，栈是变量，元空间存的是类元信息，对象引用在虚拟机栈，对象实例在堆。

**点评**

**评分：8.5/10**

主干正确。栈不要只说“变量”，更准确是“方法调用产生的栈帧”，栈帧里包含局部变量表、操作数栈、动态链接和方法返回地址。

**标准回答**

| 区域 | 存放内容 | 面试重点 |
| --- | --- | --- |
| 堆 | 对象实例、数组 | GC 主战场，缓存膨胀、大对象、内存泄漏主要看这里 |
| 虚拟机栈 | 栈帧、局部变量表、操作数栈、动态链接、返回地址 | 递归过深、栈帧过大可能导致栈溢出 |
| 方法区 / 元空间 | 类元信息、方法元数据、字段信息、运行时常量池 | 动态代理、CGLIB、热部署、类加载器泄漏可能导致元空间 OOM |

对象引用如果是方法里的局部变量，通常在栈帧的局部变量表中；`new` 出来的对象实例在堆中。

**面试表达**

> 堆主要存对象实例和数组，是 GC 主要管理区域。虚拟机栈跟线程方法调用有关，每次方法调用会创建一个栈帧，里面有局部变量表、操作数栈、动态链接和返回地址。方法区在 JDK 8 后主要由元空间实现，存放类元信息、方法元数据、字段信息、运行时常量池等。对象引用如果是方法里的局部变量，通常在栈帧里；对象本体在堆上。

### 3. Java 对象从 new 到可用经历哪些步骤？

**你的回答**

> 检查、加载，初始化，分配内存，引用。

**点评**

**评分：5.5/10**

方向有一点，但顺序和关键步骤要补齐。对象创建建议按“类加载检查 -> 分配内存 -> 零值初始化 -> 设置对象头 -> 执行构造方法”来讲。

**标准回答**

1. **类加载检查**：检查对象所属类是否已经被加载、解析、初始化。
2. **分配内存**：在堆上为对象分配内存，可能使用指针碰撞或空闲列表；并发分配时可能用 CAS 或 TLAB。
3. **初始化零值**：把字段设置成默认零值，例如 `int = 0`，引用类型为 `null`。
4. **设置对象头**：设置 Mark Word、类型指针，数组对象还会设置数组长度。
5. **执行构造方法**：执行 `<init>`，完成业务意义上的初始化。

对象创建完成后，栈上的局部变量引用会指向堆中的对象实例。

**面试表达**

> Java 对象创建时，JVM 会先做类加载检查，确认类已经加载、解析和初始化。然后在堆上为对象分配内存，并把字段初始化为默认零值。接着设置对象头，包括 Mark Word、类型指针等信息。最后执行构造方法 `<init>`，完成对象的业务初始化。对象创建完成后，局部变量中的引用会指向堆中的对象实例。

### 4. 对象头里有什么？和 synchronized 锁升级有什么关系？

**你的回答**

> Markword 类型指针，如果是偏向锁会记录这个线程。

**点评**

**评分：8/10**

答到了核心。需要补上数组长度，以及 Mark Word 在不同锁状态下存储的内容。

**标准回答**

对象头通常包括：

1. **Mark Word**：存储 hashCode、GC 年龄、锁标志位、偏向线程 ID、锁记录指针或 Monitor 指针等。
2. **Klass Pointer 类型指针**：指向类元信息，用于确定对象属于哪个类。
3. **数组长度**：只有数组对象才有。

| 锁状态 | Mark Word 里可能存什么 |
| --- | --- |
| 无锁 | hashCode、GC 年龄、锁标志位 |
| 偏向锁 | 偏向线程 ID、epoch、GC 年龄、锁标志位 |
| 轻量级锁 | 指向线程栈中 Lock Record 的指针 |
| 重量级锁 | 指向 Monitor 对象的指针 |

**面试表达**

> Java 对象头主要包括 Mark Word 和类型指针，如果是数组对象还会有数组长度。类型指针用于定位对象属于哪个类；Mark Word 用来存储对象运行时数据，比如 hashCode、GC 年龄、锁标志位等。`synchronized` 的锁状态也体现在 Mark Word 里：偏向锁时记录偏向线程 ID，轻量级锁时记录指向线程栈中 Lock Record 的指针，重量级锁时记录指向 Monitor 的指针。所以锁升级本质上会伴随 Mark Word 内容变化。

### 5. 类加载生命周期有哪些阶段？

**你的回答**

> 这题不会。

**评分：0/10（未作答）**

**标准回答**

类加载重点记前五个阶段：

```text
加载 -> 验证 -> 准备 -> 解析 -> 初始化
```

| 阶段  | 作用                      | 关键点                             |
| --- | ----------------------- | ------------------------------- |
| 加载  | 把 `.class` 字节码加载到 JVM   | 生成类元信息和 `Class<?>` 对象           |
| 验证  | 校验字节码是否合法、安全            | 防止恶意字节码破坏 JVM                   |
| 准备  | 给 `static` 类变量分配内存并赋默认值 | `static int count = 10` 此时是 `0` |
| 解析  | 把符号引用转换成直接引用            | 从“名字”变成可定位的内存引用                 |
| 初始化 | 执行 `<clinit>()`         | 执行 `static` 变量显式赋值和静态代码块        |

注意：

- 普通 `static` 变量在准备阶段是默认值，初始化阶段才赋代码里的值。
- `static final` 编译期常量可能在准备阶段就被赋值。

**面试表达**

> 类加载主要包括加载、验证、准备、解析、初始化五个阶段。加载阶段是把 class 字节码读入 JVM，生成类元信息和 Class 对象；验证阶段校验字节码是否合法安全；准备阶段给 static 类变量分配内存并赋默认值；解析阶段把常量池中的符号引用转换成直接引用；初始化阶段执行 `<clinit>()` 方法，也就是执行 static 变量显式赋值和 static 静态代码块。需要注意，普通 static 变量在准备阶段是默认值，真正的代码赋值发生在初始化阶段。

### 6. 什么是双亲委派机制？

**你的回答**

> 双亲委派机制就是把类的加载交给最顶级父类去加载，避免加载错误。

**点评**

**评分：5/10**

方向对，但表述不准确。不是“交给最顶级父类去加载”，而是当前类加载器先委托父加载器，父加载器加载不了时，当前加载器才自己加载。

**标准回答**

类加载器层级通常可以理解为：

```text
Bootstrap ClassLoader
    ↓
Extension / Platform ClassLoader
    ↓
Application ClassLoader
    ↓
自定义 ClassLoader
```

加载流程：

1. 当前类加载器收到加载请求。
2. 先委托父加载器。
3. 父加载器继续向上委托。
4. 如果父加载器能加载，就直接返回。
5. 如果父加载器都加载不了，才由当前加载器自己加载。

设计目的：

- 保证 JDK 核心类安全，例如自定义 `java.lang.String` 不会替换 JDK 自带类。
- 避免同一个类被重复加载。
- 维护类加载层级和隔离边界。

打破双亲委派的典型场景：

- Tomcat：每个 Web 应用使用独立类加载器，实现应用隔离。
- SPI：例如 JDBC 驱动，通过线程上下文类加载器加载厂商实现。
- OSGi、插件化、热部署：需要模块隔离和动态加载。
- 自定义 ClassLoader：从网络、数据库、加密文件加载 class。

**面试表达**

> 双亲委派是类加载器的一种委托机制。一个类加载器收到类加载请求后，先委托给父加载器，父加载器继续向上委托，直到 Bootstrap ClassLoader。如果父加载器能加载就直接返回，只有父加载器都加载不了，当前类加载器才自己加载。这样可以保证 JDK 核心类不会被应用自己定义的类篡改，也能避免同一个类被重复加载。典型打破双亲委派的场景有 Tomcat 的 WebApp 类加载器、JDBC SPI 的线程上下文类加载器，以及插件化、热部署等需要隔离加载的场景。

### 7. JVM 如何判断对象是否可以被回收？什么是 GC Roots？

**你的回答**

> GC roots 不可达的对象就可以被回收。

**点评**

**评分：6/10**

方向正确，但要说出“可达性分析”和 GC Roots 的具体例子。

**标准回答**

主流 JVM 使用可达性分析判断对象是否存活：

```text
从 GC Roots 出发，沿着引用链搜索。
能被 GC Roots 直接或间接访问到的对象是存活对象；
到 GC Roots 没有任何引用链的对象，是不可达对象，具备被回收条件。
```

常见 GC Roots：

- 虚拟机栈中局部变量表引用的对象。
- 方法区 / 元空间中类静态变量引用的对象。
- 方法区中常量引用的对象。
- 本地方法栈中 JNI 引用的对象。
- 正在运行的线程对象。
- 被 `synchronized` 锁持有的对象。
- 类加载器引用的对象。

**面试表达**

> JVM 主要通过可达性分析判断对象是否存活。它会从一组 GC Roots 对象出发，沿着引用链向下搜索，能被 GC Roots 直接或间接访问到的对象认为是存活对象；如果一个对象到 GC Roots 没有任何引用链，就认为是不可达，具备被回收的条件。常见 GC Roots 包括栈帧中的局部变量引用、类静态变量、常量、JNI 引用、正在运行的线程、被 synchronized 持有的对象以及类加载器等。

### 8. 强引用、软引用、弱引用、虚引用有什么区别？

**你的回答**

> 强引用不会被垃圾回收，软引用如果内存不足会被垃圾回收，弱引用只要触发垃圾回收就会回收，虚引用直接回收。

**点评**

**评分：8/10**

前三个基本正确。虚引用不是“直接回收”，而是几乎不影响对象生命周期，主要用于对象回收通知，通常配合 `ReferenceQueue`。

**标准回答**

| 引用类型 | 回收时机 | 典型场景 |
| --- | --- | --- |
| 强引用 | 只要引用链可达，就不会被回收 | 普通对象引用 |
| 软引用 | 内存不足时才可能被回收 | 内存敏感缓存 |
| 弱引用 | 只要发生 GC 且没有强引用，就会被回收 | `WeakHashMap`、`ThreadLocal` key |
| 虚引用 | 不决定对象存活，只用于回收通知 | 堆外内存释放、资源清理跟踪 |

ThreadLocal 的 key 是弱引用，但 value 是强引用。线程池场景如果不 `remove()`，可能出现：

```text
Thread -> ThreadLocalMap -> Entry -> value
```

导致 value 泄漏。

**面试表达**

> 强引用是最普通的引用，只要从 GC Roots 可达就不会回收；软引用在内存不足时会被回收，可以用于内存敏感缓存；弱引用只要发生 GC 且没有强引用关联就会被回收，典型如 WeakHashMap、ThreadLocal 的 key；虚引用不影响对象生命周期，主要配合 ReferenceQueue 感知对象回收，用于堆外内存或资源清理。需要注意 ThreadLocal 的 key 是弱引用，但 value 是强引用，线程池场景用完必须 remove。

### 9. Minor GC、Major GC、Full GC 有什么区别？频繁 Full GC 怎么排查？

**你的回答**

> Minor GC 回收新生代，Major 回收老年代，Full GC 都会回收，会触发 stop the world。

**点评**

**评分：7/10**

主干正确，但要补两个细节：Minor GC 也会 STW，只是通常更短；Major GC 在不同资料里有时和 Full GC 混用，面试时要主动说明。

**标准回答**

| 类型 | 回收范围 | 特点 |
| --- | --- | --- |
| Minor GC / Young GC | 新生代 | 触发频繁，速度较快，也会 STW |
| Major GC / Old GC | 老年代 | 老年代空间不足或晋升失败时触发，成本较高 |
| Full GC | 整个堆，通常还包括方法区 / 元空间相关区域 | 停顿时间长，对线上影响大 |

频繁 Full GC 排查步骤：

1. **看 GC 日志**：确认触发原因、停顿时间、Full GC 前后老年代是否下降。
2. **看内存趋势**：用 `jstat -gcutil <pid> 1000` 看老年代、元空间、YGC/FGC 趋势。
3. **导出堆快照**：用 `jmap -dump:format=b,file=heap.hprof <pid>`，再用 MAT 分析。
4. **结合业务场景**：检查大对象、缓存无限增长、ThreadLocal 未清理、队列积压、元空间类加载过多、显式 `System.gc()` 等。

**判断重点**

- Full GC 后老年代明显下降：更像瞬时大对象、内存配置偏小或对象晋升过快。
- Full GC 后老年代不下降：重点怀疑内存泄漏或对象长期被引用。

**面试表达**

> Minor GC 主要回收新生代，也会 STW，只是通常停顿较短。Major GC 通常指老年代回收，但不同资料里有时会和 Full GC 混用。Full GC 一般会回收整个堆以及方法区/元空间相关区域，停顿时间更长，对线上影响更明显。频繁 Full GC 我会先看 GC 日志确认触发原因和回收效果，再用 jstat 看老年代、元空间和 YGC/FGC 趋势。如果 Full GC 后老年代不下降，重点怀疑内存泄漏，导出 heap dump 用 MAT 分析大对象、静态集合、缓存、ThreadLocal、队列积压和类加载器泄漏；如果 Full GC 后能下降，则更可能是内存配置、瞬时大对象或对象晋升过快。

### 10. 线上 CPU 100% 或接口突然超时，怎么判断是不是 JVM 问题？

**你的回答**

> 查看 dump 文件。

**点评**

**评分：4/10**

Dump 文件更适合 OOM 或内存泄漏，不是 CPU 100% 的第一步。CPU 高优先看高 CPU 线程栈；接口超时要分层看 JVM、线程池、连接池、DB、Redis、RPC 和网关。

**标准排查链路**

1. **找 Java 进程**

```bash
top
ps -ef | grep java
```

2. **找高 CPU 线程**

```bash
top -Hp <pid>
```

3. **线程 ID 转 16 进制**

```bash
printf "%x\n" <tid>
```

4. **用 jstack 定位线程栈**

```bash
jstack <pid> > thread.dump
grep -n "<hex_tid>" thread.dump
```

判断：

- 线程在跑业务代码：可能是死循环、复杂计算、热点逻辑。
- 线程在 GC：结合 `jstat` 和 GC 日志判断是否频繁 GC。
- 大量线程 `BLOCKED`：可能是锁竞争或死锁。
- 大量线程 `WAITING/TIMED_WAITING`：可能是线程池、连接池、RPC、Redis、DB 等下游等待。

5. **看 GC 是否异常**

```bash
jstat -gcutil <pid> 1000
```

关注：

- `YGC/YGCT`：Young GC 次数和耗时。
- `FGC/FGCT`：Full GC 次数和耗时。
- `O`：老年代占用。
- `M`：元空间占用。

6. **必要时导出 heap dump**

```bash
jmap -dump:format=b,file=heap.hprof <pid>
```

用于分析 OOM、内存泄漏、大对象、缓存膨胀、ThreadLocal、队列积压等。

**面试表达**

> CPU 100% 我不会一上来就看 heap dump，而是先用 `top` 找到 Java 进程，再用 `top -Hp pid` 找到占用 CPU 高的线程，把线程 ID 转成 16 进制后，用 `jstack` 在线程栈里定位它正在执行什么。如果是业务代码，重点看死循环和复杂计算；如果是 GC 线程，再结合 `jstat -gcutil` 和 GC 日志看是否频繁 GC；如果大量线程 BLOCKED 或 WAITING，要看锁竞争、线程池、连接池和下游依赖。接口超时也是先分层看，结合监控、Trace、jstack、GC 指标和慢 SQL 判断是不是 JVM 问题。heap dump 更适合 OOM 或内存泄漏分析，不是 CPU 高的第一步。

### 本轮薄弱点与复习建议

**薄弱点**

- 类加载阶段需要重点背：加载、验证、准备、解析、初始化。
- 双亲委派要说清楚“先委托父加载器，父加载器加载不了才自己加载”。

### 2. ArrayList 是怎么扩容的？为什么建议能预估容量时指定初始容量？

**你的回答**

> 当数组添加新元素的时候，如果已经达到了数组的最大长度，则进行扩容，生成一个长度为数组长度+1/2长度的数组，把数据复制过去。因为如果不指定初始容量，容易触发扩容，一旦扩容便会涉及到数组的拷贝。

**点评**

**评分：9/10**

核心正确：容量不足时扩容为 1.5 倍，并复制旧数组。更严谨的说法是“新增后所需容量超过当前数组长度时触发扩容”，不是“最大长度”。

**标准回答**

`ArrayList` 底层是数组。JDK 8 以后无参构造创建的是空数组，第一次 `add` 时才初始化为默认容量 10。后续每次添加元素前都会检查容量，如果新增后所需容量超过当前数组长度，就会扩容。

扩容公式：

```java
int newCapacity = oldCapacity + (oldCapacity >> 1);
```

也就是旧容量的 1.5 倍。扩容后通过 `Arrays.copyOf` 创建新数组并复制旧元素，底层会用到 `System.arraycopy()`。

**面试表达**

> `ArrayList` 底层是数组。JDK 8 以后无参构造是懒加载，第一次 `add` 才初始化为默认容量 10。后续添加元素前会检查容量，如果新增后所需容量超过当前数组长度，就按 1.5 倍扩容，并通过 `Arrays.copyOf` 复制旧数组。扩容会申请新数组并拷贝已有元素，所以如果能预估数量，建议构造时指定初始容量，减少扩容和数组拷贝成本。

### 3. HashMap.put 的核心流程是什么？发生 hash 冲突时怎么处理？

**你的回答**

> 根据 hashcode 高低位异或去寻找桶的位置，如果该位置有值了，则生成 node 链，如果当前链路长度到达 8 且数组长度大于 64，则转成红黑树。

**点评**

**评分：7.5/10**

答到了扰动函数、定位桶、冲突链表和树化条件。但 `put` 流程还要补全：桶头匹配覆盖、红黑树插入、链表尾插、`size` 增加和扩容判断。

**标准回答**

```text
1. 计算 key 的扰动 hash。
2. 如果 table 未初始化，先 resize 初始化数组。
3. 通过 (n - 1) & hash 定位桶下标。
4. 如果桶为空，直接新建 Node 放入。
5. 如果桶不为空：
   - 先判断桶头节点是否是同一个 key，是则覆盖 value；
   - 如果桶头是红黑树节点，走红黑树插入；
   - 否则遍历链表，找到相同 key 就覆盖，找不到就尾插新节点；
   - 链表长度达到阈值后尝试树化。
6. 插入新节点后 size++。
7. 如果 size > threshold，触发 resize 扩容。
```

树化条件：

```text
链表长度 >= 8 且 table 容量 >= 64
```

如果链表长度到 8 但数组容量小于 64，会优先扩容。

**面试表达**

> `HashMap.put` 会先计算 key 的扰动 hash，通过 `(n - 1) & hash` 定位数组下标。如果 table 还没初始化，会先初始化数组。如果桶为空，直接放入新 Node；如果桶不为空，先比较桶头节点的 hash 和 key，如果 key 相同就覆盖 value。如果桶是红黑树，就走红黑树插入；否则遍历链表，找到相同 key 就覆盖，找不到就尾插新节点。链表长度达到 8 且数组容量至少 64 时，会转成红黑树，否则优先扩容。插入新节点后 size 增加，如果超过阈值 `capacity * loadFactor`，触发扩容。

### 4. 为什么重写 equals 时必须重写 hashCode？

**你的回答**

> HashMap 对比值的时候，会先判断 hashcode，如果不重写 hashcode，那么一开始就可能会不相等，两个相等的值就会存在不同的桶中。

**点评**

**评分：9/10**

回答很好，核心原因准确。需要补一句契约：`equals` 为 true 的两个对象，`hashCode` 必须相同。

**标准回答**

`HashMap`、`HashSet` 等哈希集合判断元素是否相同一般分两步：

1. 先根据 `hashCode()` 定位桶。
2. 再在桶内用 `==` 或 `equals()` 判断是否是同一个 key。

如果两个对象 `equals()` 返回 true，但 `hashCode()` 不同，它们可能落到不同桶里，后续根本不会走到 `equals()` 比较，导致重复插入或查询不到。

**面试表达**

> `equals` 表示业务语义上是否相等，`hashCode` 用于哈希集合定位桶。`HashMap`、`HashSet` 会先根据 `hashCode` 找桶，再用 `equals` 判断桶内 key 是否相同。如果只重写 `equals` 不重写 `hashCode`，业务相等的对象可能落到不同桶，导致重复插入或查询不到。所以重写 `equals` 时必须同时重写 `hashCode`，保证 `equals` 为 true 的对象 `hashCode` 一定相同。

### 5. 为什么 HashMap 数组长度必须是 2 的幂？扩容时为什么不用重新计算 hash？

**你的回答**

> 方便扩容后，不需要重新计算桶的位置。

**点评**

**评分：5.5/10**

答到扩容高效这一点，但还要补两个关键原因：位运算寻址和分布更均匀。

**标准回答**

HashMap 数组长度是 2 的幂，主要有三个原因：

1. **位运算替代取模**：当 `n` 是 2 的幂时，`hash % n` 等价于 `(n - 1) & hash`，位运算效率更高。
2. **哈希分布更均匀**：`n - 1` 的低位全是 1，可以更充分利用 hash 低位；如果长度不是 2 的幂，某些桶位很难命中，冲突会变多。
3. **扩容迁移更简单**：每次扩容容量翻倍，新下标只比旧下标多看一位 hash。

扩容时节点只有两种情况：

```text
(hash & oldCap) == 0：留在原位置
(hash & oldCap) != 0：移动到 原位置 + oldCap
```

**面试表达**

> HashMap 数组长度是 2 的幂，主要有三个原因。第一，可以用 `(n - 1) & hash` 替代 `hash % n`，位运算效率更高。第二，长度为 2 的幂时，`n - 1` 的低位全是 1，能更充分利用 hash 的低位，让元素分布更均匀。第三，扩容时容量翻倍，新下标只比旧下标多看一位 hash，节点要么留在原位置，要么移动到 `原位置 + oldCap`，不需要重新计算 hash，迁移效率更高。

### 6. HashMap 链表什么时候树化？为什么不是链表长度达到 8 就一定树化？

**你的回答**

> 数组长度大于 64 并且链表达到 8，如果数组长度太小，容易产生冲突，导致链表太长。

**点评**

**评分：8.5/10**

主干正确。细节上应改为数组容量 **大于等于 64**，不是大于 64。

**标准回答**

树化条件：

```text
链表长度 >= 8，并且数组容量 >= 64
```

如果链表长度达到 8，但数组容量小于 64，HashMap 会优先扩容，而不是树化。原因是数组太小时链表变长可能只是容量不足导致的；扩容后元素重新分布，链表可能自然变短。红黑树维护成本比链表高，只有容量足够大但冲突仍然严重时，才适合树化。

补充阈值：

```text
TREEIFY_THRESHOLD = 8
MIN_TREEIFY_CAPACITY = 64
UNTREEIFY_THRESHOLD = 6
```

**面试表达**

> HashMap 链表树化需要两个条件：桶内链表长度达到 8，并且数组容量至少是 64。如果链表长度达到 8 但数组容量还小于 64，HashMap 会优先扩容，而不是直接树化。因为容量太小时冲突可能只是数组太小导致的，扩容后元素重新分布，链表可能变短。红黑树虽然能把极端查询复杂度从 O(n) 降到 O(log n)，但维护成本比链表高，所以只有在容量已经足够大但冲突仍然严重时，才适合树化。

### 7. JDK 1.8 的 ConcurrentHashMap 只是加了 synchronized 吗？

**你的回答**

> 在桶的位置加了 synchronized，只有在插入数组位置冲突时才会锁住。

**点评**

**评分：7/10**

答到了桶级锁和冲突时加锁，但要补全：JDK 1.8 的 `ConcurrentHashMap` 是 `CAS + synchronized + volatile + 协助扩容` 的组合。

**标准回答**

JDK 1.8 的 `ConcurrentHashMap` 不是简单在 `HashMap` 外面套一层 `synchronized`。它的并发控制机制包括：

1. `table` 延迟初始化，初始化时用 `sizeCtl + CAS` 控制只有一个线程初始化。
2. 空桶插入用 CAS，不加锁。
3. 桶不为空时，用 `synchronized(f)` 锁住当前桶头节点，锁粒度是桶级别。
4. 如果发现桶头是 `MOVED`，说明正在扩容，当前线程会协助迁移。
5. `size` 统计用 `baseCount + CounterCell`，降低并发竞争。
6. `table`、节点 `val`、`next` 等字段通过 `volatile` 保证可见性。

**面试表达**

> JDK 1.8 的 ConcurrentHashMap 不是在 HashMap 外面简单套 synchronized。它是 CAS 和 synchronized 结合：空桶插入直接 CAS，发生 hash 冲突时才 synchronized 锁住当前桶头节点，所以锁粒度是桶级别。扩容时多个线程还可以协助迁移，降低单线程扩容压力。再配合 volatile 保证 table、value、next 的可见性，以及 baseCount、CounterCell 做 size 统计，所以它是一个专门为并发设计的哈希容器。

### 8. 什么是 LRU 缓存？如何用 HashMap + 双向链表实现 get 和 put 都是 O(1)？

**你的回答**

> LRU 是最近最少使用缓存，因为最少使用的都会在链表的最后，所以用双向链表直接取 tail 的值就可以了。

**点评**

**评分：6.5/10**

答到了“最近最少使用”和“淘汰尾部”，但缺少 HashMap 定位，以及 `get`/`put` 后移动到头部的流程。

**标准回答**

LRU 是最近最少使用，缓存满时淘汰最久没有访问的数据。要做到 `get` 和 `put` 都是 O(1)，通常使用：

```text
HashMap：key -> Node，用于 O(1) 定位节点
双向链表：维护访问顺序，头部是最近使用，尾部是最久未使用
```

`get(key)` 流程：

1. 从 HashMap 中 O(1) 找到节点。
2. 不存在则返回空。
3. 存在则把节点移动到链表头。
4. 返回 value。

`put(key, value)` 流程：

1. key 已存在：更新 value，并移动到链表头。
2. key 不存在：新建节点，写入 HashMap，并插入链表头。
3. 容量超限：删除尾部最久未使用节点，并从 HashMap 删除对应 key。

**面试表达**

> LRU 是最近最少使用，缓存满时淘汰最久没有访问的数据。要做到 `get` 和 `put` 都是 O(1)，一般用 HashMap + 双向链表。HashMap 保存 key 到链表节点的映射，负责 O(1) 定位节点；双向链表维护访问顺序，头部表示最近使用，尾部表示最久未使用。每次 get 命中或 put 更新，都把节点移动到链表头；新增节点也插入链表头；容量超限时删除尾部最久未使用节点，并从 HashMap 中删除对应 key。因为查找、移动、删除都是 O(1)，所以整体平均 O(1)。

### 9. Java 泛型为什么说是伪泛型？什么是类型擦除？

**你的回答**

> 因为编译期间会做类型擦除，运行期间是同一个类型。

**点评**

**评分：8.5/10**

主干正确。可以补上“上限类型”和“读取时自动强转”。

**标准回答**

Java 泛型主要在编译期提供类型检查，运行期会发生类型擦除：

```java
List<String>
List<Integer>
```

编译后都会变成原始类型：

```java
List
```

类型参数擦除规则：

```text
T -> Object
T extends Number -> Number
```

读取泛型值时，编译器会自动插入强制类型转换：

```java
String s = list.get(0);
```

类似变成：

```java
String s = (String) list.get(0);
```

所以 `List<String>` 和 `List<Integer>` 运行期是同一个 `Class`。

**面试表达**

> Java 泛型是伪泛型，因为它主要在编译期提供类型检查，运行期会发生类型擦除。比如 `List<String>` 和 `List<Integer>` 编译后都会被擦除成原始类型 `List`，运行期拿到的是同一个 `Class` 对象。泛型类型参数如果没有上限会被擦除成 `Object`，如果有上限会擦除成上限类型。为了保证取值类型正确，编译器会在读取泛型数据的位置自动插入强制类型转换。这样设计主要是为了兼容 JDK 1.4 以前没有泛型的字节码。

### 10. JDK 动态代理和 CGLIB 动态代理有什么区别？

**你的回答**

> JDK 动态代理基于接口，CGLIB 代理是根据类生成对应的子类。

**点评**

**评分：8/10**

核心区别正确。需要补上调用机制、`final` 限制和 Spring AOP 默认选择逻辑。

**标准回答**

| 对比点 | JDK 动态代理 | CGLIB 动态代理 |
| --- | --- | --- |
| 代理基础 | 基于接口 | 基于继承 |
| 生成方式 | 运行期生成实现接口的代理类 | 运行期生成目标类的子类 |
| 调用方式 | 通过 `InvocationHandler.invoke()` 反射调用目标方法 | 通过 `MethodInterceptor.intercept()` 拦截方法 |
| 是否要求接口 | 要求目标类实现接口 | 不要求接口 |
| 限制 | 不能代理没有接口的类 | 不能代理 `final` 类和 `final` 方法 |
| Spring AOP 默认选择 | 有接口优先 JDK 动态代理 | 无接口时用 CGLIB |

为什么 JDK 动态代理必须有接口？

```java
class $Proxy0 extends Proxy implements UserService {
    // ...
}
```

JDK 生成的代理类本质上是实现目标接口的代理类，而不是继承目标类的子类。如果目标类没有接口，JDK 动态代理就不知道要生成一个实现什么接口的代理对象。

CGLIB 生成的是目标类子类：

```java
class UserService$$EnhancerByCGLIB extends UserService {
    // ...
}
```

所以 CGLIB 不要求接口，但不能代理 `final` 类和 `final` 方法。

**面试表达**

> JDK 动态代理是基于接口的，运行期生成一个实现目标接口的代理类，调用时通过 `InvocationHandler` 反射转发到目标对象，所以目标类必须实现接口。CGLIB 是基于继承的，运行期通过字节码技术生成目标类的子类，并重写父类方法进行增强，所以它不要求接口，但不能代理 final 类和 final 方法。Spring AOP 默认有接口时优先使用 JDK 动态代理，没有接口时使用 CGLIB。

### 1. synchronized 锁的是什么？实例方法、静态方法、代码块分别锁谁？

**你的回答**

> 实例方法锁对象、静态方法所得是 class 类，代码块锁的是代码块括号中的对象。

**点评**

**评分：9/10**

回答正确。实例方法建议明确说“锁当前实例对象 `this`”。另外要强调：`synchronized` 锁的是对象，不是代码，只有竞争同一个锁对象才会互斥。

**标准回答**

| 用法 | 锁对象 |
| --- | --- |
| `synchronized` 实例方法 | 当前实例对象 `this` |
| `static synchronized` 静态方法 | 当前类的 `Class` 对象，例如 `Demo.class` |
| `synchronized(obj)` 代码块 | 括号里的对象 `obj` |

同一个实例上的多个 `synchronized` 实例方法会互斥，因为它们都锁 `this`；普通实例方法和静态同步方法不一定互斥，因为一个锁 `this`，一个锁 `Class` 对象。

**面试表达**

> `synchronized` 锁的是对象，不是代码。实例同步方法锁当前实例 `this`；静态同步方法锁当前类的 `Class` 对象；同步代码块锁括号里指定的对象。多个线程只有竞争同一个锁对象时才会互斥，所以同一个对象上的多个 synchronized 实例方法会互斥，不同对象之间不会互斥。

### 2. synchronized 和 volatile 有什么区别？volatile 能不能保证 count++ 的线程安全？

**你的回答**

> volatile 只能保证可见性和有序性，不能保证原子性。

**点评**

**评分：8/10**

关键点正确。需要补上和 `synchronized` 的对比，以及 `count++` 为什么不是原子操作。

**标准回答**

| 对比  | `synchronized`    | `volatile`         |
| --- | ----------------- | ------------------ |
| 可见性 | 能保证               | 能保证                |
| 有序性 | 能保证               | 能保证，通过内存屏障禁止特定指令重排 |
| 原子性 | 能保证临界区互斥执行        | 只能保证单次读/写，不能保证复合操作 |
| 阻塞  | 会阻塞               | 不阻塞                |
| 场景  | 保护复合逻辑、临界区、共享资源修改 | 状态标记、开关、配置刷新、单写多读  |

`count++` 包含三步：

```text
1. 读取 count
2. count + 1
3. 写回 count
```

多个线程可能同时读到旧值再写回，导致丢失更新，所以 `volatile` 不能保证 `count++` 线程安全。计数场景应使用 `AtomicInteger`、`LongAdder` 或加锁。

**面试表达**

> `synchronized` 可以保证互斥、可见性和有序性，它通过同一把锁让临界区同一时刻只有一个线程执行。`volatile` 只能保证变量修改对其他线程可见，并通过内存屏障禁止特定指令重排，但它不保证复合操作的原子性。`count++` 包含读、加一、写回三个步骤，多个线程并发执行仍然会丢失更新，所以 volatile 不能保证 `count++` 线程安全。

### 3. synchronized 和 ReentrantLock 怎么选？ReentrantLock 多了哪些能力？

**你的回答**

> 如果需要自己解锁，需要中断，需要公平锁，可以使用 ReentrantLock。

**点评**

**评分：8/10**

方向正确。可以再补 `tryLock(timeout)`、多个 `Condition` 队列，以及普通场景优先 `synchronized`。

**标准回答**

| 对比点 | `synchronized` | `ReentrantLock` |
| --- | --- | --- |
| 实现层面 | JVM 内置锁 | JUC API，基于 AQS |
| 释放锁 | 自动释放，异常也会释放 | 手动 `unlock()`，通常放 `finally` |
| 可中断 | 不支持等待锁时中断 | 支持 `lockInterruptibly()` |
| 超时获取锁 | 不支持 | 支持 `tryLock(timeout)` |
| 公平锁 | 不支持配置 | 构造方法可指定公平/非公平 |
| 条件队列 | 一个对象等待队列，配合 `wait/notify` | 支持多个 `Condition` 队列 |

标准写法：

```java
lock.lock();
try {
    // 临界区
} finally {
    lock.unlock();
}
```

**面试表达**

> 普通互斥同步我优先用 `synchronized`，因为语义简单，JVM 自动释放锁，异常场景也不容易忘记解锁。`ReentrantLock` 更适合需要精细控制的场景，比如等待锁时可中断、超时获取锁、公平锁、多个 Condition 条件队列等。使用 `ReentrantLock` 时必须在 `finally` 中手动释放锁，否则容易造成死锁。

### 4. yield、sleep、join、wait 有什么区别？哪些会释放锁？

**你的回答**

> 只有 wait 才会释放锁，yield 是让出 CPU、sleep 是休眠对应时间，join 是阻塞当前主线程，等 join 的线程执行完再继续执行。

**点评**

**评分：8.5/10**

关键点基本正确。`join` 不是只阻塞主线程，而是阻塞调用 `join()` 的当前线程。

**标准回答**

| 方法 | 作用 | 线程状态 | 是否释放锁 | 是否必须在 synchronized 中 |
| --- | --- | --- | --- | --- |
| `yield()` | 提示调度器让出 CPU | `RUNNABLE` | 不释放 | 否 |
| `sleep(ms)` | 休眠指定时间 | `TIMED_WAITING` | 不释放 | 否 |
| `join()` | 当前线程等待目标线程结束 | `WAITING/TIMED_WAITING` | 不释放当前持有的锁 | 否 |
| `wait()` | 当前线程进入对象等待队列 | `WAITING/TIMED_WAITING` | 释放当前对象锁 | 是 |

`wait/notify` 必须在 `synchronized` 中调用，因为它们操作的是对象 Monitor 的等待队列。`notify()` 只是唤醒等待线程，不会立刻释放锁，被唤醒线程还要重新竞争锁。

**面试表达**

> `yield` 是提示调度器让出 CPU，但不保证一定让出，线程仍是 RUNNABLE；`sleep` 是让当前线程睡眠指定时间，进入 TIMED_WAITING；`join` 是当前线程等待目标线程执行完成；`wait` 是当前线程进入对象等待队列。`yield`、`sleep`、`join` 都不会主动释放当前线程已经持有的 synchronized 对象锁；只有 `wait` 会释放当前对象锁，并且 `wait/notify` 必须在 synchronized 中调用。

### 5. CAS 是什么？ABA 问题是什么？怎么解决？

**你的回答**

> CAS 是比较并交换，ABA 就是在 CAS 执行之前改变过其中的值，又改回来了，可以用 LongAdder。

**点评**

**评分：6.5/10**

CAS 和 ABA 的定义方向正确，但解决方案需要修正：`LongAdder` 不是解决 ABA 的，它是降低高并发计数场景下 CAS 热点竞争的。ABA 应使用版本号或标记位。

**标准回答**

CAS 是 Compare And Swap，比较并交换。它有三个操作数：

```text
V：内存当前值
A：期望值
B：要更新的新值
```

只有当 `V == A` 时，才把 V 更新成 B，否则更新失败。

ABA 问题：

```text
线程 1 读取 A，准备 CAS(A -> C)
线程 2 把 A 改成 B
线程 2 又把 B 改回 A
线程 1 执行 CAS，发现还是 A，于是更新成功
```

线程 1 以为数据没变过，但中间其实发生过修改。普通计数场景影响可能不大，但在无锁栈、无锁队列等依赖节点引用关系的数据结构中，可能导致节点被错误复用或链表结构异常。

解决方案：

```text
A(version=1) -> B(version=2) -> A(version=3)
```

使用版本号或标记位，例如：

```java
AtomicStampedReference
AtomicMarkableReference
```

**面试表达**

> CAS 是比较并交换，它会比较内存当前值是否等于期望值，如果相等就更新为新值，否则更新失败。它是乐观锁的一种实现，底层依赖 CPU 原子指令。ABA 问题是指线程读到 A 后，其他线程把 A 改成 B 又改回 A，当前线程再 CAS 时发现值还是 A，于是误以为没有变化。解决 ABA 通常要引入版本号或标记位，比如使用 `AtomicStampedReference` 或 `AtomicMarkableReference`。`LongAdder` 不是解决 ABA 的，它主要用于降低高并发计数场景下的 CAS 热点竞争。

### 6. JMM 是什么？可见性、原子性、有序性分别怎么理解？

**你的回答**

> Java 内存模型。可见性就是主内存的值改动到，其他子内存都必须能感知到，原子性就是所有的操作要么成功要么失败，有序性就是按照对应的代码顺序执行。

**点评**

**评分：7/10**

方向正确，但要把“子内存”改成“工作内存”。原子性不要套数据库事务口径，有序性也不是简单按代码顺序执行，而是约束指令重排。

**标准回答**

JMM 是 Java Memory Model，Java 内存模型。它不是 JVM 运行时内存区域，而是一套多线程读写共享变量的规范。

JMM 抽象出：

```text
主内存：共享变量所在位置
工作内存：每个线程自己的变量副本
```

三大特性：

| 特性 | 含义 | 常见实现 |
| --- | --- | --- |
| 可见性 | 一个线程修改共享变量后，其他线程能及时看到 | `volatile`、`synchronized`、Lock |
| 原子性 | 一个操作或一组操作不可被线程调度打断 | `synchronized`、Lock、CAS、Atomic 类 |
| 有序性 | 约束编译器和 CPU 指令重排，保证多线程结果符合预期 | `volatile`、锁、happens-before、内存屏障 |

`count++` 不是原子操作；双重检查锁中的 `volatile` 可以防止对象初始化过程中的指令重排。

**面试表达**

> JMM 是 Java 内存模型，它不是 JVM 运行时内存区域，而是一套多线程读写共享变量的规范。JMM 抽象出主内存和线程工作内存，定义了可见性、原子性和有序性。可见性是一个线程修改共享变量后，其他线程能及时看到；原子性是一个操作或一组操作不可被中断；有序性是指在多线程环境下要约束编译器和 CPU 指令重排，保证程序结果符合预期。Java 中可以通过 `volatile`、`synchronized`、Lock、CAS、happens-before 规则等来保证这些语义。

### 7. 线程池的 7 个核心参数是什么？任务提交后线程池的执行流程是什么？

**你的回答**

> 核心线程数、最大线程数、超时时间和单位、线程工厂、队列和拒绝策略。流程为有线程需要时先创建核心线程，如果到达了核心线程数，则放到队列，如果队列也满了，则继续新增线程数直到到达最大线程数时，走拒绝策略。

**点评**

**评分：8.5/10**

参数和流程基本正确。建议明确 `keepAliveTime` 是非核心线程空闲存活时间，并补充“非核心线程优先执行新提交任务”和“核心线程默认不回收”。

**标准回答**

```java
new ThreadPoolExecutor(
    int corePoolSize,                    // 核心线程数
    int maximumPoolSize,                 // 最大线程数
    long keepAliveTime,                  // 非核心线程空闲存活时间
    TimeUnit unit,                       // 时间单位
    BlockingQueue<Runnable> workQueue,   // 阻塞队列
    ThreadFactory threadFactory,         // 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)
```

任务执行流程：

```text
1. 当前线程数 < corePoolSize
   -> 创建核心线程执行任务

2. 当前线程数 >= corePoolSize，且队列未满
   -> 放入 workQueue 等待

3. 队列已满，且当前线程数 < maximumPoolSize
   -> 创建非核心线程执行当前新任务

4. 队列已满，且当前线程数 >= maximumPoolSize
   -> 执行拒绝策略
```

非核心线程空闲超过 `keepAliveTime` 会被回收，核心线程默认不会回收，除非开启 `allowCoreThreadTimeOut(true)`。

**面试表达**

> 线程池有 7 个核心参数：核心线程数、最大线程数、空闲线程存活时间、时间单位、阻塞队列、线程工厂和拒绝策略。任务提交后，如果当前线程数小于核心线程数，就创建核心线程执行；如果核心线程已满，就尝试放入阻塞队列；如果队列也满了，并且线程数还没到最大线程数，就创建非核心线程执行当前新任务；如果线程数已经达到最大线程数，就触发拒绝策略。非核心线程空闲超过 keepAliveTime 会被回收，核心线程默认不会回收。

### 8. 线程池拒绝策略怎么选？生产上有没有更好的自定义实现？

**你的回答**

> 拒绝策略建议线上自定义实现，如果是核心流程，可以保存到本地，重新通过 MQ 执行，如果是非核心流程，可以打印日志，整个过程做成可监控性。

**点评**

**评分：9/10**

回答很好，是生产口径。可以补充 JDK 四种拒绝策略和 `CallerRunsPolicy` 的反压价值及风险。

**标准回答**

| 策略 | 行为 | 适用性 |
| --- | --- | --- |
| `AbortPolicy` | 直接抛 `RejectedExecutionException` | 默认策略，适合快速暴露问题 |
| `CallerRunsPolicy` | 提交任务的线程自己执行 | 可以形成反压，避免任务直接丢失 |
| `DiscardPolicy` | 静默丢弃 | 风险大，只适合完全可丢任务 |
| `DiscardOldestPolicy` | 丢弃队列最老任务，再提交新任务 | 风险大，可能破坏顺序和业务完整性 |

生产建议：

- 核心任务不能直接丢：记录日志、上报监控、触发告警，必要时落库、本地消息表或投递 MQ，由补偿任务处理。
- 非核心任务可以降级或丢弃，但拒绝次数必须可观测。
- 入口请求线程慎用 `CallerRunsPolicy`，否则可能把 Tomcat 业务线程拖慢。

**面试表达**

> 线程池拒绝策略没有绝对最优，要看任务是否允许丢失。JDK 默认有 `AbortPolicy`、`CallerRunsPolicy`、`DiscardPolicy`、`DiscardOldestPolicy`。核心业务我不会用静默丢弃策略，通常会自定义拒绝策略：记录日志、上报监控、触发告警，必要时把任务落库、本地消息表或投递 MQ，由补偿任务后续处理。非核心任务可以降级或丢弃，但拒绝次数必须可观测。`CallerRunsPolicy` 可以形成反压，但如果调用线程是 Tomcat 请求线程，要谨慎使用，避免把入口线程拖慢。

### 9. ThreadLocal 为什么可能内存泄漏？为什么 key 是弱引用、value 是强引用？

**你的回答**

> 因为使用线程池后，有些使用 ThreadLocal 的线程可能没回收，所以 ThreadLocal 的 value 值一直有引用，不会被回收。key 是弱引用，如果没有其他强引用，threadLocal 就会被回收，而 value 不能这么做，不然有可能使用期间被回收了。

**点评**

**评分：9/10**

回答很好。补上引用链和 `finally remove()`，就是标准答案。

**标准回答**

每个线程都有自己的 `ThreadLocalMap`：

```text
Thread -> ThreadLocalMap -> Entry
```

其中：

```text
Entry.key   = WeakReference<ThreadLocal>
Entry.value = 业务对象强引用
```

如果业务代码中的 `ThreadLocal` 强引用断开，GC 后 key 会变成 `null`，但 value 仍然存在引用链：

```text
Thread -> ThreadLocalMap -> Entry -> value
```

只要线程不结束，value 就无法被 GC。普通线程结束后问题不大；线程池线程会长期复用，`Thread` 一直是 GC Root，所以 value 可能长期残留。

key 设计成弱引用，是为了避免 `ThreadLocal` 对象本身泄漏；value 必须是强引用，否则业务使用期间可能被 GC 回收。

正确写法：

```java
try {
    threadLocal.set(value);
    // 业务逻辑
} finally {
    threadLocal.remove();
}
```

**面试表达**

> ThreadLocal 可能泄漏主要发生在线程池场景。每个线程内部都有 ThreadLocalMap，Entry 的 key 是 ThreadLocal 的弱引用，value 是业务对象的强引用。如果 ThreadLocal 外部强引用断开，GC 后 key 会变成 null，但 value 仍然通过 `Thread -> ThreadLocalMap -> Entry -> value` 这条链路被线程引用。线程池线程长期存活，不会退出，所以 value 可能一直无法回收。key 设计成弱引用是为了避免 ThreadLocal 对象本身泄漏；value 必须是强引用，否则业务使用期间可能被 GC 回收。最佳实践是在 finally 中调用 `remove()`，主动清理 value。

### 10. AQS 的核心结构是什么？state、CLH 队列、哨兵节点分别有什么作用？

**你的回答**

> state + CLH 队列。

**点评**

**评分：4/10**

回答太短，只答出两个关键词。面试需要把 `state`、CLH 队列、Node 状态、哨兵节点、CAS 和 `LockSupport` 串起来。

**标准回答**

AQS 核心结构可以概括为：

```text
1. volatile int state
2. CLH 双向同步队列
3. CAS + LockSupport
```

`state` 是同步状态，不同同步器含义不同：

| 同步器 | state 含义 |
| --- | --- |
| ReentrantLock | 锁状态和重入次数，0 表示未加锁，大于 0 表示被占用 |
| Semaphore | 可用许可证数量 |
| CountDownLatch | 剩余计数 |

线程获取同步状态失败后，会被包装成 `Node` 加入 CLH 双向队列：

```text
head(dummy) -> node1 -> node2 -> tail
```

Node 中包含：

```text
thread：等待线程
prev / next：前驱和后继
waitStatus：等待状态
```

常见 `waitStatus`：

```text
SIGNAL = -1：后继节点需要被唤醒
CANCELLED = 1：当前节点取消排队
CONDITION = -2：在 Condition 队列中
PROPAGATE = -3：共享模式传播
```

哨兵节点 `head` 不代表真实等待线程，主要作用是稳定队列头部、简化入队出队和唤醒逻辑。真正等待的线程从 `head.next` 开始；当前节点前驱是 `head` 时，说明自己有资格尝试获取锁；释放锁时从 `head` 开始唤醒后继节点。

AQS 用 CAS 修改 `state`、`head`、`tail`；线程抢锁失败后用 `LockSupport.park()` 挂起，前驱释放锁后用 `LockSupport.unpark()` 唤醒。

**面试表达**

> AQS 是 JUC 同步器的基础，核心结构可以概括为 `state + CLH 同步队列 + CAS/LockSupport`。`state` 是同步状态，不同组件含义不同，比如 ReentrantLock 中表示锁状态和重入次数，Semaphore 中表示许可证数量，CountDownLatch 中表示剩余计数。线程获取同步状态失败后，会被包装成 Node 加入 CLH 双向队列，通过 `park` 挂起等待；释放锁时再从 head 开始唤醒后继节点。队列里的 head 是哨兵节点，不代表真实线程，主要用于稳定队列边界，真正等待的线程从 `head.next` 开始。AQS 通过 CAS 修改 state、head、tail，保证并发入队和状态修改的线程安全。

---

### 本轮薄弱点与复习建议

**表现较好的点**

- `synchronized` 锁对象、`volatile`、`ReentrantLock`、线程协作方法回答较稳。
- 线程池参数、执行流程、拒绝策略有生产意识。
- `ThreadLocal` 泄漏原因和弱引用设计理解比较清楚。

**需要补强的点**

- CAS 的 ABA 解决方案要记成版本号、`AtomicStampedReference`、`AtomicMarkableReference`，不要说 `LongAdder`。
- JMM 的原子性和有序性要按并发语义讲，不要套数据库事务口径。
- AQS 不能只答 `state + CLH`，要补 `Node`、`waitStatus`、哨兵节点、CAS、`LockSupport.park/unpark`。

**下一轮建议练习**

1. 只练 AQS：`state`、CLH 队列、加锁入队、解锁唤醒、丢失唤醒。
2. 只练 CAS：ABA、版本号、LongAdder 和 AtomicLong 区别。
3. 只练 JMM：可见性、原子性、有序性、happens-before、volatile 内存屏障。

### 2. RocketMQ 的整体架构是什么？NameServer、Broker、Producer、Consumer 分别负责什么？

**你的回答**

> 整体架构是 NameServer、Broker、Producer、Consumer，NameServer 作为注册中心、Broker 保存消息、Producer 发送消息、Consumer 消费消息。

**点评**

**评分：7/10**

四个核心角色答对了，但还要补路由关系、Topic、MessageQueue、ConsumerGroup，避免只停留在名词解释。

**标准回答**

RocketMQ 主要由 `NameServer`、`Broker`、`Producer`、`Consumer` 组成。

- `NameServer`：轻量级路由中心，Broker 向它注册 Topic 和队列信息，Producer/Consumer 从它获取路由。
- `Broker`：真正保存和投递消息，负责消息存储、消费进度、重试、死信等能力。
- `Producer`：根据 Topic 路由把消息发送到 Broker。
- `Consumer`：从 Broker 拉取消息并执行业务消费。

核心模型：

- `Topic`：业务主题。
- `MessageQueue`：消息分区、顺序性和消费并发的基本单位。
- `ConsumerGroup`：消费者组，同组共同消费一份消息，不同组可以各自消费一份消息。

**面试表达**

> RocketMQ 的架构可以按四个角色讲：NameServer 是轻量路由中心，Broker 会向 NameServer 注册 Topic 和队列信息；Producer 发消息前先从 NameServer 获取路由，再把消息发送到对应 Broker；Broker 负责消息存储、投递、消费进度、重试和死信；Consumer 从 Broker 拉取消息并执行业务消费。需要注意，NameServer 只提供路由，不保存消息，真正存储消息的是 Broker。

### 3. RocketMQ 如何保证消息不丢？你会从哪几段链路来讲？

**你的回答**

> 生产者要等消息发送完再返回数据，结束事务。broker 要保存数据避免丢失，consumer 要保证 ack。

**点评**

**评分：7/10**

能按生产者、Broker、消费者三段链路讲，这是正确方向。但“结束事务”表述不够准确，普通消息发送成功不等于本地事务和消息发送已经一致。

**标准回答**

消息可靠性可以按三段链路讲：

1. 生产者端：核心消息使用同步发送，发送失败要重试；如果涉及本地事务和消息发送一致性，可以使用本地消息表或 RocketMQ 事务消息。
2. Broker 端：消息写入 Broker 后依赖持久化机制，根据业务重要程度选择同步刷盘、主从同步复制等方式，降低 Broker 宕机导致消息丢失的风险。
3. 消费者端：只有业务处理成功后才返回消费成功；处理失败要返回失败或抛异常，让 RocketMQ 重试；消费端必须做好幂等和补偿。

**面试表达**

> RocketMQ 防止消息丢失，我会分生产者、Broker、消费者三段讲。生产者端核心消息用同步发送，发送失败要重试；如果本地事务和发消息需要一致，就用本地消息表或事务消息。Broker 端依靠持久化、刷盘策略和主从复制降低宕机丢消息风险。消费者端要在业务真正处理成功后再 ACK，失败就让 RocketMQ 重试，同时消费逻辑必须幂等，配合死信和补偿机制。

### 4. RocketMQ 为什么会重复消费？消费端幂等怎么设计？

**你的回答**

> 可能因为网络抖动等原因，或者是没收到 ack，消息有重新发送了。幂等设计可以用分布式锁，或者是唯一主键来做。

**点评**

**评分：7/10**

ACK 失败导致重复消费答对了，唯一主键也是正确方向。但分布式锁只能作为并发保护，不能替代幂等记录。

**标准回答**

RocketMQ 默认是至少投递一次，重复消费是正常现象。常见原因：

1. 消费者业务处理成功，但返回 ACK 时网络抖动或消费者宕机，Broker 没收到成功确认。
2. 消费超时、返回失败、抛异常，Broker 触发重试。
3. Producer 发送重试可能产生重复消息。
4. Consumer 重平衡、重试、异常恢复时可能再次投递。

幂等设计：

1. 使用业务唯一键，不要只依赖 `msgId`。
2. 通过唯一索引、消费记录表、状态机、版本号、条件更新保证只生效一次。
3. 幂等记录和业务变更尽量放在同一个本地事务。
4. 分布式锁可以作为并发保护，但不能替代幂等。

**面试表达**

> RocketMQ 是至少投递一次语义，所以重复消费是正常情况。比如业务处理成功了，但 ACK 失败，Broker 没收到成功确认，就会重新投递。消费端幂等要基于业务唯一键设计，比如订单号、工单号、支付单号，通过唯一索引、消费记录表、状态机或条件更新保证只处理一次。分布式锁只能降低并发冲突，不能替代幂等记录。

### 5. RocketMQ 顺序消息怎么实现？全局顺序和局部顺序有什么区别？

**你的回答**

> 顺序消费是通过把消息发送到同一个队列里。

**点评**

**评分：6.5/10**

抓住了核心，但还要说明按什么维度进同一个队列，以及消费端如何保证同一队列内顺序消费。

**标准回答**

RocketMQ 顺序消息通常是把同一个业务维度的消息发送到同一个 `MessageQueue`。比如同一个订单号、工单号、用户 ID，通过 Hash 选择固定队列，使同一业务对象的事件按顺序进入同一个队列。

消费端使用顺序消费模式，同一个队列同一时间由一个消费者线程顺序处理，避免队列内并发导致乱序。

全局顺序和局部顺序：

- 全局顺序：整个 Topic 只有一个队列，所有消息严格有序，但吞吐量很低。
- 局部顺序：同一个业务 key 内有序，不同 key 之间并发处理，是生产中更常用的方式。

**面试表达**

> RocketMQ 顺序消息本质是把同一个业务 key 的消息固定路由到同一个 MessageQueue，比如同一个订单或工单的状态事件都发到同一个队列，消费端再使用顺序消费模式保证队列内按顺序处理。全局顺序要求整个 Topic 只有一个队列，吞吐量很低；实际项目更常用局部顺序，只保证同一个业务 key 内有序。

### 6. RocketMQ 事务消息的原理是什么？它能不能替代分布式事务？

**你的回答**

> 先发送消息，此时未存到 broker，等本地事务提交后，再把消息存到 broker。可以作为一种分布式事务使用。

**点评**

**评分：5.5/10**

这里要纠正两个点：半消息已经保存到 Broker，只是对消费者不可见；事务消息不能直接替代强一致分布式事务，它解决的是本地事务和消息发送的一致性。

**标准回答**

RocketMQ 事务消息流程：

1. Producer 先发送半消息到 Broker。
2. Broker 持久化半消息后返回成功，但半消息对 Consumer 不可见。
3. Producer 执行本地事务。
4. 本地事务成功，Producer 向 Broker 发送 Commit，消息变为可消费。
5. 本地事务失败，Producer 向 Broker 发送 Rollback，Broker 丢弃半消息。
6. 如果 Broker 长时间没有收到 Commit/Rollback，会回查 Producer，Producer 根据本地事务表或业务表返回事务状态。

事务消息不能完全替代强一致分布式事务。它适合解决生产者侧“本地事务执行成功，但消息发送不确定”的最终一致问题；下游消费仍然要靠重试、幂等、补偿和状态校验保证。

**面试表达**

> RocketMQ 事务消息是先发半消息到 Broker，Broker 已经保存但消费者不可见。然后 Producer 执行本地事务，本地事务成功就 Commit，消息对消费者可见；失败就 Rollback。如果 Broker 没收到明确结果，会回查 Producer，Producer 根据本地事务状态返回 Commit、Rollback 或 Unknown。它不能完全替代强一致分布式事务，主要解决本地事务和消息发送之间的最终一致问题。

### 7. RocketMQ 延迟消息怎么用？它一定准时吗？业务上怎么兜底？

**你的回答**

> 不一定准时，会根据需要消费的消息决定。

**点评**

**评分：4.5/10**

“不一定准时”判断正确，但要补使用场景、不准时原因和业务兜底方案。

**标准回答**

延迟消息常见场景：

1. 订单超时关闭。
2. 支付超时检查。
3. 预约库存释放。
4. 失败后延迟重试。

1. 消费延迟消息时先查业务当前状态，不能无条件操作。

### 8. RocketMQ 消息积压怎么排查和处理？

**你的回答**

> 确认是消费者消费太慢了还是生产者发送太多了，如果是消费太慢了，可以增加消费者机器或者改为异步消费。

**点评**

**评分：6.5/10**

能先区分生产过快还是消费过慢，也能想到扩消费者。但“改为异步消费”不够准确，消费链路本身已经是异步的，更应该说优化消费逻辑、提升并发、拆分慢操作和削峰降级。

**标准回答**

排查消息积压：

**面试表达**

### 9. RocketMQ 的重试和死信队列是什么？死信消息应该怎么处理？

**你的回答**

> 重新消费。

**点评**

**评分：3/10**

回答过短，只说到了死信可以重新消费，没有说明重试触发条件、死信含义和治理方式。

**标准回答**

RocketMQ 消费失败后，比如消费者返回失败、抛异常或消费超时，Broker 会按照重试策略重新投递。重试多次仍失败后，消息会进入死信队列，也就是 DLQ。死信队列通常按 ConsumerGroup 维度隔离，用来保存长期无法成功消费的问题消息。

死信处理：

1. 先告警，不能静默堆积。
2. 根据业务 key、日志、Trace 定位失败原因。
3. 可恢复问题，比如下游超时、数据库异常，修复后重新投递或人工补偿。
4. 不可重试问题，比如数据格式错误、业务状态不合法，要修数据、人工处理或业务关闭。
5. 避免无脑重投导致死循环。

**面试表达**

> RocketMQ 消费失败会先重试，比如消费者抛异常、返回失败或消费超时。超过最大重试次数后，消息会进入死信队列。死信队列不是垃圾桶，而是异常消息隔离区。进入死信后要先告警，再根据业务 key、日志和链路追踪定位原因，判断是修复后重投、人工补偿、修数据，还是业务关闭，不能简单无脑重新消费。

### 10. 本地消息表和 RocketMQ 事务消息怎么选？本地消息表怎么落地？

**你的回答**

> 新增本地消息数据表。

**点评**

**评分：3.5/10**

回答只说了一个动作，没有说明本地消息表如何和业务事务绑定，也没有讲和 RocketMQ 事务消息的选型差异。

**标准回答**

本地消息表和 RocketMQ 事务消息都用于解决本地事务和消息发送之间的一致性问题。

选型：

1. RocketMQ 事务消息适合本地事务完成后发送一条消息，链路相对简单的场景。
2. 本地消息表适合复杂核心链路，比如需要审计、人工补偿、多次重试、失败原因记录、跨多个下游通知。

本地消息表落地：

1. 在同一个数据库事务里，同时写业务表和消息表。
2. 消息表保存业务 key、消息体、状态、重试次数、下次重试时间、失败原因等字段。
3. 事务提交后，由定时任务或异步任务扫描待发送消息。
4. 发送成功后更新为成功；发送失败则记录失败原因并按策略重试。
5. 超过阈值后进入人工补偿。
6. 消费端仍要基于业务唯一键、唯一索引或状态机做好幂等。

**面试表达**

> 本地消息表和事务消息都是解决本地事务和消息发送一致性的问题。简单链路可以用 RocketMQ 事务消息，通过半消息、本地事务、Commit/Rollback 和回查保证最终一致。复杂核心链路我更倾向本地消息表，因为它方便审计、重试、记录失败原因和人工补偿。落地时在同一个本地事务里写业务表和消息表，事务提交后由任务扫描待发送消息，发送成功改成功，失败记录原因并重试，超过阈值进入人工补偿。消费端仍然必须做幂等。

### 1. Redis 为什么快？Redis 真的是单线程吗？

**你的回答**

**评分：6.5/10**

1. 数据主要在内存中，读写不需要频繁访问磁盘。
2. 核心命令执行是单线程，避免多线程锁竞争和上下文切换。

**面试表达**

> Redis 快不是只因为单线程，而是因为它基于内存操作，命令执行模型简单，单线程避免锁竞争，配合 IO 多路复用处理大量连接，同时数据结构实现很高效。Redis 6 之后支持多线程 IO，但核心命令执行仍然是单线程。

### 2. Redis 常用数据类型有哪些？String、Hash、ZSet 底层大致怎么实现？

### 3. Redis 的 RDB、AOF、混合持久化分别是什么？线上一般怎么选？

**你的回答**

> rdb 是数据快照存储方式，aof 是命令追加存储方式，混合是指两种一起使用，线上一般是用混合持久化。

**点评**

**评分：7/10**

RDB 和 AOF 的核心区别答对了。混合持久化不是简单“两种一起使用”，而是 AOF 重写后的文件前半部分是 RDB 快照，后半部分追加增量 AOF 命令。

**标准回答**

RDB 是快照持久化，会在某个时间点把 Redis 内存中的数据生成二进制快照文件。优点是文件小、恢复快，适合备份和灾难恢复；缺点是两次快照之间如果 Redis 宕机，可能丢失一段数据。

AOF 是追加日志持久化，会把 Redis 执行过的写命令追加到 AOF 文件中，Redis 重启时通过重放命令恢复数据。优点是数据安全性更高，常见配置是 `appendfsync everysec`，通常最多丢约 1 秒数据；缺点是文件更大，恢复相对慢，需要 AOF rewrite 控制文件大小。

混合持久化是在 AOF 重写后，文件前半部分使用 RDB 格式保存全量数据，后半部分使用 AOF 保存重写期间的增量命令。这样兼顾 RDB 恢复快和 AOF 数据更安全。

**面试表达**

> RDB 是快照持久化，保存某一时刻的全量数据，恢复快但可能丢失两次快照之间的数据。AOF 是追加写命令日志，数据安全性更好，常用 everysec，通常最多丢约 1 秒数据，但文件更大、恢复更慢。混合持久化是在 AOF 重写后，前半部分用 RDB 保存全量数据，后半部分用 AOF 保存增量命令。线上一般会开启 AOF everysec 和混合持久化，RDB 更多用于备份和灾难恢复。

### 4. Redis 过期 key 是怎么删除的？内存满了以后又是怎么淘汰 key 的？

**你的回答**

> 定时删除 + 惰性删除。内存满了有随机淘汰，最少使用和最少频率等淘汰策略。

**点评**

**评分：7.5/10**

主干正确，但建议把“定时删除”修正为“定期删除”。Redis 不是给每个 key 单独挂一个定时器，而是周期性抽样检查过期 key。

**标准回答**

Redis 过期 key 删除主要是惰性删除 + 定期删除。

1. 惰性删除：访问某个 key 时，Redis 先判断它是否过期，如果过期就删除并返回空。
2. 定期删除：Redis 周期性抽样检查一批设置了过期时间的 key，把过期 key 删除掉。

这样设计是为了平衡内存和 CPU：只靠惰性删除，过期 key 如果一直没人访问会继续占内存；如果给每个 key 都做定时删除，又会消耗太多 CPU。

当 Redis 达到 `maxmemory` 后，会根据 `maxmemory-policy` 淘汰 key。常见策略：

1. `noeviction`：不淘汰，写入直接报错。
2. `allkeys-random` / `volatile-random`：随机淘汰所有 key 或带 TTL 的 key。
3. `allkeys-lru` / `volatile-lru`：淘汰最近最少使用的 key。
4. `allkeys-lfu` / `volatile-lfu`：淘汰访问频率最低的 key。
5. `volatile-ttl`：优先淘汰 TTL 最短的 key。

**面试表达**

> Redis 过期删除是惰性删除加定期删除。访问 key 时发现过期会删除，这是惰性删除；同时 Redis 会周期性抽样检查一批带过期时间的 key，把过期的删掉，这是定期删除。内存达到 maxmemory 后，再按淘汰策略处理，比如 allkeys-lru、allkeys-lfu、volatile-lru、volatile-ttl 或 noeviction。纯缓存场景常用 allkeys-lru 或 allkeys-lfu；如果有核心数据不能被淘汰，要用 noeviction 配合容量告警和扩容。

### 5. 缓存穿透、缓存击穿、缓存雪崩分别是什么？怎么处理？

**你的回答**

> 缓存不存在导致数据库被打穿，可以设置短暂的非空值缓存。击穿指的是缓存过期了，大量请求，可以设置本地缓存或者业务过期判断。缓存雪崩在同一个时间很多缓存同时失效，给缓存设置随机的过期时间。

**点评**

**评分：8/10**

三个概念都能区分出来，也给了对应处理思路。需要修正两点：穿透一般是缓存和数据库都不存在；空值缓存通常是缓存空对象或特殊标记，不是“非空值”。

**标准回答**

缓存穿透：查询的数据在缓存和数据库中都不存在，请求每次都绕过缓存打到数据库，常见于恶意请求不存在的 ID。处理方式有参数校验、布隆过滤器、空值缓存。空值缓存 TTL 要短，避免长期缓存错误结果。

缓存击穿：某个热点 key 过期瞬间，大量并发请求同时查不到缓存，一起打到数据库。处理方式有互斥锁、逻辑过期、热点 key 提前续期、本地缓存等。

缓存雪崩：大量 key 在同一时间失效，或者 Redis 故障，导致大量流量直接打到数据库。处理方式有 TTL 加随机值、缓存预热、多级缓存、限流降级、Redis 高可用。

**面试表达**

> 穿透、击穿、雪崩分别对应“不存在数据反复查”“热点 key 过期瞬间被打爆”“大量 key 同时失效或 Redis 故障”。穿透用布隆过滤器和空值缓存；击穿用互斥锁、逻辑过期或热点续期；雪崩用 TTL 随机化、缓存预热、多级缓存、限流降级和 Redis 高可用。

### 6. Redis 和数据库怎么保证缓存一致性？为什么一般建议先更新数据库，再删除缓存？

**你的回答**

> 使用旁路策略，先删除数据再删缓存可以避免大概率的脏数据。

**点评**

**评分：5/10**

能说出旁路策略是正确方向，但“先删除数据再删缓存”表述有问题。推荐说法是写请求先更新数据库，再删除缓存。

**标准回答**

常见做法是 Cache Aside：

1. 读请求先查缓存，缓存未命中再查数据库，并回填缓存。
2. 写请求先更新数据库，再删除缓存。

不建议先删缓存再更新数据库，因为如果先删缓存，数据库还没更新完成时有读请求进来，发现缓存不存在，就会读取旧数据库值并回填到缓存。之后写请求再把数据库更新成功，缓存里却已经是旧值，产生脏缓存。

建议先更新数据库，再删除缓存。数据库更新成功后删除缓存，后续读请求缓存未命中，会从数据库读到新值并回填。即使删除缓存失败，也可以通过重试、MQ、Canal 订阅 binlog、定时校验等方式补偿。

**面试表达**

> 缓存和数据库一致性我一般用 Cache Aside。读时先读缓存，未命中查 DB 后回填；写时先更新 DB，再删除缓存。缓存不要直接更新，因为复杂缓存可能是聚合结果，并发更新容易乱序覆盖。删除缓存失败要有重试、MQ 或 Canal binlog 订阅兜底，核心链路还可以做延迟双删或版本号校验，但最终要接受缓存和数据库是最终一致，不是强一致。

### 7. Redis 分布式锁怎么实现才安全？Redisson 的看门狗机制是什么？

**你的回答**

> 可以使用 redission 作为分布式锁，看门狗机制是锁时间自动续期。

**点评**

**评分：5.5/10**

方向对，但还缺基础实现、安全释放、唯一 value、Lua 解锁，以及 Redisson 看门狗什么时候生效。

**标准回答**

Redis 分布式锁基础写法：

```text
SET lock_key unique_value NX PX ttl
```

其中：

1. `NX`：key 不存在时才能加锁。
2. `PX ttl`：设置过期时间，避免业务异常导致锁永远不释放。
3. `unique_value`：唯一值，比如 UUID + 线程标识，用来证明锁是自己的。

释放锁时必须先校验 value 再删除，建议使用 Lua 保证原子性：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

Redisson 看门狗：如果使用 `lock()` 或不指定固定 `leaseTime` 的加锁方式，默认会启用看门狗。只要当前 JVM 进程存活、业务线程还持有锁，Redisson 会定期给锁续期，避免业务执行时间超过锁 TTL 导致锁提前释放。如果指定固定 `leaseTime`，通常不依赖无限续期，到时间锁会自动释放。

**面试表达**

> Redis 锁基础写法是 `SET key value NX PX ttl`，value 必须唯一，释放锁时用 Lua 先校验 value 再删除，防止误删别人的锁。生产里我更倾向用 Redisson，因为它封装了可重入、Lua 解锁、阻塞等待和看门狗续期。看门狗会在不指定固定 leaseTime 时自动给锁续期，避免业务还没执行完锁就过期。但分布式锁不能替代数据库唯一约束、状态机、幂等和补偿，核心业务最终还要靠业务兜底。

### 8. Redis Lua 脚本有什么用？为什么库存扣减、限流、解锁常用 Lua？

**你的回答**

> 能保证原子性。

**点评**

**评分：5/10**

关键词正确，但太短。要说明 Lua 保证的是 Redis 内部多命令组合执行的原子性，不是分布式事务，也不保证数据库那一段成功。

**标准回答**

Redis Lua 脚本可以把多条 Redis 命令放到服务端一次执行。由于 Redis 执行 Lua 脚本时不会被其他命令插入打断，所以适合处理“先判断再修改”的并发场景。

常见场景：

1. 库存扣减：判断库存是否大于 0、判断用户是否已抢过、扣减库存、记录抢购标记。
2. 限流：读取当前计数、判断是否超过阈值、递增计数、设置过期时间。
3. 解锁：判断锁 value 是否是自己的，是自己的才删除。

Lua 解决的是 Redis 层多命令组合的原子性，不是分布式事务。比如秒杀预扣可以用 Lua 保证 Redis 库存扣减不超卖，但后续订单落库、MQ 投递、支付超时回补，还要靠数据库条件更新、幂等和补偿保证最终一致。

**面试表达**

> Lua 的价值是把 Redis 多命令组合变成一个原子操作，特别适合库存扣减、限流、解锁这类“判断 + 修改”的场景。它解决的是 Redis 层的原子性，不是分布式事务。比如秒杀预扣可以用 Lua 保证 Redis 库存扣减不超卖，但后续订单落库、MQ 投递、支付超时回补，还要靠数据库条件更新、幂等和补偿保证最终一致。

### 9. BigKey 和 HotKey 分别是什么？线上怎么发现和处理？

**你的回答**

> bigkey 是大 key，用 del 删除会有问题。hotkey 是热点 key。

**点评**

**评分：4.5/10**

概念答到了，但要补为什么危险、怎么发现、怎么处理。

**标准回答**

BigKey 是 value 很大的 key，可能是一个很大的 String，也可能是元素特别多的 Hash、List、Set、ZSet。它的问题是网络传输慢、内存占用大、删除释放内存慢、持久化和主从同步成本高。删除 BigKey 不建议直接用 `DEL`，因为可能在主线程同步释放内存导致 Redis 卡顿，更推荐用 `UNLINK` 异步释放；批量查找用 `SCAN`，不要在线上用 `KEYS`。

HotKey 是访问特别集中的热点 key，比如秒杀商品库存、热门商品详情、首页配置等。它的问题是大量请求集中打到某个 Redis 节点甚至某个 key，导致单点 QPS 飙高、延迟上升，严重时 Redis 被打挂。

发现方式：

1. BigKey：`redis-cli --bigkeys`、`MEMORY USAGE`、离线 RDB 分析、监控大对象。
2. HotKey：业务日志、访问统计、Redis 慢日志、客户端埋点、代理层统计、云厂商热点 key 分析。

处理方式：

1. BigKey：按业务维度拆分成多个小 key；删除使用 `UNLINK` 或分批删除集合元素。
2. HotKey：本地缓存、Caffeine、多级缓存、热点 key 副本、库存分桶、限流降级、提前预热。

**面试表达**

> BigKey 重点是“大”，会带来内存、网络、删除、持久化和主从同步压力；HotKey 重点是“热”，会带来单 key 或单节点高 QPS。BigKey 发现后要拆分和异步删除，HotKey 要通过本地缓存、多级缓存、分桶、限流和预热把流量打散。

### 10. Redis Cluster 为什么是 16384 个槽？请求是怎么路由的？多 key 操作要注意什么？

**你的回答**

> 不清楚。

**点评**

**评分：1/10**

Redis Cluster 这块需要重点补。面试时至少要答出槽位、CRC16、客户端路由、MOVED/ASK、多 key 同槽和 hash tag。

**标准回答**

Redis Cluster 把整个 key 空间划分成 16384 个 hash slot。每个 key 会通过 CRC16 计算后，对 16384 取模，得到它属于哪个槽。集群中的每个主节点负责一部分槽位，所以数据不是按节点直接 hash，而是先映射到槽，再由槽映射到节点。

请求路由：

1. 客户端缓存槽位到节点的映射关系。
2. 访问 key 时，客户端先计算 key 所属 slot，再把请求发到负责这个 slot 的节点。
3. 如果发错节点，Redis 会返回 `MOVED` 或 `ASK` 重定向，客户端更新路由或临时跳转。

为什么是 16384：

1. 槽太少，数据分布和迁移粒度不够细。
2. 槽太多，节点之间维护槽位元数据和集群心跳传播成本更高。
3. 16384 是分片粒度和集群通信成本之间的工程折中。

多 key 操作注意：

1. `mget`、`mset`、Lua、多 key 事务等操作要求相关 key 在同一个 slot。
2. 需要同槽时可以使用 hash tag，比如 `order:{1001}:info`、`order:{1001}:items`，Redis 只对 `{1001}` 这部分计算 slot。

**面试表达**

> Redis Cluster 通过 16384 个槽做分片，key 先经过 CRC16 计算映射到槽，槽再分配到不同主节点。客户端会缓存槽位路由，如果访问错节点，Redis 会返回 MOVED 或 ASK 重定向。多 key 操作要注意同槽问题，必要时用 hash tag 让相关 key 落到同一个槽，比如把订单相关 key 都写成包含 `{orderId}` 的形式。

### 1. MySQL InnoDB 为什么常用 B+Tree 做索引？聚簇索引和非聚簇索引有什么区别？

**你的回答**

> 用 b+ 树层级比较低，一般三级树就能存储千万级主键索引数据。聚集索引的叶子节点存放的是完整数据，非聚集索引叶子节点存放的是索引值。

**点评**

**评分：7.5/10**

B+Tree 层级低、聚簇索引叶子节点存整行这两个核心点都说到了。需要修正：非聚簇索引叶子节点不是只存索引值，而是存索引列和主键值，再通过主键回表到聚簇索引取整行。

**标准回答**

InnoDB 常用 B+Tree 做索引，主要是因为 B+Tree 是多路平衡树，树高比较低，磁盘 IO 次数少。非叶子节点只存索引键和指针，叶子节点之间有双向链表，适合范围查询和排序。通常三到四层 B+Tree 就能支撑千万级甚至更大规模的数据定位。

聚簇索引通常就是主键索引，叶子节点存放完整行数据，所以通过主键查询可以直接拿到整行。

非聚簇索引也叫二级索引，叶子节点存放的是索引列值和主键值。如果查询字段不在二级索引里，就需要先通过二级索引找到主键，再回到聚簇索引查询完整行，这就是回表。

InnoDB 表最好有短小、递增、稳定的主键。因为二级索引叶子节点会保存主键值，主键太大会让所有二级索引变大；主键无序会导致页分裂更频繁。

**面试表达**

> InnoDB 用 B+Tree 做索引，是因为 B+Tree 层级低，磁盘 IO 少，叶子节点有序并通过链表连接，适合范围查询。聚簇索引叶子节点存整行数据，通常就是主键索引；二级索引叶子节点存索引列和主键值，如果查询字段不在二级索引里，就要拿主键回表到聚簇索引查整行。所以主键最好短小、递增、稳定，避免二级索引膨胀和页分裂。

### 2. 什么是最左前缀原则？哪些情况会导致索引失效？

**你的回答**

> 联合索引要从最左边的值开始命中索引，否则无法命中索引。索引失效的场景比较多，比如函数计算，类型不一致，全表扫描性能更高的场景。

**点评**

**评分：7/10**

核心方向正确：联合索引要从最左列开始匹配，也能说出函数计算、类型不一致、优化器认为全表扫描更划算。建议补范围查询后的列、`like` 前置通配符、`or` 条件、隐式字符集转换、低选择性字段。

**标准回答**

最左前缀原则是指联合索引会按照索引定义的列顺序建立有序结构，比如 `(a, b, c)`，查询条件要从最左边的 `a` 开始连续匹配，才能充分利用索引。可以使用 `(a)`、`(a,b)`、`(a,b,c)`，但如果跳过 `a` 只查 `b` 或 `c`，通常不能有效命中这个联合索引。

遇到范围查询后，范围列可以用到索引，但范围列后面的列通常不能继续用于索引有序匹配。比如 `where a = ? and b > ? and c = ?`，一般 `a` 和 `b` 能用上，`c` 很难继续用于索引定位。

常见索引失效场景：

1. 对索引列使用函数或表达式，比如 `date(create_time)`、`id + 1`。
2. 隐式类型转换，比如字符串字段用数字查询。
3. `like '%xxx'` 前置通配符。
4. 联合索引不满足最左前缀。
5. `or` 两边不是都能用索引。
6. 字符集或排序规则不一致导致隐式转换。
7. 查询返回数据比例太高，优化器认为全表扫描更划算。

**面试表达**

> 最左前缀原则本质是联合索引按照列顺序排序，查询要从最左列开始连续匹配。索引失效常见原因有函数计算、隐式类型转换、like 前置百分号、跳过联合索引左列、范围查询后续列无法继续用于定位、or 条件不合理，以及优化器判断全表扫描成本更低。实际排查不能只凭感觉，要用 EXPLAIN 看 type、key、rows 和 Extra。

### 3. MySQL 事务的 ACID 是什么？InnoDB 分别靠什么机制实现？

**你的回答**

> 原子性、持久性，隔离性和一致性。mvcc。

**点评**

**评分：5/10**

ACID 四个词答出来了，但 InnoDB 靠什么实现只说了 MVCC，不够完整。要把 undo log、redo log、锁、MVCC、约束和 ACID 对应起来。

**标准回答**

ACID 分别是原子性、一致性、隔离性、持久性。

对应机制：

1. 原子性主要靠 undo log。事务执行失败或回滚时，可以根据 undo log 把数据恢复到修改前。
2. 持久性主要靠 redo log。事务提交后，即使数据库宕机，也可以通过 redo log 做崩溃恢复。
3. 隔离性主要靠锁和 MVCC。当前读通过锁控制并发修改，快照读通过 MVCC 读取历史版本。
4. 一致性是最终目标，依赖原子性、隔离性、持久性，以及数据库约束、唯一索引、外键、业务规则共同保证。

**面试表达**

> InnoDB 实现 ACID 可以按机制对应来讲：undo log 保证原子性，事务失败可以回滚；redo log 保证持久性，提交后的数据宕机后能恢复；锁和 MVCC 保证隔离性，当前读靠锁，快照读靠版本链；一致性是事务最终要达到的目标，依赖 A、I、D 以及约束、唯一索引和业务规则共同保证。

### 4. MySQL 的事务隔离级别有哪些？可重复读是怎么解决幻读的？

**你的回答**

> 可重复读，串行化，可提交读，未提交读。通过 mvcc 解决幻读。

**点评**

**评分：6/10**

四个隔离级别答出来了，但顺序和幻读解决机制要补细。MVCC 主要解决快照读下的幻读；当前读下防幻读依赖 next-key lock，也就是记录锁加间隙锁。

**标准回答**

MySQL 常见四个隔离级别从低到高是：读未提交、读已提交、可重复读、串行化。

它们分别解决的问题：

1. 读未提交可能读到未提交数据，会有脏读。
2. 读已提交解决脏读，但同一个事务内两次读取可能结果不同，会有不可重复读。
3. 可重复读解决不可重复读，是 MySQL InnoDB 默认隔离级别。
4. 串行化隔离最强，但并发性能最差。

可重复读解决幻读要分两种读：

1. 普通 `select` 是快照读，InnoDB 通过 MVCC 和 Read View 保证同一个事务内多次读取看到的是同一份一致性视图，所以不会看到其他事务新插入的数据。
2. `select ... for update`、`update`、`delete` 这类当前读，需要读取最新数据并加锁，InnoDB 通过 next-key lock，也就是记录锁 + 间隙锁，锁住索引记录以及记录之间的间隙，阻止其他事务在范围内插入新记录。

**面试表达**

> InnoDB 可重复读下防幻读要区分快照读和当前读。普通 select 是快照读，靠 MVCC 和 Read View 保证事务内读到一致视图；当前读比如 for update、update、delete，需要读最新数据并加锁，所以通过 next-key lock，也就是记录锁加间隙锁，锁住范围，防止其他事务插入新的幻影记录。

### 5. MVCC 是什么？Read View 里主要有哪些字段？它怎么判断一条记录是否可见？

**你的回答**

> 多版本并发控制。

**点评**

**评分：2/10**

只答出了 MVCC 的全称。MySQL 高频核心题至少要说出 undo log 版本链、Read View 和可见性判断。

**标准回答**

MVCC 是多版本并发控制。InnoDB 会为每行记录维护隐藏字段，比如事务 ID 和回滚指针。每次修改数据时，会生成 undo log，多个历史版本通过回滚指针形成版本链。快照读时不会直接加锁，而是基于 Read View 判断当前事务能看到版本链中的哪个版本。

Read View 里常见关键字段：

1. `m_ids`：生成 Read View 时，当前活跃未提交事务 ID 列表。
2. `min_trx_id`：活跃事务 ID 中最小的事务 ID。
3. `max_trx_id`：下一个将要分配的事务 ID，可以理解为生成 Read View 时系统中已出现事务 ID 的上界。
4. `creator_trx_id`：创建这个 Read View 的事务 ID。

可见性判断：

1. 如果记录版本的 `trx_id` 小于 `min_trx_id`，说明这个版本在 Read View 创建前已经提交，对当前事务可见。
2. 如果 `trx_id` 大于等于 `max_trx_id`，说明这个版本是 Read View 创建后才产生的，不可见。
3. 如果 `trx_id` 在 `min_trx_id` 和 `max_trx_id` 之间，就看它是否在 `m_ids` 活跃事务列表中：在列表中说明还没提交，不可见；不在列表中说明已经提交，可见。
4. 如果当前版本不可见，就沿着 undo log 版本链继续找上一个历史版本。

RC 和 RR 的区别：读已提交下，每次快照读都会生成新的 Read View；可重复读下，事务第一次快照读生成 Read View，后续复用同一个 Read View，所以能做到可重复读。

**面试表达**

> MVCC 是 InnoDB 用来实现一致性非锁定读的机制，本质是 undo log 版本链加 Read View。每行记录有事务 ID 和回滚指针，修改时生成历史版本；快照读时根据 Read View 判断版本是否可见。RC 每次查询生成新的 Read View，RR 一个事务内复用第一次快照读的 Read View，所以 RR 能保证同一事务内多次读取结果一致。

### 6. InnoDB 有哪些常见锁？行锁、间隙锁、next-key lock 分别解决什么问题？

**你的回答**

> 行锁 表锁 间隙锁。

**点评**

**评分：3.5/10**

答出了几个锁名，但还缺 next-key lock、共享锁/排他锁、意向锁，也没有说明每种锁解决什么问题。MySQL 锁一定要带上“锁什么范围、为什么锁”。

**标准回答**

InnoDB 常见锁可以从几个角度理解：

1. 按粒度：表锁、行锁。
2. 按读写：共享锁 S 锁、排他锁 X 锁。
3. 表级辅助锁：意向共享锁、意向排他锁。
4. 可重复读防幻读相关：记录锁、间隙锁、next-key lock。

重点解释：

1. 行锁锁的是索引记录，不是直接锁物理行。如果 SQL 没有走索引，可能扫描大量记录并加锁，导致锁范围扩大。
2. 记录锁锁住已经存在的索引记录，比如锁住 `id = 10` 这条记录。
3. 间隙锁锁住两个索引记录之间的范围，不锁具体记录，目的是阻止其他事务往这个范围插入新记录。
4. next-key lock = 记录锁 + 间隙锁，既锁住已有记录，也锁住记录前面的间隙，用来防止当前读下的幻读。

**面试表达**

> InnoDB 锁要按范围讲。记录锁锁已存在的索引记录，间隙锁锁索引记录之间的空隙，next-key lock 是记录锁加间隙锁，用来在可重复读下防止当前读出现幻读。行锁本质锁的是索引，如果 SQL 不走索引，锁范围会扩大，甚至造成大量阻塞。所以线上更新和加锁查询必须尽量命中高选择性索引。

### 7. redo log、undo log、binlog 分别有什么作用？redo log 和 binlog 为什么需要两阶段提交？

**你的回答**

> redo log 用于写入数据恢复，undolog 用于回滚数据，binlog 用于同步数据。

**点评**

**评分：6.5/10**

三个日志的基本作用都抓到了。但两阶段提交为什么需要还没答。建议把 redo prepare -> 写 binlog -> redo commit 这条链路背熟。

**标准回答**

redo log 是 InnoDB 的物理日志，记录数据页的修改，用于崩溃恢复，保证事务持久性。事务提交后即使数据库宕机，也可以通过 redo log 恢复已提交的数据。

undo log 记录数据修改前的旧版本，用于事务回滚，也用于 MVCC 快照读的历史版本。

binlog 是 MySQL Server 层的逻辑日志，记录 SQL 或行变更，主要用于主从复制、数据恢复和审计。

redo log 和 binlog 需要两阶段提交，是因为一次事务提交既要写 InnoDB 的 redo log，又要写 Server 层的 binlog。如果二者不一致，就可能导致主库崩溃恢复和从库复制结果不一致。

MySQL 使用两阶段提交：

1. 先写 redo log，并标记为 prepare。
2. 再写 binlog。
3. 最后把 redo log 标记为 commit。

崩溃恢复时：

1. 如果 redo 没有 prepare，说明事务没提交，回滚。
2. 如果 redo 是 prepare，但 binlog 不完整，说明 binlog 没写成功，回滚。
3. 如果 redo 是 prepare 且 binlog 完整，说明事务可以提交，恢复为已提交状态。

**面试表达**

> redo log 保证 InnoDB 崩溃恢复，undo log 支持回滚和 MVCC，binlog 用于主从复制和数据恢复。两阶段提交是为了解决 redo log 和 binlog 一致性：事务提交时先写 redo prepare，再写 binlog，最后写 redo commit。这样即使中间宕机，恢复时也能根据 redo 状态和 binlog 是否完整判断事务该提交还是回滚，避免主库恢复和从库复制结果不一致。

1. 先从监控和慢查询日志定位具体 SQL。
2. 看 QPS、RT、慢 SQL 数量、连接数、CPU、磁盘 IO、Buffer Pool 命中率等指标。
3. 确认是单条 SQL 问题，还是整体数据库压力问题。
4. 拿到 SQL 后，用 EXPLAIN 分析执行计划，再结合表结构、索引、数据量和业务条件判断优化方案。

EXPLAIN 重点看：

1. `type`：访问类型，通常从好到差有 `const`、`eq_ref`、`ref`、`range`、`index`、`ALL`，如果是 `ALL` 要重点关注是否全表扫描。
2. `key`：实际使用的索引。
3. `possible_keys`：可能使用的索引。
4. `rows`：预估扫描行数。
5. `filtered`：过滤比例。
6. `Extra`：重点看是否有 `Using filesort`、`Using temporary`、`Using index`、`Using where`。

常见优化方向：索引是否失效、联合索引顺序是否合理、是否回表过多、是否排序和分组导致临时表或文件排序、是否返回字段太多、是否深分页、是否可以覆盖索引、拆 SQL、冷热归档、缓存或搜索引擎承接复杂查询。

**面试表达**

> 慢 SQL 排查我会先用监控和慢日志定位 SQL，再用 EXPLAIN 看 type、key、rows、filtered 和 Extra。重点关注全表扫描、索引没命中、扫描行数过多、Using filesort、Using temporary、回表过多和深分页。优化不一定只是加索引，也可能是改 SQL、覆盖索引、拆查询、归档、缓存或用 ES 承接复杂检索。

### 9. MySQL 深分页为什么慢？怎么优化？游标分页和延迟关联有什么区别？

**你的回答**

> 可以用游标分页或者是延迟关联优化。延迟关联还是存在扫描行数多的问题。

**点评**

**评分：7/10**

能说出游标分页和延迟关联两个核心方案，并知道延迟关联仍然有扫描大量 offset 的问题。需要补为什么慢和两个方案的适用场景。

**标准回答**

深分页慢是因为 `limit offset, size` 需要先扫描并丢弃前面大量数据，再返回目标页。例如 `limit 1000000, 20`，MySQL 要先找到前 1000020 条，再丢掉前 1000000 条。即使走索引，也可能扫描大量索引记录，并伴随大量回表。

游标分页：

```sql
select *
from order_info
where id < #{lastId}
order by id desc
limit 20;
```

游标分页通过上一页最后一条记录作为游标，避免大 offset，适合 C 端列表、信息流、订单流水这种连续翻页场景。但它不适合任意跳页。

延迟关联：

```sql
select o.*
from order_info o
join (
    select id
    from order_info
    where user_id = 100
    order by id desc
    limit 1000000, 20
) t on o.id = t.id;
```

延迟关联先通过覆盖索引查出目标页主键，再回表查完整数据，可以减少大量无效回表，但 offset 扫描成本仍然存在。它适合必须保留 `pageNo/pageSize` 的后台管理系统。

**面试表达**

> 深分页慢是因为数据库要扫描并丢弃大量前置记录，还可能大量回表。游标分页从根上避免大 offset，适合 C 端连续翻页；延迟关联是先用覆盖索引查主键，再回表查详情，减少无效回表，但 offset 扫描还在，适合必须支持页码跳转的后台列表。

### 10. 高并发库存扣减 SQL 怎么设计？如何避免超卖和重复扣减？

**你的回答**

> 需要在 sql 判断是否还有库存。

**点评**

**评分：4/10**

在 SQL 里判断库存是否充足是避免超卖的关键，但还需要补一条原子条件更新、影响行数判断、扣减流水唯一索引和幂等。

**标准回答**

高并发库存扣减不能先查库存再扣库存，因为查询和扣减分成两步，中间会被其他事务插入，容易超卖。更推荐把库存判断和扣减合并成一条条件更新 SQL。

```sql
update sku_stock
set stock = stock - #{num}
where sku_id = #{skuId}
  and stock >= #{num};
```

根据影响行数判断：

1. 影响行数是 1，说明扣减成功。
2. 影响行数是 0，说明库存不足或商品不存在。

这样利用 MySQL 行锁和条件更新保证单行库存不会扣成负数。

避免重复扣减，需要设计库存扣减流水表，用业务单号或 `order_id + sku_id` 做唯一索引。扣减前或同一事务中写入扣减流水，重复请求命中唯一索引后直接返回幂等结果，避免同一个订单重复扣库存。

```sql
create unique index uk_order_sku on stock_deduct_log(order_id, sku_id);
```

**面试表达**

> 普通交易场景我会用 MySQL 条件更新防止超卖，把库存判断和扣减合并成一条 SQL：`update stock set stock = stock - num where sku_id = ? and stock >= num`，再根据影响行数判断是否扣减成功。为了避免重复扣减，还要有库存扣减流水表，并用订单号、商品 ID 做唯一索引保证幂等。秒杀这类热点库存不适合所有请求都打 MySQL 单行锁，更适合 Redis Lua 预扣、MQ 削峰和异步落库。

### 本轮薄弱点与复习建议

**表现较好的点**

- B+Tree 层级低、聚簇索引叶子节点存整行这两个核心点比较清楚。
- 最左前缀原则、函数计算、类型不一致、优化器选择全表扫描这些索引失效场景有基础认识。
- redo log、undo log、binlog 的基本职责能区分。
- 深分页能想到游标分页和延迟关联，也知道延迟关联仍然有 offset 扫描成本。

**需要补强的点**

- ACID 要能和 InnoDB 机制对应：undo log、redo log、锁、MVCC、约束。
- 隔离级别要按从低到高记忆：读未提交、读已提交、可重复读、串行化。
- MVCC 是当前最大短板，要重点背 undo log 版本链、Read View 字段和可见性判断。
- 锁机制要补记录锁、间隙锁、next-key lock、意向锁，以及“行锁锁索引”。
- 慢 SQL 排查要形成完整链路：监控/慢日志 -> EXPLAIN -> 索引/回表/排序/分页/业务改造。
- 库存扣减要补一条条件更新 SQL、影响行数判断、扣减流水唯一索引和幂等。

**下一轮建议练习**

1. 只练 MVCC：隐藏字段、undo log 版本链、Read View、RC/RR 区别。
2. 只练锁：记录锁、间隙锁、next-key lock、死锁排查、索引和锁范围。
3. 只练日志：redo、undo、binlog、两阶段提交、主从复制和主从延迟。
